Modern C - Current Open Questions

Purpose

This file is no longer a first-pass questionnaire.
The core language semantics now live in:
- docs/arch/language/language_spec.txt
- docs/arch/runtime/runtime_spec.txt
- docs/arch/language/canonical_programs.txt
- docs/plan/backlog.txt
- docs/plan/decision_log.txt

The major language-design decisions are now settled for v0.2:
- integer semantics and reduced UB
- pointer aliasing and provenance rules
- array, slice, buffer, and string model
- explicit `(value, Error)` error handling
- allocator shape and ownership rules
- `defer` behavior
- tagged unions and C interop boundary
- minimal concurrency and atomic model
- no hidden language runtime

What remains open is smaller and mostly about packaging, conventions, and what v0.2 should standardize early.

Use docs/plan/backlog.txt for follow-up work and recurring implementation themes.
Use docs/plan/decision_log.txt for intentional limited solutions and deferred fuller fixes.

1. Error-Handling Ergonomics Follow-Up

The semantic decision is settled: explicit returns, no hidden propagation. The remaining question is whether a very small convenience pattern is justified later.

Current repository status:
- the repository now admits a phase-1 opaque `Error` surface with kind or code inspection and named subsystem constructors at the stdlib boundary
- the Phase 47 full error-treatment wave now also standardizes the small source-level vocabulary and the named module-local code pattern across the admitted examples, tests, and narrow stdlib helpers
- Phase 138 now also admits one compiler-known fatal primitive, `panic(code)`, for non-recoverable invariant failure while keeping recoverable flows on explicit `Error` returns

Open questions:
- After more real programs are written, is plain `if !cond { panic(code) }` sufficient for invariant-heavy code without admitting a separate `require` surface?
- If a convenience helper is ever added, can it remain purely explicit and avoid becoming a second hidden control-flow system or a disguised recoverable-flow shortcut?

Why this is still open:
- The next possible follow-up is no longer whether the language needs any fatal primitive at all; it is whether real programs eventually need richer failure detail than kind or code alone, or a separate invariant helper beyond plain `if` plus `panic(code)`, without reintroducing hidden control flow.

2. Procedure-Value Dispatch Tables

The semantic decision is partly settled: procedure types and callable values
already exist. The remaining question is whether Modern C should explicitly
standardize table-shaped storage of named procedures as ordinary typed data.

Current repository status:
- procedure types already belong to the maintained semantic type surface
- ordinary call syntax already accepts callable values
- the compiler already proves basic procedure-value passing and indirect calls
- repeated real-code pressure still reports that the language lacks an obvious
	clean dispatch-table pattern for explicit typed handler maps

Open questions:
- Should explicit typed fixed arrays and structs containing named procedures be
	part of the maintained source contract for v0.2?
- If admitted, should the feature stay limited to exact-signature non-capturing
	procedure values stored in explicit typed aggregates?
- Can this remain a pure data-shape admission without reopening closures,
	callback registries, or dynamic service routing?

Why this is still open:
- the repeated pressure is real, but the narrowest honest answer may be either
	a small compiler proof wave for explicit typed procedure tables or an
	explicit deferral; the repository should decide that question directly
	instead of letting ad hoc switch-heavy code stand in for the design choice

Reference:
- docs/arch/c-lang/language/procedure_value_dispatch_tables_note.txt

Non-open items

The following are not current open questions for v0.2 unless real program pressure reopens them:
- signed overflow behavior
- strict aliasing policy
- UTF-8 and string ownership model
- `Buffer<T>` versus `Slice<T>` roles
- volatile versus atomic separation
- thread, mutex, condvar, and atomic baseline
- C interop spelling and explicit ABI annotations
- `distinct` scalar types
- test discovery convention and ordinary `func() Error` test shape
- `mem.default_allocator() *Allocator` as the hosted default allocator accessor
- explicit allocator parameters for allocating library APIs in v0.2
- no broad hosted convenience wrappers that silently choose the default allocator in v0.2
- canonical stdlib packaging: `mem`, `utf8`, `strings`, `io`, `fs`, `net`, `sync`, `time`, `os`, and `hal`, plus toolchain companion `testing`
- portable evented I/O surface in `io`, with raw backends such as `epoll`, `kqueue`, `io_uring`, and IOCP kept under `os` or implementation boundaries
- portable networking surface in `net`, above `os`
- one portable `File` handle type for ordinary stream I/O and readiness polling in v0.2, rather than split `File`, `Socket`, and `Listener` handle families
- concrete binary `IpAddr` and `IpEndpoint` portable networking values, rather than text-only or opaque address handles
- fallible `io.close(file) Error` in the portable surface, with explicit discard when callers intentionally ignore shutdown errors
- no omnibus `core` module or hidden prelude in v0.2
- exclusion of exceptions, GC, hidden init, and compile-time execution
