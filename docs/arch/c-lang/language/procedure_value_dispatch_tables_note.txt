Modern C - Procedure-Value Dispatch Tables
==========================================

Purpose
-------
Record the admitted narrow callable-data surface for explicit procedure-value
dispatch tables.

This note is intentionally narrower than a general function-pointer, closure,
or callback-framework proposal. Modern C now admits explicit typed fixed arrays
and structs containing exact-signature named procedures as ordinary typed data.

Admitted surface
----------------
1. procedure values remain non-capturing callable values with exact signature
  identity
2. named non-generic procedures may appear as constant ingredients when the
  destination type is an explicit matching procedure type
3. explicit typed aggregate literals may contain procedure values when every
  element or field matches the declared procedure type exactly
4. top-level `const` values, local values, struct fields, and fixed-array
  elements may use procedure values under the same explicit typing rules
5. imported constant aggregates containing procedure values remain valid across
  module-surface export, import rewriting, MIR lowering, build-unit
  namespacing, and executable emission
6. calling through a procedure value loaded from such a table uses the
  ordinary call rules already admitted for callable values

The feature is: explicit typed data that happens to contain procedure values.

Suggested spellings
-------------------
The intended shape is small and explicit.

Example A -- Struct table with named handlers

```mc
struct ShellRoutes {
    log: func(Message) Effect,
    kv: func(Message) Effect,
}

const ROUTES: ShellRoutes = ShellRoutes{
    log: route_log,
    kv: route_kv,
}
```

Example B -- Fixed array of one exact signature

```mc
const OPS: [3]func(u8) u8 = [3]func(u8) u8{
    op_inc,
    op_dec,
    op_flip,
}
```

Example C -- Explicit table lookup followed by ordinary call

```mc
handler: func(u8) u8 = OPS[index]
return handler(value)
```

Why this surface exists
-----------------------
- It reuses existing procedure-type and ordinary-call semantics instead of
  inventing a second callable model.
- It matches the repository's preference for explicit typed data rather than
  hidden registries or callback frameworks.
- It gives user-space and library code a clean table-shaped pattern without
  forcing the kernel architecture to prefer tables over explicit owner-local
  routing.
- It stays inside one narrow parser/sema/constant-evaluation/import/codegen
  slice rather than widening into a broader callable feature wave.

Non-goals
---------
- closures or captured environments
- lambda syntax or anonymous procedure literals
- callback registries, service discovery tables, or dynamic broker layers
- implicit coercion between distinct procedure signatures
- generic procedure instantiation as a constant ingredient
- erased `void*`-style callback conventions
- method promotion, object-model dispatch, or hidden vtable semantics
- using language support for procedure tables as an argument to replace the
  kernel's explicit routing style wholesale

Design guardrail
----------------
Admitting explicit procedure-value tables should be understood as a data-shape
decision, not as an architecture decision. The language may admit the shape
while the kernel and reset-lane system still prefer flat owner-local routing,
switches, and topology-owned explicit dispatch.

Current boundary
----------------
The admitted surface is intentionally limited to named procedures with exact
signatures stored in explicit typed aggregates. It does not imply procedure
equality, ordering, closures, erased callbacks, or a broader callback
framework.

Imported-module note
--------------------
The admitted cross-module surface includes both fixed arrays and struct-shaped
constant aggregates containing procedure values. Imported struct constants may
be copied into local constants, projected through fields, and called through
those projected procedure values under the same exact-signature rules.
