Modern C - Memory Model

Purpose

This document freezes the storage, aliasing, lifetime, and pointer model for v0.2.
It exists to keep the compiler, MIR, backend, and standard library aligned on what memory operations are allowed to mean.

Authority and scope:
- `language_spec.txt` remains the semantic authority for the language.
- `types_spec.txt` remains the authority for semantic type forms.
- `allocation_spec.txt` remains the allocation-convention note.
- `runtime/memory_architecture_note.txt` records staged runtime and kernel memory-strategy guidance outside the core language contract.
- this document freezes the memory model, pointer provenance rules, aliasing rules, lifetime rules, and checked-invalid-operation boundaries for v0.2.
- this document does not define a hidden runtime, GC, or ownership-tracking system.
- this document defines the language contract for admitted behavior; the current bootstrap compiler may diagnose or dynamically trap only a subset of these rules.

1. Core Position

Modern C memory rules should be:
- explicit
- small
- deterministic for defined behavior
- honest about invalid behavior

Rules:
- the compiler must not invent non-aliasing facts not stated by the program
- the language must not require a garbage collector or hidden lifetime extension
- byte-wise access should remain available for inspection and interop
- the memory model should preserve the machine-level reality of storage without exposing undefined convenience

2. Storage And Lifetime

Storage kinds:
- locals
- globals
- heap allocations
- array elements
- aggregate fields
- string literal storage
- imported or externally provided storage

Rules:
- every storage object has a lifetime
- a value stored in memory is valid only while its storage is live
- storage that outlives a scope must be returned or stored in an owning form
- borrowed views do not extend lifetime

Consequences:
- `Slice<T>` and the canonical string type are non-owning views
- `Buffer<T>` owns its storage
- returning a view into temporary internal storage is invalid design

Storage invalidation events:
- storage may end through scope exit, explicit free, region or arena reset, pool-slot return, mapping teardown, or any equivalent runtime operation that ends the underlying storage contract
- when storage ends, every pointer, slice, string view, or other borrowed view into that storage becomes invalid immediately
- ending one storage region does not implicitly extend or relocate the lifetime of values borrowed from it
- checked builds may trap some invalid uses after such events, but the language rule does not depend on dynamic trapping

3. Initialization And Temporary Values

Initialization rules:
- top-level `var` declarations without an initializer are zero-initialized
- local storage declarations without an initializer create live storage with indeterminate contents until the program writes a value
- reading local or heap-backed storage before it has been initialized is invalid behavior
- aggregate fields and array elements follow the same rule: any part not yet initialized is indeterminate until written
- checked builds may trap some obvious uninitialized uses, but v0.2 does not require a full definite-assignment proof in the bootstrap checker

String literal and static storage:
- string literals reside in static read-only storage for the lifetime of the program
- imported or external storage follows the lifetime contract of the defining runtime or FFI boundary, not a hidden language-owned extension rule

Temporary values:
- v0.2 does not introduce a separate temporary-storage ownership category
- unnamed temporary values produced during expression evaluation live only for the enclosing full expression unless materialized into named or returned storage by ordinary language rules
- taking a pointer or borrowed view into a temporary does not extend that temporary's lifetime
- using a pointer, slice, or string view after the temporary storage it referred to has ended is invalid behavior

4. Pointer Model

Pointer validity:
- a pointer is valid only while it still refers to live storage under the program rules
- dereferencing an invalid pointer is invalid behavior
- nil is the null pointer value

Pointer arithmetic:
- pointer arithmetic is valid only within a single live allocation, array, or one-past the end of that object
- pointer arithmetic may not cross into unrelated allocations

Pointer comparison:
- pointer comparison is defined only for pointers into the same allocation or against nil
- comparisons across unrelated allocations are not meaningful enough to rely on

Integer and pointer conversion:
- pointer-to-integer and integer-to-pointer conversions are explicit and unsafe
- a round trip preserves usability only if the numeric value is unchanged and the source allocation remains live
- these conversions are intended for FFI, MMIO, and low-level runtime code

Provenance guidance:
- pointers derived from a live object remain tied to that allocation for arithmetic, comparison, and dereference purposes
- forming a pointer from an unrelated integer does not create valid ownership or lifetime facts by itself
- outside explicit low-level APIs such as MMIO and FFI, forged pointers should be treated as unsafe and invalid to dereference unless they designate live storage established by the program or runtime contract
- pointers into mapped or runtime-provided storage remain valid only while that storage remains live and accessible under the relevant runtime contract

5. Object Representation And Aliasing Rules

Object representation:
- the object representation of a live value consists of its declared field or element bytes plus any target-defined padding bytes required by layout
- field order is the declared order; size, alignment, and padding follow the target ABI unless an admitted representation attribute says otherwise
- padding bytes may contain unspecified contents unless the program has explicitly written them through byte-wise operations
- byte-wise copy of a live object's storage copies its full representation, including padding
- semantic value equality and type identity are not defined by raw-byte equality alone

No strict-aliasing UB:
- Modern C does not adopt C-style strict-aliasing undefined behavior
- different pointer types may alias the same storage
- byte-wise inspection through `ui8` pointers or byte slices is always allowed

Explicit non-aliasing:
- non-aliasing assumptions must be stated explicitly by the program
- `@noalias` is the semantic escape hatch for parameters that promise non-aliasing
- the compiler may rely on `@noalias` only where the semantic checker has accepted it

Rules:
- aliasing facts come from the program, not from backend folklore
- the backend must not strengthen aliasing beyond the semantic contract

6. Constness And Mutability

Const model:
- `const` is part of the type
- const on aggregates is deep for the reachable value through that binding
- pointer constness must be written at the pointer layer

Rules:
- `const` controls mutation through a binding
- `const` does not magically change storage ownership
- `const` does not imply noalias
- `const` does not create a separate lifetime class

7. Allocation And Ownership

Allocation boundary:
- allocation is a library feature, not a language primitive
- allocation failure is explicit
- memory must be freed through the same allocator family that produced it

Rules:
- `Buffer<T>` owns heap-backed contiguous storage
- `Slice<T>` and string views borrow storage
- fixed arrays are values, not heap owners
- freeing storage invalidates all pointers and views into it
- bulk-release operations such as arena reset, region release, pool return, and unmap invalidate all pointers and borrowed views into the released storage just as ordinary free does

Consequences:
- ownership transfers must be explicit in APIs
- callers must not keep using a pointer, slice, or view after the underlying storage is released
- allocator APIs should remain separate from ordinary value semantics
- the language does not require arena, pool, slab, or page-allocation syntax; those remain library or runtime design choices

8. Checked, Invalid, And Diagnosable Behavior

Defined arithmetic behavior:
- integer overflow wraps
- signed and unsigned integer arithmetic use modular wrap

Checked faults:
- division by zero
- shift counts outside the bit width
- checked bounds violations where the compiler inserts explicit checks

Invalid behavior:
- read of uninitialized storage
- use-after-free
- out-of-bounds access
- invalid pointer dereference
- use of a slice after its underlying storage becomes invalid
- conflicting unsynchronized non-atomic accesses from multiple threads

Rules:
- semantic analysis should diagnose statically provable violations where feasible
- checked builds may trap additional invalid operations
- checked behavior does not redefine release semantics
- the compiler should preserve explicit faults instead of silently recovering

9. Volatile And Atomics

Volatile:
- volatile access is an explicit low-level access boundary
- volatile is not a substitute for ownership, aliasing, or synchronization rules

Atomics:
- atomics are explicit synchronization primitives
- atomics are not interchangeable with ordinary loads and stores
- atomics are not interchangeable with volatile operations

Admitted ordering vocabulary for v0.2:
- `Relaxed`
- `Acquire`
- `Release`
- `AcqRel`
- `SeqCst`

Ordering model:
- relaxed atomic operations guarantee atomicity on the addressed atomic object, but do not by themselves establish ordering for unrelated ordinary memory accesses
- a release operation on an atomic object synchronizes with an acquire operation that reads the value written by that release, establishing a happens-before edge for ordinary memory effects ordered around those operations
- acqrel operations provide both acquire and release behavior on read-modify-write operations
- seqcst operations additionally participate in one global sequentially consistent order for seqcst atomic operations
- volatile operations preserve observable device or boundary accesses but do not by themselves establish synchronization between threads

Rules:
- volatile and atomic behavior should remain distinct in sema and MIR
- memory ordering should be carried explicitly where the language and library expose it
- backend lowering must preserve the explicit boundary recorded by the front end

10. View Types And Lifetime Guidance

Borrowed views:
- slices and strings are borrowed views
- views are cheap to copy but do not own storage

Guidance:
- functions that return storage whose lifetime outlives the callee should return `Buffer<T>` or another owning type
- functions that return views should return `Slice<T>` or the canonical string type as appropriate
- libraries should not return borrowed views into temporary internal allocations
- libraries should not return borrowed views tied to request-local, region-local, pool-local, or mapped storage unless that lifetime is part of the explicit API contract and remains valid after return

11. Non-Goals

This document does not add:
- GC
- reference counting as a language rule
- automatic lifetime extension
- hidden borrow checking
- a strict-aliasing model copied from C
- a second ownership system layered on top of the language

Bottom line:
- storage is explicit
- aliasing is explicit
- lifetimes matter
- views do not own
- the compiler must preserve the memory behavior the program actually states
