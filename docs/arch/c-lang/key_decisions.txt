Modern C - Key Decisions

Purpose

This file records major settled language and compiler-boundary decisions that benefit from a short rationale in addition to the normative spec text.

Use this file for:
- decisions that materially affect parser, sema, MIR, runtime, or stdlib boundaries
- concise rationale for why the repository chose one design over nearby alternatives

Do not use this file for:
- unresolved questions
- exhaustive language reference text
- implementation status tracking

1. Explicit Conversion Syntax

Decision:
- Explicit conversions use parenthesized type-expression syntax: `(<type-expr>)(value)`.

Examples:
- `(UserId)(raw)`
- `(i64)(count)`
- `(Slice<i32>)(values)`
- `(*u8)(addr)`

Why:
- keeps function calls and conversions in different syntactic categories
- avoids scope-sensitive reinterpretation of `Foo(x)` between function and type meanings
- gives the parser an explicit boundary instead of deferring ambiguity into semantic analysis
- scales cleanly to raw pointer, generic, const, and other richer type-expression targets

Consequence:
- ordinary `foo(x)` remains expression-call syntax only
- bare named forms like `UserId(x)` are not conversion syntax in v0.2
- MIR lowering still classifies conversions semantically, but the frontend no longer depends on a bare-name conversion fallback

2. Builtin Aggregate Ownership

Decision:
- Builtin aggregate shapes are owned by the semantic type layer, not redefined independently in MIR or LLVM lowering.
- The canonical builtin aggregates are `string`, `Slice<T>`, and `Buffer<T>`.
- Surface spellings like `str`, `string`, and `cstr` normalize through sema type canonicalization before later phases consume them.

Why:
- keeps builtin field/layout rules in one place instead of duplicating `ptr`/`len`/`cap`/`alloc` knowledge across sema, MIR, and backend code
- prevents drift where one layer treats string-like spellings as equivalent and another treats them as distinct
- makes aggregate init, field access, and executable lowering consume the same contract

Consequence:
- MIR validation and LLVM lowering must consult semantic builtin aggregate metadata instead of carrying private tables
- new builtin aggregate behavior should be added in sema type helpers first, then consumed downstream
- executable support for aggregate field/index effects depends on this normalized sema-owned metadata rather than source spelling

3. Phase-1 Error Representation

Decision:
- keep the language-level explicit `(value, Error)` model unchanged
- keep the hosted ABI for `Error` scalar at the stdlib/runtime boundary for now
- treat `Error` as an opaque encoded value classified through `errors.kind(err)` and `errors.code(err)` rather than as a raw integer sentinel

Why:
- real programs now need more meaning than anonymous `fail_code(1)` style sentinels provide
- moving directly to heap-backed error objects would widen runtime ABI, allocation policy, and ownership questions all at once
- a scalar kind/code encoding gives immediate classification wins without reintroducing hidden control flow or premature object-lifetime policy

Consequence:
- callers should construct errors through helpers such as `errors.fail(...)` and `errors.fail_errno(...)` instead of inventing raw numeric layouts
- hosted runtime failures should preserve errno-classification through the encoded scalar channel
- richer message, source-context, cause-chaining, or heap-backed error objects remain a later follow-up rather than part of the current admitted slice

4. Fatal Primitive Admission Boundary

Decision:
- admit one compiler-known fatal primitive, `panic(code)`, using ordinary call syntax
- require exactly one non-negative compile-time integer fault code
- treat `panic(code)` as no-return control flow in sema, MIR, and backend lowering
- keep recoverable failure on the explicit `(value, Error)` path
- do not admit `require`, `expect`, a panic keyword, unwinding, or a broader runtime panic object in the same slice

Why:
- real freestanding and kernel-shaped code now needs one explicit fatal path that is stronger than a helper convention but narrower than a second failure system
- ordinary call syntax keeps the parser and source contract small while still letting the compiler own fatal semantics directly
- preserving the fatal edge explicitly in sema and MIR is more mechanically honest than lowering it immediately into an ordinary library call
- admitting `require` at the same time would mostly be spelling sugar until real invariant-heavy code proves it carries distinct semantic value

Consequence:
- internal invariants can now use `if !cond { panic(code) }` without changing the recoverable `Error` model
- MIR and backend ownership now include one deterministic fatal edge that lowers through the existing narrow trap or abort path
- any later proposal for `require`, richer panic payloads, or recoverable convenience helpers must justify itself against the landed `panic(code)` baseline in real code

5. Module Visibility And Package-Internal Sharing

Decision:
- ordinary top-level declarations are public to legal importers by default
- `@private` is the only declaration-level hiding mechanism
- a source file named `internal.mc` is the only package-internal module kind
- package identity is explicit and build-owned rather than inferred from raw
	directory adjacency
- direct-source workflows do not admit `internal.mc` because they do not carry
	explicit package identity

Why:
- removes the curated export-list contract and replaces it with one simpler
	semantic rule the parser, sema, `.mci`, and project workflow can all share
- keeps package sharing module-scoped rather than inventing a second
	declaration-level package-private lattice
- avoids path-based heuristics that would make import legality depend on local
	filesystem accidents instead of declared build state
- keeps direct-source mode honest about what it can and cannot prove without
	project metadata

Consequence:
- grammar and examples remove `export {}` from the v0.2 source contract
- build descriptions must name package identity explicitly for project-driven
	workflows
- `.mci` must summarize the non-private semantic surface plus enough metadata
	to validate `internal.mc` imports
- later implementation phases should reject stale export-block assumptions
	rather than carrying compatibility shims

6. Struct Embedding Deferred — Explicit Composition Is The Default

Decision:
- struct embedding (anonymous embedded fields with promoted field lookup) is not
  admitted in the current language.
- explicit composition through named fields is the only struct nesting model.
- with-helpers (e.g. `kvwith`, `logwith`, `bootwith_log`, `pathwith`) are the
  standard relief for the "update one field, carry the rest" pattern.

Why:
- Phase 164 audit measured real wrapper pressure in the reset-lane kernel and
  found that every identified pressure site was the "update one field, carry
  the rest" struct-update antipattern — already solved by the `kvwith`/`logwith`
  pattern in the existing codebase.
- Adding with-helpers to `boot.mc` and `serial_shell_path.mc` removed all
  repeated struct literals in `kernel_dispatch.mc` without any compiler change.
- The delegating accessor functions in `serial_shell_path.mc` are intentional
  encapsulation, not boilerplate: callers must not need to know which inner
  struct owns a reply field.
- Plan 9 clarity means simplicity of structure and flat readable code, not
  mandatory adoption of Plan 9 language conveniences like embedding.
- Promoted-field lookup with ambiguity rules, layout implications, and
  disambiguation costs adds language complexity that the audit did not justify.

Consequence:
- wrapper struct pressure should be resolved by: (a) with-helpers for
  single-field updates; (b) smaller records or better owner factoring if the
  record has too many unchanging fields; (c) explicit delegation functions if
  callers need to forward through a contained sub-struct's accessors.
- struct embedding proposals must prove repeated pressure that neither
  with-helpers nor owner refactoring can address before reopening this question.
- if reopened: any admitted form must be strictly structural, shallow, explicit
  in docs, and ambiguity-rejecting; no method promotion, subtyping, or implicit
  conversion may arise.

Revisit trigger:
- a wrapper record where with-helpers and owner refactoring would themselves
  require more boilerplate than the embedding they replace, and the lookup
  semantics can be proved unambiguous in that use case.
