Modern C - Ownership-Based Build And Test Selection Note

Purpose

This note records the intended next step for smarter local validation in the
repository: keep ordinary CMake incremental rebuilds, but make test selection
explicitly ownership-based so small changes do not force unrelated local test
runs.

Current state

- `cmake --build build/debug -j4` is already incremental for compiled sources
- local friction now comes more from test selection than from compilation
- repository tests are increasingly organized around explicit ownership slices:
  parser, sema, MIR, codegen, tool workflow, tool build-state, tool
  real-project, and the higher-level smoke/audit helpers
- the grouped tool workflow and real-project suites are now split into
  focused CTest targets rather than one monolithic workflow target and one
  monolithic real-project target
- `make select-tests` is the low-friction wrapper for
  `tools/select_tests.py` during narrow local iteration
- the retired freestanding proof harness has been removed from the live
  repository surface and is no longer part of the active ownership-based
  selection contract

Design position

- do not start with content-addressed remote caching or a Bazel-like graph
- start with an explicit changed-path to owning-CTest mapping
- keep the mapping conservative and reviewable
- prefer broad fallback over hidden false negatives

Why this is the right first step

- the repository already has named CTest slices with semantic ownership
- recent freestanding cleanup made kernel ownership narrower and more stable
- explicit ownership mapping matches the test design principles better than a
  heuristic or opaque cache
- local users mostly need a faster answer to "what should I rerun now?"

Canonical first tool

- `tools/select_tests.py`
- `make select-tests` as the repository-facing wrapper

Behavior

- discover changed paths from an explicit path list or, by default, from the
  current branch merge-base against `origin/main`, `main`, `origin/master`, or
  `master`
- include untracked files in that changed-path set
- map changed paths to owning CTest targets
- print the selected targets and the reason each target was selected
- optionally run an incremental build and the selected tests
- optionally skip selected tests whose owned inputs fingerprint matches the
  last successful local cached run

Important rule

- the selector is a local iteration helper, not a release gate by itself
- broad gates still matter before landing non-trivial changes

Ownership mapping guidance

- syntax-layer compiler changes should route to parser, sema, MIR, and narrow
  codegen validation
- driver and support-layer changes should route to the grouped tool suites
- active `kernel/` changes should route to `mc_tool_workflow_kernel_reset_lane_unit`, which now
  builds and runs the repository-owned `kernel/build.toml` directly
- unknown executable-path changes should fall back conservatively rather than
  silently selecting nothing

Examples

- changing `kernel/src/kernel_dispatch.mc` should select
  `mc_tool_workflow_kernel_reset_lane_unit`
- changing `tests/tool/tool_real_project_suite.cpp` should select
  `mc_tool_real_project_.*_unit`
- changing `compiler/driver/` should select the grouped tool suites

Current focused rerun commands

- workflow family: `ctest --test-dir build/debug -R 'mc_tool_workflow_.*_unit' --output-on-failure`
- build-state family: `ctest --test-dir build/debug -R mc_tool_build_state_unit --output-on-failure`
- real-project family: `ctest --test-dir build/debug -R 'mc_tool_real_project_.*_unit' --output-on-failure`

Current bounded cache model

- the selector may keep a local cache file under the active build tree
- cache entries are per CTest target and record the last successful owned-input
  fingerprint
- cache misses or unknown ownership should rerun tests rather than guessing

Non-goal:

- a fully general graph engine with hidden dependency discovery

Bottom line

- use CMake for incremental compilation
- use explicit ownership mapping for selective local test execution
- use per-target owned-input fingerprints only as a conservative local rerun
  skip, not as a replacement for broader gates
