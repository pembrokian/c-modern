Modern C - Freestanding Kernel Test Architecture Contract

Purpose

This document records the current repository contract for freestanding kernel
tests.

It is no longer a migration sketch. It describes the active public surfaces,
the supported local selectors, the ownership model for expectations, and the
remaining gaps that are still intentionally outside the fully descriptor-only
runtime shape.

Current contract

The freestanding kernel test architecture now has four public top-level
surfaces:

- runtime
- synthetic
- docs
- artifacts

These map directly to the canonical CTest targets:

- `mc_tool_freestanding_kernel_runtime_unit`
- `mc_tool_freestanding_kernel_synthetic_unit`
- `mc_tool_freestanding_kernel_docs_unit`
- `mc_tool_freestanding_kernel_artifacts_unit`

Those four surfaces are the public contract. Shard-based targets, shard-based
selectors, and shard-owned source files are no longer part of the active
freestanding kernel interface.

Selector contract

The supported freestanding kernel selectors are intentionally small and
explicit:

- `kernel-runtime`
- `kernel-runtime:<label>`
- `kernel-synthetic`
- `kernel-synthetic:<label>`
- `kernel-docs`
- `kernel-artifacts`

Related top-level freestanding selectors that remain valid because they are
separate surfaces, not compatibility shims:

- `bootstrap`
- `system`
- `all`

Decision: no broader compatibility layer is preserved.

That means:

- no shard selectors
- no `kernel-case:` aliases
- no wildcard selector contract beyond the exact label-based selectors above
- no compatibility affordances that describe historical shard ownership rather
  than current surface ownership

Local narrow runs should use the label selectors directly. Broader local runs
should use the top-level surface selectors.

Surface ownership

Runtime surface

- phases105 through 147 are descriptor-driven under
  `tests/tool/freestanding/kernel/runtime/phase.../`
- each runtime phase directory owns:
  - `phase.toml`
  - adjacent run expectations
  - adjacent MIR projection expectations
- runtime descriptor loading and runtime-case lookup live in
  `tests/tool/tool_suite_common.*`

Synthetic surface

- phases85 through 88 live permanently under
  `tests/tool/freestanding/kernel/synthetic/`
- they remain separate because they generate standalone proof projects rather
  than validating the shared `kernel/build.toml` runtime path
- they are selected through `kernel-synthetic` and
  `kernel-synthetic:<label>` only

Docs surface

- publication and documentation assertions live under
  `tests/tool/freestanding/kernel/docs/...`
- each docs audit owns its descriptor and checked text expectations in that
  surface

Artifacts surface

- relink and artifact assertions live under
  `tests/tool/freestanding/kernel/artifact_specs/...`
- each artifact audit owns its descriptor and adjacent checked expectations in
  that surface

Legacy standalone runtime proofs

The old phases97 through 104 standalone C++ proof files have been retired.
Their preserved narrow MIR proof slices still live under
`tests/tool/freestanding/kernel/runtime/legacy_goldens/` as repository-owned
regression references inside the runtime surface. They do not justify any
extra selector vocabulary.

Expectation ownership

The repository now uses surface-owned expectation placement:

- runtime phase expectations live adjacent to each descriptor under
  `tests/tool/freestanding/kernel/runtime/phase.../`
- earlier standalone runtime MIR expectations live under
  `tests/tool/freestanding/kernel/runtime/legacy_goldens/`
- synthetic expectations live under
  `tests/tool/freestanding/kernel/synthetic/`
- artifact expectations live adjacent to each artifact descriptor under
  `tests/tool/freestanding/kernel/artifact_specs/.../`
- docs expectations live adjacent to each docs descriptor under
  `tests/tool/freestanding/kernel/docs/.../`

The old shared `tests/tool/freestanding/kernel/goldens/` tree is retired and
is not part of the active ownership model.

MIR contract

The freestanding kernel runtime MIR contract is explicit and stable:

- `mir_contract = "first-match-projection-v1"`

Each runtime descriptor owns:

- `selectors = [...]`
- `mir_golden = ...`

Literal full-MIR diffs are intentionally out of scope until the compiler
publishes a stable full-text MIR contract.

Changed-path selection

`tools/select_tests.py` is part of the repository contract.

It routes kernel-related changes by top-level surface ownership:

- runtime
- synthetic
- docs
- artifacts

The selector intentionally stays explicit and reviewable. The repository is
not using opaque dependency discovery or hidden grouping for local ownership
selection.

Current remaining gaps

The remaining gap is not shard cleanup or selector cleanup. Those are done.

The remaining architectural gaps are:

- phases97 through 104 are still compiled standalone runtime proofs rather than
  per-phase descriptor directories
- runtime is therefore not yet a pure "add one directory, add one phase"
  filesystem-only model across all runtime proofs
- docs and artifacts remain separate surfaces by design rather than being
  collapsed into one generic runtime tree

These are intentional current boundaries, not accidental leftovers.

Contributor workflow

When adding or updating a freestanding kernel test, use this order:

1. Decide which top-level surface owns the behavior.
2. Update the owned descriptor or proof file in that surface.
3. Update the owned expectations adjacent to that surface.
4. Update `tools/select_tests.py` only if surface ownership changes.

Normal ownership rules:

- add new shared-runtime phases under `kernel/runtime/phase.../`
- update `kernel/synthetic/` only for phases85 through 88
- update `kernel/docs/...` only for publication and documentation audits
- update `kernel/artifact_specs/...` only for artifact and relink contracts
- update `runtime/legacy_goldens/` only when touching the earlier standalone
  runtime proof files for phases97 through 104

Validation

Focused validation commands:

- `cmake --build build/debug -j4`
- `ctest --test-dir build/debug -R 'mc_tool_freestanding_kernel_(runtime|synthetic|docs|artifacts)_unit' --output-on-failure`

Useful local selector examples:

- `mc_tool_freestanding_tests <source-root> <binary-root> kernel-runtime`
- `mc_tool_freestanding_tests <source-root> <binary-root> kernel-runtime:phase109_first_running_kernel_slice_audit`
- `mc_tool_freestanding_tests <source-root> <binary-root> kernel-synthetic`
- `mc_tool_freestanding_tests <source-root> <binary-root> kernel-synthetic:phase85_endpoint_queue`

Summary

The active contract is now straightforward:

- four public kernel surfaces
- explicit narrow selector vocabulary
- no preserved shard compatibility layer
- surface-owned expectations
- explicit descriptor-owned MIR projection contract

The only substantial remaining runtime gap is that phases97 through 104 are
still standalone compiled proofs rather than descriptor directories.

2. Runtime proof discovery moved out of handwritten per-phase tables

- the first late-runtime conversion point landed as a folder-based phase
  descriptor layout
- phases105 through 147 now use the runtime-folder descriptor layout
- the former shard1 runtime phases105 and 106 also use the runtime-folder
  descriptor path

3. Stable outer workflow stayed in place during the hybrid migration

- `kernel-runtime:<label>` is now the direct local runtime selector
- `kernel-synthetic:<label>` now owns the separate synthetic proof selector

4. Changed-path routing now follows top-level surfaces

- `tools/select_tests.py` routes by surface ownership boundaries
- runtime phase directories, synthetic proofs, docs descriptors, and artifact
  descriptors now map directly to their top-level surfaces

5. Registry pressure was reduced and the shard runtime layer was removed

- runtime-folder cases now resolve through descriptor-driven lookup rather than
  a broad compiled per-phase table
- phase108 manifest and phase114 plan-note checks now live on the descriptor-
  driven artifact and docs surfaces
- the synthetic surface keeps only the standalone-project proofs for phases85
  through 88 as a separate surface
- the old shard2 through shard9 source files and shard-named run goldens were
  removed without changing the active public interface

Historical near-term execution order

The steps below are now effectively completed for the hybrid migration and are
retained only as a record of how the repository moved from the earlier shard2
experiment to the current runtime-folder layout.

Focused validation commands for this sequence

- `cmake --build build/debug -j4`
- `ctest --test-dir build/debug -R mc_tool_freestanding_kernel_runtime_unit --output-on-failure`
- `ctest --test-dir build/debug -R 'mc_tool_freestanding_kernel_(runtime|synthetic|docs|artifacts)_unit' --output-on-failure`

Future simplification

If the repository continues simplifying the runtime surface, the next step is
not selector work and not shard cleanup. Those are complete.

The next simplification target is phases97 through 104.

If those proofs are ever moved into descriptor-owned runtime directories, the
resulting architecture would be:

- one runtime surface where every runtime proof is directory-owned
- one synthetic surface for phases85 through 88
- one docs surface
- one artifacts surface

That would remove the only remaining runtime-specific exception in the current
contract.

Until that happens, the current architecture described above is the repository
contract.
