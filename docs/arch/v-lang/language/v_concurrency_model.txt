V - Concurrency Model

Purpose

This document defines the intended service-level concurrency model for V.

Design center

V is intended for message-driven system services.
Its concurrency model should therefore make the following shape natural:
- start tasks explicitly
- communicate through channels or equivalent explicit queues
- transfer ownership through those communication boundaries
- avoid default shared mutable state

The model should support CSP-shaped composition without turning the language
into a hidden scheduler runtime or a framework-driven actor system.

1. Core stance

The current design direction is:
- tasks are explicit
- channels are explicit
- shared memory is allowed only deliberately
- ownership transfer is the normal way data crosses task boundaries

This means V concurrency is not primarily about:
- futures everywhere
- implicit async rewriting as the main execution story
- background runtimes inventing work on the programmer's behalf

2. Tasks

V should provide an explicit task concept.

Baseline expectations:
- task creation is visible in source
- task entrypoints are ordinary functions or a narrowly defined equivalent
- task lifecycle should be observable enough for service shutdown and error
  handling

Important design pressure:
- services need clear startup, ready, stop, and exit paths
- the concurrency model should support structured service bring-up without
  forcing a large framework

3. Channels

Channels should be the primary built-in coordination primitive for service
composition.

Design role:
- channels coordinate execution between tasks
- channels carry ordinary protocol values
- channels are not themselves the protocol definition

Desired baseline behavior:
- send is explicit
- receive is explicit
- ownership transfer on send is the default rule for owned values
- closing or shutdown behavior should be explicit and predictable

4. Ownership transfer across tasks

The concurrency model depends on the memory model:
- sending an owned value transfers ownership
- the sender may not keep using a moved value
- the receiver becomes the new owner

This is one of the most important V design rules because it aligns:
- memory behavior
- concurrency behavior
- service protocol behavior

5. Shared memory

Shared memory should be possible, but not the default service-composition
story.

If V admits shared memory:
- it should require explicit shared types or synchronization primitives
- it should not arise accidentally from ordinary assignment or capture
- visibility and mutation rules must remain explicit

Reason:
- the target architecture wants explicit protocols and capability flow
- default shared mutable state makes service boundaries harder to reason about

6. Scheduling

V should not require a hidden green-thread scheduler as a core language
feature.

Open implementation space may include:
- one task maps to one host thread in early runtimes
- a later runtime may multiplex tasks if it can do so honestly

But the language-level commitment should remain narrow:
- task semantics must not depend on opaque scheduler cleverness
- blocking, wakeup, and shutdown behavior should stay understandable

7. Blocking and readiness

V services will eventually need both blocking coordination and event-driven
readiness.

Design direction:
- channels handle in-process coordination
- readiness from the kernel or service runtime should appear through explicit
  APIs, not through magical async transformation

This allows the language to support event-oriented services without making all
code coroutine-shaped by default.

8. Cancellation and shutdown

V should specify a simple, explicit shutdown model.

The language and runtime should support:
- intentional service stop requests
- channel closure or equivalent end-of-stream signaling
- deterministic cleanup after task exit

V should avoid:
- arbitrary asynchronous task destruction
- hidden cancellation injection into ordinary code

A service language needs shutdown clarity more than clever interruption.

9. Ordering and visibility

The concurrency model must define what task communication guarantees.

Baseline direction:
- successful channel send or receive operations create a clear ownership and
  visibility handoff
- synchronization primitives, if admitted, must define memory-order effects
  explicitly
- plain unsynchronized shared mutation should not gain accidental guarantees

The exact low-level memory-order vocabulary can stay narrow at first, but the
spec must still be explicit about what communication establishes visibility.

10. Capability flow in concurrency

If tasks exchange capabilities or handles, the transfer must be explicit.

This implies:
- capability movement through channels is visible in program structure
- duplicated authority, if permitted, must be explicit
- task startup APIs should not silently inherit broad ambient authority beyond
  what the runtime or service launcher documents

11. What the concurrency model should make easy

V should make these patterns straightforward:
- a request loop that receives, handles, and replies
- a dispatcher that sends work to worker tasks
- a service that owns its local mutable state and shares little else
- a shutdown path that closes channels, drains work, and releases resources
- a readiness loop that integrates explicit event sources without infecting the
  whole language with async syntax

12. Non-goals

The first V concurrency model should not promise:
- a universal actor framework
- distributed transparency
- lock-free container breadth in the core language
- implicit process supervision trees baked into syntax
- fully general realtime scheduling guarantees

13. Summary

The intended V concurrency shape is:
- explicit tasks
- explicit channels
- transfer-oriented communication
- deliberate and limited shared memory
- explicit shutdown
- narrow scheduling assumptions

This keeps the language aligned with a protocol-first operating-system design
instead of drifting toward framework-owned concurrency.
