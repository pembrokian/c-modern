Veya sys v0 — First std Wrapper Layer

Status: companion to sys v0
Scope: thin Modern C wrapper over the status-based kernel contract
Audience: stdlib + application implementers

⸻

1. Purpose

The kernel boundary should stay status-based and pointer-oriented.
The first std wrapper layer exists to make that boundary usable in Modern C without adding policy.

The wrapper should:
	•	preserve explicit control flow
	•	translate Status into the repository's explicit Error model
	•	replace raw pointer-plus-length output patterns with ordinary slices and small result structs
	•	not add allocation, retries, serialization, or namespace lookup

⸻

2. Layering

Recommended split:
	•	sys.raw: exact syscall-shaped boundary
	•	sys: thin explicit wrapper for ordinary user code

Responsibilities:

sys.raw:
	•	exposes the kernel-shaped signatures
	•	uses Status directly
	•	accepts raw buffers and output pointers

sys:
	•	maps Status to Error
	•	presents value-plus-Error returns
	•	keeps move semantics for transferred capabilities explicit

⸻

3. Error Mapping

Status values should map into ordinary explicit Error values.

Recommended rule:
	•	Status.Ok means no error
	•	all other Status values map to module-local sys error codes
	•	the wrapper should preserve the original Status code for inspection

Example conceptual mapping:

enum SysCode {
    Invalid,
    PermissionDenied,
    NotFound,
    Exhausted,
    TooLarge,
    WouldBlock,
    Unknown,
}

func map_status(status: sys.raw.Status) Error

This remains ordinary explicit `(value, Error)` handling.
No hidden propagation is introduced.

⸻

4. Public Wrapper Shape

The first wrapper should stay narrow.

Conceptual API:

func endpoint_create() (Endpoint, Error)

func send(ep: Endpoint, data: []u8, caps: []Capability) Error

type RecvResult {
    bytes: u64
    caps: u64
}

func recv(ep: Endpoint, data: []u8, caps: []Capability) (RecvResult, Error)

type ProcSpec {
    image: Capability
    args: []str
    caps: []Capability
}

func spawn(spec: ProcSpec) (Process, Error)

func wait(p: Process) (i32, Error)

func kill(p: Process) Error

func now() Time

func sleep(ms: u64) Error

⸻

5. Wrapper Semantics

5.1 send

`send` is a destructive transfer for attached capabilities.
On success:
	•	capabilities in `caps` have moved out of the caller's authority set
	•	callers must not continue treating them as usable local handles

On failure:
	•	no capability transfer occurred
	•	the original handles remain valid for the caller

The wrapper must not hide this with implicit cloning.

5.2 recv

`recv` does not allocate.
The caller provides storage for both bytes and capability handles.
The returned `RecvResult` reports how much of that storage was populated.

If buffers are too small, `recv` returns `Error` corresponding to `TooLarge`.
The wrapper should not silently retry or allocate a larger message.

5.3 spawn

`spawn` is driven by `ProcSpec.image`, which is a launchable program capability.
The wrapper must not accept path strings and perform loader lookup internally.
That policy belongs above sys.

⸻

6. Example Wrapper Sketch

Conceptual implementation outline:

func endpoint_create() (Endpoint, Error) {
    ep: Endpoint
    status := sys.raw.endpoint_create(&ep)
    if status != sys.raw.Status.Ok {
        return Endpoint{}, map_status(status)
    }
    return ep, Error.ok()
}

func recv(ep: Endpoint, data: []u8, caps: []Capability) (RecvResult, Error) {
    raw := sys.raw.RecvMessage {
        data: data.ptr
        len: data.len
        caps: caps.ptr
        cap_count: caps.len
        recv_len: 0
        recv_caps: 0
    }
    status := sys.raw.recv(ep, &raw)
    if status != sys.raw.Status.Ok {
        return RecvResult{}, map_status(status)
    }
    return RecvResult {
        bytes: raw.recv_len,
        caps: raw.recv_caps,
    }, Error.ok()
}

The exact spelling can follow the admitted Modern C syntax once this layer is implemented.
The important part is the contract shape, not the temporary notation.

⸻

7. Deliberate Omissions

The first std wrapper should not include:
	•	dynamic message allocation helpers
	•	message serialization formats
	•	service discovery
	•	filesystem-backed program lookup
	•	automatic capability duplication
	•	async adapters

Those belong in higher layers.

⸻

8. Recommended Near-Term Structure

The smallest useful split is:
	•	sys.raw for compiler/runtime bring-up and tests
	•	sys for most low-level user code
	•	higher libraries above that for loaders, services, and protocols

That keeps the kernel ABI honest while still matching Modern C's explicit error model.
