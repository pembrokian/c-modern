Veya Lite v0 - First Bring-Up Architecture

Purpose

This document defines a simplified first-bring-up architecture for Veya.
It preserves the direction of the fuller Veya model while removing enough
mechanism and ambition to make early implementation realistic.

This is a first-system contract, not the final architecture.
If a later fuller Veya architecture is restored, it should grow from this
document rather than bypassing it.

Design stance

- keep the kernel small
- keep policy in user space
- keep authority explicit
- prefer message passing as the primary coordination model
- avoid locking in hosted Unix or POSIX semantics as the long-term design
- defer elegance that multiplies moving parts before the system boots

Architectural invariant

Veya Lite v0 is a small microkernel with explicit endpoint-based IPC,
explicit capability handles, explicit address-space management, and
user-space-owned policy.

The first system must be explainable as:

- tasks
- address spaces
- endpoints
- capability handles
- memory mappings
- one init process

If a feature cannot be explained in those terms, it is probably too early.

1. What This Version Is Trying To Achieve

The first version aims to prove:

- the kernel can boot and enter user space
- the kernel can create tasks and isolated address spaces
- user space can exchange structured messages through kernel-mediated endpoints
- user space can hold and transfer explicit capability handles
- one small init process can launch and coordinate a minimal service set

This version is not trying to prove the full eventual Veya graph model,
full namespace rewiring, zero-copy IPC, or a sophisticated runtime scheduler.

2. Structural Model

Kernel-owned responsibilities in v0:

- task creation and destruction
- address-space creation and mapping
- endpoint creation
- capability-handle validation
- message send or receive mediation
- interrupt delivery into explicit kernel-owned endpoints
- minimal scheduler and timer support

User-space-owned responsibilities in v0:

- init and service startup
- process orchestration policy
- naming or discovery policy
- filesystem policy
- network policy
- logging and diagnostics policy

Rule:
- the kernel owns mechanisms
- user space owns policy

3. Process And Task Model

The first system should adopt a simple process model:

- a process owns one address space
- a task is an executing thread of control inside one address space
- the first bring-up may start with one task per process if that reduces
  implementation complexity

Required lifecycle operations:

- create process
- create initial task
- exit task
- exit process
- wait for child completion

Initial simplification:
- detached tasks, structured task trees, and rich shutdown protocols are
  deferred until ordinary boot and service management are already working

4. IPC Model

IPC is the central coordination primitive for the first system, but the first
version should use the smallest honest form.

Required properties:

- kernel-mediated endpoint send
- kernel-mediated endpoint receive
- explicit request or reply shape in user protocols
- bounded message size
- blocking semantics are acceptable in the first version

Initial simplifications:

- copy-based IPC is acceptable
- zero-copy transport is deferred
- async completion frameworks are deferred
- multi-party channel frameworks are deferred
- distributed transparency is deferred

Rule:
- prefer one simple IPC primitive that works over several clever variants

5. Capability Model

Authority should be explicit from the start, but the first representation may
be simpler than the final Veya capability graph.

v0 capability model:

- a capability is a kernel-validated handle naming an allowed object or action
- capabilities are stored in a per-process handle table
- handles are unforgeable because only the kernel can create valid entries
- handles may be transferred explicitly over IPC where policy permits

Admitted capability targets in v0:

- endpoints
- processes or child-process wait handles
- memory objects or mapping rights
- interrupt-routing or device-facing handles only if needed for bring-up

Deferred from v0:

- a fully generalized graph-theoretic authority model
- revocation algebra beyond simple close or destroy rules
- capability derivation taxonomies beyond what real services need to boot

6. Namespace And Discovery

The first system should not begin with the full rewritable graph namespace.

Instead, v0 should admit one of these simple discovery models:

- a fixed init-known service table
- a small registry service
- a small mount or bind table owned by one namespace service

Required property:
- name resolution or service lookup must still result in explicit capability
  acquisition rather than ambient global reach

Deferred from v0:

- arbitrary graph rewiring as a first-class system-wide primitive
- remote and local namespace unification
- general mount or bind algebra

Rule:
- prove service discovery first
- prove namespace rewriting later

7. Scheduler Model

Scheduling exists, but it is a kernel mechanism, not the architecture story.

v0 scheduler goals:

- run ready tasks
- block tasks on receive or reply or wait
- wake tasks on message arrival, child completion, or timer events
- keep implementation simple and inspectable

Initial simplifications:

- one scheduler policy is enough
- per-CPU scheduling is deferred
- work stealing is deferred
- scheduler activations are deferred
- runtime-visible task frameworks are deferred

Rule:
- do not turn scheduling sophistication into an early blocker

8. Memory Model

The first kernel needs explicit memory ownership and mapping, but not the full
eventual memory architecture on day one.

Required kernel mechanisms:

- create address space
- map memory into an address space
- unmap memory
- transfer or share mapping rights explicitly when needed

Initial simplifications:

- copy-based IPC buffers are acceptable
- shared-memory regions may be absent or narrowly scoped at first
- region or arena discipline may be used internally where it helps bring-up,
  but it should not prevent ordinary debugging and iteration

Deferred from v0:

- zero-copy message transport as the default path
- a full region-only public memory object model
- advanced page-cache or VM policy

9. Device And Interrupt Boundary

The first version should keep hardware reality explicit and narrow.

Kernel responsibility:

- own interrupt routing
- expose explicit device-facing handles or messages only where needed

Architecture rule:

- MMIO, volatile access, and architecture-specific barriers belong in a
  hardware-facing boundary, not in general `mem` or `sync` APIs

Bring-up bias:

- start with the minimum device set required for boot, timer, console, and one
  storage or transport path
- defer a broad device framework

10. Observability

The first system should be observable, but observability must stay simple.

Required v0 observability:

- boot log output
- explicit service start or fail reporting
- deterministic boot phase markers where practical
- a small kernel or init diagnostic path that does not grant hidden authority

Deferred from v0:

- a broad tracing framework
- distributed introspection model
- full reproducibility guarantees for every artifact

11. Explicit Non-Goals For v0

Veya Lite v0 does not attempt to provide:

- a full graph of macro state machines as the initial implementation unit
- all interaction as ownership-moving channels with zero-copy guarantees
- a complete rewritable namespace graph
- a policy-rich kernel
- Unix syscall compatibility as a design goal
- a language-level scheduler, futures model, or hidden async runtime
- a fully self-hosting system as the first milestone

12. Recommended First Bring-Up Slice

The first implementation slice should look roughly like this:

- kernel boots
- kernel starts one init process
- init can create one child process
- processes communicate through one IPC endpoint primitive
- processes hold explicit capability handles
- one registry or namespace service exposes a tiny service-discovery path
- one logging path proves observability

Good enough first proof:

- init starts a log service and one echo or ping service
- a client sends a request through IPC
- the service replies
- capability checks and address-space isolation are exercised honestly

13. Growth Path After v0

Only after the first slice is stable should the architecture widen toward:

- richer capability transfer rules
- shared-memory and zero-copy optimization
- broader namespace and bind or mount behavior
- device-service expansion
- network-service expansion
- multicore-aware scheduling
- stronger reproducibility and self-hosting goals

Growth rule:
- add one new mechanism family only when a running system makes the next gap
  obvious

14. Relationship To Modern C

This architecture fits Modern C best if the language and stdlib remain small:

- no language-level scheduler
- no hidden runtime
- `sync` remains low-level and explicit
- `os` and later `hal` carry target-facing reality
- richer concurrency and service frameworks live in user space

The operating system should not force Modern C to absorb policy that belongs
in system services or userland libraries.

Bottom line

Veya Lite v0 keeps the soul of the original Veya direction:

- microkernel
- explicit authority
- explicit IPC
- policy in user space

But it rejects the attempt to prove the whole whiteboard at once.
The first system should be small enough to boot, inspect, debug, and revise
without needing the full final graph model on day one.
