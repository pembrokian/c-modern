Canopus Kernel End-State Architecture
=====================================

Status
------
PROPOSED

Purpose
-------
Describe the intended long-term internal kernel structure for Canopus after
the first bring-up slices are already working.

This note is not a phase closeout and not a claim that the repository has
already reached this architecture.
It records the target subsystem layout and ownership boundaries that later
kernel growth should converge toward.

Relationship to companion notes
-------------------------------
This document should be read after:

- `docs/arch/canopus/canopus_lite_v0.txt`
- `docs/arch/canopus/canopus_kernel_blueprint_v0.txt`
- `docs/arch/canopus/canopus_kernel_implementation_blueprint_v0.txt`
- `docs/arch/canopus/canopus_sys_interface_v0.txt`
- `docs/arch/canopus/canopus_syscall_mapping_v0.txt`
- `docs/plan/admin/canopus_kernel_bringup_speculative_roadmap.txt`

Those notes define the first admitted Canopus shape and one plausible bring-up
ordering.
This note answers a different question: once the first kernel path is real,
what stable internal architecture should later implementation work grow into?

Scope
-----
This document is about kernel-internal structure:

- subsystem boundaries
- directory and file-family ownership
- layering rules
- object-placement rules
- long-lived non-goals

This document is not:

- a promise that the full end-state kernel already exists
- a full user-visible ABI definition
- a complete driver model
- a full multicore design
- a package, image, or distribution specification

Architectural stance
--------------------
The end-state Canopus kernel should remain a small capability-based
microkernel.
Even after more features land, it should still optimize for:

- explicit authority
- user-space-owned policy
- inspectable control flow
- narrow subsystem interfaces
- deterministic bounded kernel mechanisms

It should not drift into:

- an in-kernel policy framework
- Unix-compatibility-first subsystem growth
- a large ambient namespace kernel
- hidden resource inheritance
- a driver-heavy monolithic-kernel shape

Core design rule
----------------
If a new feature cannot be assigned cleanly to one kernel subsystem without
smearing policy across several others, the design is probably not ready.

Top-level target layout
-----------------------
The kernel should converge toward a layout of this form:

```txt
kernel/
├── arch/
├── core/
├── sched/
├── ipc/
├── mm/
├── cap/
├── sys/
├── interrupt/
├── device/
├── init/
├── debug/
└── include/
```

The exact file names may evolve, but the ownership split should remain
recognizable.

1. `arch/` -- Hardware-Specific Mechanism
-----------------------------------------
`arch/` owns the architecture-local code required to make the kernel run on a
real target.

Typical contents:

- reset or boot entry
- trap entry and exit
- saved-context layout
- context-switch assembly glue
- MMU or paging implementation
- interrupt-controller hookup
- timer hardware hookup
- CPU-local initialization

Recommended structure:

```txt
arch/
└── <target>/
    ├── boot/
    ├── cpu/
    ├── mmu/
    ├── trap/
    ├── timer/
    └── context.S
```

Hard rules:

- Nothing outside `arch/` should manipulate registers or architecture control
  state directly.
- Generic kernel code should depend on architecture-neutral helpers, not on
  architecture-private structures.
- Page-table policy belongs in generic VM ownership; page-table mechanics
  belong here.

2. `core/` -- Shared Kernel Object Definitions
----------------------------------------------
`core/` owns the small shared types that several subsystems must agree on.

Typical contents:

- object ids
- kernel-wide status or error enums
- task or process state enums
- shared object headers or slot-state helpers
- small common structs that are truly cross-cutting

`core/` should stay definition-heavy and logic-light.

Hard rules:

- Do not let `core/` become a grab bag for random helpers.
- If behavior is meaningfully owned by one subsystem, put it there instead of
  in `core/`.
- Shared structs are acceptable here; cross-subsystem policy is not.

3. `sched/` -- Runnable, Blocked, And Wakeup State
--------------------------------------------------
`sched/` owns scheduler state transitions and runnable-task selection.

Responsibilities:

- ready-queue ownership
- runnable-task selection
- block and unblock paths
- sleep and wake bookkeeping
- preemption trigger integration with interrupt or timer sources

It may know:

- task states
- wake reasons
- deadlines or tick values

It should not know:

- IPC payload layout
- capability transfer mechanics
- address-space mapping policy

Required boundary:

- `ipc/`, `mm/`, `sys/`, and `interrupt/` may ask `sched/` to block, wake, or
  enqueue tasks, but they should not embed scheduler policy locally.

4. `ipc/` -- Endpoints And Message Transport
--------------------------------------------
`ipc/` owns endpoint state and message delivery semantics.

Responsibilities:

- endpoint creation and destruction
- bounded endpoint queues
- send and receive mechanics
- message storage and message lifetime rules
- delivery of byte payload plus attached-capability payload

It may depend on:

- `cap/` for handle-transfer validation and reinstall
- `sched/` for block or wake hooks
- `core/` for shared task or object identifiers

It should not own:

- global scheduling policy
- user-visible service protocols
- VM-backed shared-memory transport policy by default

Required rule:

- `ipc/` may trigger block or wake transitions, but the actual runnable-state
  policy still belongs to `sched/`.

5. `mm/` -- Address Spaces And Mapping Ownership
------------------------------------------------
`mm/` owns architecture-neutral virtual-memory state.

Responsibilities:

- address-space objects
- mapping records
- image, stack, and static-data mapping ownership
- page-fault classification and top-level handling
- process-teardown unmap paths
- higher-level memory or mapping-right logic

Split rule:

- generic address-space and mapping policy belongs in `mm/`
- architecture-specific page-table implementation belongs in `arch/*/mmu/`

It should not grow into:

- a filesystem cache layer
- a general kernel object allocator framework unless that is clearly required
- a hidden IPC transport policy owner

6. `cap/` -- Capability Handles And Rights
------------------------------------------
`cap/` owns the kernel-side representation of explicit authority.

Responsibilities:

- process-local handle-table mechanics
- handle lookup and validation
- rights checks
- install, remove, duplicate, or move helpers where admitted
- authority mediation for cross-process transfer

It should remain mechanical and explicit.

It should not own:

- service-discovery policy
- namespace policy
- user-space object naming
- high-level launch policy

Rule:

- handles are kernel-validated references and rights records, not user-space
  metadata conventions.

7. `sys/` -- User-Kernel Boundary
---------------------------------
`sys/` owns the syscall entry surface and dispatch glue.

Responsibilities:

- syscall decode
- argument validation
- copying or marshalling at the kernel boundary
- dispatch into the owning subsystem
- status-code normalization for the user-visible ABI

It should stay thin.

Hard rules:

- Real IPC semantics belong in `ipc/`, not in `sys/`.
- Real spawn or wait ownership belongs with the owning task or process
  subsystem, not in `sys/`.
- `sys/` should not become a policy layer or an alternate home for kernel
  business logic.

8. `interrupt/` -- Interrupt Routing And Timer Trigger Paths
------------------------------------------------------------
`interrupt/` owns generic interrupt-facing control flow above the
architecture-local entry layer.

Responsibilities:

- interrupt dispatch routing
- generic IRQ bookkeeping
- timer tick handoff into kernel time or scheduler logic
- interrupt-to-kernel-event bridging

Split rule:

- raw trap entry belongs in `arch/`
- generic interrupt dispatch and wakeup integration belongs here

It should remain narrow.
If an interrupt source can later be represented as a user-space service with a
small explicit kernel mediation layer, do not eagerly absorb its policy here.

9. `device/` -- Minimal Kernel-Resident Device Support
------------------------------------------------------
`device/` owns only the device code that must remain in the kernel for the
current system slice.

Likely early contents:

- serial or console output
- timer device support
- one narrow boot-critical device path if later required

Default rule:

- if a device function can live in user space without breaking the current
  bring-up goal, it should not be placed in the kernel

This subsystem should stay deliberately small.
The end-state Canopus direction is still user-space-heavy policy, not a large
driver-resident kernel.

10. `init/` -- Boot Sequencing And First User Handoff
-----------------------------------------------------
`init/` owns kernel-side bring-up orchestration.

Responsibilities:

- boot sequencing after low-level architecture entry
- subsystem initialization order
- first process or first task construction
- bootstrap capability seeding
- handoff to the first user-space entry path

This subsystem is about one-time kernel bring-up sequencing, not long-term
system policy.

Rule:

- keep `init/` focused on getting the kernel to a stable running baseline and
  entering user space
- once the system is running, ongoing policy belongs in user-space init or
  services, not in this directory

11. `debug/` -- Diagnostics, Panic, And Trace Hooks
---------------------------------------------------
`debug/` owns the observability tools needed to make a small kernel livable.

Responsibilities:

- panic paths
- assertions
- early logging
- debug counters or trace hooks
- targeted state dumps used during bring-up

This code is not optional convenience.
For the Canopus kernel shape, explicit diagnostics are part of the
implementation strategy because the system intentionally prefers inspectable
small mechanisms over opaque abstractions.

12. `include/` -- Cross-Subsystem Header Boundary
-------------------------------------------------
`include/` owns the headers intentionally shared across subsystem boundaries.

Recommended structure:

```txt
include/
└── kernel/
    ├── types.h
    ├── task.h
    ├── syscall.h
    └── ...
```

Rules:

- expose only the types and function declarations that another subsystem
  genuinely needs
- prefer forward declarations and narrow interfaces over internal-structure
  leakage
- do not turn `include/` into a second copy of every private header tree

Dependency Rules
----------------
The main value of this architecture is not the directory names by themselves.
The value is the dependency discipline they enforce.

Preferred dependency shape:

- `arch/` provides low-level target hooks upward
- `core/` provides small shared definitions sideways
- `sys/` depends on owning subsystems
- `ipc/`, `mm/`, `cap/`, `sched/`, and `interrupt/` depend on `core/`
- `device/` depends on the narrow generic support it needs
- `init/` orchestrates subsystem initialization but should not permanently own
  their business logic
- `debug/` may be used broadly, but it should not become a hidden policy owner

Unwanted dependency patterns:

- `ipc/` embedding scheduler policy
- `mm/` embedding architecture-private register or page-table manipulation
- `sys/` duplicating kernel object logic
- `device/` accumulating user-space service policy
- `core/` becoming an all-purpose utility layer

Object Ownership Summary
------------------------
The kernel should be explainable through a small number of object families and
clear owners:

- process and task state: `core/` plus `sched/` and process or task owners
- address spaces and mappings: `mm/`
- endpoints and queued messages: `ipc/`
- handle tables and rights: `cap/`
- syscall entry and argument validation: `sys/`
- trap entry and machine context: `arch/`
- interrupt routing and tick integration: `interrupt/`
- boot sequencing and first handoff: `init/`

If an object has no obvious owning subsystem, either the object is too vague
or the architecture boundary needs to be simplified before code lands.

Non-Goals For This End-State Note
---------------------------------
This document intentionally does not define:

- a final multicore architecture
- a final revocation algebra
- a full driver framework
- a full filesystem architecture
- a full namespace or service-discovery algebra
- a full image, loader, or package distribution contract
- a promise that Canopus should grow into a broad monolithic kernel

Repository Guidance
-------------------
As repository-owned kernel code grows under `kernel/`, changes should move the
tree toward this ownership split gradually rather than forcing premature
churn.

Recommended rule:

- prefer the smallest honest change that preserves or improves subsystem
  boundaries
- do not reorganize code into the full end-state layout before real
  implementation pressure makes the new ownership meaningful

Bottom line
-----------
The intended Canopus end-state is still a small microkernel with explicit
authority, explicit IPC, explicit address-space ownership, and user-space
policy.
This note makes that target concrete by naming the subsystem boundaries that
future kernel work should preserve as the implementation grows from the v0
bring-up slices into a more durable long-lived kernel layout.
