Canopus sys v0 — System Interface

Status: v0 bring-up
Scope: minimal user-visible system interface
Audience: kernel + userland implementers

⸻

1. Purpose

sys defines the only supported interface by which user programs interact with the Canopus kernel.

It provides:
	•	a small, stable API
	•	explicit access to kernel mechanisms
	•	a thin abstraction over endpoints, processes, time, and capabilities

sys is not:
	•	a service framework
	•	a messaging system
	•	a runtime environment

It is a minimal boundary layer.

	•	docs/arch/runtime/canopus_std_sys_wrapper_v0.txt

2. Design Principles

2.1 Minimal surface

Only include operations required for system bring-up.

2.2 Explicit authority

All operations requiring authority must use capabilities or capability-backed handles.

2.3 No policy

sys exposes mechanisms only. Policy lives in user space.

2.4 No hidden behavior

No implicit resource access. No ambient authority. No hidden namespace lookup.

2.5 Stable core

The API should remain small and stable over time.

⸻

3. Relationship to Canopus Lite

sys sits above the kernel and below user libraries:

[ applications ]
        ↓
[ std / user libraries ]
        ↓
[ sys ]
        ↓
[ kernel: tasks, endpoints, time, caps ]

sys maps directly to kernel primitives:

Kernel concept	sys abstraction
endpoint	Endpoint
task/process	Process
capability	Capability
IPC		SendMessage / RecvMessage
time		Time

⸻

4. Contract Shape

The signatures below describe the kernel-facing contract, not a required Modern C surface syntax.

The stable rule for v0 is:
	•	operations return a status code
	•	callers provide output storage
	•	user libraries may wrap this in a richer result or error model later

This keeps the kernel boundary simple without forcing language-level error semantics too early.

⸻

4.1 Status

enum Status {
    Ok,
    Invalid,
    PermissionDenied,
    NotFound,
    Exhausted,
    TooLarge,
    WouldBlock,
    Unknown,
}

⸻

4.2 Opaque Handles

All kernel objects are opaque process-local handles:

type Endpoint
type Process
type Capability

Handles are:
	•	created by the kernel
	•	validated by the kernel
	•	process-local
	•	not constructible by user code except through sys

⸻

5. Capability Model

5.1 Overview

Capabilities represent explicit authority.

Properties:
	•	created by kernel
	•	stored in the process handle table
	•	transferable via IPC
	•	required for privileged operations

⸻

5.2 Rules
	•	no ambient authority exists
	•	capabilities must be passed explicitly
	•	invalid or unauthorized handle use returns PermissionDenied
	•	capability transfer is explicit in the IPC surface, not implicit in byte payloads

⸻

6. IPC Interface

IPC is the primary primitive exposed by sys.

⸻

6.1 Message Model

v0 messages have two parts:
	•	a byte payload
	•	a capability payload

type SendMessage {
    data: *u8
    len: u64
    caps: *Capability
    cap_count: u64
}

type RecvMessage {
    data: *u8
    len: u64
    caps: *Capability
    cap_count: u64
    recv_len: u64
    recv_caps: u64
}

No schema or encoding is enforced for the byte payload.

⸻

6.2 Operations

func send(ep: Endpoint, msg: SendMessage) -> Status

func recv(ep: Endpoint, msg: *RecvMessage) -> Status

⸻

6.3 Capability Transfer Semantics

	•	bytes and attached capabilities are delivered atomically as one message
	•	on successful send, attached capabilities are removed from the sender and reinstalled in the receiver as receiver-local handles
	•	on failed send, attached capabilities remain valid in the sender
	•	recv writes the received byte count to recv_len and the received capability count to recv_caps
	•	if the provided byte or capability buffer is too small, recv returns TooLarge and the message remains queued
	•	kernel mediates all delivery; user code cannot forge received capabilities from raw bytes

⸻

6.4 General Semantics

	•	blocking is acceptable in v0
	•	byte payloads are copied
	•	message size is bounded
	•	attached capability count is bounded

⸻

7. Endpoint Management

func endpoint_create(out: *Endpoint) -> Status

Created endpoints are initially usable only by the creating process.
Other processes gain access only through explicit capability transfer.

Future extensions may include:
	•	endpoint rights refinement
	•	transfer restrictions
	•	multi-party endpoint patterns

Not required for v0.

⸻

8. Process Interface

8.1 Process Creation

sys does not resolve executable paths or consult a global namespace.
Process creation is driven by an explicit program capability.

type ProcSpec {
    image: Capability
    args: []str
    caps: []Capability
}

func spawn(spec: ProcSpec, out: *Process) -> Status

⸻

8.2 Process Control

func wait(p: Process, out_exit_code: *i32) -> Status

func kill(p: Process) -> Status

⸻

8.3 Semantics
	•	new process receives only the capabilities listed in ProcSpec.caps
	•	no implicit inheritance occurs
	•	image identifies an executable object or launchable program handle supplied by a loader or parent process
	•	process isolation is enforced by the kernel

⸻

9. Memory Boundary

sys v0 does not define alloc or free.

Heap allocation policy belongs in user libraries or a bootstrap runtime layer, not in the core system interface.
If later kernel primitives are needed for virtual memory management, they should be exposed as raw memory-mapping mechanisms rather than heap policy.

⸻

10. Time Interface

type Time = u64

func now() -> Time

func sleep(ms: u64) -> Status

⸻

11. What sys Does NOT Include

The following are explicitly excluded from v0:

❌ No filesystem API

❌ No networking API

❌ No service registry

❌ No namespace system

❌ No structured messaging (ASDL)

❌ No async runtime

❌ No threads API

❌ No heap allocator API

These belong in user space or later revisions.

⸻

12. Example Usage

12.1 Simple IPC

ep: Endpoint
status = sys.endpoint_create(&ep)
if status != Status.Ok {
    return status
}

out_buf: [128]u8
out_caps: [1]Capability

send_msg := SendMessage {
    data: &out_buf[0]
    len: n
    caps: &out_caps[0]
    cap_count: 1
}

status = sys.send(ep, send_msg)
if status != Status.Ok {
    return status
}

in_buf: [128]u8
in_caps: [4]Capability
recv_msg := RecvMessage {
    data: &in_buf[0]
    len: 128
    caps: &in_caps[0]
    cap_count: 4
    recv_len: 0
    recv_caps: 0
}

status = sys.recv(ep, &recv_msg)

⸻

12.2 Spawn Process

p: Process
status = sys.spawn({
    image: echo_image_cap,
    args: ["hello"],
    caps: []
}, &p)
if status != Status.Ok {
    return status
}

exit_code: i32
status = sys.wait(p, &exit_code)

⸻

13. Bring-Up Requirements

A valid v0 system must support:
	•	creating an endpoint
	•	sending bytes
	•	receiving bytes
	•	transferring at least one capability through IPC
	•	spawning a process from an explicit program capability
	•	waiting for process completion

Minimal proof:
	•	init creates endpoint A
	•	init transfers endpoint A or a service capability to another process
	•	client sends a request message
	•	service receives bytes and any attached capabilities
	•	service replies

⸻

14. Evolution Path

After v0 stabilizes, sys may grow:
	•	capability rights refinement APIs
	•	shared memory primitives
	•	structured IPC helpers above raw transport

Higher-level features should be added to:
	•	user libraries (std)
	•	service frameworks

⸻

15. Design Constraints

Future changes must obey:

15.1 No implicit authority

15.2 No hidden global state

15.3 No policy in sys

15.4 Prefer addition in userland over kernel/API growth

15.5 Keep capability transfer explicit in the interface contract

⸻

16. Summary

sys v0 provides a minimal, explicit interface:
	•	endpoint-based IPC
	•	explicit capability transfer
	•	process control driven by explicit program handles
	•	basic time access

It intentionally avoids:
	•	high-level abstractions
	•	service frameworks
	•	hidden namespace lookup
	•	runtime policy such as heap allocation

The goal is:

to make the system usable without constraining its evolution

⸻

Final Note

This design gives you a cheap, repeatable way to use the kernel
without committing to:
	•	full service architecture
	•	SDL (service domain langauge and interface)
	•	distributed abstractions
	•	a premature language-level error model
