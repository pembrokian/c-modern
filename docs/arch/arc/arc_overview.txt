Arc - Overview

Purpose

This document defines the intended role of Arc in the broader Canopus
direction and explains how the Arc spec set is partitioned.

Arc is not a general replacement for Modern C.
Modern C remains the language for the kernel, startup path, and the smallest
hardware-facing support stubs.
Arc is the planned main language for long-lived operating-system services,
servers, daemons, and user-space policy.

Design position

Arc is intended to be:
- explicit rather than magical
- message-oriented rather than shared-state-oriented
- ownership-based rather than GC-based
- service-friendly rather than application-framework-oriented
- small enough to implement and reason about in stages

Arc is not intended to be:
- a hidden-runtime managed language
- an exception-driven or reflection-heavy environment
- a language whose safety story depends on pervasive global GC
- a broad research vehicle for type-system cleverness

Architectural role

The current intended split is:
- Modern C owns kernel code, target bring-up, and hardware stubs
- Arc owns user-space services and policy-rich system components

This split exists for practical reasons:
- kernel and hardware work need the narrowest possible runtime assumptions
- long-lived services benefit from clearer ownership and message boundaries
- the operating-system design is explicitly protocol-first rather than
  shared-memory-first

Primary design pressure

Arc should fit a system where:
- services communicate through explicit protocol messages
- ownership transfer is common
- zero-copy handoff matters for larger payloads
- latency and shutdown behavior must stay predictable
- capability flow is visible and not ambient

Spec set

Arc should start with a small spec stack rather than one giant document:

- `arc_core_spec.txt`
    - syntax categories, modules, types, control flow, and basic program
      meaning
- `arc_memory_model.txt`
    - ownership, moves, borrowing, destruction, arenas, and explicit sharing
- `arc_concurrency_model.txt`
    - tasks, channels, transfer rules, visibility, cancellation, and ordering
- `arc_runtime_ffi_spec.txt`
    - runtime stance, service process assumptions, C interop, and kernel or
      syscall boundary rules

Separation rule

Keep these concerns separate on purpose:
- the core spec says what Arc code means
- the memory spec says how values live and move
- the concurrency spec says how tasks communicate and observe each other
- the runtime and FFI spec says what execution support and foreign boundaries
  exist

This separation matters because Arc is being designed for system services.
If ownership, concurrency, or capability flow become vague, the language will
drift toward hidden policy instead of making the operating-system design more
explicit.

Core principles

Arc should be guided by these rules:
1. Protocols are ordinary data, not magic channel objects.
2. Channels are an execution tool, not the wire format.
3. Values are owned by default.
4. Transfer should be cheaper and more common than sharing.
5. Shared ownership must be explicit and visibly more expensive.
6. Allocation and runtime support must stay visible.
7. Capabilities should be explicit values, not ambient process privilege.

Current status

These documents are a first-pass design scaffold.
They define intended boundaries and default direction.
They do not yet claim a finished grammar, finished runtime ABI, or a completed
capability system.

Immediate next-step guidance

Use this spec set to answer early design questions in this order:
1. Is Arc specifically a message-driven systems language rather than a general
   host language? This document assumes yes.
2. Are ownership transfer and predictable lifetime the default model?
   `arc_memory_model.txt` assumes yes.
3. Are tasks and channels the main service-level concurrency shape?
   `arc_concurrency_model.txt` assumes yes.
4. Is the runtime intentionally narrow and replaceable?
   `arc_runtime_ffi_spec.txt` assumes yes.

Capability note

Capabilities are central to the Canopus direction, but they should not be
smuggled into Arc as hidden ambient authority.
The language should support capability-oriented APIs cleanly, while the fuller
capability model remains an operating-system and service-architecture concern.
That fuller policy can be specified separately and referenced from Arc once the
kernel and service boundary is sharper.
