Plan Doc Specification
======================

Purpose
-------
This document defines the normative structure for per-phase plan documents in
`docs/plan/active/`.

The goal is not to replace the repository's current planning style. The goal
is to make that style more explicit, more spec-driven, and more reviewable by
requiring each phase plan to state:

- the problem being solved
- what the phase will not do
- the invariants that must remain true
- the required and forbidden surface changes
- the main files, navigations, patterns, and owner-local invariants that will
	reduce implementation search time
- the relevant test surfaces that should be used for narrow validation
- the test obligations and acceptance criteria
- the expected future follow-up

This spec is for planning documents first. Landed documents may append
implementation notes, validation performed, and exit status, but those do not
replace the planning obligations.

Normative words
---------------
- MUST: required for a valid phase plan
- SHOULD: strongly preferred unless there is a documented reason not to
- MAY: optional when it improves clarity

Scope
-----
This spec applies to new repository-owned phase plans and to major rewrites of
existing active phase plans.

This spec is not a requirement for:

- backlog entries
- decision-log entries
- critique-response notes
- archive summaries or closeout-only notes

Core principles
---------------
Every phase plan MUST behave like an executable contract, not just a narrative
essay.

Each phase plan MUST:

- describe one bounded problem
- identify the smallest honest slice that solves that problem
- protect existing architectural boundaries explicitly
- extract the planning-time information that will shorten later navigation and
	implementation work
- define what evidence will count as success
- define what changes are out of scope even if they look adjacent

Each phase plan MUST NOT:

- read like a generic wishlist
- hide key constraints inside prose only
- rely on implied non-goals or implied invariants
- substitute tests or proofs for a real implementation when the phase claims a
	real implementation landing
- quietly widen compiler, runtime, tool, or kernel scope without stating that
	widening as an explicit surface change

Required document order
-----------------------
Each phase plan MUST use the following top-level section order.

1. `Phase N -- <concise title>`
2. `Status`
3. `Purpose`
4. `Repository status note`
5. `Problem being solved`
6. `Non-goals`
7. `Invariants`
8. `Required surface changes`
9. `Forbidden changes`
10. `Phase role in the larger sequence`
11. `Relevant backlog items this phase intentionally brings under pressure`
12. `Questions this phase must answer`
13. `Preferred implementation shape`
14. `Implementation accelerators`
15. `Test obligations`
16. `Acceptance criteria`
17. `Future follow-up`

The document MAY include additional sections after these if they materially
improve clarity, but the required sections MUST appear and MUST remain easy to
find.

Required section semantics
--------------------------

Status
------
The `Status` section MUST state the current phase state in a short explicit
form.

Allowed values include:

- `PROPOSED`
- `IN PROGRESS`
- `LANDED`
- `COMPLETED`
- `IMPLEMENTED`

If the plan is no longer purely forward-looking, the status SHOULD include the
date or short qualifier.

Purpose
-------
The `Purpose` section MUST explain the bounded capability or repair being
admitted.

It MUST answer:

- what capability the phase is introducing or repairing
- why this phase exists now
- why this slice is the next honest move rather than a broader framework step

This section SHOULD stay high-level and short.

Repository status note
----------------------
This section MUST summarize the local repository context that makes the phase
necessary.

It MUST:

- name the immediately relevant prior phases, files, or decisions
- describe what is already true
- describe what is still missing

It MUST NOT become a general repository history.

Problem being solved
--------------------
This section is REQUIRED.

It MUST describe the concrete contradiction, gap, or pressure point that the
phase resolves.

It SHOULD explain:

- what is insufficient in the current repo state
- why leaving the problem unresolved is dishonest or destabilizing
- why the problem is real and not merely aesthetic

The problem statement MUST be written as a current-repo problem, not as a
generic industry problem.

Non-goals
---------
This section is REQUIRED.

It MUST list the tempting adjacent expansions that this phase explicitly will
not do.

Each non-goal SHOULD be concrete. Good examples:

- no scheduler framework
- no generic mailbox system
- no capability algebra
- no compiler widening beyond one stated follow-up trigger

This section MUST make it easy for implementers and reviewers to reject scope
creep.

Invariants
----------
This section is REQUIRED.

It MUST list the architectural or behavioral truths that must remain true after
the phase lands.

Invariants SHOULD be stated as obligations, not vague intentions. They SHOULD
include boundaries such as:

- ownership boundaries
- transport-width limits
- phase-boundary responsibilities
- determinism requirements
- existing public-surface constraints

Each invariant SHOULD be falsifiable.

Examples:

- dispatch remains a routing and gating boundary only
- the four-byte shell frame stays fixed
- semantic truth stays in sema rather than moving into MIR
- reset-lane full coverage remains one maintained grouped case

Required surface changes
------------------------
This section is REQUIRED.

It MUST enumerate the minimal admitted surfaces that are expected to change.

This is the normative replacement for the older softer pattern around
`Chosen slice for this phase`.

Each item SHOULD identify one of:

- a new owner or service
- a bounded new command family
- a required file or module touch set
- a required public or user-visible behavior change
- a required scenario or fixture addition

This section MUST describe what the phase is allowed and expected to change.

Forbidden changes
-----------------
This section is REQUIRED.

It MUST describe changes that reviewers should reject even if they appear to
solve the problem.

This section differs from `Non-goals`:

- `Non-goals` explains adjacent areas the phase is not trying to solve
- `Forbidden changes` explains implementation moves that would violate the
	phase contract

Examples:

- do not widen shell transport width
- do not move owner-local truth into boot or dispatch merely to simplify code
- do not add generic registries, dashboards, or frameworks
- do not solve slowness by adding manual shard maintenance when reuse can be
	repaired in the owning runner

Phase role in the larger sequence
---------------------------------
This section MUST explain how the phase fits the surrounding plan chain.

It SHOULD describe:

- what earlier phase pressure it consumes
- what later phases it is preparing
- what follow-on work becomes possible only if this phase lands honestly

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
This section SHOULD remain in the current repository style.

It MUST identify the backlog themes this phase is intentionally testing,
forcing, or narrowing.

It SHOULD reference concrete backlog entries or bounded themes rather than
generic categories.

Questions this phase must answer
--------------------------------
This section is REQUIRED.

It MUST list the real design questions whose answers will determine whether the
phase remains honest.

Good questions are:

- repo-local
- architecturally meaningful
- answerable by implementation and tests

Bad questions are:

- vague brainstorming prompts
- product strategy questions unrelated to the implementation slice

Preferred implementation shape
------------------------------
This section is REQUIRED.

It MUST guide implementers toward the preferred bounded shape without forcing a
line-by-line design.

It SHOULD identify:

- the intended owner
- where truth should live
- where composition should happen
- which existing surfaces should be reused
- which boundaries should not be crossed

Implementation accelerators
---------------------------
This section is REQUIRED.

Its purpose is to reduce implementation-time search, orientation, and
re-derivation work by forcing the plan to extract the most useful repo-local
accelerators while the problem is still being framed.

This section MUST stay implementation-facing rather than contractual. It does
not replace `Invariants`, `Required surface changes`, or `Acceptance criteria`.

It MUST include four parts:

1. `Main files and navigations`
2. `Relevant cookbook patterns`
3. `Relevant file-local invariants`
4. `Relevant test surfaces`

Main files and navigations
~~~~~~~~~~~~~~~~~~~~~~~~~~
This subsection MUST identify the likely controlling owner and the first-hop
navigation set an implementer should read before editing.

It MUST name:

- the primary controlling file or module
- the small set of adjacent files that usually move with it
- the expected first validation target when the owner changes

It SHOULD also name nearby prompts, fixtures, or declarations accelerators
when they materially reduce search time, for example:

- `docs/agent/prompts/patterns_cookbook.txt`
- `docs/agent/prompts/compiler_file_invariants.txt`
- `docs/agent/prompts/kernel_file_invariants.txt`
- `tools/miscl/extracted_c_funcs/`
- `tools/miscl/extracted_decls/`

This subsection MUST stay selective. It is not a full file inventory.

Relevant cookbook patterns
~~~~~~~~~~~~~~~~~~~~~~~~~~
This subsection MUST identify any existing cookbook pattern that cleanly fits
the phase slice.

It SHOULD name:

- the relevant pattern title
- why it applies here
- any important way this phase intentionally differs from the default pattern

If no current cookbook pattern fits honestly, the plan SHOULD say so
explicitly rather than forcing a weak analogy.

Relevant file-local invariants
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
This subsection MUST capture the owner-local or file-local truths that an
implementer is likely to need during editing but that are more granular than
the broader phase invariants.

These invariants SHOULD answer questions such as:

- which file is the single owner of this truth
- what nearby truth must not be moved across a boundary
- which helper or sibling file usually changes in the same slice
- what narrow validation is normal for edits in that owner

These MAY reference repository prompt docs or restate the needed facts in a
shorter phase-local form.

This subsection MUST NOT duplicate the entire contents of a file-invariants
document. It should extract only the invariants relevant to the current phase.

Relevant test surfaces
~~~~~~~~~~~~~~~~~~~~~~
This subsection is REQUIRED.

It MUST identify the smallest honest validation surface an implementer should
start with for the phase.

It MUST name:

- the primary owning test directory or grouped suite family
- the narrowest expected first executable validation target
- any fixture convention or output-layout rule that materially affects how the
	phase should be tested

It SHOULD also point to the repository testing prompt when that reduces search
time, for example:

- `docs/agent/prompts/test_overview.txt`

This subsection SHOULD explain whether the phase is primarily a compiler,
tool, workflow, smoke, system, or hosted reset-lane validation problem.

This subsection MUST stay selective. It is not a full test inventory.

Recommended accelerator extraction workflow
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
When developing a new phase plan, authors SHOULD extract this section in the
following order so the plan solves implementation-routing work early rather
than leaving it to the coding pass.

1. Identify the controlling owner.
	Start from the file, module, service, or test owner that directly decides
	the behavior named in the problem statement.
2. Extract the first-hop navigation set.
	Record the controlling file, the small adjacent touch set, and the
	narrowest expected first validation target.
3. Check for a relevant cookbook pattern.
	Consult `docs/agent/prompts/patterns_cookbook.txt` and record any pattern
	that fits honestly.
4. Check for owner-local invariant material.
	Consult the relevant file-invariants prompt such as
	`docs/agent/prompts/compiler_file_invariants.txt` or
	`docs/agent/prompts/kernel_file_invariants.txt` and extract only the local
	truths that matter for this phase slice.
5. Identify the owning test surface.
	Consult `docs/agent/prompts/test_overview.txt`, the nearest checked-in
	fixtures, and the active grouped suite owners to record the narrowest honest
	validation target plus any fixture or output-layout rule that matters.
6. Use declaration summaries only to reduce search, not to define ownership.
	`tools/miscl/extracted_c_funcs/` and `tools/miscl/extracted_decls/` are
	navigation aids for discovering nearby files, declarations, and helper
	families. They MUST NOT override the ownership defined by code and
	architecture docs.
7. Compress the result into the plan.
	The final accelerator content should be short, selective, and executable by
	a future implementer. It should answer where to start, what to read next,
	which pattern applies, which local truths must be preserved, and which test
	surface should be run first.

Authors SHOULD prefer extraction over restatement. If an existing prompt or
repo-local helper already captures the truth well, the plan should point to it
and summarize only the phase-relevant parts.

Test obligations
----------------
This section is REQUIRED.

It replaces the looser older pattern around `Validation expectations`.

It MUST state the minimum test evidence required before the phase can be
claimed complete.

It MUST distinguish between:

- build obligations
- focused scenario or fixture obligations
- grouped-suite or workflow obligations when applicable

It SHOULD call out performance-sensitive obligations when relevant. Examples:

- run the owning target twice to confirm build reuse
- inspect per-scenario cache summaries if suite time changes materially
- prefer the narrowest owning grouped target before broad full-suite runs

Acceptance criteria
-------------------
This section is REQUIRED.

It replaces the older looser pattern around `Exit criteria`.

It MUST define the observable conditions that must all be true for the phase to
count as landed.

Acceptance criteria MUST:

- be checkable
- correspond to the problem statement
- reflect the invariants and non-goals
- avoid hidden criteria that appear only in narrative prose

Future follow-up
----------------
This section is REQUIRED.

It MUST state what should happen next after this phase lands, without turning
the current phase into a framework placeholder.

This section SHOULD distinguish between:

- likely immediate next pressure
- deferred adjacent work
- conditions that would justify reopening a bounded non-goal later

If the phase exposes a language or tooling limitation, this section SHOULD say
where that follow-up belongs, for example:

- `docs/plan/backlog.txt`
- `docs/plan/decision_log.txt`
- a later phase plan

Recommended optional sections
-----------------------------
The following sections are OPTIONAL but recommended when they materially help:

- `Files expected to change`
- `Accelerator source notes`
- `Concrete examples this phase should make possible`
- `Failure modes to watch for`
- `Migration or compatibility notes`

These MAY be used, but they MUST NOT replace any required section above.

Post-landing appendices
-----------------------
Once a phase lands, the plan MAY append landing-specific sections such as:

- `Implemented shape`
- `Implementation outcome`
- `Implementation notes`
- `Files landed`
- `Validation performed`
- `Exit status`
- `Decision notes`

These are useful, but they are post-landing appendices. They MUST NOT be used
to avoid writing the forward-looking planning contract.

Section writing rules
---------------------
Every required section SHOULD be concise, concrete, and reviewable.

Phase plans SHOULD:

- prefer short bullets over long diffuse paragraphs when enumerating contract
	points
- name actual files, services, commands, and tests when the repo already knows
	them
- describe one bounded slice rather than multiple loosely-related landings
- extract accelerator material while drafting rather than expecting the coding
	pass to rediscover it
- keep the contract honest to the repository's current architecture

Phase plans MUST NOT:

- claim completion of a larger roadmap phase when the repo has only landed a
	repository-bounded milestone
- describe framework-scale future dreams as if they are part of the current
	acceptance criteria
- bury the real constraint in a late note while the earlier sections imply a
	broader scope

Mapping from current structure
------------------------------
Existing repository plans often use these headings:

- `Purpose`
- `Repository status note`
- `Problem being solved`
- `Core rule`
- `Chosen slice for this phase`
- `Questions this phase must answer`
- `Preferred implementation shape`
- `Non-goals`
- `Validation expectations`
- `Exit criteria`

New plans SHOULD preserve that familiar voice, but MUST normalize the contract
into the required structure above.

Recommended mapping:

- `Core rule` content should move into `Invariants` and `Forbidden changes`
- `Chosen slice for this phase` content should move into `Required surface changes`
- planning-time file routing, cookbook-pattern selection, and owner-local
	editing notes should move into `Implementation accelerators`
- `Validation expectations` should move into `Test obligations`
- `Exit criteria` should move into `Acceptance criteria`

Authors MAY still keep a short `Core rule` subsection if it materially improves
readability, but it is optional and cannot substitute for the required
sections.

Review checklist
----------------
A phase plan is ready for use only if the answer to every question below is
yes:

1. Does the document state a concrete repo-local problem?
2. Does it state what the phase will not do?
3. Does it protect the critical invariants explicitly?
4. Does it enumerate the required surface changes rather than hinting at them?
5. Does it extract the main files, navigations, relevant patterns, and
	owner-local invariants that will shorten implementation work?
6. Does it enumerate forbidden implementation moves?
7. Does it define concrete test obligations?
8. Does it define acceptance criteria that can actually be checked?
9. Does it identify the likely next follow-up without widening the current
	 phase?
10. Would a reviewer be able to reject scope creep using this document alone?

If any answer is no, the plan is not yet specific enough.
