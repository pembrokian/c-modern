Phase 56 -- Post-Phase-55 Frontier Audit And Critique-Driven Follow-Through Selection
======================================================================================

Status
------
COMPLETED

Implementation result
---------------------
- Phase 56 closed the owner-local language-implementation gap for logical
  `||`, logical `&&`, and the already-admitted indexed-access surface by
  landing explicit short-circuit MIR lowering and executable proof instead of
  leaving those forms as roadmap-only debt.
- Parser and basic semantic support for these forms already existed; the real
  missing contract was runtime-faithful lowering for `||` and `&&`.
- MIR now lowers short-circuit boolean operators through explicit conditional
  control flow and a hidden bool result local so the right-hand side is only
  evaluated when required by the language spec.
- Indexed access `a[i]` remains within the admitted surface with explicit MIR
  bounds checks and existing fixture coverage, so this phase did not widen the
  language further than the named owner-local closure.

Repository status note
----------------------
- The repository already closed several Phase 51 stdlib critique items through
  the recent stdlib wave: the `io.write` and `io.write_line` error contract,
  the invalid `testing.fail()` sentinel, the `utf8.codepoint_len` naming bug,
  the `net.IpAddr` naming drift, the `parse_port` digit magic numbers, and the
  absence of `path` and `time`.
- The repository also now has sharper `path` and `time` semantics plus focused
  executable proof on those modules.
- Several remaining critique items are implementable now, but they are no
  longer all the same kind of work:
  - some are owner-local language or compiler closure for forms the written
    grammar already names, such as `||`, `&&`, and indexed access `a[i]`
  - some are adjacent stdlib or runtime widenings that are now feasible but
    still broader than a small cleanup, such as richer filesystem error
    channels or argv-shaped subprocess APIs
  - some remain blocked by live compiler or ABI boundaries, such as exported
    imported scalar constants, nominal `Arena` wrapping, and honest atomic
    storage layout
- The repository needs one explicit route-to-owner decision now so those three
  classes of work are not mixed together under one vague "cleanup" label.

Problem being solved
--------------------
- After the module-model reset series was inserted and the recent stdlib wave
  landed, the old speculative roadmap numbering and frontier ranking no longer
  reflect the real repository state.
- Without one fresh audit, planning could blur together:
  - syntax and expression forms already promised by the written grammar
  - adjacent library widenings that are now honest but still optional
  - genuinely blocked fronts that still require compiler or ABI closure first
- The repository needs to separate those classes clearly and choose one active
  owner boundary rather than reopening all remaining critique items in
  parallel.

Core rule
---------
- If the written grammar already names a form, close it at the language-owner
  boundary before normalizing the gap in stdlib or example code.
- Distinguish implementable-now adjacent widenings from still-blocked fronts.
- Prefer one active owner at a time, with one explicit second-wave candidate
  named behind it.
- Keep the blocked fronts visible with their real owner boundary instead of
  mislabeling them as ordinary library cleanup.

Chosen slice for this phase
---------------------------
Targeted proof shape: one documentation-backed frontier audit across the
current admitted language, stdlib, runtime, tool workflow, the recent Phase 51
stdlib critique, and the new module-model phase series.

The audit should answer four questions directly:
- which remaining critique items are already closed and therefore should not
  still appear as ambient roadmap debt?
- which remaining fronts are owner-local language or compiler closure because
  the written grammar already promises them?
- which remaining stdlib or runtime items are honest next widenings now, even
  if they require callsite churn or hosted runtime support?
- which fronts are still blocked by the current compiler, MIR, or backend
  boundaries and therefore should not be scheduled as near-term library work?

Fronts audited in this phase
----------------------------
1. Short-term language-owner closure for logical `||`, logical `&&`, and
   indexed access `a[i]`, which are already named by the concrete grammar.
2. Remaining filesystem and cross-module error-contract convergence after the
   recent `fs.file_size(...)` and `fs.is_dir(...)` widening.
3. Narrow process-surface widening beyond the current fixed-argv subprocess
   stub.
4. `testing` result-shape and helper hardening beyond the now-landed minimal
   diagnostic output.
5. `sync` and `mem` follow-on work, split between real adjacent widening and
   still-blocked compiler-owned representation problems.
6. Remaining low-risk strings and UTF-8 clarity or performance follow-through.

What this phase should try to prove
-----------------------------------
- the next active frontier after Phase 55 is not a generic "stdlib cleanup"
  bucket but a specific owner-local language closure: make the implementation
  match the already-written grammar for `||`, `&&`, and `a[i]`
- the strongest next adjacent widening after that language-owner closure is a
  narrower and more reviewable stdlib or runtime wave, not a broad reopening
  of all deferred families
- the still-blocked fronts can be named precisely enough that future plans do
  not accidentally treat them as ordinary library tasks
- the speculative roadmap can now be renumbered cleanly so it starts after the
  concrete Phase 56 owner-selection document and the inserted Phase 52 through
  Phase 55 module-model series
