Purpose
-------
Turn the broad non-bootstrap Phase 3 roadmap into an execution-ready plan that
can be implemented in reviewable waves without blurring ownership boundaries.

After Phase 9:

- sema is the authoritative checker for the admitted v0.2 core rather than a
  bootstrap-valid subset that later stages quietly constrain
- imported type identity, layout-sensitive truth, and exported semantic facts
  are precise enough that `.mci` work becomes serialization rather than
  semantic reconstruction
- explicit conversions, compile-time rules, representation rules, and semantic
  control-flow contracts are enforced in sema with deterministic diagnostics
- tests under `/Users/ro/dev/c_modern/tests/sema/` cover every admitted rule
  that the repository claims for the deeper Phase 3 surface

Problem being solved
--------------------
- The repository can already parse, resolve imports, perform bootstrap
  semantic checking, produce a semantic module, and lower an admitted slice to
  MIR.
- That semantic layer is still incomplete relative to the long-term Phase 3
  contract in `plan.txt` and the repository's own non-bootstrap roadmap.
- The main remaining risk is not absence of a checker, but partial authority:
  some type rules, conversion rules, layout rules, constant-evaluation rules,
  control-flow contracts, and export facts are still bootstrap-shaped.
- Without an explicit execution plan:
  - type and conversion gaps will continue to leak into MIR or backend-shaped
    workarounds
  - layout-sensitive behavior can drift away from sema-owned truth
  - future `.mci` work can harden around incomplete export facts
  - Phase 8 pressure findings will remain anecdotal instead of converting into
    narrow semantic regressions

Core rule
---------
- Fix semantic gaps in sema, not in MIR, the backend, the driver, or the
  stdlib.
- Deepen the admitted semantic surface before widening target or workflow
  breadth.
- Prefer a narrower honest semantic contract over a broader bootstrap
  convenience rule.
- Keep semantic dumps and diagnostics deterministic so the fixture layer stays
  authoritative.

Phase 9 execution waves
-----------------------
Phase 9A -- Type and Conversion Foundation
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Finish the type rules and explicit conversion rules first so later layout,
compile-time, control-flow, and export work can depend on stable semantic
identity instead of shifting bootstrap assumptions.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 3A -- Type-system completion
- 3B -- Explicit conversion and coercion closure

Tasks
~~~~~
1. Re-audit the current canonical type model against the admitted primitive,
   pointer, array, slice, buffer, procedure, struct, and distinct-type surface.
2. Close remaining holes where sema accepts or rejects code only because later
   stages are narrower.
3. Tighten imported user-defined type identity so qualified imported types and
   exported semantic identities survive module boundaries precisely.
4. Centralize explicit conversion checks and enforce the full no-implicit-
   numeric-conversions rule consistently across assignments, returns, calls,
   aggregate initialization, and explicit conversion expressions.
5. Extend distinct-type conversion rules only where the admitted Phase 3
   contract already authorizes them.
6. Add sema fixtures covering pass and fail cases for imported named types,
   distinct types, procedure signatures, and conversion-heavy expressions.

Phase 9B -- Representation and Compile-Time Authority
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Make sema the authoritative owner for representation-sensitive truth and the
admitted compile-time subset so later stages consume already-checked meaning.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 3C -- Layout and ABI correctness
- 3D -- Constant evaluation and compile-time rules

Tasks
~~~~~
1. Extend layout computation so sema owns size, alignment, field offsets,
   padding, packed behavior, `@abi(c)` behavior, nested aggregate layout, and
   imported layout-sensitive facts across the admitted slice.
2. Ensure representation-sensitive semantic facts are exported in deterministic
   form rather than recomputed later by the backend.
3. Implement deterministic evaluation for top-level constants and required
   compile-time expressions within the admitted subset.
4. Convert invalid compile-time forms into stable semantic diagnostics rather
   than latent runtime behavior or backend crashes.
5. Add layout and constant-evaluation fixtures for packed aggregates,
   `@abi(c)` declarations, nested aggregates, imported layout-sensitive types,
   and invalid compile-time expressions.

Phase 9C -- Semantic Contracts and Interface Authority
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Complete the semantic contracts that remain bootstrap-shaped and make the
semantic export surface trustworthy enough for later interface-artifact and
project-mode work.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 3E -- Control flow and semantic contracts
- 3F -- Export and interface authority

Tasks
~~~~~
1. Deepen semantic checking for `defer` beyond the current bootstrap-valid
   subset.
2. Implement explicit error-return semantics where the architecture texts
   already require them.
3. Implement semantic `@noalias` checking where the admitted surface claims it.
4. Strengthen return, break, continue, and lifetime-sensitive diagnostics in
   multi-branch and nested-scope cases.
5. Tighten exported-module surface construction so later `.mci` work is a
   serialization problem rather than a meaning-reconstruction problem.
6. Ensure imported semantic facts preserve callable signatures, type identity,
   and layout-sensitive truth without downstream repair.
7. Add multi-module fixtures that prove exported semantic truth is complete
   enough for later phases to consume directly.
