Import design decisions and trade-offs that need explicit closure
-----------------------------------------------------------------
Pressure-program packaging:

- Decision: whether substantial Phase 8 programs live under one shared repo project or under dedicated per-program project fixtures.
- Recommendation: use dedicated build.toml-driven project fixtures for substantial multi-module programs.
- Trade-off: a shared project reduces duplication, but it couples unrelated programs and hides per-program import assumptions; dedicated projects are noisier but make the import and build graph explicit.

Explicit roots versus convenience fallback:

- Decision: whether multi-module Phase 8 proofs may rely on repository-local stdlib or source-root fallback behavior.
- Recommendation: multi-module pressure programs should declare their roots explicitly through search_paths.modules; automatic repository-local stdlib discovery should remain a direct-source convenience only.
- Trade-off: convenience fallback is easier short term, but it weakens the Phase 8 goal of validating the explicit Phase 7 build and import model.

Cross-example helper reuse:

- Decision: whether programs may import shared helper modules outside their own project root.
- Recommendation: keep programs self-contained by default; only share helpers when the helper is itself intended as reusable library surface.
- Trade-off: shared helpers reduce duplication, but they blur which program is proving which language or tooling property.

Vendored roots:

- Decision: whether any Phase 8 program may rely on implicit vendor/ behavior.
- Recommendation: do not admit implicit vendored search; if a pressure program needs vendored modules, they must be named explicitly in search_paths.modules.
- Trade-off: implicit vendoring is convenient, but it drifts toward hidden package behavior and weakens the explicitness bias frozen in build_spec.txt.

Test imports:

- Decision: whether ordinary tests import the real program modules or use duplicated test-only copies.
- Recommendation: tests should import the real project modules through configured roots so the actual build graph is exercised.
- Trade-off: duplicated test copies simplify fixtures, but they stop proving the real project and import workflow.

Module identity and naming:

- Decision: whether to change the current ordered-root one-file-one-module lookup if Phase 8 programs feel awkward.
- Recommendation: preserve the current model first and capture friction as a pressure finding before changing the import design.
- Trade-off: changing the model early may smooth a few examples, but it can also bend the architecture to the first awkward case instead of validating the existing design honestly.
