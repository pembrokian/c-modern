Hosted Slice Release-Hardening Note
===================================

Status
------
PUBLISHED (repository-bounded support freeze, install-path hardening, supported-slice re-audit, public-cut audit, release-readiness re-audit, and `v0.2` gate)

Purpose
-------
Record the current supported hosted slice, frozen support matrix, installation
path, usage path, checked-versus-release posture, and known limitations for
the repository's current capability plateau.

Phase 65 now makes one additional planning point explicit: the admitted v0.2
core is semantically complete enough for this repository-bounded release-
hardening posture. The remaining open work is therefore support breadth,
adjacent-family admission, portability, and workflow trust, not a known
missing parser/sema/MIR ownership gap inside the admitted core slice.

This note is the repository-bounded realization of the original bootstrap-plan
Phase 9 release-hardening work. It is not a claim that the entire language,
toolchain, or long-term architecture roadmap is finished.

Frozen support matrix
---------------------
- supported compiler host and produced executable target: Darwin arm64 only
- supported runtime environment: hosted only
- supported direct-source commands: `mc check` and `mc build`
- supported project commands: `mc check`, `mc build`, `mc run`, and `mc test`
- admitted project target boundary: hosted `exe` targets, hosted checked-test
  targets, and one admitted hosted `staticlib` target consumed inside one
  project by executable targets and ordinary tests
- admitted adjacent helper boundary: narrow hosted `time` `Duration` plus
  `monotonic()` and pure slash-separated `path` `join(...)`, `basename(...)`,
  and `dirname(...)`
- admitted richer real-project proof set: `examples/real/issue_rollup/`,
  `examples/real/worker_queue/`, `examples/real/pipe_handoff/`,
  `examples/real/pipe_ready/`, `examples/real/line_filter_relay/`, and
  `examples/real/evented_echo/`, with
  `worker_queue` scoped as the low-level shared-memory sync proof,
  `pipe_handoff` as the direct handle-first communication proof,
  `pipe_ready` as the poller-coupled pipe-readiness proof,
  `line_filter_relay` as the narrow subprocess-plus-stdio proof, and
  `evented_echo` as the stronger networking proof over the same handle model
- supported workflow guarantee on the admitted richer proof: deterministic
  selected-target same-build-dir reuse without non-selected-target churn,
  now exposed directly on the shipped `issue_rollup` project through its
  default and report executable consumers over the admitted static-library
  boundary
- unsupported today: non-hosted targets, cross-compilation, shared library
  targets, external system-library links in project manifests, package
  management, and any portability claim beyond Darwin arm64

Installation and usage
----------------------
Supported host assumptions:

- host platform: Darwin arm64 only
- runtime boundary: hosted only
- installation story: build from source only
- toolchain assumption: CMake plus the ordinary host Clang toolchain

Build from source on the supported host:

```sh
cmake -S . -B build/debug -DCMAKE_BUILD_TYPE=Debug
cmake --build build/debug -j4
```

The produced compiler entrypoint on the supported path is:

```sh
build/debug/mc
```

First-use smoke validation on the supported path:

```sh
build/debug/mc --help
build/debug/mc check tests/cases/hello.mc
build/debug/mc run --project examples/real/issue_rollup/build.toml --build-dir build/debug/phase32_issue_rollup -- examples/real/issue_rollup/tests/sample.txt
build/debug/mc test --project examples/real/issue_rollup/build.toml --build-dir build/debug/phase32_issue_rollup
```

Repo-owned convenience entrypoint for that same narrow supported path:

```sh
make first-use-smoke
```

Repo-owned convenience entrypoint for the broader current public-cut smoke
audit:

```sh
make public-cut-smoke
```

Repo-owned convenience entrypoint for the current release-readiness re-audit:

```sh
make release-readiness-audit
```

Repo-owned convenience entrypoint for the current `v0.2` gate:

```sh
make v0_2_gate
```

Supported command patterns:

- direct-source inspection or build: `build/debug/mc check <file.mc>` and
  `build/debug/mc build <file.mc> --build-dir <dir>`
- project inspection, build, run, or test:
  `build/debug/mc check --project <build.toml>`,
  `build/debug/mc build --project <build.toml> --build-dir <dir>`,
  `build/debug/mc run --project <build.toml> --build-dir <dir> -- <args>`, and
  `build/debug/mc test --project <build.toml> --build-dir <dir>`

Artifact expectations:

- `build/debug/mc` is the produced compiler entrypoint for the supported path
- `build/` is disposable generated state
- command-local `--build-dir` trees are disposable generated state and should
  be treated as build outputs, not source inputs
- `make build` and `make test` remain convenience wrappers around the same
  CMake-based path; they are not a separate supported installation story
- `make first-use-smoke` is the repo-owned convenience wrapper for the narrow
  documented first-use validation path, not a broader public-cut certification
- `make public-cut-smoke` is the repo-owned convenience wrapper for the current
  broader public-cut smoke audit on the admitted hosted slice
- `make release-readiness-audit` is the repo-owned convenience wrapper for the
  current aggregate release-readiness re-audit on the admitted hosted slice
- `make v0_2_gate` is the repo-owned convenience wrapper for the final
  documented `v0.2` gate outcome on the admitted hosted slice

Checked versus debug audit
--------------------------
- project executable targets in the admitted real-project fixtures currently
  default to `mode = "debug"`
- ordinary tests in admitted real-project fixtures currently run in
  `mode = "checked"`, matching the repository-owned `mc test` contract
- debug-mode proof is narrow: it proves the admitted executable-run workflow
  on the supported hosted slice, not a broad release-mode certification across
  the whole language surface
- checked-mode proof is the main safety-validation surface for the current
  repository: `mc test` ordinary-test coverage is the strongest admitted proof
  for workflow trust on the supported slice
- checked behavior is treated as additional runtime validation on top of the
  same language semantics; it does not redefine release semantics

First public-cut hosted statement
---------------------------------
The current repository can honestly claim a hosted Darwin arm64 bootstrap
compiler and toolchain slice with direct-source semantic check and executable
build, project-mode debug executable runs, checked ordinary tests, grouped
internal package layouts, one admitted in-project static-library archive
boundary, deterministic selected-target same-build-dir reuse without non-
selected-target churn on the admitted real `issue_rollup` project proof, and the admitted
real `issue_rollup`, `worker_queue`, `pipe_handoff`, `pipe_ready`,
`line_filter_relay`, and `evented_echo` examples, with `worker_queue` scoped
as the low-level shared-memory sync proof, `pipe_handoff` as the direct
handle-first communication proof, `pipe_ready` as the poller-coupled
pipe-readiness proof, and `line_filter_relay` as the narrow hosted
subprocess-plus-stdio proof.

It does not honestly claim broad release-mode certification across the full
canonical-program surface, portability beyond Darwin arm64, package
distribution, non-hosted targets, or broader target-kind and linker-policy
coverage.

Release-readiness re-audit
--------------------------
The current release-readiness outcome is:

- yes for a source-only Darwin arm64 hosted `v0.2` support claim
- no for any broader portability, distribution, or target-kind claim than the
  frozen support matrix above

In practical terms, the remaining deferred fronts now cluster around broader
portability, package distribution, wider linker policy, and still-deferred
adjacent families rather than around hidden breakage inside the admitted
source-only hosted slice itself.

That means the remaining gaps are now narrow enough to tolerate in a source-
only `v0.2` release cut if the repository chooses to tag at that boundary.
Phase 79 still owns the final tag-or-defer decision and any one remaining
plateau statement needed for that cut.

`v0.2` gate
-----------
The current `v0.2` gate outcome is:

- no in-repository tag is created at this point
- the one remaining plateau blocking that tag is maintainer-owned release
  execution: selecting the final reviewed commit snapshot and applying the tag
  from committed repository state rather than from a live working tree

Within the repository itself, the technical gate is otherwise satisfied by the
current first-use smoke, supported-slice doc audit, public-cut smoke, and
aggregate release-readiness audit.

Known limitations
-----------------
- the supported host target set is still one hosted platform: Darwin arm64
- the current networking slice is intentionally narrow: IPv4 TCP listen,
  accept, and connect plus a portable poller surface; no UDP, DNS, TLS, async
  runtime, or scheduler claims are made
- the current portable I/O surface now also includes a minimal blocking
  `io.Pipe` plus `io.pipe()` admission proved by `examples/real/pipe_handoff/`;
  it also now proves poller-coupled pipe readiness through
  `examples/real/pipe_ready/`; channel libraries, richer IPC policy,
  and non-blocking pipe semantics beyond the current readiness proof remain
  out of scope
- the repository now admits one narrow hosted `os` subprocess slice with
  explicit argv-slice `spawn_argv(...)`, additive `spawn_pipe_argv(...)` for
  the admitted one-input-pipe plus one-output-pipe relay shape, additive
  `spawn_pipe_argv_merged_output(...)` for bounded merged child diagnostics on
  that same pipe family, additive `spawn_pipe_argv_split_output(...)` for
  bounded separate stdout and stderr capture on that same family, the legacy
  fixed `spawn(...)` compatibility wrapper, blocking `wait(...)`, and bounded
  stdin/stdout pipe redirection proved by `examples/real/line_filter_relay/`;
  shell command strings, broader stderr routing policy, environment APIs, cwd
  mutation, signals, and broader process management remain out of scope
- the repository now admits a narrow hosted `sync` slice: typed thread spawn
  or join plus mutex init, destroy, lock, unlock, condvar init, destroy,
  wait, signal, and broadcast, plus a narrow `MemoryOrder` and `Atomic<T>`
  load or store
  surface proved by the shared-counter, producer-consumer, and atomic-
  publication canonical executables plus the real `examples/real/worker_queue/`
  project as an explicit low-level shared-memory proof; atomic read-modify-
  write operations and broader portability claims remain out of scope
- the repository also admits one narrow hosted `time` plus `path` helper slice:
  `time.Duration`, `time.monotonic()`, and pure slash-separated
  `path.join(...)`, `path.basename(...)`, and `path.dirname(...)`; sleep,
  wall-clock, calendar, timezone, normalization, and broader platform APIs
  remain out of scope
- the repository does not yet claim `hal` as an admitted module family, and it
  does not claim `os` beyond the narrow subprocess slice recorded in the phase
  documents
- the repository does not yet claim self-hosting, package management, or a
  stable public distribution format beyond build-from-source on the supported
  host

Practical support statement
---------------------------
The current repository can honestly be used as a hosted Darwin arm64 bootstrap
compiler and toolchain slice for direct-source programs, bounded project-mode
workflows, deterministic checked ordinary tests, grouped internal package
layouts, one admitted in-project static-library archive boundary, deterministic
selected-target same-build-dir reuse without non-selected-target churn on the
admitted richer project proof, and the admitted real-project examples.
Everything outside that slice should be treated as explicitly unsupported until
a narrower owner document and proof layer admit it.

Within that bounded support claim, the admitted v0.2 core should now be read
as semantically complete enough for repository-bounded release hardening. New
core-language closure work should therefore reopen only when a fresh admitted
parser, sema, MIR, or backend distortion is found on the already admitted
slice.
