Phase 48 -- Post-Phase-47 Frontier Audit
========================================

Status
------
IMPLEMENTED

Purpose
-------
Audit the remaining deferred fronts after the completed Phase 47 error-
treatment wave and name the next honest active owner from current repository
evidence, not from roadmap inertia or abstract completeness pressure.

This document is a repository execution plan, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- Phase 47 completed the current explicit `Error` vocabulary wave without
  widening into message-owned errors, wrapped causes, or hidden propagation.
- Phase 42 already closed imported structural aggregate constants at the
  sema-plus-`.mci` boundary, but executable top-level aggregate const global
  emission remains a separate backend follow-up that is still not part of the
  admitted supported slice.
- The project workflow and linker surface still stop intentionally at hosted
  `staticlib`, selected-target reuse, and in-project archive consumption.
- Phase 34 already kept `time` deferred, and the later admitted projects have
  still not produced stronger cross-subsystem timing pressure than that audit
  required.
- The repository therefore needs one explicit route-to-owner decision before
  starting another capability or cleanup wave.

Problem being solved
--------------------
- After Phase 47, the repository has several still-explicit deferred fronts,
  but they no longer have equal weight.
- Without one frontier audit now, planning could drift into whichever family
  happens to sound broad or roadmap-adjacent rather than whichever one still
  distorts the admitted repository slice most directly.
- The repository needs to distinguish between:
  - live distortions inside the current admitted slice
  - optional follow-throughs that remain real but are not yet required by the
    admitted proof surface
  - broader capability families that still lack enough consumer pressure to
    reopen honestly

Core rule
---------
- Prefer the remaining owner-local distortion that already affects an admitted
  public surface over deferred breadth that still lacks pressure.
- Do not reopen `time`, broader linker policy, or richer error objects without
  new admitted consumer evidence.
- Do not force a new real project or synthetic probe merely to justify a front
  that still is not the narrowest current blocker.
- Keep non-winning fronts explicit and deferred rather than quietly dropping
  them.
- Use this phase to choose the next owner, not to implement that next owner's
  full closure in parallel.

Chosen slice for this phase
---------------------------
Targeted proof shape: one documentation-backed frontier audit across the
current admitted slice, the preserved limited-solution register, and the
remaining deferred capability notes.

The audit should answer four questions directly:
- does the next honest frontier already distort an admitted public surface
  today, or would reopening it require manufacturing new pressure first?
- which remaining front still belongs clearly to one owner boundary?
- which previously deferred fronts remain real but optional because the
  admitted proof set still does not require them?
- what should the repository do next without implying that several fronts are
  now open in parallel?

- whether the non-winning fronts should remain deferred unchanged, or with a
