- Fix compiler-owned limitations only when Phase 22 pressure justifies them.
- Prefer one closed limited solution over several partially loosened families.

- If no new compiler gap is materially exposed, do not force this phase into a
  speculative widening exercise.

Primary candidate
-----------------
Retire the compiler-owned imported generic `thread_spawn<T>` rewrite only if
Phase 22 lands a second real, independent imported generic procedure-call
consumer beyond the current hosted `sync` surface.

Why this candidate is primary
-----------------------------
- It is the sharpest remaining compiler workaround already named in the
  decision log and backlog.
- It already has a clear revisit trigger rather than a vague "later cleanup"
  note.
- Removing it would close a real bootstrap-shaped compiler compromise instead
  of widening a new feature family speculatively.

Secondary acceptable candidates if Phase 22 exposes them directly
-----------------------------------------------------------------
- fuller preservation of generic nominal type-argument identity across import
  and interface paths
- fuller accepted `@noalias` contract handling where a real imported API or
  ABI boundary needs it
- tuple or multi-result lowering work only if a real project or canonical
  proof becomes materially cleaner and more honest with that support

Admission rule for candidate work
---------------------------------
- A candidate should land only if at least one admitted real project or
  canonical proof is materially cleaner, more honest, or more deterministic
  after the compiler-owner change.
- If the limitation is still only theoretical, keep it explicit in
  `/Users/ro/dev/c_modern/docs/plan/backlog.txt` or
