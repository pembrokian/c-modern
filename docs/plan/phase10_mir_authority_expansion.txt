Repository-specific scope note
------------------------------
- The architecture texts stop at Phase 8 and already state that later pressure
  findings must route back to the smallest correct owner.
- This repository uses the Phase 10 label only to package the next execution
  wave as a concrete plan document under `docs/plan/`.
- In substance, Phase 10 is the implementation plan for Workstreams 4A through
  4D from the non-bootstrap roadmap, with the first execution wave explicitly
  prioritizing MIR surface closure and validator hardening before broader
  backend deepening.

Purpose
-------
Turn the broad non-bootstrap Phase 4 roadmap into an execution-ready plan that
can be implemented in reviewable waves without blurring the boundary between
sema-owned meaning, MIR-owned lowered truth, and backend-owned realization.

After Phase 10:

- the admitted non-bootstrap semantic surface lowers to MIR through explicit,
  typed, deterministic representations rather than bootstrap fallbacks
- semantic-boundary operations required by the admitted language surface are
  represented directly in MIR instead of being left to backend defaults,
  runtime accidents, or library-shaped call conventions
- MIR validation is strong enough to reject malformed or semantically
  impossible typed MIR before backend work begins
- the deterministic fixture layer under `/Users/ro/dev/c_modern/tests/mir/`
  acts as the main stable proof surface for MIR behavior rather than only a
  bootstrap smoke check

Problem being solved
--------------------
- The repository already has a real typed MIR layer, deterministic MIR dumps,
  a growing validator, and representative MIR fixtures for the currently
  supported bootstrap slice.
- That MIR layer is still incomplete relative to the long-term Phase 4
  contract in `plan.txt` and the repository's own non-bootstrap roadmap.
- The main remaining risk is not absence of MIR, but partial authority:
  some lowering paths are still bounded by bootstrap coverage, some semantic-
  boundary forms are only covered for the current admitted subset, validator
  legality is stronger than before but not yet fully instruction-complete, and
  dump/debug quality still carries bootstrap-oriented compromises.
- Without an explicit execution plan:
  - backend work can harden around a narrower MIR subset than the admitted sema
    surface really needs
  - unsupported or partially explicit boundaries can drift back toward generic
    call shaping, display-string recovery, or backend-local policy
  - validator coverage can remain strongest only on the currently emitted
    subset instead of the full admitted MIR contract
  - Phase 8 findings that should become MIR regressions can remain anecdotal
    or get routed directly into backend fixes

Core rule
---------
- Fix lowered semantic-boundary gaps in MIR, not in the backend, the driver,
  or the stdlib.
- Do not force MIR to invent meaning that sema does not yet own.
- Prefer a narrower honest MIR contract over a broader bootstrap convenience
  encoding.
- Keep MIR dumps, validation failures, and fixture outputs deterministic so the
  MIR test layer remains authoritative.

Phase 10 execution waves
-----------------------
Phase 10A -- Lowering Surface Closure
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Freeze and complete the MIR lowering matrix for the admitted semantic surface
so supported constructs lower through explicit, typed MIR without backend-
shaped fallback behavior.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 4A -- MIR surface completion

Tasks
~~~~~
1. Build and maintain a concrete lowering-coverage matrix from the admitted
   sema surface, the current bootstrap status note, and the current MIR
   fixtures.
2. Separate true MIR gaps from upstream sema gaps so lowering work does not
   accidentally absorb semantic ownership.
3. Complete lowering for the remaining admitted aggregates, conversions,
   address-sensitive operations, imported symbol/type-sensitive accesses, and
   other non-bootstrap forms that Phase 3 already makes meaningful.
4. Remove any remaining supported-path dependence on generic call shaping,
   display-string identity recovery, or bootstrap-only fallback encodings.
5. Ensure unsupported constructs fail crisply at the MIR boundary when the
   repository is still intentionally narrower than the long-term architecture.
6. Add representative fixtures proving that widened lowering paths stay typed,
   deterministic, and free of bootstrap recovery artifacts on success.

Phase 10B -- Semantic-Boundary Instruction Completion
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Finish the explicit MIR instruction and metadata surface for semantic-boundary
operations that the admitted language already requires.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 4B -- Semantic-boundary instruction completion

Tasks
~~~~~
1. Audit the admitted language surface for every operation whose semantics must
   be explicit in MIR rather than inherited from LLVM, the host ABI, or a
   library call convention.
2. Complete any remaining checked-fault, pointer-int, array/slice, buffer/
   slice, volatile, atomic, or other low-level boundary forms that are already
   semantically admitted but not yet fully explicit in MIR.
3. Make sure each dedicated instruction family carries enough structured
   metadata that validation and backend lowering can reason from MIR directly
   rather than reparsing dump text.
4. Reject fallback encodings for instruction families that now have a dedicated
   MIR representation.
5. Keep boundaries honest where repository support is still intentionally
   incomplete: unsupported low-level forms should fail narrowly instead of
   silently lowering through generic mechanisms.

Phase 10C -- Validator Legality Completion
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Make MIR validation a full legality check for the admitted typed MIR surface,
not only a guardrail around the currently emitted bootstrap subset.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 4C -- Validation hardening

Tasks
~~~~~
1. Re-audit the validator against every admitted instruction and terminator
   family instead of only the currently common emitted paths.
2. Strengthen type-consistency, operand-shape, result/effect-shape, aggregate,
   target-metadata, block-termination, and control-flow legality checks so the
   validator describes the MIR contract explicitly.
3. Validate cross-instruction invariants wherever MIR requires structured
   relationships between instructions, metadata, or checked-operation helpers.
4. Preserve legitimate imported or sema-backed references without weakening the
   legality rules for locally materialized MIR facts.
5. Expand negative coverage so each major MIR family has direct invalid-module
   or mutation-based tests proving the validator rejects malformed input.

Phase 10D -- Dump, Fixture, and Debugging Quality
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Keep MIR inspectable enough that lowering bugs and contract regressions can be
localized at the MIR layer before they spread into backend or executable tests.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 4D -- MIR debugging quality

Tasks
~~~~~
1. Preserve deterministic dump ordering and formatting as the MIR surface
   widens.
2. Prefer structured, reviewable MIR dump content over lookup schemes that only
   work because human-readable display strings happen to be stable.
3. Expand fixture coverage with representative semantic cases, imported-module
   cases, and pressure-derived cases rather than only narrow synthetic inputs.
4. Keep the fixture harness auditing key MIR-quality properties such as stable
   re-lowering behavior and absence of placeholder typing on supported paths.
5. Promote repeated Phase 8 findings into MIR fixtures or MIR validator tests
   when MIR is the clearest stable contract for the bug.

Out of scope
------------

- broad backend deepening that belongs to Phase 5 executable closure

  keep MIR dumps, validation, and debug workflows usable
- widening the target matrix or using target-specific behavior to hide MIR gaps
- adding speculative MIR instruction families that are not justified by the
  admitted semantic surface or repeated repository pressure
