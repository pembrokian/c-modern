Phase 8 -- Findings Log
=======================

Purpose
-------
- Record what each admitted pressure program exposed while it was being implemented.
- Keep implementation fixes, bootstrap gaps, and post-v0.2 design questions separate.

Taxonomy
--------
- implementation bug: the intended repository-owned capability exists, but the compiler, stdlib, runtime, or proof harness behaves incorrectly.
- bootstrap capability gap: the admitted repository slice does not yet contain a needed capability, but the gap is still within the current bootstrap direction.
- stdlib or runtime gap: the compiler works, but the library or hosted runtime boundary is too thin for the admitted pressure program.
- tooling gap: build, run, test, project, or artifact wiring is insufficient even though language and library support are otherwise adequate.
- spec-level follow-up: repeated pressure exposed a language or build-model awkwardness that should not be reopened until multiple programs point at the same problem.

Decision rule
-------------
- Fix implementation, stdlib, runtime, or tooling truth first.
- Reopen language or build-model questions only after repeated multi-program pressure.

Cross-program routing summary
-----------------------------
- sema: owns distinct-type authority, import-visible type preservation, and any
	semantic rule that examples expose as incorrect language meaning rather than
	awkward library shape
- MIR: owns typed lowered truth for explicit conversions, aggregate and enum
	lowering, checked or low-level instruction forms, and validator coverage when
	programs expose a mismatch between checked semantics and lowered structure
- backend: owns executable realization on the first hosted target when an
	admitted program already has correct sema and MIR truth but still fails to
	build or run honestly
- stdlib: owns API honesty, allocation visibility, borrowed versus owned text
	boundaries, directory traversal, future poller and network surfaces, and any
	`sync` or `hal` module admission needed by blocked canonical programs
- runtime: owns hosted startup and narrow low-level support when a blocked or
	awkward pressure program truly needs process-entry, thread, or hosted support
	rather than a broader library redesign
- driver and test harness: own project roots, `.mci` usage, `mc run`, `mc test`,
	timeout behavior, and deterministic workflow reporting when a real utility
	exposes workflow friction instead of semantic or library gaps

Phase 12 and Phase 13 pressure links
------------------------------------
- Phase 12 link: repeated pressure around streaming I/O, richer buffer helpers,
	allocator protocol honesty, poller and net admission, `sync`, and `hal`
	should route into the Phase 12 stdlib and runtime maturity backlog rather
	than being normalized inside utilities
- Phase 13 link: repeated pressure around project-mode imports, `.mci`
	qualification, ordinary tests, or deterministic project execution should
	route into the Phase 13 toolchain maturity backlog rather than being hidden
	in per-example fixture glue

Design-signal capture rules
---------------------------
- A single awkward program is not yet a design signal; first try to fix the
	owning implementation, stdlib, runtime, or toolchain truth.
- Treat a pattern as a design signal only when multiple independent programs
	expose the same friction after the current owner truth is otherwise honest.
- Do not reopen language, import-model, or allocator-model questions merely
	because a larger example would prefer a convenience shortcut.
- When a repeated pattern is still owner-local, record it as backlog or an
	enabling gap rather than as spec-level follow-up.
- Any real design signal should name the repeated programs, the attempted
	owner-local fixes, and the reason the remaining pressure is no longer just an
	implementation gap.
