Phase 107 -- Real User-To-User Capability Transfer
=================================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the completed Phase 106 real echo-service request-reply proof
into the next remaining Band C owner: prove that one real user-space
component can transfer one explicit endpoint authority to another over the
landed real kernel IPC boundary, that the receiver gains that authority only
through the transfer rather than ambient installation, and that the receiver
can use the transferred handle for one bounded action without widening into a
general service framework, multi-hop capability routing, or broader user-space
policy.

This document is both the Phase 107 execution note and its closeout.
It does not claim a general service registry, multi-service startup policy,
wait-handle transfer, or a running general user-space system.

Problem being solved
--------------------
- The speculative Veya kernel roadmap names explicit user-to-user
  capability transfer as the next bounded user-policy owner after the landed
  real echo-service proof.
- The repository already had the necessary ingredients spread across the
  owned kernel lane: one explicit attached-handle transfer message shape, one
  real service spawn path, one explicit service-visible control endpoint, and
  one retained init-visible endpoint capability that could observe follow-
  through behavior on the transferred endpoint.
- The landed syscall transfer helper still removed the transferred sender
  handle only when receive completed against the sender-owned table, which was
  sufficient for the earlier bounded same-owner proof but not for a real
  user-to-user transfer.
- Without this step, later service or authority phases could overclaim that
  the repository already preserves user-to-user capability movement over the
  real kernel boundary when the actual landed path still stopped at one same-
  owner follow-through.

Core rule
---------
- Keep the phase to one explicit init-to-service endpoint transfer only.
- Fix the transfer boundary at the syscall owner so sender-side removal occurs
  on successful send and receiver-side installation occurs against the real
  receiver table.
- Reuse the already-landed spawn, wait, endpoint, handle, and IPC owners.
- Add only the smallest helper module needed to keep the service-local
  transfer observation explicit and inspectable.
- Stop before general authority routing, multi-service policy, wait-handle
  transfer, or a broader runtime framework.

Chosen slice for this phase
---------------------------
Targeted proof shape:

1. one syscall-boundary correction that removes an attached sender handle on
   successful send and installs the received handle into the actual receiver-
   local table at receive time
2. one bounded `transfer_service` helper module owning service-local transfer
   state, copied emit payload construction, and final observation records
3. one repository-owned kernel follow-through that retains one init-visible
   endpoint capability locally, transfers one duplicate endpoint handle to one
   spawned service task over the landed real IPC path, and proves that the
   transferred sender slot disappears immediately after send success
4. one service-issued bounded follow-through send on the transferred endpoint
   so the receiver uses the newly acquired authority for one real action with
   no ambient shortcut
5. one explicit service-exit and wait-handle reap follow-through so the proof
   cleans up the reused child slot after the transfer succeeds
6. one focused freestanding regression and MIR inspection proving that the
   slice stays inside ordinary imported structs, enums, aggregate
   initialization, fixed-array payload construction, and explicit helper
   boundaries while preserving the landed Phase 106 proof

Chosen outcome
--------------
Chosen outcome: yes.

The repository should cross this user-to-user boundary now because the earlier
attached-handle work already proved the queued message shape and the later
service phases already proved one real spawned receiver. The next honest owner
is not a general service framework. It is one real transfer from one user
component to another with the smallest necessary syscall correction.

What this phase proves
----------------------
- one real Veya kernel artifact now removes one transferred endpoint handle
  from the sender on successful send and installs it only into the real
  receiver-local table at receive time
- init can transfer one explicit endpoint capability to one spawned service
  task over the landed real kernel IPC path without ambient reinstallation
- the receiving service can then use that transferred handle for one bounded
  send over the transferred endpoint, and init can observe that follow-through
  through its separately retained local handle
- the current language, freestanding build path, and kernel-owned proof lane
  can carry the required service-local transfer state and final observation
  without new compiler work
- the repository can stop honestly at one bounded user-to-user endpoint
  transfer without implying general authority routing or a running multi-
  service user-space system
