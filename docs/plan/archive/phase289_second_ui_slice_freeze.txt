Phase 289 -- Second UI Slice Freeze
===================================

Status
------
COMPLETED

Purpose
-------
State exactly what "basic UI" means in the repository after Band E and what
still remains intentionally closed.

This phase exists now because Band E deliberately broadens the first UI slice
only a little. The repository needs one explicit freeze point before later
platform or system-direction decisions reuse these UI results as if they were a
complete app or desktop model.

Repository status note
----------------------
- Phase 281 already froze the first UI slice after the `issue_rollup` proof.
- Phases 282 through 288 are expected to admit, reject, or defer one additional
  composition, interaction, control, proof, persistence, and workflow layer.
- After Band E, the repository should have a clearer but still bounded account
  of user-facing capability.
- What remains missing must be stated as intentionally closed rather than left
  ambiguous.

Problem being solved
--------------------
- UI waves are especially vulnerable to accidental overclaim: once two app-like
  proofs exist, readers can easily assume the repository now has a broader UI
  platform than it actually does.
- Without one explicit freeze document, later planning could treat bounded Band
  E owners as if they solved windows, widgets, accessibility, sessions, or app
  packaging.
- The current-repo problem is therefore one missing honesty checkpoint on the
  post-Band-E UI surface.

Non-goals
---------
- no new UI feature work
- no app framework, window system, or desktop shell
- no widget toolkit or theming platform
- no accessibility, text-system, or session framework
- no broader product or roadmap reset than the UI slice itself requires

Invariants
----------
- The freeze must describe actual landed repository truth only.
- Deferred UI capabilities must stay explicitly deferred.
- Existing owners must be named clearly so later phases do not reopen boundary
  confusion.
- The freeze must distinguish admitted user-facing capability from broader
  platform expectations.

Required surface changes
------------------------
1. Audit the actual landed results of Phases 282 through 288.
2. State the exact meaning of "basic UI" after Band E.
3. State the major still-closed areas explicitly.
4. Cross-link the owning validation and workflow surfaces that support the
   frozen claim set.

Forbidden changes
-----------------
- do not add new UI features while writing the freeze
- do not rewrite deferred work as implied future promises
- do not leave major closed areas implicit
- do not describe speculative platform direction as already admitted repo truth

Phase role in the larger sequence
---------------------------------
- This phase is the Band E closeout and honesty checkpoint.
- It prepares the broader post-UI platform reassessment by giving later phases
  a precise current baseline.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep roadmap claims aligned with landed owner boundaries and maintained test
  surfaces
- keep future platform direction grounded in what Band E actually proved

Questions this phase must answer
--------------------------------
1. After Band E, what exact user-facing capabilities are now admitted in the
   repo?
2. Which owners define those capabilities?
3. What larger UI or platform areas remain intentionally closed?
4. Which tests and workflows provide the supporting evidence for the frozen
   claim set?

Preferred implementation shape
------------------------------
- Follow the style of the first UI audit from Phase 281: short, explicit, and
  ownership-focused.
- Enumerate only landed truths and still-closed areas.
- Tie each admitted claim to an owner and an evidence surface.

Relevant cookbook patterns

Relevant file-local invariants
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- `docs/plan/active/phase281_first_ui_slice_audit.txt` is the immediate style

Dependency-ordered touch sequence
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- Audit the landed Phase 282 through Phase 288 outcomes first.
- Land the freeze document second.
- Update only the narrow supporting backlog, decision-log, or roadmap note that
  reflects a newly durable fact third.
- Repair an evidence gap in the owning earlier phase before widening the freeze
  claims if the audit exposes one.

Common breakage map
~~~~~~~~~~~~~~~~~~~
- Claimed UI capability has no owning evidence -> return to the earlier phase
  doc or owning test surface before freezing the claim.
- Deferred scope restated as admitted truth -> backlog and decision-log drift,
  roadmap overclaim.
- Freeze language stops naming owners -> later phases lose the boundary map and
  start treating Band E as a platform instead of a bounded slice.
- Post-green sidecar drift -> extra admin or architecture docs move even though
  the freeze only clarified already-landed Band E truth.

Relevant test surfaces
~~~~~~~~~~~~~~~~~~~~~~
- Primary owning validation surface: the already-landed test surfaces from
  Phases 282 through 288.
- Narrowest expected first executable validation target: none new unless the
  freeze discovers a missing evidence gap that must be repaired first.
- Routing note: if an evidence gap is test-owned, repair it in the narrowest
  owning target named by the earlier phase rather than rerunning broad grouped
  coverage first.
- Low-signal note: if the narrow owner-local validation target fails on a
  known unrelated assertion, record that target as low-signal and step to the
  next closest owner-local surface that still exercises the frozen claim.

Test obligations
----------------
- no new executable validation by default
- if the freeze exposes a missing evidence gap, repair that gap in the owning
  phase surface before treating the slice as frozen
- do not turn the freeze into a broad validation sweep; only rerun the narrow
  owner that supports the missing claim

Acceptance criteria
-------------------
- the repository states exactly what "basic UI" means after Band E
- the admitted scope is tied to explicit owners and evidence surfaces
- still-closed UI and platform areas are called out explicitly
- later planning can reuse this document as an honest baseline instead of a
  vague impression

Implementation outcome
----------------------
- After Band E, the repository's admitted "basic UI" means exactly one
  launcher-selected foreground app rendered on one fixed four-cell visible
  surface, with one admitted key-only input family and one optional fixed
  two-region composition seam.
- The maintained owner set for that frozen slice is:
  - `kernel/src/services/display_surface.mc` for one fixed four-cell visible
    target and present/query semantics
  - `kernel/src/services/surface_composition.mc` for one fixed two-cell body
    plus two-cell status composition path on the same surface
  - `kernel/src/input_event.mc` for the admitted `I K <byte> !` input family
  - `kernel/src/foreground_input_route.mc` for one foreground-app routing seam
  - `kernel/src/services/launcher_service.mc` for single-foreground program
    choice, direct `review_board` launch, and the existing installed
    `issue_rollup` launch classification path
  - `kernel/src/issue_rollup_app.mc` for app-local render, focus, and
    interaction truth on the first UI proof
  - `kernel/src/review_board_app.mc` plus
    `examples/real/review_board/src/review_board_ui.mc` for the second
    directly usable UI proof and its app-local interaction truth
  - `kernel/src/services/journal_service.mc` for one bounded persisted
    `review_board` view-state lane
- The admitted user-facing capabilities after Band E are now:
  - one directly usable installed foreground app path for `issue_rollup`
    with app-local rendering and the launch-only `FRSH` or `RSUM` or `INVD`
    overlay
  - one directly usable second foreground app path for `review_board` through
    the same launcher/display/input seams without a shared app framework
  - one optional body-versus-status composition path on the custom-manifest
    `issue_rollup` slice instead of a general layout or compositor model
  - one bounded persisted `review_board` view-state slice that reloads through
    the existing journal plus boot plus launch path instead of a session or
    document framework
  - one canonical local workflow on the ordinary `mc` project path plus the
    grouped reset-lane workflow path already documented in `README.md` and
    `tests/tool/README.md`
- The still-closed UI and platform areas remain explicit:
  - no second input family beyond the current key-only path
  - no window system, multi-surface policy, compositor, or general layout
    engine
  - no shared control vocabulary, widget toolkit, theming system, or
    accessibility framework
  - no text-system, document editor framework, or generalized app framework
  - no session restore, generic document store, save framework, or sync model
  - no UI-specific package manager, installer stack, desktop shell, or second
    build or run workflow outside the ordinary `mc` and grouped test paths
- The supporting evidence surfaces for the frozen claim set are:
  - the Band E phase docs from Phase 282 through Phase 288
  - the hosted UI workflow and commands documented in `README.md`
  - the hosted owning test targets
    `mc_tool_real_project_issue_rollup_unit` and
    `mc_tool_real_project_review_board_unit`
  - the reset-lane grouped UI workflow targets
    `mc_tool_workflow_kernel_reset_lane_unit` and
    `mc_tool_workflow_kernel_reset_lane_full_unit`
  - the focused restart-sensitive persistence proof under
    `tests/system/kernel_reset_lane_phase287_document_or_view_state_persistence_follow_through/`

Validation
----------
- no new executable validation was required because Phase 289 is a bounded
  freeze document and the owning executable evidence already landed in Phases
  282 through 288

Phase review against the spec
-----------------------------
- Preserved invariants:
  - the freeze records only landed repository truth from the Band E owners and
    proofs
  - deferred UI and platform areas remain explicit and reviewable
  - admitted capability is separated clearly from broader desktop or platform
    expectations
- Ownership boundaries did not weaken:
  - `display_surface.mc` remains a fixed-surface owner only
  - `surface_composition.mc` remains one tiny region assembler rather than a
    layout framework
  - `input_event.mc` and `foreground_input_route.mc` remain the admitted
    input-shape and foreground-routing owners rather than broader event policy
  - `launcher_service.mc` remains single-foreground and launch-classification
    owner rather than a UI-state or package framework
  - `issue_rollup_app.mc` and `review_board_app.mc` keep render and
    interaction semantics app-local instead of promoting a shared toolkit
  - `journal_service.mc` remains the bounded durable owner instead of a UI
    persistence framework
- Accidental abstractions and centers of gravity were avoided:
  - no desktop shell, app platform, widget family, session framework, or
    second workflow stack was introduced by the freeze
  - no display-owned render policy, launcher-owned UI policy, or durable-owner
    app framework was claimed
- Honest non-decisions and deferred work:
  - alternate input, shared controls, broader composition, richer persistence,
    and broader platform direction remain explicit later pressure rather than
    implied current capability

Future follow-up
----------------
- The next platform-direction band should consume this freeze as the current
  user-facing baseline.
- Reopen broader UI work only when later phases identify one bounded pressure
  that is stronger than the still-closed areas recorded here.
