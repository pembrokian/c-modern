Phase 292 -- Post-UI Kernel Ownership Audit
===========================================

Status
------
COMPLETED

Purpose
-------
Classify which new pressures after the second UI slice belong to kernel,
launcher, display, workflow, persistence, or app-local owners, and reject
convenience migrations that would blur those boundaries.

This phase exists now because once a small user-facing system demo exists, the
repository will naturally feel pressure to centralize behavior for convenience.
The next honest move is to audit those pressures explicitly before they become
boundary erosion.

Repository status note
----------------------
- Phase 289 freezes the current basic-UI owner map.
- Phase 290 composes launcher, update, UI, and app owners into one
  integrated small-system demo.
- The current kernel already has explicit owners for launcher choice, input
  parsing, foreground routing, fixed-surface presentation, optional small
  composition, durable journal state, and app-local interaction truth.
- This phase now records where new convenience pressure should and should not
  move.

Problem being solved
--------------------
- Integrated user-facing behavior creates strong temptation to move more logic
  into launcher, display, dispatch, or boot simply because those files are
  already on the main path.
- Without one explicit ownership audit, later work can easily widen core kernel
  seams into a hidden framework while still appearing locally convenient.
- The current-repo problem is therefore one missing post-UI ownership audit,
  not the absence of another runtime feature.

Non-goals
---------
- no new kernel service by default
- no launcher redesign, window system, or desktop policy
- no app framework or shared UI state machine
- no boot or dispatch refactor for convenience alone
- no new persistence lane unless the audit proves an existing owner is truly
  insufficient

Invariants
----------
- Boot continues to assemble state, not own runtime policy.
- `kernel_dispatch.mc` continues to route and chase effects, not absorb
  service-local or app-local state machines.
- Launcher remains the foreground-program owner, not a general UI coordinator.
- Display and composition owners remain bounded visible-surface owners.
- App-local interaction truth remains app-local unless a shared owner is
  stronger and explicit.

Required surface changes
------------------------
1. Audit the post-Phase-290 integrated flow for convenience migrations and
   boundary pressure.
2. Classify which pressures belong to existing kernel or app owners.
3. Record which tempting migrations should be rejected even if they simplify
   local code.
4. Identify only the bounded ownership gaps that remain genuinely unresolved.

Forbidden changes
-----------------
- do not turn the audit into an implementation phase for new owners
- do not widen `kernel_dispatch.mc`, boot, or launcher merely to simplify the
  integrated path
- do not restate app-local UI behavior as kernel-owned just because it is now
  exercised more often
- do not hide unresolved ownership tension by describing it vaguely

Phase role in the larger sequence
---------------------------------
- This phase is the post-UI ownership honesty check for the kernel and app
  boundary map.
- It complements Phase 291 by giving the next-wave decision a system-side
  classification alongside the compiler-side one.
- It also prepares Phase 293 by clarifying which user-facing claims are
  supported by stable owner boundaries.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- preserve the current kernel owner split as user-facing behavior becomes more
  integrated
- reject convenience migrations into boot, dispatch, launcher, or display that
  would turn bounded owners into hidden frameworks

Questions this phase must answer
--------------------------------
1. Which new pressures exposed by the integrated demo belong to launcher,
   display, workflow, persistence, or app-local owners?
2. Which tempting migrations into boot or dispatch should be rejected
   explicitly?
3. Are there any genuinely unresolved ownership gaps after Band E, or are the
   strongest remaining tensions still outside current repo scope?
4. Which still-deferred areas remain intentionally closed rather than missing?

Preferred implementation shape
------------------------------
- Use the frozen Band E owner map and the integrated Phase 290 flow as the two
  baseline inputs.
- Evaluate each pressure at the current controlling owner first before
  considering any boundary shift.
- Prefer explicit rejection of convenience migrations over vague calls for
  later cleanup.

Common breakage map
~~~~~~~~~~~~~~~~~~~
- Pressure assigned to boot or dispatch for convenience -> hidden framework
  drift and later repair churn.
- App-local interaction pressure mislabeled as launcher or display work -> UI
  boundary erosion.
- Audit forgets the frozen Band E baseline -> already-settled owners get
  reopened without new evidence.
- Unresolved ownership gaps left vague -> later phases cannot tell whether to
  add an owner or keep the pressure deferred.

Acceptance criteria
-------------------
- the repository has one explicit post-UI account of where the strongest new
  pressures belong in the kernel and app owner map
- tempting convenience migrations into boot, dispatch, launcher, or display

- Phase 294 can reuse this audit as one bounded system-side input to the next-

Implementation outcome
----------------------
- The integrated Phase 290 flow does create new convenience pressure, but the
  strongest pressures still fit the existing owner map cleanly rather than
  arguing for a new kernel framework.
- Launcher-side pressure remains narrow and already owned by
  `kernel/src/services/launcher_service.mc`: foreground selection, installed
  versus direct-launch choice, and launch-time fresh-versus-resumed-versus-
  invalidated classification stay there. The integrated demo does not justify
  moving app-local focus, body-versus-status choice, render refresh triggers,
  or persisted app view state into launcher.
- Display-side pressure remains bounded to
  `kernel/src/services/display_surface.mc` plus
  `kernel/src/services/surface_composition.mc`: visible four-cell storage and
  the tiny two-region assembly seam stay display-owned, while token meaning,
  overlay clearing, active-region truth, and app-specific redraw policy remain
  app-local. The demo does not justify a compositor, layout engine, shared
  render scheduler, or multi-surface policy.
- Workflow and persistence pressure also stays on the current owners instead
  of shifting into launcher or boot. `kernel/src/services/update_store_service.mc`
  remains the durable owner of installed bytes and the installed-launch record,
  `kernel/src/services/workflow/service.mc` remains the retained apply path,
  and `kernel/src/services/journal_service.mc` remains the bounded persisted
  `review_board` view-state lane. The integrated flow does not justify a new
  launcher-owned status cache, a boot-owned persistence shim, or a shared app-
  state store.
- App-local interaction pressure remains strongest in the existing app wrappers
  and app sources. `kernel/src/issue_rollup_app.mc` still owns the launch-only
  overlay spend, manifest-shaped present policy, and active-region truth for
  the first UI proof. `kernel/src/review_board_app.mc` plus
  `examples/real/review_board/src/review_board_ui.mc` still own the second
  app's direct interaction and view-state semantics. The integrated demo does
  not justify a shared UI-state machine, common focus owner, or kernel-owned
  app controller.
- The most tempting convenience migrations into boot or dispatch should be
  rejected explicitly:
  centralizing app-specific redraw or overlay-clear policy in
  `kernel/src/kernel_dispatch.mc`;
  moving foreground-input branching or active-region policy into dispatch just
  because all app traffic passes through it;
  moving installed-versus-direct launch coordination into `kernel/src/boot/boot.mc`
  or restart logic just because those files assemble state; and
  treating boot as the place to synthesize durable app summaries for launcher
  or display consumption. Those moves would turn routing or assembly seams into
  hidden coordinators.
- No genuinely unresolved kernel ownership gap outranks the existing deferred
  areas. After the integrated demo, the strongest remaining tensions are still
  intentionally closed rather than missing owners: richer input families,
  multi-surface policy, a broader session or persistence model, second-app
  install policy, and any general app or widget framework. The current repo
  plateau remains a bounded launcher-plus-display-plus-app system, not a
  partially hidden platform skeleton.
- The only repeated awkwardness that still shows through the integrated path
  without demanding a kernel owner shift is source-level readability around raw
  UI byte constants and byte scanning; that remains the compiler-side pressure
  recorded in Phase 291, not a reason to widen kernel boundaries here.

Validation
----------
- No executable validation was needed because Phase 292 lands as an ownership
  audit note only; the owning validation surface is editor diagnostics on the
  touched documentation.

Phase review against the spec
-----------------------------
- Preserved invariants:
  - boot remains an assembly owner, not a runtime policy owner
  - `kernel_dispatch.mc` remains a routing and effect-chasing seam rather than
    a service-local or app-local coordinator
  - launcher remains the single-foreground and launch-classification owner
  - display and composition remain bounded visible-surface owners
  - app-local interaction truth remains app-local
- Ownership boundaries did not weaken:
  - no new service, registry, framework, or persistence lane was introduced
  - the audit rejects convenience migrations into boot, dispatch, launcher,
    and display rather than normalizing them
  - unresolved pressure is named as intentionally deferred scope, not hidden as
    vague future cleanup
- No accidental center of gravity emerged:
  - this phase records the current owner split instead of moving more policy
    into the main-path kernel files
  - the next-wave decision can now compare one explicit system-side plateau
    against the compiler-side pressure from Phase 291
