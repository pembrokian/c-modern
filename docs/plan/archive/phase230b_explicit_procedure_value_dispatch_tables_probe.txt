Phase 230b -- Explicit Procedure-Value Dispatch Tables Probe
============================================================

Status
------
LANDED

Purpose
-------
Admit one narrow callable-data surface: explicit typed procedure-value dispatch
tables as ordinary data, and close only the owner boundaries needed to make
that shape compile honestly end to end.

This phase is intentionally smaller than a general callable-feature expansion.
It is a data-shape and proof-surface probe, not a closure, callback-framework,
or dynamic-dispatch architecture phase.

Repository status note
----------------------
- The language already has procedure types and callable values.
- The compiler already proves parameter passing, struct-field procedure values,
  and indirect calls in at least one narrow slice.
- The kernel and reset-lane system still prefer flat if-chains and switches,
  which is an architectural choice rather than proof that procedure-value
  tables are unwanted at the language level.
- Repeated real-code pressure now reports one narrower gap: there is no
  clearly maintained, obviously supported pattern for explicit typed dispatch
  tables of named procedures.

Landed outcome
--------------
- explicit typed fixed arrays and structs containing exact-signature named
  non-generic procedures are now admitted as ordinary typed data
- top-level `const` aggregates, local explicit typed aggregates, indexed lookup,
  field lookup, and ordinary indirect call all compile honestly through the
  owned pipeline
- imported constant aggregates containing procedure values survive module
  surface export, import rewriting, and build-unit namespacing
- the compiler change stayed narrow to sema constant evaluation, constant
  serialization, imported-symbol rewriting, and executable global emission
- the phase deliberately did not widen into closures, callback registries,
  erased callbacks, procedure equality, ordering, or generic procedure
  constants

Core rule
---------
- Keep the slice limited to explicit typed procedure-value tables.
- Prefer exact-signature named procedures only.
- Reuse existing ordinary call and aggregate-literal rules where possible.
- Fix only the owner boundary that rejects the admitted shape.
- Do not widen into closures, lambda syntax, callback registries, erased
  callback conventions, or kernel dispatch architecture changes.

Chosen slice for this phase
---------------------------
1. publish one maintained language note that states the narrow proposed surface
   for explicit procedure-value dispatch tables
2. determine whether the following shapes already work honestly end to end:
   - top-level `const` fixed arrays of named procedures with one exact
     signature
   - top-level `const` structs containing named procedures
   - local explicit typed aggregate literals containing named procedures
   - indexed or field-based lookup followed by ordinary indirect call
3. if any shape fails, repair only the rejecting owner boundary in parser,
   sema, MIR, constant handling, or executable lowering
4. add one compiler proof fixture and one executable or MIR proof using a real
   explicit dispatch-table shape rather than a synthetic callable-only probe
5. if the phase finds that the shape is still not justified, publish an
   explicit deferral and keep the kernel routing style unchanged

Planned outcome
---------------
- the repository will either admit explicit typed procedure-value dispatch
  tables honestly or document a clear bounded deferral
- authors will no longer need to infer the answer indirectly from scattered
  indirect-call or struct-field tests
- the language decision will stay narrow and will not be allowed to imply a
  broader callback or framework direction

Preferred implementation shape
------------------------------
Language-note path
- keep the semantics explicit: procedure-value tables are typed data, not a
  second dispatch system
- place the maintained note under the c-lang language docs and route the open
  question through `docs/arch/c-lang/open_decisions.txt`

Compiler-proof path
- prefer one fixed-array proof and one struct-shaped proof over a broad test
  matrix
- if a failure is found, fix the first rejecting owner instead of layering
  temporary rewrites around it
- keep signature matching exact; do not introduce coercion or erased callback
  policy

Kernel-boundary path
- do not treat language admission as a mandate to rewrite shell, boot, or
  service routing into tables
- keep any kernel-facing demonstration small and optional

Implemented repair
------------------
1. sema constant evaluation now treats named non-generic procedures as valid
  constant ingredients for explicit typed aggregates
2. `.mci` artifacts now serialize and parse procedure-valued constants
3. imported constant aggregates now rewrite procedure symbols through both the
  sema import surface and build-unit namespacing path
4. LLVM executable emission now renders procedure-valued global constants as
  function-symbol pointer constants

Validation expectations
-----------------------
- `Build debug`
- one focused sema or MIR fixture proving explicit typed procedure-value table
  construction
- one focused executable or codegen proof exercising lookup and indirect call
  through that table
- no regressions in existing callable-value or aggregate-literal tests

Validation completed
--------------------
- `Build debug`
- `mc_sema_fixture_unit`
- `mc_codegen_executable_core_unit`
- one direct imported-module build probe copying an imported procedure table
  into a local `const` and calling through it

Exit criteria
-------------
- the repository has a maintained answer on explicit typed procedure-value
  dispatch tables
- at least one real fixed-array and one struct-shaped table compile honestly
  through the owned pipeline
- the admitted shape stays limited to named procedures, exact signatures, and
  explicit typed aggregates
- no closure system, callback registry, or kernel routing framework is
  introduced

Non-goals
---------
- closures or captured environments
- anonymous function literals
- general callback framework admission
- dynamic service registries or table-driven kernel dispatch as policy
- procedure equality or ordering unless a real admitted consumer forces it
- erased `void*` callback conventions or ABI polymorphism

Why this is the narrowest honest next move
------------------------------------------
- The repository already has enough callable support that the remaining
  pressure is no longer "do procedure values exist" but "is one obvious table
  shape officially supported?"
- A narrow probe resolves that uncertainty directly.
- It keeps the language question small while preserving the repository's
  explicit, owner-local architectural style.
