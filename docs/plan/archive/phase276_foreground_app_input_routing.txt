Phase 276 -- Foreground App Input Routing
=========================================

Status
------
LANDED

Purpose
-------
Route the newly explicit input path to exactly one foreground app on the first
display path with one honest focus truth and no multi-window or session
framework.

This phase exists now because Band D should not leave input and display as
parallel mechanisms. The next honest slice is one explicit foreground-routing
owner that spends the current launcher truth on one human-facing app path.

Repository status note
----------------------
- Phase 268 already admitted one bounded launcher owner with single-foreground
  truth.
- Phase 274 should admit one explicit input-event path.
- Phase 275 should admit one explicit display owner.
- The repository still has no explicit routing seam that binds those two paths
  to exactly one foreground UI app.

Problem being solved
--------------------
- Input and display remain disconnected unless one owner decides where live UI
  events go.
- The first human-facing path needs one explicit focus or foreground answer so
  later repaint and app proof work do not invent hidden routing policy.
- Without this slice, later UI phases would either widen launcher into a
  general window manager or duplicate focus policy in app-local code.

Non-goals
---------
- no multi-window focus stack
- no background app input delivery
- no session manager or task-switching framework
- no pointer capture, selection graph, or modal routing framework
- no launcher-owned UI persistence beyond current bounded foreground truth

Invariants
----------
- `launcher_service.mc` remains the single foreground policy owner.
- Input routing stays single-foreground and explicit.
- The display owner remains a surface owner, not a focus-policy owner.
- Shell and dispatch remain routing seams only.
- No multi-app routing, sessions, or windows may be introduced.

Required surface changes
------------------------
1. Define one explicit route from the Phase 274 input owner to the current
   foreground app.
2. Reuse `launcher_service.mc` for the foreground-app decision.
3. Define one bounded no-foreground or unsupported-target outcome.
4. Add one fixture proving that only the foreground app receives the admitted
   input path.

Forbidden changes
-----------------
- do not introduce a second focus owner beside `launcher_service.mc`
- do not widen the display owner into routing policy
- do not add background delivery, subscriptions, or app registries
- do not push app-routing truth into `kernel_entry.mc` or scenarios by
  convenience

Phase role in the larger sequence
---------------------------------
- This phase composes the first two Band D anchors into one real UI path.
- It prepares the explicit repaint contract by defining which app may request
  or receive visible updates.
- It is the narrowest honest focus slice before any richer composition or
  second-input family is considered.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep launcher as the bounded app-facing policy seam
- keep the first UI path single-foreground and framework-free

Questions this phase must answer
--------------------------------
1. Where should the foreground-routing decision live without duplicating
   launcher truth?
2. What is the bounded failure or idle outcome when no foreground app is
   available?
3. What is the narrowest proof that input is reaching the right foreground app
   and no other consumer?
4. What later repaint or present contract does this routing slice need to make
   explicit?

Preferred implementation shape
------------------------------
- Keep `launcher_service.mc` as the source of current foreground-app truth.
- Keep event parsing or event-shape ownership in the Phase 274 input owner.
- Add one thin routing seam that asks launcher for the foreground target and
  forwards the event to exactly that app path.
- Keep the first route flat and explicit rather than introducing queues,
  registries, or modal routing state.

Implementation summary
----------------------
- Kept `input_event.mc` as the owner of admitted input-frame meaning by making
  it parse the `I K <byte> !` family into one compact parsed-input record
  instead of deciding the foreground app target directly.
- Added `kernel/src/foreground_input_route.mc` as the explicit routing seam
  that asks `launcher_service.mc` for the current foreground app and routes the
  admitted input family to exactly that one app path.
- Kept `kernel_dispatch.mc` as a thin caller that invokes the new routing seam
  before the existing shell route rather than absorbing focus policy or app
  routing tables.
- Preserved one explicit bounded no-foreground outcome and one explicit
  unsupported-target outcome without widening the first path into background
  delivery, sessions, or a focus stack.
- Added `tests/system/kernel_reset_lane_phase276_foreground_app_input_routing`
  as the focused reset-lane proof and wired it into the maintained grouped
  reset-lane workflow suite.

Implemented decisions
---------------------
1. The foreground-routing decision now lives in one dedicated seam,
   `foreground_input_route.mc`, rather than being folded into input parsing or
   launcher policy.
2. `launcher_service.mc` remains the sole owner of single-foreground truth;
   the new seam consumes that truth but does not store or derive a second focus
   model.
3. The bounded idle failure remains one explicit no-foreground reply on the
   admitted input family.
4. The first proof that only the foreground app receives input is one focused
   reset-lane scenario where `issue_rollup` stays foreground while
   `review_board` becomes only the selected program, and the routed input still
   lands on `issue_rollup` alone.

Preserved invariants review
---------------------------
- `launcher_service.mc` remains the single foreground policy owner.
- Input parsing stays in `input_event.mc`; display ownership stays in
  `display_surface.mc`; dispatch remains a routing seam only.
- No background delivery, multi-window focus, sessions, or registries were
  introduced.
- No central file became a new center of gravity: the new routing work lives in
  one small dedicated module rather than in launcher, dispatch, or display.
