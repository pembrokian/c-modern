Phase 294 -- Next-Wave Decision Note
====================================

Status
------
COMPLETED

Purpose
-------
Record which major next-wave pressure is strongest after the first integrated
user-facing system plateau, instead of assuming the next expansion follows
automatically from recent success.

This phase exists now because Band F is explicitly supposed to stop and decide.
After the integrated demo, the compiler audit, the kernel ownership audit, and
the support statement, the repository should choose its next bounded direction
deliberately rather than drifting into the most convenient adjacent work.

Repository status note
----------------------
- The speculative roadmap names the candidate next branches explicitly:
  broader app packaging, richer UI composition, networking, durable file-like
  growth, or another compiler follow-through wave.
- `docs/plan/backlog.txt` already carries active deferred themes for broader
  package or install policy, richer UI growth, file-like service growth,
  durable-owner boundaries, networking-style ingress, and compiler-local
  follow-through.
- Phases 291 through 293 are expected to produce the local evidence needed to
  choose among those branches.
- The repository still lacks one explicit recorded decision about which branch
  currently wins and why the others do not.

Problem being solved
--------------------
- After a visible successful wave, adjacent next steps can all look plausible.
  Without one explicit decision note, later planning can drift toward the most
  convenient code surface rather than the strongest evidence-backed pressure.
- The repository also risks reopening already-deferred branches prematurely if
  the reason they lost is not recorded while the comparison is fresh.
- The current-repo problem is therefore one missing next-wave decision record,
  not an absence of possible work.

Non-goals
---------
- no new implementation landing by default
- no broad roadmap rewrite beyond recording the chosen next-wave branch
- no speculative ranking of branches that have no current-repo evidence
- no attempt to solve ties by combining multiple major waves into one band

Invariants
----------
- The decision must be driven by current maintained evidence, not by generic
  product ambition.
- Branch candidates must be compared as bounded next-wave options, not as open-
  ended strategic visions.
- The note must record why the chosen branch wins and why the strongest losers
  remain deferred.
- The decision must preserve current owner boundaries and the frozen post-Band-
  E baseline.

Required surface changes
------------------------
1. Compare the main next-wave candidates against the evidence from Phases 290
   through 293 and the current backlog.
2. Record the winning bounded next-wave branch.
3. Record the key reasons the non-winning branches remain deferred.
4. Update only the narrow roadmap or planning surface that must reflect the
   decision.

Forbidden changes
-----------------
- do not choose multiple major next-wave branches at once just to avoid making
  a decision
- do not restate deferred branches as implied commitments
- do not pick a branch solely because it feels more visible or more fun
- do not widen the note into a broad architecture treatise

Phase role in the larger sequence
---------------------------------
- This phase closes the speculative post-254 roadmap horizon with one recorded
  decision.
- It gives the repository a clean handoff into the next concrete planning wave.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- compare deferred compiler, UI, packaging, durable file-like, and networking
  branches against the current integrated-system evidence
- keep next-wave selection evidence-based and bounded instead of convenience-
  driven

Questions this phase must answer
--------------------------------
1. Which bounded next-wave branch is strongest after the integrated demo and
   the neighboring audits?
2. Why does that branch outrank broader packaging, richer UI, networking,
   durable file-like growth, and the competing compiler options?
3. What exact evidence would have to change for one of the losing branches to
   win later?
4. Which planning surfaces must be updated so the decision stays durable and
   visible?

Preferred implementation shape
------------------------------
- Start from the ranked evidence in Phase 291 and Phase 292, then cross-check
  the active backlog themes and the human-facing support plateau from Phase 293.
- Prefer naming one winning branch with a short loser matrix over a vague
  multi-direction note.
- Keep the decision recorded in one durable planning note plus only the
  narrowest roadmap sync required.

Common breakage map
~~~~~~~~~~~~~~~~~~~
- Decision note names no winner -> the repo remains drift-prone and the band
  does not actually close.
- Winning branch has no stronger evidence than a loser -> the note becomes
  preference-driven instead of repo-driven.
- Losing branches are not explained -> the same arguments get rediscovered in
  the next planning cycle.
- Roadmap sync widens too far -> speculative admin docs churn more than the
  bounded decision requires.

Implementation outcome
----------------------
- The repository now records one explicit next-wave winner: a bounded richer
  UI-composition-and-interaction wave on the existing launcher-selected,
  fixed-surface, app-local owner map.
- That branch wins because the current supported slice has already reached a
  human-facing plateau with two directly usable apps, one maintained integrated
  small-system demo, one fixed four-cell visible contract, one optional
  two-region composition seam, and one bounded persisted `review_board`
  view-state path. The strongest present ceiling in that admitted slice is no
  longer launcher truth, installed-slot truth, or kernel owner confusion; it is
  how little expressive room the current visible surface and app-local
  interaction vocabulary provide above that already-stable substrate.
- Phase 292 is the decisive system-side input: it shows that boot, dispatch,
  launcher, display, workflow, persistence, and app wrappers already have an
  honest owner split, and that the remaining meaningful pressure is not another
  ownership repair but what bounded user-facing layer should be admitted next
  without turning the kernel into a toolkit or framework.
- Phase 291 remains the strongest compiler-side loser input, but it loses
  narrowly and explicitly. One byte-literal or character-literal slice is now
  credible because raw ASCII bytes make current UI and input code harder to
  read, yet that pressure is still a readability cost rather than the main
  blocker on the admitted user-facing path. The repository can continue to ship
  the current slice honestly without that compiler follow-through, while the UI
  ceiling is now the clearest limitation of what users can do.
- Broader app packaging remains deferred because the current maintained install
  story still has only one installed-program path (`issue_rollup`) plus one
  direct-launch second app (`review_board`). The evidence still does not justify
  package-manager semantics, multi-app install policy, richer launcher-owned
  metadata, or a broader app execution platform.
- Networking remains deferred because the repository already has bounded hosted
  and reset-lane ingress proofs, but the current human-facing plateau is not
  blocked on richer network ownership. No maintained user-facing app or
  launcher/update path currently depends on another networking wave to become
  more coherent or more usable.
- Durable file-like growth remains deferred because `file_service` is still an
  isolated pressure probe rather than the controlling owner of any current
  supported user-facing workflow. Nothing in the integrated demo, support
  statement, or current app pair argues that filesystem-style growth now beats
  the visible UI ceiling.
- The winning branch stays bounded. This note does not choose a desktop shell,
  widget toolkit, package manager, filesystem platform, or broad app framework.
  It chooses only the next major direction: one more deliberate UI-facing wave
  on the already-frozen owner map.
- The narrow roadmap sync is now recorded in
  `docs/plan/admin/speculative_roadmap_post_phase254.txt`, which names richer
  UI growth as the Band-F winner and leaves packaging, networking, durable
  file-like growth, and the byte-literal compiler slice explicitly deferred.

Future follow-up
----------------
- The next concrete planning band should be drafted as one bounded richer
  UI-facing wave above the current fixed-surface and app-local owner split,
  not as a packaging or compiler wave by default.
- Reopen broader app packaging only when a second real installed-program path
  exists or when launcher/update callers need app-facing truth that cannot stay
  honest on the current installed-slot plus direct-launch split.
- Reopen the byte-literal compiler branch only when the current raw-byte source
  awkwardness becomes a stronger blocker than the admitted UI-growth pressure,
  or when a narrower UI-facing phase depends directly on that readability repair.
- Reopen networking or durable file-like growth only when a maintained
  repository-owned consumer makes one of those owners the clearest next blocker
  on the supported user-facing slice.
