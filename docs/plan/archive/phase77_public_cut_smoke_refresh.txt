Phase 77 -- Public-Cut Smoke Refresh
====================================

Status
------
IMPLEMENTED

Purpose
-------
Refresh the broader public-cut smoke path after the supported-slice re-audit so
the documented smoke commands and the repo-owned executable proof match the
current admitted hosted slice exactly.

This document is a repository execution plan and closeout note, not a new
architecture phase after Phase 8.

Repository status note
----------------------
- Phase 75 already turned the narrow first-use path into one repo-owned smoke
  entrypoint.
- Phase 76 then re-audited the README and hosted release note so the supported
  slice statement matched the actual admitted surface.
- The remaining gap after those waves was broader proof repeatability: the
  public-cut smoke section still mixed older Phase 33 command inventory with
  newer proof claims and newer admitted examples.

Problem being solved
--------------------
- The repository already documented a public-cut smoke set, but that command
  list no longer matched the full set of proof claims attached to it.
- In particular, the docs claimed explicit same-build-dir target switching on
  the shipped `issue_rollup` project while the command list still omitted the
  additive report-target run that now carries that proof.
- The docs also blurred the separate Phase 75 first-use smoke path into the
  broader public-cut proof bullets instead of keeping those two validation
  layers distinct.
- Without one explicit refresh wave, the repository risked having a correct
  supported-slice statement and a stale broader smoke path.

Core rule
---------
- Keep the public-cut smoke broader than the first-use path, but still narrow
  enough that one repo-owned command can be audited by hand.
- Update the smoke list to reflect the current admitted proof set exactly;
  do not widen the supported slice in order to justify stale commands.
- Reuse the admitted real examples and the existing first-use smoke boundary
  rather than inventing synthetic public-cut fixtures.

Chosen slice for this phase
---------------------------
Targeted proof shape: one repo-owned public-cut smoke automation path plus one
documentation refresh aligning the listed public-cut commands with that same
path.

Chosen implementation:
- add one focused `mc_public_cut_smoke` CTest that runs:
  - direct-source `mc check` on `tests/cases/hello.mc`
  - direct-source `mc build` on `tests/codegen/smoke_return_zero.mc`
  - default-target `issue_rollup` run on one shared build directory
  - explicit `issue-rollup-report` run on that same build directory
  - ordinary `mc test` workflows for `issue_rollup`, `worker_queue`,
    `pipe_handoff`, `pipe_ready`, `line_filter_relay`, and `evented_echo`
- add `make public-cut-smoke` as the top-level convenience entrypoint for that
  same public-cut path
- refresh README and the hosted release-hardening note so the documented
  public-cut commands match the repo-owned smoke exactly

Explicit non-choices:
- no release-mode certification expansion
- no broad new fixture family beyond the current admitted real projects
- no attempt to collapse the narrow first-use smoke and the broader public-cut
  smoke into one indistinguishable workflow
