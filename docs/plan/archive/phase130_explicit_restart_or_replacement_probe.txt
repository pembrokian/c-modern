Phase 130 -- Explicit Restart Or Replacement Probe
=================================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the completed Phase 129 partial failure propagation step by
proving one bounded explicit restart or replacement story over the same
already-landed Veya kernel artifact.

This document is both the Phase 130 execution note and its closeout.
It does not claim kernel-side supervision, automatic restart policy,
dynamic namespace repair, capability rebinding, a broader failure framework,
or a compiler wave.
It records one narrow explicit replacement step over the real kernel artifact
that earlier phases already kept inspectable.

Repository status note
----------------------
- Phase 129 already proved that one bounded service death remains a partial
  failure because one second bounded service path still replies and exits
  cleanly through the same fixed directory and shared control-endpoint model.
- The same landed running-system slice already carried one explicit service-
  death observation answer, one explicit fixed service-directory answer, and
  one explicit init-owned multi-service orchestration answer over the same
  admitted artifact.
- The next honest gap is not a broader recovery framework or a compiler
  reopening. It is publishing one explicit statement that init can choose one
  bounded replacement path over one dead service through the retained program
  capability and the same fixed control endpoint while the kernel still does
  not become a supervisor.

Problem being solved
--------------------
- The repository already had one explicit dead-service observation and one
  explicit partial-failure statement, but it still lacked one published proof
  that the current model can express one bounded restart or replacement story
  without importing kernel-side recovery policy.
- Without this step, later failure, IPC-composition, or device-boundary work
  could blur whether the current model already keeps one bounded replacement
  path explicit or whether it would need supervision semantics first.
- The repository needed one small phase that reuses the landed log-service
  helper, retained program capability, fixed directory key, shared control
  endpoint, and init-owned orchestration model rather than inventing a new
  service manager, rebinding layer, or compiler-owned mechanism.

Core rule
---------
- Keep the current kernel owners, fixed directory, and shared control
  endpoint intact.
- Keep the compiler, sema, MIR, backend, ABI, and `hal` surfaces closed.
- Reuse the landed log-service execution helper and retained init-owned
  program capability rather than inventing a broader restart framework.
- Prove one bounded replacement path remains explicit because init chooses to
  rerun the same bounded service through the same fixed service key after the
  first run exits.
- Do not widen into kernel-side supervision, automatic recovery, dynamic
  namespace repair, capability rebinding, or another compiler wave.
