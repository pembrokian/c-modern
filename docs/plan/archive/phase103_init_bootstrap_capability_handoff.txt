Phase 103 -- Init Bootstrap Capability Handoff
==============================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the completed Phase 102 timer sleep and scheduler block-wake
proof into the next remaining Band C owner: prove that the repository-owned
kernel artifact can start one isolated `init` process, hand it one explicit
bounded bootstrap capability set, and keep root authority kernel-only without
yet claiming real service launch, namespace policy, or a broader running
user-space stack.

This document is both the Phase 103 execution note and its closeout.
It does not claim a running log service, loader service, service registry,
ambient authority discovery, or broader init-owned orchestration.

Repository status note
----------------------
- Phase 81 through Phase 95 produced the bounded freestanding `v0.3` proof
  ladder and froze that lane into the first admitted freestanding support
  slice.
- Phase 96 then made one real kernel-owned boot entry and deterministic state
  skeleton concrete.
- Phase 97 then widened that owned kernel target into one bounded address-
  space and first-user-entry handoff.
- Phase 98 then completed the remaining Band A object-core owner by making
  explicit endpoint tables, per-process handle tables, and bounded queued-
  message lifetime concrete inside the same repository-owned kernel artifact.
- Phase 99 then made the first real syscall-owned byte-only send and receive
  boundary honest over that landed endpoint and handle object core.
- Phase 100 then made one attached-handle transfer concrete across that same
  syscall IPC boundary.
- Phase 101 then made the first explicit program-capability spawn and wait
  path concrete.
- Phase 102 then made the first explicit timer-backed sleep, wake, and ready-
  restoration path concrete.
- The strongest next kernel question was therefore no longer whether the
  kernel could boot, reach first user entry, and drive one bounded child
  lifecycle. It was whether that same real kernel artifact could now show
  exactly what authority `init` receives at boot and prove that root authority
  stays kernel-only.

Problem being solved
--------------------
- The speculative Veya kernel roadmap names init bootstrap capability
  handoff as the next remaining owner after the landed timer block-wake path.
- The repository already had one init-owned endpoint handle, one launchable
  program capability later consumed by spawn, and one root capability kept in
  kernel state, but it still had no explicit boot contract describing which of
  those authorities actually become visible to `init`.
- Without this step, later service-launch phases could overclaim that the
  kernel already enforces explicit capability discipline at boot when the
  repository still only had the individual ingredients spread across globals
  and proof-local validation.

Core rule
---------
- Keep the phase to one explicit bounded bootstrap capability set.
- Reuse the already-landed init-visible endpoint handle and one launchable
  program capability rather than inventing a broader authority taxonomy.
- Make the handoff explicit in one ordinary helper module and keep the kernel-
  owned act of installation visible in the root proof module.
- Prove that wait handles are not ambient at boot and that the root capability
  remains kernel-only.
- Stop before service launch, registry policy, loader generalization, or
  broader capability refinement.

Chosen slice for this phase
---------------------------
Targeted proof shape:

1. one widened `init` helper module that owns one bounded bootstrap-
   capability-set record and one deterministic handoff-observation record
2. one repository-owned kernel follow-through that materializes the explicit
   init bootstrap set from the landed init endpoint handle plus one launchable
   program capability
3. one validation path proving that the bootstrap set contains exactly those
   explicit authorities, that no wait handle is ambient at boot, and that the
   root capability remains kernel-only
4. one continued child-spawn, timer-sleep, wake, and reap follow-through that
   now runs on top of the explicit handoff contract rather than implicit boot-
   global knowledge
5. one focused freestanding regression and MIR inspection proving that the
   slice stays inside ordinary imported structs, aggregate initialization,
   enum matching, and explicit global-state writes

Chosen outcome
--------------
Chosen outcome: yes.

The repository should cross this boot-authority boundary now because Phase 102
already proved that the kernel can hand off execution and drive one bounded
child lifecycle. The next honest owner is not real service launch yet. It is
one explicit statement of what authority `init` starts with.

What this phase proves
----------------------
- one real Veya kernel artifact now installs one explicit bounded bootstrap
  capability set for `init`
- that set is intentionally small: one init-visible bootstrap endpoint handle
  plus one launchable program capability already used by the landed spawn path
- wait handles are not ambient at boot and still appear only after explicit
  spawn
- root authority remains kernel-owned and is not visible through the init
  bootstrap set
- the current language, freestanding build path, and runtime startup can carry
  the necessary handoff records without new compiler work
- the repository can stop honestly at one explicit init bootstrap handoff
  without implying that real service launch or broader user-space policy is
  already complete

Exit criteria
-------------
- one repository-owned kernel target materializes one explicit bounded init
  bootstrap capability set
- that set contains the landed init-visible endpoint handle plus one
  launchable program capability and no ambient wait handle
- root authority remains kernel-owned and absent from the init bootstrap set
- the runtime proof exits with exactly one bounded Phase 103 marker only after

  follow-through
- the merged MIR preserves imported bootstrap-handoff record types, helper
  functions, and explicit global writes for the new init-owned capability-set
  state
- the planning surface records Phase 103 as concrete rather than speculative
