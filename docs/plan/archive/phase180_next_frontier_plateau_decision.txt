Phase 180 -- Next Frontier Plateau Decision
===========================================

Status
------
COMPLETE

Outcome
-------
Branch A landed: the service-system frontier remains the stronger next owner,
and bounded second-target portability work stays deferred.

The comparison set from Phases 176 through 179 is now explicit:

- Phase 176 found one more real service-owned pressure point and closed it
  with one flat boot-wired `echo_service` plus explicit restart.
- Phase 177 found one real hostile-user shell truth gap and closed it with
  compact shell-owned invalid classification rather than wider payloads or a
  broader error model.
- Phase 178 found one real growth friction in topology and restart readability
  and closed it with flat lookup and endpoint-driven restart helpers rather
  than a routing table or supervisor.
- Phase 179 found one real restart-with-state pressure point and closed it
  with one explicit `log_service` snapshot and reload path rather than a
  persistence or packaging framework.

Phase 174, by contrast, named portability seams honestly but did not expose a
real admitted consumer whose narrowest next blocker was second-target support.

Phase result
------------
The next wave should stay on service growth, service identity, retained state,
and user-facing interaction inside the current running-system slice.

Bounded second-target work remains deferred until both of these are true:

1. a real admitted consumer needs Linux arm64 or another second target for a
   behavior Darwin arm64 cannot supply honestly
2. the strongest next blocked path is no longer an owner-local service-system
   question that can be answered inside `kernel/src/` without target work

What did not change:

- no second-target implementation work was admitted
- no compiler, sema, MIR, backend, ABI, runtime, or `hal` expansion was
  admitted for this phase
- no service-system question was declared permanently settled; only the next
  ordering decision is settled

Purpose
-------
Decide, after the first post-Phase-174 external-pressure wave, whether the
next honest frontier is continued service-system growth or bounded second-
target portability work.

This phase is a decision phase, not an implementation family.
It exists so the repository does not pursue both fronts at once without
evidence.

Repository status note
----------------------
- Phase 174 names the portability distance but does not yet justify second-
  target implementation work.
- Phases 176 through 179 now provide the comparison set for a different
  question: whether the stronger near-term pressure is still inside the
  service system once it has faced real growth, hostile use, and one bounded
  retained-state reload path.
- The repository should not open both portability and richer service-system
  work simultaneously unless one front clearly stops being the narrowest next
  owner.

Problem being solved
--------------------
- Without an explicit plateau decision, the repository risks mixing two
  plausible but different next waves:
  one about making the service system more real under constraint,
  and one about making the whole toolchain and runtime less Darwin-specific.
- Those waves stress different owners and should not be pursued together by
  default.

Core rule
---------
- Compare only real evidence from Phases 174 and 176 through 179.
- Do not authorize both fronts at once by vague language.
- If service growth still exposes the stronger real pressure, portability work
  remains deferred.
- If service growth stabilizes and Phase 174's portability seams are now the
  clearest next blocker, then authorize one bounded portability follow-through.

Chosen slice for this phase
---------------------------
Branch A -- service-system frontier remains stronger
- one explicit statement that the next wave should stay on service growth,
  identity, retained state, or user-facing interaction
- one refreshed trigger condition for reopening bounded second-target work

Branch B -- bounded portability follow-through now justified
- one explicit statement that the next wave should move to one bounded
  second-target seam or hosted portability probe
- one explicit statement of what service-system questions are now sufficiently
  stable to leave deferred for that work

Why this is the right placement
--------------------------------
- After Phase 179: the repository has finally tested the current system under
  real post-coherence pressure instead of more internal refinement.
- After Phase 174: portability distance is already named and can be weighed
  honestly against the new system pressure.

Planning rule for this phase
----------------------------
- Decide based on the narrowest blocked path, not on architectural taste.
- Prefer the front that currently has the clearest real consumer pressure.
- Preserve explicit deferral for whichever front loses.

What this phase should prove
----------------------------
- that the repository can choose one next frontier deliberately rather than by
  drift
- that portability and system-growth work stay evidence-driven and mutually
  ordered
- that the post-174 wave ends with a real plateau statement instead of five
  disconnected follow-on ideas
