Phase 106 -- Real Echo-Service Request-Reply Proof
==================================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the completed Phase 105 real log-service handshake into the
next remaining Band C owner: prove that the repository-owned kernel artifact
can launch one second real bounded service task from one explicit program
capability, drive one deterministic request-reply exchange whose reply bytes
are derived from the client request over the landed real syscall IPC path,
and stop before claiming multi-service startup policy, service registries,
or a running general user-space system.

This document is both the Phase 106 execution note and its closeout.
It does not claim broader service orchestration, namespace discovery,
user-to-user capability transfer beyond the landed Phase 100 slice, or a
general init-owned runtime.

Problem being solved
--------------------
- The speculative Veya kernel roadmap names a real echo-service request-
  reply proof as the next bounded service-policy owner after the landed real
  log-service handshake.
- The repository already had the necessary ingredients spread across the
  owned kernel lane: one explicit init-visible endpoint handle, one real send
  and receive syscall path, one consumed-on-success program-capability spawn
  path, one explicit wait-handle reap path, and one demonstrated single-
  service follow-through.
- Without this step, later service phases could overclaim that the repository
  already preserves request-dependent reply semantics across more than one
  bounded service shape when the landed proof still stopped at one fixed
  acknowledgment protocol.

Core rule
---------
- Keep the phase to one second real bounded service task only.
- Reuse the already-landed syscall, endpoint, handle, spawn, and wait owners.
- Add only the smallest helper module needed to keep the echo-service
  protocol state explicit and inspectable.
- Keep the protocol to one init-issued request and one service-issued reply
  whose bytes are copied from that request.
- Stop before broader service startup policy, registries, restart semantics,
  or a running multi-service user-space system claim.

Chosen slice for this phase
---------------------------
Targeted proof shape:

1. one bounded `echo_service` helper module owning explicit request and reply
   state, reply-payload construction, and final exchange observation
2. one repository-owned kernel follow-through that seeds one dedicated
   echo-service program capability, spawns one echo-service task, and
   installs one explicit service-visible endpoint handle
3. one deterministic init-issued two-byte request and one service-issued
   two-byte reply over the landed real syscall IPC path, where the reply bytes
   are copied from the request payload rather than hardcoded
4. one explicit service-exit and wait-handle reap follow-through so the proof
   cleans up the bounded child slot it reuses
5. one focused freestanding regression and MIR inspection proving that the
   slice stays inside ordinary imported structs, enums, aggregate
   initialization, fixed-array payload construction, and explicit helper
   boundaries while preserving the landed Phase 105 log-service proof

Chosen outcome
--------------
Chosen outcome: yes.

The repository should cross this second service boundary now because Phase 105
already proved the first real service launch and reap path. The next honest
owner is not a general service framework. It is one bounded real request-
dependent reply proof through a second helper-owned service shape.
