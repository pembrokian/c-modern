Phase 258 -- Procedure Dispatch Table Follow-Through
====================================================

Status
------
COMPLETED

Purpose
-------
Spend the already-admitted exact-signature procedure-table surface on one real
repository-owned consumer so the repository stops carrying that capability only
as an isolated compiler proof and starts maintaining it as a living table-owned
pattern.

This phase exists now because Phase 230b already closed the language and
constant-emission admission for explicit procedure-value dispatch tables, but
the repository still lacks one maintained post-230b consumer that proves where
procedure tables are actually honest and where flat if-chains must remain.

Repository status note
----------------------
- Phase 230b already admitted exact-signature named procedures in explicit
  typed arrays and structs, including imported constant aggregates.
- The backlog explicitly keeps broader callable-feature widening deferred.
- Kernel dispatch, shell routing, and restart control intentionally remain flat
  conditional owners and should not be rewritten into tables by convenience.
- The repository now has one maintained answer to a narrower question: the
  fixed render-kind to writer mapping in `examples/real/issue_rollup/src/render/rollup_render.mc`
  is an honest same-module procedure-table owner.

Problem being solved
--------------------
- The compiler now supports explicit procedure-value tables, but the feature is
  still maintained mostly as capability proof rather than as a living,
  repository-owned pattern.
- Without one honest consumer, later app or launcher or UI work will either
  avoid procedure tables entirely out of caution or misuse them in owners that
  should remain explicit if-chains.
- The real current-repo problem is not another callable feature gap. It is the
  lack of one maintained example that draws the line between an honest table
  owner and a framework-shaped abuse.

Non-goals
---------
- no closure or callback-framework widening
- no generic procedure constants
- no kernel-dispatch, shell, or boot routing-table rewrite
- no service registry, broker, or discovery framework
- no equality or ordering for procedures
- no dynamic plugin or scriptable dispatch surface

Invariants
----------
- Procedure tables remain ordinary typed data holding exact-signature named
  non-generic procedures only.
- Kernel dispatch, shell decode, and boot topology remain explicit owner-local
  conditionals where that boundary is already intentional.
- Sema owns callable legality; MIR and backend lower admitted callable truth.
- Deterministic constant transport and executable lowering must remain stable.
- The phase must prove one honest table owner, not imply a platform-wide
  callback direction.

Required surface changes
------------------------
1. Choose one repository-owned consumer where a fixed exact-signature procedure
   table is more honest than the current handwritten branch family.
2. Keep the consumer outside kernel dispatch, shell decode, and other owners
   whose flat explicit shape is already intentional.
3. Add focused sema, MIR, and executable regressions for the chosen procedure-
   table consumer if existing 230b coverage is not sufficient.
4. Document why the chosen consumer is honest while the preserved flat owners
   remain intentionally non-table-driven.
5. Preserve the already-admitted exact-signature, named-procedure-only limit.

Forbidden changes
-----------------
- do not use this phase to justify table-driven kernel dispatch
- do not widen into generic callbacks, closures, or dynamic service routing
- do not treat one honest table owner as evidence for registry or plugin
  frameworks
- do not reopen callable syntax or constant-data semantics already settled in
  Phase 230b unless a real regression is found
- do not move owner-local policy into tables merely because it compresses code

Phase role in the larger sequence
---------------------------------
- This phase turns the already-landed procedure-table compiler capability into
  a maintained consumer pattern before Band B and Band C reach more app-facing
  dispatch pressure.
- It complements Phase 257 by proving one table family that includes callable
  behavior without reopening broad callable work.
- It also sharpens the architectural line around where table-driven source is
  and is not honest in this repository.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep broader callable-feature widening deferred after Phase 230b now that
  imported struct procedure tables lower honestly across module boundaries

Questions this phase must answer
--------------------------------
1. What is the narrowest honest procedure-table consumer in the current tree:
   app-local command decode, launcher-local action selection, or another fixed
   exact-signature table?
2. Which current flat owner must remain flat even after the new consumer lands,
   and how should that boundary be stated explicitly?
3. Do the existing 230b compiler regressions fully cover the chosen consumer,
   or is one new executable proof needed?
4. Can the chosen consumer stay same-module, or does it rely on imported
   callable tables that should be coordinated with Phase 259?

Implementation outcome
----------------------
- Chosen consumer: `examples/real/issue_rollup/src/render/rollup_render.mc`
  now routes one classified render kind through a fixed `[4]func() i32`
  `ROLLUP_WRITERS` table.
- The consumer stayed same-module, fixed-size, and explicitly typed. It did
  not reopen imported callable tables, callback registries, descriptor-plus-
  procedure table composition, or any kernel routing owner.
- Focused proofs now cover the same shape in `tests/compiler/sema/issue_rollup_render_dispatch_table_ok.mc`,
  `tests/compiler/mir/issue_rollup_render_dispatch_table_ok.mc`, and
  `tests/compiler/codegen/issue_rollup_render_dispatch_table.mc`, while the
  existing `issue_rollup` project workflow continues to build and run the real
  consumer.

Preferred implementation shape
------------------------------
- Reuse the exact procedure-table contract admitted in Phase 230b.
- Keep the first consumer fixed-size, explicitly typed, and repository-owned.
- Prefer a consumer in support or app-local code over any core kernel routing
  owner.
- If the consumer needs labels or metadata beside procedures, compose it with
  the descriptor-table work from Phase 257 rather than inventing a second table
  model.
