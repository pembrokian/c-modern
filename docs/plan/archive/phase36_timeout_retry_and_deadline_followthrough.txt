Phase 36 -- Timeout, Retry, And Deadline Follow-Through
=======================================================

Status
------
PLANNED

Purpose
-------
Use the narrow elapsed-time slice from Phase 35 to land one real consumer-
facing follow-through wave: timeout, retry, backoff, or deadline behavior on
the admitted project that justified Lane C.

This document is a repository execution plan, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- Phase 34 must authorize Lane C through a real consumer matrix.
- Phase 35 must then land the minimal `Duration` plus monotonic-clock slice.
- The repository still should not assume that sleep is part of the first `time`
  admission; it must be justified by the chosen consumer follow-through.
- This phase is therefore about applying the admitted time slice to one real
  project honestly, not about broadening the `time` family for its own sake.

Problem being solved
--------------------
- A narrow time API only becomes meaningful if a real admitted project uses it
  to close an actual runtime-facing gap.
- Without a follow-through phase, the repository could land `Duration` and a
  monotonic clock yet still fail to prove why they belong in the admitted slice.
- The repository needs one concrete consumer outcome: retry budget, elapsed
  deadline, bounded wait, or backoff logic that is clearer with `time` than
  without it.

Core rule
---------
- Land one real consumer follow-through, not several unrelated time features.
- Prefer deadline or retry logic expressed through elapsed-time comparison
  before admitting sleep.
- Add narrow sleep only if the chosen consumer cannot stay honest without it.
- Keep behavior explicit in project or stdlib code rather than hiding it in
  tooling.
- Record any remaining broader `time` desires as deferred instead of widening
  the family opportunistically.

- sleep is either justified narrowly by the consumer or kept deferred
- the repository can say clearly what its admitted hosted `time` slice now
  covers and what it still does not

3. Narrow sleep only if the chosen project proves it is required.
4. Deterministic project and ordinary-test proof for the new behavior.
5. Documentation updates that keep the hosted support statement honest.

Exit criteria
-------------
- one admitted project now proves the hosted `time` slice through honest
  timeout, retry, backoff, or deadline behavior
- the repository can describe the admitted `time` surface concretely and
  narrowly
- any broader `time` growth remains explicit and deferred rather than implied
- the next plateau after Lane C is chosen from clearer evidence, not from the

Non-goals

- scheduler work, async runtime work, or platform escape hatches
- admitting `os` or `hal` because the chosen project wants convenience helpers
- turning ordinary test timeouts into runtime semantics implicitly
