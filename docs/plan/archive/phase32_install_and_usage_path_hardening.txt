Phase 32 -- Install And Usage Path Hardening
===========================================

Status
------
COMPLETED (repository-bounded install and usage hardening)

Purpose
-------
Record the clearer first-user installation and usage path built on top of the
 frozen Phase 31 supported slice on the supported Darwin arm64 hosted slice,
 without widening into distribution, package management, or portability
 claims.

This document is a repository closeout note, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- The repository now has a stronger admitted hosted plateau than the original
  published release note covered: grouped package roots, one in-project
  `staticlib` target, and deterministic selected-target reuse all exist on the
  supported host.
- Phase 31 froze that supported boundary into one explicit support matrix.
- After that freeze, the next release-hardening leverage was no longer "what
  is supported?" but "how does a first user build, invoke, and verify it
  without depending on repository folklore?"

What landed
-----------
- `README.md` now presents one explicit supported-host workflow instead of a
  loose mix of build commands and historical examples.
- The README now states the supported host assumptions directly: Darwin arm64,
  hosted runtime only, and build-from-source through CMake plus the ordinary
  host Clang toolchain.
- The README now separates the supported build-from-source path from broader
  developer convenience commands and gives one first-use smoke sequence that a
  new contributor can run from `build/debug/mc`.
- The README now makes the direct-source versus project-mode command boundary
  explicit and spells out the supported target and workflow surface.
- The README now records build-directory expectations clearly: `build/` is
  disposable, per-command `--build-dir` outputs are disposable, and `build/debug/mc`
  is the produced compiler entrypoint for the supported path.
- The hosted release-hardening note now mirrors the same host assumptions,
  build path, smoke validation path, artifact policy, and unsupported
  expectations.
- The slice register now records the release-hardening family as including the
  Phase 32 install-and-usage hardening wave rather than only the Phase 31
  support-matrix freeze.

What did not land
-----------------
- This phase did not admit binary distribution, package registries, installer
  tooling, or cross-platform setup guidance.
- This phase did not widen the supported compiler, runtime, stdlib, or
  project-graph surface.
- This phase did not add a broad public-cut smoke audit or checked-versus-
  debug completion claim; that remains Phase 33 work.
- This phase did not require backlog or decision-log updates because the
  install-path audit exposed no new consciously limited compromise beyond the
  already-frozen support boundary.

Hardened first-user path
------------------------
- supported host assumption: Darwin arm64 only
- supported installation story: build from source only
- required toolchain assumption: CMake plus the ordinary host Clang toolchain
- produced compiler entrypoint on the supported path: `build/debug/mc`
- supported direct-source usage: `mc check` and `mc build`
- supported project usage: `mc check`, `mc build`, `mc run`, and `mc test`
- supported target boundary: hosted `exe`, hosted checked-test targets, and
  one admitted in-project hosted `staticlib`
- disposable artifact policy: the repository `build/` tree and command-local
  `--build-dir` outputs are disposable generated state
- explicit unsupported expectations: installers, package management,
  shared-library targets, external system-library links in project manifests,
  cross-compilation, non-hosted targets, and portability claims beyond Darwin
  arm64

Validation
----------
- `cmake -S . -B build/debug -DCMAKE_BUILD_TYPE=Debug`
- `cmake --build build/debug -j4`
- `build/debug/mc --help`
- `build/debug/mc check tests/cases/hello.mc`
- `build/debug/mc run --project examples/real/issue_rollup/build.toml --build-dir build/debug/phase32_issue_rollup -- examples/real/issue_rollup/tests/sample.txt`
- `build/debug/mc test --project examples/real/issue_rollup/build.toml --build-dir build/debug/phase32_issue_rollup`

Closeout statement
------------------
The repository-bounded Phase 32 wave is complete. The repository now documents
one clear supported path from build-from-source to compiler entrypoint to
focused smoke validation on Darwin arm64, with explicit disposable-artifact
policy and unsupported expectations, without widening the admitted hosted
slice.
