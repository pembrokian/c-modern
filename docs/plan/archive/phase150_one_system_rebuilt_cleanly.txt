Phase 150 -- One System, Rebuilt Cleanly
========================================

Status
------
COMPLETE

Purpose
-------
Continue from the completed Phase 149 restart-contract step by rebuilding the
same bounded shell, log, and key-value system using only the surviving
lessons from landed Phase 146 through Phase 149 and measuring whether the
second version is simpler than the first.

This document records the landed Phase 150 slice.
It defines one bounded clean-rebuild phase that lands as a clearer system
shape over the real kernel artifact before the repository widens into broader
subsystem work.
It does not claim a new platform wave, more services, or a general framework
layer.

Phase result
------------
Phase 150 is complete as one bounded clean rebuild of the admitted
interactive-service system.

The landed repository slice now includes:

- one explicit rebuilt-system helper shape in `bootstrap_services` that owns
  the repeated shell-plus-log-plus-key-value state threading previously kept
  phase-local in `main.mc`

  it removes the hand-threaded five-state choreography from the root kernel

- one debug-owned Phase 150 audit and one focused freestanding regression over

Repository status note
----------------------
- Earlier phases in this wave extracted service shape, classified IPC and
  authority ergonomics, and formalized restart truth.
- Those results only become a convincing system-building method if the same
  small system can be rebuilt more clearly on a second pass.
- The repository therefore needed one explicit rebuild discriminator before it
  widens into filesystem, networking, scheduling richness, or any broader
  platform ambition.

Problem being solved
--------------------
- A series of locally correct patches is not the same as a repeatable system
  pattern.
- Without one clean-rebuild phase, later growth could assume the previous
  audits produced a reusable method when they may only have produced a
  successful one-off system.
- The repository therefore needed one bounded phase that removes ad hoc glue,
  rebuilds the same system using the surviving lessons, and records honestly
  whether the second version is simpler.

Core rule
---------
- Rebuild the same system, not a bigger one.
- Phase success is measured by improved clarity of ownership story, service
  shape, or restart explanation, not by feature count.
- If the rebuilt system is not simpler, record that as a model problem rather
  than hiding it behind more work.
- Debug audits, MIR projections, and tool regressions are acceptance artifacts
  only.
- Keep compiler, sema, MIR, backend, ABI, target, runtime, and `hal`
  surfaces closed unless the earlier phases already proved one concrete owner
  gap.

Planning rule for this band
---------------------------
Use this sanity check on every Phase 150 change:

- removing ad hoc structure while preserving the admitted shell, log, key-
  value, and restart behavior: admissible
- using the rebuild as an excuse to add new services or widen platform scope:
  wrong layer

Do not answer complexity by changing the problem. Keep the phase bounded to a
clean second pass over the same small system and an honest comparison against
the first version.

Chosen outcome
--------------
Chosen outcome: yes.

The landed rebuild is simpler.
The decisive simplification is that the repeated serial, shell, log, echo,
and key-value state threading no longer lives as phase-local choreography in
`kernel/src/main.mc`; it now lives behind one bounded
`bootstrap_services.Phase150RebuiltSystemState` plus one bounded
`execute_phase150_rebuilt_system_workflow(...)` helper that preserves the
existing owner boundaries and restart truth.

What this phase proved
----------------------
- the shell, log, and key-value system can be rebuilt using the surviving
  lessons from the earlier post-145 phases
- at least one earlier ad hoc pattern can be removed without losing the
  admitted bounded behavior
- the rebuilt system is simpler in a way the repository can state explicitly
- compiler, sema, MIR, backend, ABI, target, runtime, and `hal` surfaces can
  remain closed while landing that simplification

Outcome
-------
- all exit criteria above are now satisfied for the bounded Phase 150 slice
- the landed rebuild removes the earlier hand-threaded five-state shell-plus-
  log-plus-key-value choreography from `kernel/src/main.mc` and replaces it
  with one bounded rebuilt-system helper inside `kernel/src/bootstrap_services`
- the second version is simpler because it centralizes the repeated system
  state transition pattern without introducing a runtime framework, kernel
  supervisor, dynamic registration model, or compiler expansion
- the rebuilt workflow still preserves the admitted bounded system envelope:
  one explicit log append path, one explicit log tail path, one explicit
  key-value set or get path, one explicit unavailable consequence, one

  remain closed; the only supporting acceptance adjustments were the bounded
  boot log capacity increase needed to record the additional late-phase marker
  and the freestanding bootstrap pass-line shim refresh needed to recognize
