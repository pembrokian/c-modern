Phase 15 -- Post-Phase-14 Cleanup Closeout
==========================================

Status
------
COMPLETED (repository-bounded cross-phase cleanup wave)

Purpose
-------
Record the large post-Phase-14 execution wave now landed in the repository so
its implementation truth does not live only in scattered plan edits, example
changes, and focused regression additions.

This is a repository closeout note, not a new architecture phase.
It records the bounded cleanup and hardening wave that followed the first
Phase 14 closeout and the initial networking plus `sync` admissions. It does
not claim that the long-term Phase 6, Phase 7, or Phase 8 architecture work is
fully complete.

Scope of this closeout
----------------------
- This document covers the cross-phase follow-on work that the ranked backlog
  in `/Users/ro/dev/c_modern/docs/plan/admin/post_phase14_execution_plan.txt`
  had elevated ahead of any third capability-family expansion.
- It packages together repository work that belonged to different owners but
  one execution wave: current-surface proof closure, narrow `sync` follow-on
  honesty, workflow cleanup for the real networking project, compiler-debt

- It does not introduce a new language layer or replace the standing Phase 12,
  Phase 13, or Phase 14 planning documents.

Why this phase existed
----------------------
- After the first Phase 14 closeout, the repository had a real hosted
  networking family, a first narrow hosted `sync` family, and several current-
  surface candidate proofs ready to land.
- The highest-leverage next work was no longer a fresh capability-family
  admission. It was finishing the current-surface candidate set, narrowing the
  compiler limitations those admissions exposed, and bringing the support and
  planning statements back into alignment with repository truth.
- Without one closeout note, the repository risked treating this large cleanup
  wave as ambient follow-up instead of a completed execution slice with its own
  proof and documentation obligations.

What landed
-----------
- current-surface Phase 14 follow-on proofs are now complete for the named
  next-wave candidates that required no new broad module family: bounded ring
  buffer, distinct-type safety, fixed array/slice/buffer conversion, and a
  standalone hosted default-allocator proof.
- the hosted `sync` slice is now materially more honest than the first mutex-
  only admission: producer-consumer condition-variable proof landed, atomic
  publication landed, and lifecycle cleanup now includes `mutex_destroy` plus
  `condvar_destroy`.
- the atomic-publication proof no longer relies on a typed-local workaround for
  literal-valued stores; the backend now emits the admitted atomic store shape
  honestly enough for direct literal use in the canonical proof.
- the real evented-echo project no longer depends on local hosted ABI
  declarations in its main server path. Ordinary project code now uses the
  admitted imported stdlib wrappers directly, while the workflow proof stays on
  the real project graph.
- the merged project-MIR path now deduplicates imported type declarations,
  globals, and duplicate extern wrappers instead of surfacing project-shaped
  duplicate-symbol failures for already-admitted code.
- the compiler-owned typed `sync.thread_spawn<T>` rewrite remains explicit but
  is now narrower and more honest: MIR lowering checks the intended imported
  `func(*T), *T` surface and preserves the raw-helper rewrite through deferred
  calls instead of silently falling back to generic imported-call lowering.
- direct field-address lowering is now proven more honestly through executable
  coverage for both local and global aggregate fields, and MIR lowering now
  handles qualified global field address and value paths without falling back
  to invalid generic symbol references.
- atomic-order handling is now tighter at the MIR boundary and more honest in
  the executable backend for the already-admitted low-level operations:
  invalid order combinations are rejected in MIR, and the backend now accepts
  the admitted `SeqCst` and `AcqRel` cases for supported load, store,
  exchange, and fetch-add operations.
- borrowed-text cleanup is now active through `strings.bytes(...)`, and the
  covered hashing, grep-lite, file-walker, and evented-echo paths no longer
  spell ad hoc `str` to `Slice<u8>` conversions by hand.
- support notes and planning docs no longer describe the evented-echo project
  workaround or the earlier atomic literal limitation as if they were still the
  current repository truth.
