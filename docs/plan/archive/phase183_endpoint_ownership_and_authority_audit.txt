Phase 183 -- Endpoint Ownership And Authority Audit
==================================================

Status
------
COMPLETE

Outcome
-------
Phase 183 landed as one explicit ownership classification of boot-wired
endpoint ids, one bounded follow-through in the dispatcher, and one
dedicated behavior-first test fixture.

The classification established in code:

- Ownership: `service_topology.mc` is the sole definer of endpoint id
  constants.  No other module introduces new endpoint id values.
- Routing: `kernel_dispatch.mc` routes on endpoint ids.  `kernel_dispatch_step`
  now has an explicit `endpoint_is_boot_wired` guard before per-service
  routing.  Unknown endpoint ids return `InvalidEndpoint` at this gate
  rather than falling through the per-service chain implicitly.
- Public name vs authority: endpoint ids are stable public names that
  double as routing keys.  Knowing a boot-wired endpoint id is sufficient
  to address that service.  No capability check is performed.
- Authority surface: handle transfer lives in
  `syscall.ReceiveObservation.received_handle_slot` and
  `received_handle_count`.  That is the intended future authority gate,
  separate and distinct from routing by name.
- Conclusion: endpoint ids are stable public names, not elevated
  authority-bearing values.  The distinction from capability-based authority
  is now stated in code comments and enforced by the explicit gate.

The bounded follow-through:

- `service_topology.endpoint_is_boot_wired(endpoint: u32) bool` added as
  the explicit public-name gate.  All five wired endpoints return true;
  all other values return false.
- `kernel_dispatch_step` now calls `endpoint_is_boot_wired` before routing,
  replacing the implicit fall-through to `InvalidEndpoint` in the service
  chain with one explicit owner-local decision at the gateway.

What did not change:

- no capability model, endpoint allocator, naming service, or compiler-known
  endpoint type
- no dynamic topology, endpoint rebinding, or service registry
- no compiler, sema, MIR, backend, ABI, target, runtime, or `hal` expansion

Phase result
------------
The repository now has an explicit answer to whether endpoint ids are
names, routing keys, authorities, or a split contract:

  Endpoint ids are stable public names and routing keys.
  Knowing one is sufficient to address the service.
  They are NOT capability authority tokens.
  The authority surface for IPC is the (currently reserved) handle-slot
  transfer mechanism in `syscall.ReceiveObservation`.

Phase 184 (capability propagation and leakage) can now rely on this explicit
contract rather than inferring it from the implicit fall-through behavior.

Validation
----------
- `tests/system/kernel_reset_lane_phase183_endpoint_authority`

Purpose
-------
Clarify what endpoint ids mean in the current running system and what kind of
authority, if any, they carry.

The repository now has static boot-wired endpoint ids, ServiceRef wrappers,
handle-based authority transfer, explicit restart, and growing retained-state
pressure.  That is enough to ask a sharper question:

are endpoint ids names, routing keys, authorities, or some split combination
of those roles?

Repository status note
----------------------
- `service_topology.mc` owns fixed endpoint ids and flat lookup helpers.
- ServiceRef names boot-wired services through endpoint ids.
- Capability and handle transfer already exist as explicit authority-bearing
  surfaces.
- Restart keeps endpoint ids stable today.
- The repository still has no dynamic endpoint allocation, discovery, or
  service registry.

Problem being solved
--------------------
- Endpoint ids currently serve multiple practical roles at once:
  topology constants, routing selectors, and public service names.
- At the same time, handles and capabilities remain the intended explicit
  authority surfaces.
- Without one explicit audit, later work on restart, stale authority, service
  graph growth, or dynamic topology pressure could blur naming and authority
  into one accidental model.

Core rule
---------
- Keep naming, routing, and authority distinct unless current code proves they
  are intentionally the same.
- Work from the admitted boot-wired system, not from hypothetical dynamic
  endpoint allocation.
- Do not introduce a new capability model, endpoint allocator, naming service,
  or compiler-known endpoint type.
- Do not widen compiler, sema, MIR, backend, ABI, target, runtime, or `hal`
  surfaces.

Chosen slice for this phase
---------------------------
1. one explicit ownership classification of endpoint ids across the current
   system: who defines them, who routes on them, who may speak about them, and
   whether knowing one conveys authority
2. one comparison between ServiceRef, endpoint id, and handle-carried
   authority under the current restart and topology rules
3. one bounded follow-through only if the current source exposes a real owner
   blur, such as moving one authority check or tightening one endpoint-facing
   helper boundary
4. one explicit statement of whether endpoint ids are currently best read as
   stable public names, stable routing keys, authority-bearing values, or a
   split contract with named authority elsewhere

What this phase should not do
-----------------------------
- no dynamic topology work
- no endpoint rebinding mechanism
- no service registry or namespace framework
- no new capability algebra
- no compiler-facing endpoint feature

Why this is the right placement
-------------------------------
- After Phase 182: restart identity continuity should already be named.
- Before capability leakage and IPC restart-pressure audits: endpoint meaning
  needs to be explicit.
- This keeps the work inside the current service-system pressure rather than
  widening into subsystem growth.

Planning rule for this phase
----------------------------
- Prefer one clear classification over a new mechanism.
- If a repair is needed, keep it owner-local and flat.
- Preserve the current Ken-Thompson-simple shape: explicit constants,
  explicit checks, explicit routing.

What this phase should prove
----------------------------
- that the current endpoint model is either honest enough to carry more growth
  or exposes one specifically named blur between naming and authority
- that later retained-state, restart, and IPC phases can rely on a sharper
  endpoint contract
- that no larger topology or capability subsystem is needed by default
