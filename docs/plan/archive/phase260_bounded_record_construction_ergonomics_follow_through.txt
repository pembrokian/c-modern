Phase 260 -- Bounded Record-Construction Ergonomics Follow-Through
==================================================================

Status
------
COMPLETED

Purpose
-------
Admit one narrow source-level record-construction ergonomics slice that removes
repeated repository-owned boilerplate in aggregate literals or record updates
without widening into builder frameworks, hidden mutation, or general object-
construction redesign.

Implementation outcome
----------------------
- The landed shorthand is `field:` meaning `field: field` inside existing named
  aggregate field lists and existing `with { ... }` record-update lists.
- The shorthand is intentionally limited to simple field names; omitted values
  on dotted record-update paths such as `outer.inner:` remain rejected.
- Parser ownership stays local to field-init parsing, sema keeps field binding
  and destination-type rules, and MIR still lowers through aggregate rebuild.
- The real spend sites are `kernel/src/boot/service_cell_helpers.mc` and
  `kernel/src/boot/boot_update.mc`.

This phase exists now because the repository already admitted inferred brace
aggregates plus shallow and nested record updates, but real owner code such as
`kernel/src/boot/boot_update.mc` and related boot-state files still carries a
large amount of repeated field-name boilerplate that has become more visible as
the compiler fronts above it have closed.

Repository status note
----------------------
- Phase 191 admitted explicit typed aggregate tables.
- Phase 192 admitted inferred brace aggregates in expected-type contexts.
- Phase 195 admitted shallow `with { field: value }` record updates.
- Phase 199 admitted nested record-update paths like `base with { outer.inner:
  value }`.
- The decision log after Phase 182 explicitly recorded remaining repeated
  record-update pressure as a possible later language follow-through once the
  stronger generic-helper boundary was settled.

Problem being solved
--------------------
- Repository-owned record literals and record updates still repeat obvious
  field names and wrapper shapes in places where the destination type is
  already explicit and the source intent is mechanically clear.
- That repetition is no longer just aesthetic noise. It obscures the real
  state change in files such as `kernel/src/boot/boot_update.mc` and makes the
  compiler look less expressive than the surrounding admitted aggregate and
  update surfaces suggest.
- The current-repo problem is therefore one bounded ergonomic gap in record
  construction, not a need for builders, mutation frameworks, or broader object
  semantics.

Non-goals
---------
- no builder or object-construction framework
- no hidden mutation or in-place update semantics
- no dynamic field iteration or reflection
- no array or slice element update widening
- no broad record-pattern or destructuring redesign
- no general aggregate inference beyond owners that already carry an explicit
  destination type

Invariants
----------
- Aggregate and record meaning remain explicit and sema-owned.
- MIR continues to lower admitted record construction by rebuilding aggregates,
  not by inventing mutation instructions.
- Any new shorthand must require either an explicit destination type or an
  already-admitted expected-type owner.
- The phase must remove real repository-owned boilerplate at one or more spend
  sites.
- Existing record-update determinism and diagnostics remain stable.

Required surface changes
------------------------
1. Choose one narrow source-level shorthand for repeated record-construction or
   record-update boilerplate in already-admitted typed contexts.
2. Keep the shorthand bounded to owners where destination type and field
   identity remain explicit and reviewable.
3. Add focused parser, sema, and MIR regressions for the admitted shorthand.
4. Spend the feature in one or more real repository-owned source sites such as
   `kernel/src/boot/boot_update.mc`.
5. Preserve explicit rejection for broader builder-like or hidden-mutation
   interpretations.

Forbidden changes
-----------------
- do not widen into a record-builder DSL or helper framework
- do not introduce hidden mutation, partial object initialization, or default-
  filling semantics that are not already explicit
- do not reopen unrelated aggregate or collection inference work
- do not treat one shorthand as permission for pattern matching or reflection
- do not skip parser or sema diagnostics if the chosen shorthand introduces a
  new syntactic form

Phase role in the larger sequence
---------------------------------
- This phase converts one explicitly recorded source-noise frontier into a real
  language improvement after the stronger generic-helper phases have landed.
- It prepares Band B and later app-local code by making state and descriptor
  records cheaper to express without creating a new center of gravity.
- It should still leave broader source-level construction work explicitly
  deferred.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- repeated record-update pressure across owners may justify a narrow follow-
  through slice once stronger compiler frontiers are closed

Questions this phase must answer
--------------------------------
1. What is the narrowest honest shorthand for current-repo pressure: field-name
   shorthand inside record literals, field-name shorthand inside `with` blocks,
   or another similarly explicit form?
2. Which owner rejects the shorthand first: parser syntax, sema field binding,
   or MIR aggregate rebuild lowering?
3. Which repository-owned spend sites demonstrate the gain most clearly without
   inventing synthetic examples?
4. How can the phase keep builder-like or hidden-mutation interpretations
   explicitly closed?

Preferred implementation shape
------------------------------
- Prefer a small shorthand that reuses the existing aggregate-init or record-
  update model rather than inventing a second construction system.
- Keep parser ownership limited to the new source form if one is required.
- Keep sema ownership explicit for field resolution and destination-type rules.
- Keep MIR lowering on the existing aggregate rebuild path.
- Spend the feature immediately in `kernel/src/boot/boot_update.mc` and any
  equally strong adjacent owner sites.
