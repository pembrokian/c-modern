Phase 49 -- Imported Generic Call Closure
========================================

Status
------
IMPLEMENTED

Purpose
-------
Remove the remaining compiler-owned imported generic procedure-call rewrite on
the admitted `sync.thread_spawn<T>` surface, while keeping the closure bounded
to the erased-ABI generic call family the current admitted slice actually
needs.

This document is a repository execution plan, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- Phase 48 identified the live `sync.thread_spawn<T>` to
  `sync.thread_spawn_raw` MIR rewrite as the strongest remaining distortion on
  the admitted slice.
- Bootstrap sema already instantiates imported generic function signatures at
  the call site, and MIR already has enough direct-call metadata to represent
  ordinary imported calls.
- The compiler-core closure landed first: MIR validation and LLVM emission now
  accept the narrow erased-ABI imported generic function family needed by the
  admitted sync surface.
- The remaining supported-slice distortion was the library boundary itself:
  `stdlib/sync.mc` still exported `thread_spawn_raw` and implemented the public
  `thread_spawn<T>` API through that raw helper, so ordinary executable and
  project proofs did not yet travel through the real imported generic function
  body.
- The repository still does not claim broad generic-function completion, full
  monomorphization, or arbitrary imported generic executable support.

Problem being solved
--------------------
- The compiler-owned MIR rewrite is gone, but the admitted `sync.thread_spawn<T>`
  API still reached the runtime through a public raw helper inside
  `stdlib/sync.mc` rather than through the generic function body itself.
- That meant the supported executable and project proofs still did not confirm
  that the erased-ABI imported generic function family survived the real
  library boundary end-to-end.
- As long as `thread_spawn_raw` remained part of the supported sync slice, the
  repository still carried a sharper consumer-shaped escape hatch than the
  compiler and backend now required.

Core rule
---------
- Remove the admitted sync rewrite at the owning compiler boundary instead of
  adding another consumer-local shim.
- Widen only the erased-ABI imported generic function family needed by the
  admitted sync surface.
- Keep sema as the owner of instantiated call signatures, MIR as the owner of
  direct-call structure, and backend emission as the owner of erased executable
  representation.
- Do not claim generic monomorphization, generic interface artifacts with
  bodies, or arbitrary imported generic executable support beyond the erased
  family proven here.

Chosen slice for this phase
---------------------------
Targeted proof shape: the existing admitted `sync.thread_spawn<T>` project and
fixture surface works end-to-end as an ordinary imported direct call, without a
MIR rewrite to `sync.thread_spawn_raw`, because validation and LLVM emission now
honestly support the erased-ABI generic function family that surface uses.

What this phase should try to prove
-----------------------------------
- MIR can represent imported `sync.thread_spawn<T>` calls and deferred calls as
  ordinary direct function calls again.
- MIR validation accepts the instantiated generic direct-call metadata instead
  of requiring it to look like a non-generic callee definition.
- LLVM executable emission can declare and define the erased-ABI generic
  function symbols needed by the admitted sync surface.
- The admitted project proof still builds and runs without reintroducing a
  consumer-specific workaround.
- Generic-function breadth outside this erased family remains explicitly
  deferred.

Preferred pressure points
-------------------------
1. Imported direct call lowering for `sync.thread_spawn<T>`.
2. Deferred-call lowering for `sync.thread_spawn<T>`.
3. MIR validation of instantiated generic direct-call metadata.
4. LLVM declaration and definition emission for erased-ABI generic functions
   whose signatures and bodies remain representable without monomorphization.
5. Existing admitted `worker_queue` project proof through normal project build
   and run coverage.

Deliverables
------------
- removal of the public `sync.thread_spawn_raw` helper from the admitted sync
  slice so `thread_spawn<T>` becomes the real library implementation boundary
- focused MIR expectation updates that keep the supported imported surface on
  the direct generic call path only
- focused executable-build proof that checks the emitted LLVM IR for the real
  `sync.thread_spawn` definition and call shape while preserving the admitted
  worker-queue project proof
- planning text that records the closure as an honest library-and-executable
  follow-through rather than a still-pending compiler rewrite

Suggested execution order
-------------------------
1. Remove the public `thread_spawn_raw` export and helper body from
  `stdlib/sync.mc` so `thread_spawn<T>` performs the runtime call directly.
2. Refresh focused MIR expectations so the supported imported-module fixtures
  only model the direct generic surface.
3. Add one executable-build proof that checks the emitted LLVM IR for the real
  `sync.thread_spawn` definition and direct-call sites.
4. Re-run the admitted worker-queue project proof to keep the project workflow
  honest after the library-surface cleanup.
5. Record the closed raw-helper escape hatch and keep broader generic-function
  support deferred with a sharper boundary.

Exit criteria
-------------
- the admitted `sync.thread_spawn<T>` surface no longer depends on a compiler-
  owned rewrite to `sync.thread_spawn_raw`
- MIR and LLVM now honestly support the erased-ABI imported generic function
  family required by that surface
- existing admitted project proof survives through normal build and execution
  coverage
- broader generic-function completion remains explicitly deferred rather than
  implied by this narrower closure

Non-goals
---------
- general generic-function monomorphization
- imported generic executable support for signatures whose erased ABI still
  depends on unresolved type parameters
- generic interface artifacts carrying bodies
- reopening aggregate-constant backend follow-through, `time`, or broader
  linker work in the same phase

Acceptance criteria for this document
-------------------------------------
- Phase 49 stands alone as the bounded owner doc for removing the admitted sync
  rewrite.
- The phase is explicit that the implemented closure is erased-ABI generic call
  support, not broad generic-function completion.

What landed
-----------
- removed the public `sync.thread_spawn_raw` export and helper body from the
  admitted sync slice so `thread_spawn<T>` now performs the runtime call inside
  the generic function body itself
- preserved the existing MIR direct-call closure and tightened the imported-
  module MIR tests so the supported surface no longer models the obsolete raw
  helper as part of the sync boundary
- added executable-build proof on the canonical shared-counter sync example,
  checking the emitted LLVM IR for both the `sync.thread_spawn` definition and
  the direct call sites that target it
- preserved the existing admitted project proof path so the `worker_queue`
  project continues to build and run through normal tool coverage with the raw
  helper removed from the public slice
- kept broader generic-function executable support deferred for functions whose
  ABI or lowered body still depends on unresolved type parameters

Closeout statement
------------------
Phase 49 is complete. The admitted `sync.thread_spawn<T>` surface now travels
through sema-instantiated direct-call truth, ordinary MIR call lowering, the
real generic stdlib function body, and erased-ABI LLVM emission without the old
`sync.thread_spawn_raw` escape hatch. The repository closes the admitted
distortion at its owner boundary while keeping the broader generic frontier
explicit: this phase does not claim general generic-function completion, only
the narrower erased-ABI imported generic call family the supported sync slice
required.
