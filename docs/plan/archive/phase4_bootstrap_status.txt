Bootstrap Phase 4 Status

This repository now records a narrower bootstrap Phase 4 completion note in `docs/plan/phase4_bootstrap_finish.txt`, but it should not claim that the full Phase 4 from plan.txt is finished.

What is in place:
- typed MIR module, function, block, instruction, and terminator structures exist
- MIR lowering runs after bootstrap semantic checking in `mc check`
- MIR dumps are deterministic and covered by fixture tests under `tests/mir/`
- MIR validation checks duplicate functions, duplicate blocks, undefined operands, duplicate values, reachable block termination, branch target validity, and typed legality for locals, calls, unary/binary ops, explicit bounds checks, conversion op selection, field/index/slice base and bound rules, aggregate initializers, variant match/extract ops, return values, branch conditions, and structured field/index store targets
- MIR validation now also enforces structural legality for emitted value/effect instructions, dedicated conversion-family metadata, dedicated volatile/atomic opcode metadata, stricter symbol/store target metadata, and block/terminator shape invariants even for unreachable dead blocks
- MIR validation also checks symbol-ref type consistency where matching MIR-local declarations exist without rejecting import-backed symbol refs that are semantically known but not materialized as local MIR declarations
- bootstrap MIR lowering covers core control flow, locals, calls, aggregates, range loops, foreach loops, defer unwinding, expression switch cases, and variant switch patterns
- recent bootstrap tightening carries semantic types through indexed slice elements, known global/function symbol refs, sema-typed deferred and statement-only sink callees, variant payload extraction, synthetic range/foreach loop state, and ordinary local/expression lowering instead of leaving those MIR values as `unknown`
- MIR dumps now preserve enum variant payload metadata and structured symbol/store-target metadata so validator checks do not need to recover those facts from plain rendered strings alone
- bootstrap MIR now splits explicit conversions into dedicated MIR forms for distinct/base conversions, numeric conversions, array-to-slice conversions, and buffer-to-slice conversions instead of routing those supported cases through one generic conversion instruction
- direct, foreach-synthesized, and indexed-store bounds-sensitive paths now lower through an explicit `bounds_check` MIR instruction before the value-producing `index` or `slice` operation or accompanying indexed store effect, so checked-bounds semantics are no longer implicit backend policy for those supported paths
- integer division, remainder, and shift expressions now lower through explicit `div_check` and `shift_check` MIR instructions before the corresponding value-producing binary operation, so those checked runtime faults are no longer implicit backend policy on supported paths
- concrete integer `+`, `-`, and `*` operations now record explicit wraparound semantics in MIR binary metadata instead of leaving overflow behavior implicit
- the library-shaped `mmio_ptr` boundary now lowers through the dedicated `int_to_pointer` MIR instruction instead of generic call shaping, matching the existing explicit pointer/integer conversion family
- library-shaped volatile and atomic operations now lower through dedicated MIR instructions for `volatile_load`, `volatile_store`, `atomic_load`, `atomic_store`, `atomic_exchange`, `atomic_compare_exchange`, and `atomic_fetch_add` instead of falling back to generic call shaping
- ordinary AST expression result types are now recorded by sema and consumed by MIR lowering for literals, unary/binary expressions, calls, field/index/slice expressions, and aggregate initializers instead of being re-derived locally in MIR
- MIR instructions now carry structured target metadata for direct calls, field/index/slice accesses, store targets, and variant operations, and the validator uses that metadata directly instead of re-deriving identities from rendered display text where the structured facts are available
- MIR unit coverage now includes focused validator failures for non-bool conditions, bad unary legality, incompatible comparison operands, malformed bounds checks, missing fault checks on division/shift binaries, missing or invalid wraparound arithmetic metadata, bad volatile or atomic instruction shapes, misencoded generic conversions, duplicate aggregate fields, invalid field/index/slice bases and slice bounds, bad call argument types, aggregate field mismatches, variant payload mismatches, bad symbol-ref types, and structured store-target mismatches
- MIR fixtures now cover semantic-boundary intrinsic lowering for volatile and atomic operations in addition to the earlier bounds-check and conversion-focused fixtures
- representative supported MIR lowering tests now assert that variant switches, foreach/range loops, and typed symbol-reference paths do not emit `unknown`, and a dump audit across current accepted sample inputs finds no emitted `unknown` MIR types
- MIR fixtures now include representative canonical-style programs covering explicit conversions, checked faults, aggregates, slices, loops, variants, and low-level memory operations instead of only narrow unit-style cases
- MIR fixture coverage now includes lowering-failure diagnostics for unsupported bootstrap boundaries such as non-iterable foreach loops and variant payload bindings that would shadow existing locals

What still blocks a Phase 4 completion claim:
- MIR is materially tighter across the currently supported surface, but unsupported or not-yet-semanticized paths can still hit lowering diagnostics and recovery behavior rather than a fully explicit typed MIR form
- the validator now covers a broader typed subset including explicit conversion-family selection, core unary/comparison legality, field/index/slice base-and-bound checks, and aggregate field-shape checks, but it still does not comprehensively reject illegal typed operations across the full MIR surface
- explicit MIR forms now cover distinct/base, numeric, pointer-int, array-to-slice, and buffer-to-slice conversions, library-shaped `mmio_ptr` pointer formation, checked divide/shift faults, and the current library-shaped volatile/atomic operation subset; other semantic-boundary operations are still pending

Current structural debt worth tracking explicitly:
- some MIR lookup and naming paths still rely on rendered expression strings instead of structured semantic identities; this is acceptable for bootstrap dumps but fragile as a long-term lookup mechanism
- local typing no longer intentionally carries `unknown` through supported lowering paths, but the remaining recovery behavior is still bootstrap-oriented rather than a final long-term contract

Practical bootstrap interpretation:
- the repository has closed the narrower bootstrap Phase 4 milestone recorded in `docs/plan/phase4_bootstrap_finish.txt`
- the current MIR layer is strong enough to support continued bootstrap bring-up
- the current tree is green under the canonical local build and test flow
- the repository should continue distinguishing the narrower bootstrap milestone from full Phase 4 completion in `plan.txt`

2. keep removing the remaining bootstrap recovery paths so unsupported lowering fails crisply instead of relying on generic fallback shaping
3. keep expanding MIR validator checks from the current typed subset into fuller instruction-by-instruction legality coverage, especially generic convert, aggregate edge cases, and any future explicit fault/volatile/atomic instructions
4. add MIR fixtures for representative canonical small programs, not just targeted unit-style cases
5. reduce dependence on rendered-expression lookup paths by threading more structured semantic identity into MIR
6. only after those steps, consider writing a repository-specific `phase4_bootstrap_finish.txt` analogous to the Phase 3 bootstrap milestone note
