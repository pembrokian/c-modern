Purpose
-------
Execute the next bring-up phase by turning the repository from a compiler binary with a narrow build command into a usable local toolchain with:

- a declarative build.toml project model
- explicit target selection and build-mode selection
- deterministic separate compilation through .mci interface artifacts
- one ordinary mc run flow for executable targets
- one ordinary mc test flow for project tests and compiler regression classes
- a simple, inspectable build graph whose rebuild behavior follows interface hashes rather than reparsing the whole source tree every time

Problem being solved
--------------------
- The repository can now parse, type-check, lower, emit LLVM IR, link, and run a bounded executable slice, and it has a minimal hosted stdlib/runtime boundary.
- It still lacks the ordinary developer workflow boundary described in plan.txt and the build/test specs.
- Without a bounded Phase 7 closure pass:
  - mc build remains a single-source bootstrap command rather than a project tool
  - developers cannot describe targets, search paths, build modes, test roots, runtime selection, or link libraries declaratively
  - import resolution can drift between direct compiler invocations, future project builds, and future .mci consumption because no single Phase 7 import policy has been frozen
  - downstream modules cannot consume stable interface artifacts, so every build still trends toward whole-source recompilation
  - test discovery and runner generation remain architecture text only, leaving no single tool entrypoint for project tests and compiler regression classes
  - CTest stays the only orchestration layer, which is sufficient for repository bring-up but not sufficient evidence for the language-owned tool surface

Core rule
---------
- The compiler owns parsing, sema, MIR, codegen, and interface-artifact truth.
- The build tool owns target description, dependency ordering, rebuild decisions, runner generation, and process orchestration.
- Phase 7 may add project configuration, dependency scanning, .mci I/O, and generated test runners, but it must not hide imports, hide package resolution, or create a second module system distinct from the language import rules.

Import design decisions and trade-offs that require explicit closure
-------------------------------------------------------------------
Search-path precedence:

- Decision: choose one order among command-line overrides, build.toml target search_paths.modules, repository-local stdlib discovery, and any fallback source-relative root.
- Recommendation: CLI overrides first, build.toml target roots second, repository stdlib discovery last as a bootstrap fallback for direct-source commands only.
- Trade-off: preserving automatic stdlib discovery everywhere is convenient for the repository, but if it outranks target-declared roots it will make project builds less explicit and harder to reason about.

Module identity versus file-path lookup:

- Decision: freeze whether an import name resolves by searching ordered roots for module.mc only, or whether additional package-like path rewriting is allowed.
- Recommendation: keep the current simple ordered-root lookup model with no package aliases, no hidden module renaming, and crisp duplicate-match diagnostics.
- Trade-off: ordered search roots are simple and fit the repository, but they require an explicit ambiguity policy when two roots export the same module name.

Direct-source commands versus project commands:

- Decision: decide whether mc check and mc build continue to accept raw source paths alongside build.toml targets.
- Recommendation: keep both. Direct-source mode remains the bootstrap and fixture path; target mode becomes the canonical project workflow.
- Trade-off: dual entrypoints add some driver complexity, but removing direct-source mode would force repository fixtures, focused diagnostics, and narrow bring-up tests through unnecessary project scaffolding.

.mci lookup model:

- Decision: decide whether dependent compilation should consult source files, .mci files, or both when resolving imports inside a target build.
- Recommendation: use source files for graph discovery and owning compilation units, then use emitted .mci files for downstream semantic import consumption within the build graph.
- Trade-off: reading only .mci files improves separate compilation discipline, but graph discovery still needs a clear source-root walk unless build.toml enumerates every module.

Vendored dependency roots:

- Decision: decide whether vendor/ is purely conventional or implicitly searched.
- Recommendation: keep vendor/ conventional only; require it to be named in search_paths.modules.
- Trade-off: implicit vendor support is convenient, but it becomes hidden package behavior and conflicts with the explicitness bias in build_spec.txt.

Compiler regression tests under mc test:

- Decision: decide how much of the existing repository CTest fixture system should also be surfaced through mc test in Phase 7.
- Recommendation: Phase 7 should add a narrow compiler-regression manifest path under tests/tool/ that lets mc test drive compile-pass, compile-fail, run-output, and MIR-oriented classes without trying to replace all C++ fixture tests immediately.
- Trade-off: folding every existing C++ test into mc test would over-expand the phase; keeping mc test limited to ordinary project tests only would under-deliver on the plan.txt exit criteria.
