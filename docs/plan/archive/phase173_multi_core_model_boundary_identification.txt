Phase 173 -- Multi-Core Model Boundary Identification
======================================================

Status
------
COMPLETE

Purpose
-------
Name the first repeated constraint that the current kernel model violates
under multi-core assumptions, without implementing any SMP change.

This is a Phase-160-style identification phase for the SMP boundary.
The repository does not claim multicore support.  What this phase does is
make that deferral concrete: instead of "SMP is future work", the repository
should be able to say "these specific model assumptions break under SMP, and
here is why the current model does not need to address them yet."

Repository status note
----------------------
- The current kernel is single-core, table-driven, and fixed-size.
- All critical sections are currently expressed as single-core critical
  sections (interrupt masking or cooperative model).
- The end-state kernel architecture doc anticipates eventual SMP support but
  treats it as out of scope for the current slice.
- Common SMP failure modes for microkernel models include: endpoint-table
  mutation races, capability-handle-table ownership under concurrent send and
  receive, scheduler ready-queue races under concurrent wake, and process-slot
  lifecycle races under concurrent spawn.
- This phase surveys those candidates against the current code and names the
  one or two that would be hardest to retrofit.

Problem being solved
--------------------
- Without naming the SMP constraints explicitly, later kernel growth can
  inadvertently make SMP harder (e.g., by widening single-core assumptions into
  more places) or easier (e.g., by separating ownership in ways that compose
  cleanly with per-core data structures).
- Naming the constraints now preserves optionality without committing to an
  SMP implementation timeline.
- It also gives the supervisor and topology phases (172, 170) a boundary to
  reason against.

Core rule
---------
- Name the SMP constraints from current code, not from architecture theory.
- Do not implement any SMP change.
- Do not add locks, barriers, or per-core data structures.
- If the current model already has clean SMP-compatible ownership (e.g.,
  through per-process table ownership), note that explicitly.
- This is a documentation and classification phase, not an implementation phase.

Chosen slice for this phase
---------------------------
1. one survey of the current kernel source for single-core assumptions
2. one classification of the two or three model constraints that would be
   hardest to retrofit for SMP
3. one explicit statement of why those constraints do not currently block
   single-core work
4. one trigger condition for each constraint (e.g., "reopen this constraint
   when the scheduler ready queue needs concurrent wake from two cores")

Why this is the right placement
--------------------------------
- After Phase 172: supervisor and restart shape are understood, so the SMP
  survey can reason about process slot and lifecycle ownership concretely.
- Before any broader platform portability work (Phase 174): platform targets
  often interact with SMP assumptions, so naming SMP constraints first creates
  a clearer portability audit.

Planning rule for this phase
------------------------------
- Work from current source, not from architecture diagrams.
- Name only constraints that are real today, not hypothetical future ones.

What this phase should prove
-----------------------------
- that the repository's SMP deferral is explicit and reasoned, not silent
- that the current kernel structure is understood well enough to name the
  points of SMP fragility
- that future kernel growth is aware of those constraints even while
  single-core work continues

Survey results
--------------
Source read: kernel/src/kernel_dispatch.mc, boot.mc, init.mc, service_topology.mc,
kv_service.mc, log_service.mc, syscall.mc, scenarios.mc.

Single-core assumptions found in current source:

  1. kernel_dispatch_step(state: *boot.KernelBootState, ...)
     Takes a single mutable pointer to the entire kernel state.  Every
     dispatch — leaf, kv, effect-chain — reads *state, calls a service,
     and writes a new state back through the same pointer.  The
     read-modify-write is not atomic.  Under SMP, two cores calling
     kernel_dispatch_step with the same state pointer concurrently race
     on the write-back.

  2. kernel_dispatch_kv compound mutation
     kernel_dispatch_kv reads current.log_state, dispatches kv, dispatches
     the advisory log append, then commits both updates to *state as a
     single compound assignment.  The two dispatches are not fused into one
     atomic operation.  Under SMP, another core can observe the intermediate
     state between the kv commit and the advisory log commit.

  3. execute_effect depth guard
     MAX_EFFECT_CHAIN_DEPTH is passed as a stack-local value parameter.
     This is already SMP-safe: each call chain carries its own depth
     counter independently.  No constraint here.

  4. ServiceSlot constants and endpoint IDs
     All slots and endpoint IDs in service_topology.mc are compile-time
     constants (u32 values 10–13).  Immutable.  SMP-safe by construction.

  5. Service handler functions (kv_service.handle, log_service.handle)
     Each service takes its state by value and returns a new state value.
     Pure functions with no global side effects.  These already compose
     correctly with per-core state shards.  No SMP constraint here.

Named constraints
-----------------
Two constraints would be hardest to retrofit for SMP.

  Constraint A: Unified dispatch state pointer
  Location: kernel_dispatch.mc — kernel_dispatch_step, kernel_dispatch_leaf,
            kernel_dispatch_kv, execute_effect
  Shape: KernelBootState is a single struct containing all per-service
         retained states.  The dispatcher takes *KernelBootState and all
         paths write back through that same pointer.  There is no per-service
         lock, no per-service shard, and no version guard.
  Difficulty to retrofit: high.  Fixing this requires either a global
         serialization lock (reducing SMP to a single dispatch lane) or
         partitioning KernelBootState into per-service pointers with
         per-service ownership (structural redesign of the dispatch call site).
  Why it does not block single-core: single-core sequential dispatch is
         exactly the assumption the current model encodes.  All 22 steps in
         the scenario verify sequential composition over a shared *state
         without races.
  Trigger: reopen when a phase requires concurrent interrupt-driven dispatch
         from multiple hardware cores, or when the scheduler ready queue
         needs concurrent wake from two cores over different service shards.

  Constraint B: Advisory-first compound dispatch atomicity
  Location: kernel_dispatch.mc — kernel_dispatch_kv
  Shape: kernel_dispatch_kv executes two dependent dispatches (kv write
         then advisory log append) and commits both outcomes to *state in
         one compound assignment.  The compound is not atomic.  Under SMP
         a second core observing *state between the two sub-dispatches
         would see kv state updated without the corresponding log update.
  Difficulty to retrofit: medium.  Could be closed by a per-kv-dispatch
         lock or by making the advisory append the kv caller's explicit
         responsibility (eliminating the compound).
  Why it does not block single-core: advisory log silencing is already a
         named model gap (Phase 160); the compound is consistent from the
         single-core caller's perspective because no other dispatch can
         interleave between the two sub-dispatches on one core.
  Trigger: reopen when a phase introduces concurrent log writes from two
         sources that share the same advisory commit window, or when the
         advisory gap is closed and a stronger cross-service atomicity
         guarantee is required.

Clean pattern (SMP-compatible)
-------------------------------
The per-service handler functions (kv_service.handle, log_service.handle,
etc.) take state by value and return a new state value.  They have no
global side effects.  If the state pointer is ever partitioned into per-
service pointers, the handler functions require no change.  This is the
future clean path if SMP adoption ever becomes a real phase.
