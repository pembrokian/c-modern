Phase 194 -- Narrow Explicit Integer Conversion For Compact Protocol Packing
==========================================================================

Status
------
COMPLETE

Purpose
-------
Admit one narrow explicit integer-conversion surface so admitted low-level
owners can state intentional byte or word truncation directly in source
without reopening implicit numeric coercions, broader arithmetic widening, or
a general cast framework.

This phase should answer one recurring compiler pressure point that has now
appeared in more than one real kernel follow-through.
Compact shell and service protocols naturally carry byte-sized payloads, but
the underlying state they need to expose or compare often lives in wider
integer fields.
The language currently requires compatible operand widths and has no explicit
conversion form that says "truncate this wider integer to the target width on
purpose".

Repository status note
----------------------
- The repository already has multiple honest compact protocol owners whose
  payloads are byte-sized by design.
- Phase 188 used a bounded ticket protocol where stale-state truth and
  compact payload shape had to stay explicit without reopening transport or
  compiler surfaces.
- Phase 193 hit the same underlying limit again when the shell-visible
  generation query needed four fixed bytes but service-instance generation
  naturally lived as a wider numeric counter.
- The landed Phase 193 implementation therefore kept an owner-local tracked
  four-byte generation payload beside the numeric generation counter in
  `kernel/src/service_identity.mc` and `kernel/src/boot.mc` instead of
  widening compiler behavior.

Implemented result
------------------
Phase 194 lands one explicit builtin-integer target conversion form:

1. `u8(expr)` and the analogous builtin scalar integer targets now parse as a
  dedicated explicit conversion expression rather than an ordinary call
2. the admitted bare syntax is limited to builtin scalar integer target names
3. sema requires the source expression of that bare form to be integer-typed
  and preserves the rule that ordinary mixed-width arithmetic does not gain
  implicit narrowing
4. MIR and LLVM reuse the existing explicit numeric conversion lowering path
  so the feature remains explicit end-to-end rather than backend-only
5. the feature is spent immediately in `kernel/src/service_identity.mc` to
  pack restart-generation reply bytes directly from the numeric generation
  counter, removing the owner-local tracked four-byte shadow state from
  `ServiceMark` and `ServiceCell<T>`

The older parenthesized explicit conversion surface remains in place for the
broader preexisting explicit-conversion families already admitted elsewhere in
the compiler. Phase 194 does not widen the new bare syntax to those broader
families.

Core rule
---------
- Keep any admitted conversion explicit at the source site.
- Require the destination width or destination integer type to be named by
  the programmer.
- Preserve the current rule that the language does not perform implicit
  numeric narrowing across mixed-width operations.
- Prefer one conversion expression form over a larger arithmetic redesign.
- Do not let this phase become a general cast framework, pointer-cast system,
  or hidden coercion policy.

Chosen conversion shape
-----------------------
The landed shape is:

1. admit one explicit integer-conversion expression that names the target
   integer type directly at the conversion site
2. support only integer-to-integer width changes in the first slice
3. allow explicit narrowing and widening between the existing scalar integer
   types only
4. keep overflow or truncation semantics explicit and deterministic rather
   than implicit in ordinary arithmetic or comparison
5. spend the feature immediately on one real compact-packing owner instead of
   landing it as unused language surface

Landed syntax direction
-----------------------
The landed explicit target-typed conversion form is:

1. `u8(expr)` for integer-to-integer conversion where the target type is
  named directly
2. the same direct spelling for the other builtin scalar integer targets:
  `i8`, `i16`, `i32`, `i64`, `isize`, `u8`, `u16`, `u32`, `u64`, `usize`,
  and `uintptr`

The important constraint is not the exact token spelling.
The important constraint is that the source must state the target integer
type explicitly and that ordinary mixed-width arithmetic must continue to
reject implicit narrowing.

Why this is the right next step
-------------------------------
- It addresses a real recurring pressure point already seen in active kernel
  work rather than a speculative language neatness request.
- It would let compact protocol owners state intentional byte packing or
  width alignment directly in source instead of carrying parallel tracked
  payload state solely to satisfy current typing limits.
- It stays much smaller and more honest than a broader cast model because the
  feature can be limited to scalar integer conversions only.

Implemented surfaces
--------------------
The landed owning surfaces are:

1. `compiler/parse/parser_expr.cpp`
   - parse the narrow builtin-integer bare conversion form and mark it as the
     Phase 194 surface without widening ordinary call parsing broadly

2. `compiler/ast/ast.h`
   - carry one narrow syntax flag so sema can distinguish the new bare
     builtin-integer form from the older parenthesized explicit conversions

3. `compiler/sema/check.cpp`
   - validate that the new bare source form is integer-to-integer only while
     leaving the older parenthesized explicit-conversion path unchanged

4. `compiler/mir/`
   - reuse the existing explicit numeric conversion lowering path; no new MIR
     opcode family is required

5. `compiler/codegen_llvm/`
   - reuse the existing integer conversion backend mapping; no new backend
     cast family is admitted

6. `kernel/src/service_identity.mc`, `kernel/src/boot.mc`,
   `kernel/src/boot_update.mc`, and `kernel/src/boot_identity.mc`
   - spend the feature immediately on the restart-generation byte-packing path
     and delete the tracked shadow payload state

7. `tests/compiler/sema/` and `tests/compiler/mir/`
   - add focused positive and negative coverage for bare integer conversion
     legality, determinism, and rejection of non-integer bare uses

Chosen slice for this phase
---------------------------
1. one explicit integer-to-integer conversion form
2. explicit narrowing and widening only for scalar integer types
3. no implicit mixed-width coercions in arithmetic, comparison, or aggregate
   construction
4. one immediate kernel spend on compact protocol packing or byte comparison
5. one bounded negative test surface proving non-integer casts remain out of
   scope

What this phase should not do
-----------------------------
- no implicit numeric coercion rules
- no pointer casts, address reinterpretation, or ABI escape hatch
- no new bare float conversion model
- no general cast framework shared across all type families
- no hidden truncation in comparison or assignment
- no backend-only legalization that sema does not model explicitly

What this phase intentionally leaves unchanged
---------------------------------------------
- the older parenthesized explicit-conversion syntax remains the only source
  spelling for non-integer explicit conversions already admitted elsewhere in
  the compiler
- distinct, alias, pointer, slice, and other broader explicit conversions do
  not gain the new bare `T(expr)` syntax in this phase
- no implicit mixed-width comparison, assignment, or aggregate-construction
  rule is added

Planning rule for this phase
----------------------------
Follow-on work should reopen only when a real admitted owner is more distorted
by the remaining limits than by the language staying explicit.

If follow-on work is considered:

1. start from the smallest real owner rather than from an abstract
   numeric-type completeness goal
2. keep the semantics target-typed, explicit, and deterministic
3. stop if the design starts pressuring implicit coercion rules or a general
   cast framework

What this phase should prove
----------------------------
- that the language can admit one honest low-level integer-conversion step
  without reopening implicit coercion policy
- that compact protocol packing can be expressed directly in source when the
  programmer intentionally names the destination width
- that the real owner-local workaround pressure from Phases 188 and 193 can
  be relieved by one narrow compiler slice rather than by broader numeric or
  framework growth
