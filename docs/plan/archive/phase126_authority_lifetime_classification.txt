Phase 126 -- Authority Lifetime Classification
==============================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the completed Phase 125 invalidation and rejection audit step
by proving one explicit long-lived versus short-lived authority distinction
over the same already-landed Veya kernel artifact.

This document is both the Phase 126 execution note and its closeout.
It does not claim a broader lifetime framework, service-death handling,
restart policy, general failure semantics, or a compiler wave.
It records one narrow authority lifetime classification step over the real
kernel artifact that Phase 125 already kept explicit.

Repository status note
----------------------
- Phase 124 already proved that the same explicit-authority model survives one
  deeper two-hop delegation chain while each prior holder loses reuse rights.
- Phase 125 then proved that one explicit final-holder authority-loss path is
  rejected consistently at the current send and receive use sites while a
  separate control handle survives.
- The next honest gap is not a broader lifetime framework. It is proving that
  the current model can distinguish one bounded long-lived authority path from
  one bounded short-lived delegated path without reopening compiler work.

Problem being solved
--------------------
- The repository already had one surviving control handle and one invalidated
  delegated handle after Phase 125, but it still lacked one published
  aggregate proof that those two paths belong to different bounded lifetime
  classes rather than to one implicit authority bucket.
- Without this step, later failure, IPC-composition, or device-boundary work
  could blur whether the current model already has an honest lifetime answer
  or whether a broader framework is required immediately.
- The repository needed one small phase that classified lifetime through the
  existing syscall, capability, endpoint, bootstrap-audit, and debug owners
  rather than through a new kernel policy layer.

Core rule
---------
- Keep the current kernel owners and service topology intact.
- Keep the compiler, sema, MIR, backend, ABI, and `hal` surfaces closed.
- Reuse the landed surviving control-handle path and the existing invalidated
  delegated-handle path rather than inventing a lifetime framework.
- Prove one bounded long-lived authority path and one bounded short-lived
  authority path at the final holder of the landed Phase 124 delegation chain.
- Do not widen into a general lifetime framework, broader failure policy,
  restart logic, dynamic namespace growth, or a new service owner.

Chosen slice for this phase
---------------------------
Targeted boundary shape:

1. one canonical Phase 126 note recording the authority lifetime
   classification step and the decision to keep compiler work closed
2. one debug-owned Phase 126 audit that packages the landed Phase 125
   invalidation truth together with one explicit long-lived versus short-lived
   authority classification at the final holder
3. one narrow root-proof helper that replays the landed invalidation path,
   proves the delegated handle stays rejected as the short-lived path, and
   proves the separate control handle remains reusable as the long-lived path
4. one advance of the real kernel proof marker from Phase 125 to Phase 126
   only after that lifetime audit succeeds
5. refreshed roadmap, position note, kernel README, repository map,
   freestanding test READMEs, decision log, and backlog entries
6. one focused Phase 126 freestanding regression that checks build, run,
   publication wording, and MIR visibility of the new lifetime path

Chosen outcome
--------------
Chosen outcome: yes.

The repository should take this step now because the current authority model
already proves deeper delegation and explicit invalidation. The next narrow
pressure is publishing one explicit lifetime distinction before later phases
widen failure or IPC-composition work.

What this phase proves
----------------------
- the repository can now honestly publish one bounded authority lifetime
  classification step over the real kernel artifact
- one final holder from the landed Phase 124 delegation chain can retain one
  long-lived control handle while losing one short-lived delegated handle
- that lifetime answer still stops well short of a broader lifetime
  framework, failure framework, service manager, or compiler reopening
- the current kernel model can answer one more authority-lifecycle question
  through existing owners rather than by forcing new policy machinery
