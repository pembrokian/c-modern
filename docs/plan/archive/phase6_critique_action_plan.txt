Scope
-----
This plan covers only actionable findings that require code changes —
confirmed bugs, correctness gaps, and clear DRY violations with a
well-defined mechanical fix.  Architecture-only findings (refactoring
ValidateModule, splitting AnalyzeExpr, replacing SpecialCallKind
dispatch with sema attributes, adding a --target flag) are excluded
per the repository working rule: "prefer minimal, targeted changes."

Findings reviewed but NOT planned
----------------------------------
These were examined and found to be either non-issues or out of scope:

  - active_types stale entry after cycle-detection early return:
    ComputeTypeLayouts() calls active_types.clear() after every
    top-level type, and the recursive field-failure unwinding path
    already erases each parent's own entry.  The code is correct.

  - kHostedRuntimeSupportPath relative to CWD: backend.cpp already
    calls DiscoverRepositoryRoot and resolves the runtime source
    path as (*repo_root / kHostedRuntimeSupportPath) on line 4281.
    The critique is outdated.

  - Performance findings (SpanKey allocation, O(n^2) symbol lookup,
    RewriteImportedModuleSurfaceTypes caching): real at scale, not
    blocking at bootstrap scale.  Defer.

  - Parser defensiveness (recursion depth limits, allow_aggregate_init_
    threading): correct as-is for bootstrap; no correctness bug exists.

  - SpecialCallKind string dispatch → sema attributes: architectural
    rework, not a bug fix.

  - Breaking ValidateModule / AnalyzeExpr into sub-functions:
    valid design goal, wrong time.

TIER 1 — Bug Fixes (compiler correctness)
------------------------------------------------------------------------

Item B1: backend.cpp — LowerStructTypeInfo ignores is_packed
  Severity : HIGH — correctness bug
  File     : compiler/codegen_llvm/backend.cpp
  Function : LowerStructTypeInfo

  Problem
    The function emits struct field types as "{ f0, f1, ... }" with
    natural alignment for all structs, regardless of is_packed.  A
    packed struct should be emitted as "< { f0, f1, ... } >" (LLVM
    packed syntax) with every field_alignment forced to 1.  Without
    this, LLVM applies natural-alignment padding, producing a different
    in-memory layout than what sema computed via ComputeTypeLayout.
    Any field access by sema-computed byte offset will land at the
    wrong address in emitted IR.

  Fix
    1. In LowerStructTypeInfo, check type_decl.is_packed.
    2. If packed: set local field_alignment = 1 (ignore lowered
       field alignment); do not apply AlignTo between fields; do not
       pad the final struct size to alignment; set info.alignment = 1.
    3. When building backend_name:
         packed=false : "{ f0, f1, f2 }"
         packed=true  : "<{ f0, f1, f2 }>"
    4. Add a sema fixture and/or MIR fixture with a packed struct that
       verifies the layout offset assignments, and a codegen test that
       exercises packed struct field reads and writes.

  Test file to add or extend
    tests/codegen/ — add packed_struct.mc exercising [[packed]] structs
    tests/sema/    — verify sema-computed layout offsets for [[packed]]


Item B2: check.cpp + mir.cpp + type.cpp — unguarded std::stoul in
         ProcedureParamCount and FormatType
  Severity : HIGH — uncaught exception terminates the compiler process
  Files    : compiler/sema/check.cpp  (ProcedureParamCount, line 139)
             compiler/mir/mir.cpp     (ProcedureParamCount, line 656)
             compiler/sema/type.cpp   (FormatType, line 345)

  Problem
    All three sites call std::stoul(type.metadata) or
    std::stoul(canonical.metadata) without exception handling.  If
    type.metadata ever holds a non-numeric string (e.g. from an
    error-recovery path), std::stoul throws std::invalid_argument,
    which propagates as an unhandled exception and kills the compiler
    process instead of emitting a diagnostic.

  Fix (bundled with Item B3 below)
    1. Move ParseArrayLength to compiler/support/ (see B3).
    2. Replace every std::stoul(type.metadata) / std::stoul(metadata)
       call with:
         mc::support::ParseArrayLength(type.metadata).value_or(0)
       The existing ParseArrayLength already does a digit-only parse
       and returns nullopt on invalid input, so value_or(0) is a
       safe fallback that produces 0 param count — sema will then
       emit a downstream type error rather than crashing.
    3. Remove the local ParseArrayLength definitions in check.cpp,
       mir.cpp, and backend.cpp after moving to support/.


Item B3: check.cpp + mir.cpp + backend.cpp — ParseArrayLength
         triplication
  Severity : LOW (DRY violation, bundled with B2)
  Files    : compiler/sema/check.cpp   line 157
             compiler/mir/mir.cpp      line 410
             compiler/codegen_llvm/backend.cpp  line 191

  Problem
    An identical 12-line decimal integer parser appears three times.
    All three copies must be kept in sync if array-length handling
    changes.

  Fix
    1. Add ParseArrayLength to compiler/support/diagnostics.h (or a
       new compiler/support/parse_utils.h) as a free function:

         namespace mc::support {
         std::optional<std::size_t> ParseArrayLength(std::string_view text);
         }

    2. Add the implementation to compiler/support/diagnostics.cpp (or
       parse_utils.cpp if a new file is warranted).
    3. Delete the three local copies and add the include to each file.
    4. Verify the build.

  Note: ParseArrayLength is a pure digit-accumulator with no
  diagnostic dependency, so support/ is the right layer.  If a
  dedicated parse_utils.{h,cpp} is added to support/, update
  CMakeLists.txt to include the new translation unit in mc_support.


Item B4: ast.cpp — DumpFieldInit unguarded *init.value dereference
  Severity : LOW (diagnostic path crash risk)
  File     : compiler/ast/ast.cpp
  Function : DumpFieldInit (line 204)

  Problem
    DumpExpr(*init.value, stream, indent + 1) dereferences init.value
    (a unique_ptr) without a null check.  Every other pointer
    dereference in the dump file is guarded.  If any future code path
    produces a FieldInit with a null value pointer, this crashes with
    no diagnostic information.

  Fix
    Wrap the dereference:
      if (init.value != nullptr) {
          DumpExpr(*init.value, stream, indent + 1);
      }
    One line change.  No tests need updating; this is a defensive guard.

Item Q1: check.cpp + driver.cpp — ResolveImportPath duplication
  Severity : MEDIUM — algorithmic drift risk
  Files    : compiler/sema/check.cpp   line 1750
             compiler/driver/driver.cpp line 138

  Problem
    Both files contain a nearly identical ResolveImportPath function
    (importer-relative search, then import-root search, then error).
    The only difference is how they access the root list: check.cpp
    takes const CheckOptions& and accesses options.import_roots;
    driver.cpp takes const std::vector<std::filesystem::path>& directly.
    If the resolution logic changes in one place (e.g. adding a third
    lookup root), the other diverges silently.

  Fix
    1. Add to compiler/support/ (e.g. in a new import_resolver.h/.cpp
       or appended to an existing support header) a shared function:

         namespace mc::support {
         std::optional<std::filesystem::path> ResolveImportPath(
             const std::filesystem::path& importer_path,
             std::string_view module_name,
             const std::vector<std::filesystem::path>& import_roots,
             DiagnosticSink& diagnostics,
             const SourceSpan& span);
         }

       The root-list parameter is the common denominator of both callers.

    2. In check.cpp: replace the local implementation with a call:
         return mc::support::ResolveImportPath(
             importer_path, module_name, options.import_roots,
             diagnostics, span);

    3. In driver.cpp: replace the local implementation with a call to
       the same shared function (already takes import_roots directly).

    4. Build and run all fixture suites to confirm no behaviour change.

  Note: If Q1 is done at the same time as B3 (ParseArrayLength), adding
  import_resolver.h is the natural landing spot for both utilities, or
  they can share parse_utils.h.


Item Q2: driver.cpp — entry file compiled twice in build mode
  Severity : MEDIUM — wasted work; correct result, double pipeline cost
  File     : compiler/driver/driver.cpp
  Function : CompileToMir (include_imports_for_build = true path)

  Problem
    When RunBuild calls CompileToMir with include_imports_for_build=true:
      1. The entry file is compiled fully (lex → parse → sema → MIR →
         validate) and stored in parse_result / sema_result / mir_result.
      2. CollectBuildUnits(source_file->path, ...) is then called with an
         empty built_indices map.  Since the entry path is not in
         built_indices, CollectBuildUnits runs the full pipeline for the
         entry file a SECOND time.
    The second compilation's result is used in MergeBuildUnits; the first
    compilation's results are only used for AST/MIR dump artifacts in
    RunBuild.  Both compilations produce identical output for a correct
    source file, so the end result is correct but wasteful.

  Fix
    Restructure the include_imports_for_build branch of CompileToMir to
    avoid the redundant entry-file compile:

    Strategy: Instead of calling CollectBuildUnits(entry_path, ...) which
    immediately invokes the full pipeline for the entry file, pre-seed the
    units vector and built_indices map with the already-compiled entry unit,
    then process only the entry file's direct imports:

      // Construct a BuildUnit from the already-compiled entry data.
      // (parse_result and sema_result must not be moved from yet because
      //  CheckedProgram still needs them — use the path-mapped unit only
      //  for MIR merge.)
      const auto entry_abs =
          std::filesystem::absolute(source_file->path).lexically_normal();
      const std::string entry_key = entry_abs.generic_string();

      std::unordered_set<std::string> visiting { entry_key };
      std::unordered_map<std::string, std::size_t> built_indices;
      std::vector<BuildUnit> units;

      // Collect all transitive imports first (imports before entry in units).
      const auto entry_roots = ComputeEffectiveImportRoots(entry_abs,
                                                           options.import_roots);
      for (const auto& import_decl : parse_result.source_file->imports) {
          const auto import_path = ResolveImportPath(entry_abs,
              import_decl.module_name, entry_roots, diagnostics,
              import_decl.span);
          if (!import_path.has_value()) { return std::nullopt; }
          if (!CollectBuildUnits(*import_path, options, diagnostics,
                                 visiting, built_indices, units)) {
              return std::nullopt;
          }
      }

      // Append the pre-compiled entry unit last (matching CollectBuildUnits
      // ordering: imports before the file that imports them).
      built_indices.emplace(entry_key, units.size());
      units.push_back(BuildUnit {
          .source_path = entry_abs,
          // Share the parse/sema/mir results — safe because CheckedProgram
          // below still holds independent references to source_file and
          // the merged module pointer will be overwritten.
          .parse_result = std::move(parse_result),
          .sema_result  = std::move(sema_result),
          .mir_result   = std::move(mir_result),
      });
      // Re-obtain references for the CheckedProgram return value.
      const BuildUnit& entry_unit = units.back();

      auto merged_module = MergeBuildUnits(units, source_file->path);
      ...

    The CheckedProgram returned to RunBuild should carry references into
    entry_unit rather than into the moved-from locals.  This may require
    CheckedProgram to hold non-owning pointers/references into the last
    unit, or the design can be adjusted so RunBuild accesses dump data via
    the entry BuildUnit directly.

    Implementation note: the exact ownership structure (move vs reference)
    depends on how CheckedProgram is used downstream.  The key correctness
    invariant is that MergeBuildUnits receives the pre-compiled entry MIR
    exactly once, at the correct position (last in units, after all
    transitive imports).

  Test validation
    Run existing build-mode fixture tests after the change.  They confirm
    that the merged MIR and emitted binary are unchanged:
      ctest --test-dir build/debug -R mc_codegen_fixture --output-on-failure

Item D2: mir.cpp — document known validation gaps and conventions
  Add three targeted comments:

       rejected by the lexer.
