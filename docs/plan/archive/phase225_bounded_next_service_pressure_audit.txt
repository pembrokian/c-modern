Phase 225 -- Bounded Next Service Pressure Audit
================================================

Status
------
COMPLETED (2026-04-17)

Purpose
-------
Choose exactly one next owner-local running-system pressure step after the
Phase 220 timer/task admission, the Phase 221 file-service state reshape, and
the Phase 224 dispatch ownership cleanup, then land that step as a real
implementation.

Repository status note
----------------------
- Phase 220 admitted bounded `timer_service` and `task_service` owners.
- Phase 221 kept file-service growth owner-local through a slot-in-state
  redesign rather than compiler expansion.
- The backlog still records multiple active kernel follow-through themes:
  queue behavior, ticket-service restart-visible state, retained coordination,
  file-like growth, timer growth, and task growth.
- The repository continues to defer supervisor frameworks, durable storage,
  dynamic discovery, and broader orchestration models unless a real admitted
  consumer proves them necessary.

Problem being solved
--------------------
- Several bounded service-pressure candidates are plausible next moves, but
  the repository should pick only one.
- Without a selection phase, "next kernel work" can blur into multiple small
  widenings across queue, timer, task, ticket, or retained-state owners.
- The project needed one evidence-backed choice for the next owner-local
  service slice and explicit deferral wording for the others.
- `task_service` already defined `TASK_STATE_DONE`, but no shell-visible or
  service-local operation could ever produce it. The admitted task owner could
  submit active work, cancel it, fail it immediately, or clear it on restart,
  but it could not represent successful completion at all.

Core rule
---------
- Pick exactly one owner-local next service-pressure step.
- Prefer the candidate with the strongest real admitted consumer pressure.
- Keep the landing compact: one owner, one small command family addition or
  one bounded state consequence, one scenario, one focused reset-lane fixture.
- Defer all other candidate owners explicitly.

Decision summary
----------------
1. Chosen branch: Branch C -- task follow-through.
2. Landed owner-local addition: `JD<id>!` transitions one active task to
   `TASK_STATE_DONE`.
3. Queue, timer, ticket, and file follow-through remain explicitly deferred.

   - stronger observability or consequence surface beyond current count, peek,
     pressure, and stall classification
2. Timer-service follow-through
   - one richer temporal consequence beyond current create, cancel, query, and

3. Task-service follow-through
   - one richer work-state or consequence path beyond current submit, query,

5. File-service follow-through
   - one file-like owner-local step that remains inside retained-only semantics

Questions this phase must answer

1. Which candidate owner now has the strongest real caller-visible pressure in

2. Can that pressure be closed honestly with one owner-local addition rather

3. Which nearby candidates are still tempting but should remain explicitly

Chosen branch
-------------
Branch C -- task follow-through
- `task_service` had the strongest current caller-visible gap because its own
  state vocabulary exposed `DONE` while no executable command path could ever
  reach that state.
- The narrowest honest repair was one owner-local successful-completion
  consequence, not a process model, worker pool, scheduler, or framework.
- The landing stays inside the current task owner, compact shell protocol, and
  reset-lane workflow surface.

- Queue follow-through deferred: queue already has caller-visible count, peek,

- Ticket follow-through deferred: ticket already has explicit restart-visible
  stale and invalid outcomes.
- File follow-through deferred: file just received its Phase 221 owner-local

Selection criteria
------------------
The selected branch must satisfy all of the following:
- it pressures a current admitted owner rather than inventing a new subsystem
- it is caller-visible in the running system rather than purely aesthetic
- it can be closed with one owner-local addition and one narrow executable
  proof
- it is stronger than reopening durability, supervisor, or discovery fronts

Suggested audit method
----------------------
1. Re-survey the active running-system owners and their current compact command
   and inspection surfaces.
2. Identify where a caller today still hits a real visibility or consequence
   gap.
3. Compare the candidate owners using one pressure matrix:
   caller-visible gap, narrowest honest fix, executable proof cost, and
   framework risk.
4. Select exactly one branch and defer the others explicitly.

Post-land re-audit
------------------
- After the Phase 225 landing and full reset-lane validation, queue, timer,
  ticket, and file were re-audited again.
- The strongest next gap is now file-service bounded payload growth.
- Queue, timer, and ticket are all internally coherent on their current compact
  contracts:
  - queue already covers count, peek, pressure, stall, and retained restart
  - timer already covers create, cancel, query, expired-window, and retained
    restart
  - ticket already covers issue, consume, stale, invalid, and restart-visible
    epoch change
- File remains the thinnest current owner: its slot shape already carries
  `len` plus `data0..data3`, but the admitted behavior still stops at one-byte
  write-once storage. That is a real caller-visible limitation, not just a
  missing audit.
- The narrowest honest next move is one file-owned bounded payload growth step:
  let `FW<name><byte>` append one byte up to a fixed per-file limit of four
  bytes and let `FR<name>!` return the retained byte prefix with length `1..4`.
- This stays owner-local, uses the already-landed slot-in-state shape from
  Phase 221, does not widen into durability or a filesystem framework, and does
  not require compiler, sema, MIR, backend, runtime, stdlib, or `hal` work.

Exit criteria
-------------
- the repository has one chosen next service-pressure owner and one real
  owner-local landing
- the non-selected candidate owners are explicitly deferred rather than left as
  vague co-equal possibilities
- the chosen next move is small enough to stay owner-local and executable

Chosen next gap after Phase 225
-------------------------------
- Best next gap: file-service bounded payload growth inside the retained-only
  file owner.
- Outcome: landed in Phase 226 as bounded append-until-full plus retained
  prefix read, still inside the retained-only file owner.
- Recommended next slice:
  - keep `FILE_CAPACITY = 4` file slots unchanged
  - grow per-file retained payload from one byte to up to four bytes
  - reinterpret `FW<name><byte>` as append-one-byte-until-full instead of
    single-write-only
  - keep `FR<name>!` as the full retained read, now returning `payload_len`
    from `0..4` and bytes `data0..data3`
  - keep retained restart behavior explicit and unchanged
- Non-goals for that next slice:
  - no path graph or namespace framework
  - no durable storage
  - no seek, truncate, delete, or directory semantics
  - no compiler or protocol-width expansion

Non-goals for this phase
------------------------
- no supervisor framework
- no durable storage or filesystem framework
- no capability or ACL framework expansion
- no dynamic service discovery registry
- no scheduler, session, or orchestration framework growth
- no compiler, sema, MIR, backend, runtime, stdlib, or `hal` widening unless a
  later selected owner proves that as the narrowest blocker

Why this is the narrowest honest next move
------------------------------------------
- The repository has several plausible service fronts, but the discipline of
  recent phases has been to widen one owner at a time.
- A bounded selection audit preserves that discipline and prevents several
  adjacent small kernel ideas from landing as one unfocused wave.
- The next honest service step should be chosen by real caller-visible pressure
  rather than by taste or by which owner is easiest to edit.
- `task_service` exposed exactly that kind of honest gap: it already named a
  successful terminal state, but the running system had no way to reach it.
