Phase 273 -- App Substrate Pressure Audit
=========================================

Status
------
COMPLETED

Purpose
-------
Close Band C by stating which app-facing pressures now belong to launcher,
update, object-store, or other explicit owners, and which still remain too
weak to justify broader packaging or app-framework work.

This phase exists now because Band C intentionally mixes identity, launcher,
install, restart, asset, and demo work. The repository should stop after that
wave and classify what those slices actually proved before it starts first UI
work.

Repository status note
----------------------
- Phase 267 landed one explicit runtime `program_catalog` owner for bounded
  launch descriptors.
- Phase 268 landed one bounded `launcher_service` owner for fixed-catalog
  selection and one foreground launch consequence.
- Phase 269 through Phase 271 landed one fixed installed-program slot, one
  launcher-visible fresh-versus-resumed-versus-invalidated launch
  classification, and one `issue_rollup`-only manifest family.
- Phase 272 landed one visible installed-workflow demo path with one explicit
  launcher-owned status query over the admitted `issue_rollup` path.
- The remaining question was whether that evidence already makes Band D's
  first input/display owner stronger than another app-substrate follow-through.

Problem being solved
--------------------
- Without an explicit closeout audit, the roadmap could slide from the first
  app-shaped substrate slices straight into UI work without stating what the
  app band actually solved.
- The repository needs one honest answer on which pressures are now closed,
  which remain deferred, and whether the strongest next owner is truly in the
  UI band or still in app-facing substrate work.
- This is a current planning problem, not generic roadmap hygiene.

Non-goals
---------
- no new launcher, install, or UI feature landing by convenience
- no broad packaging or app-framework proposal hidden inside the audit
- no support statement broader than the actual Band C evidence
- no silent reinterpretation of earlier phase outcomes

Invariants
----------
- The audit must classify actual Band C outcomes only.
- Closed surfaces must remain bounded to what the earlier phases really
  proved.
- Still-open app pressures must stay concrete and owner-local.
- The audit must not claim UI readiness unless the Band C evidence actually
  supports it.

Required surface changes
------------------------
1. Re-audit the actual outcomes of Phases 267 through 272.
2. Publish which app-substrate surfaces are now maintained truth.
3. Publish which app-facing pressures remain deferred and why.
4. State whether Band D UI work now honestly wins over further app-substrate
   follow-through.
5. Tighten backlog or decision-log entries only where the audit exposes a real
   still-open pressure.

Forbidden changes
-----------------
- do not land new app features inside the audit
- do not treat one bounded installed workflow demo as proof of a full app
  platform
- do not bury still-open blockers under vague labels like "polish" or
  "productization"
- do not widen the audit into UI design beyond what is needed to classify the
  next owner

Phase role in the larger sequence
---------------------------------
- This phase closes Band C.
- It determines whether the repository is ready to start Band D's first UI
  surface honestly.
- It preserves the repository's evidence-first planning style by stopping
  after one mixed app-facing wave and stating what is actually true.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep app-facing owner growth bounded to explicit launcher, durable, and
  workflow seams
- keep broader packaging and app-framework work deferred until a real second
  consumer demands it

Questions this phase must answer
--------------------------------
1. Which Band C slices landed cleanly enough that later UI phases can rely on
   them as maintained truth?
2. Which still-open app-facing pressure, if any, remains stronger than the
   first UI band?
3. Did the installed workflow demo expose one missing owner that still belongs
   to app substrate rather than UI?
4. What exact evidence now supports starting Phase 274 and later UI work?

Preferred implementation shape
------------------------------
- Keep the phase document-focused and evidence-based.
- Reuse the repository's earlier plateau and frontier-audit style.
- Prefer short explicit classifications over broad narrative summary.

Implementation outcome
----------------------
- Band C is now classified as complete for its admitted slice.
- The maintained app-substrate truths are:
  - one fixed runtime program catalog owned by `program_catalog`
  - one bounded single-foreground launcher owner for selection, identify,
    launch, restart classification, manifest query, and installed-workflow
    status on the admitted path
  - one update-store-owned installed-program slot plus one minimal persisted
    launch-record comparison seam
  - one `issue_rollup`-only manifest family carried on the installed bytes
    instead of a broader package or filesystem surface
  - one explicit end-to-end installed-workflow demo proving that those owners
    compose without a hidden orchestrator
- The audit found no stronger still-open app-substrate owner than Phase 274's
  first input-event work.
- The still-open app-facing pressures remain real but weaker than Band D's
  first slice:
  - a second installed-program consumer
  - richer app-facing status than the current compact launcher query
  - broader app execution semantics than the current launcher-plus-update-store
    split can honestly express
  - a second asset or manifest family that cannot stay on the current
    `issue_rollup`-only path
- The installed-workflow demo did not expose a missing launcher, update,
  object-store, or workflow owner. It instead confirmed that the current
  bounded app path is sufficient for one first UI consumer to sit on top of it.

Phase review against the spec
-----------------------------
- Preserved invariants:
  - the audit classifies only the landed Phase 267 through Phase 272 outcomes
  - closed surfaces stay bounded to one catalog, one launcher owner, one fixed
    installed slot, one bounded restart classification, one `issue_rollup`-
    only manifest family, and one explicit installed-workflow demo
  - deferred pressures remain concrete and owner-local instead of being buried
    under generic productization language
- Ownership boundaries did not weaken:
  - launcher remains the app-facing policy owner
  - `update_store_service` remains the durable owner of installed bytes and the
    minimal launch-record comparison truth
  - `object_store_service`, `workflow_service`, and
    `completion_mailbox_service` were not widened into a package or app
    framework
  - shell and dispatch remain routing seams only
- Accidental centers of gravity were avoided:
  - no package manager, app framework, status registry, filesystem, or second
    orchestrator was introduced by the audit
  - no new launcher-owned durable storage or update-store-owned app policy was
    claimed
- Honest non-decisions and deferred work:
  - no second installed demo consumer yet
  - no broader process-execution semantics yet
  - no richer manifest or asset family yet
  - no session, document-restore, or save-state framework yet
  - no shell-visible dashboard or broader app metadata surface yet
