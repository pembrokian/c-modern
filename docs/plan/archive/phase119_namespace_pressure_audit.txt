Phase 119 -- Namespace Pressure Audit
=====================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the completed Phase 118 delegated request-reply follow-through
by answering the next bounded running-system question explicitly: does the
repository now need a broader namespace or service-discovery framework in
order to widen the admitted kernel slice honestly, or is one smaller
init-owned answer enough for the current pressure?

This document is both the Phase 119 execution note and its closeout.
It does not claim general namespace policy, dynamic service discovery,
ambient capability lookup, a service manager, or a compiler wave.
It records one narrow namespace-pressure answer inside the already admitted
first running Veya kernel slice.

Repository status note
----------------------
- Phase 117 already published one bounded init-orchestrated multi-service
  bring-up over the landed log, echo, and transfer owners.
- Phase 118 then published one bounded delegated request-reply path together
  with one explicit refused action on the invalidated source slot.
- That left the next running-system question narrower and clearer: the
  repository now had multiple real services and one delegated reply path, but
  it still had not published the smallest explicit answer to the next
  namespace pressure point.
- The real kernel artifact already had enough stable truth to answer that
  question without widening the kernel or compiler surface: init already owns
  the stable program capabilities, stable wait-handle slots, and the shared
  control endpoint used to start and observe the bounded service set.

Problem being solved
--------------------
- The speculative post-Phase-118 roadmap reserved Phase 119 for one
  namespace-pressure audit.
- The repository needed to state whether the next honest step was a broader
  namespace mechanism or a narrower publication decision.
- Leaving that pressure implicit would have made later work too easy to blur
  into premature claims about service discovery, naming frameworks, or new
  compiler-owned machinery.

Core rule
---------
- Keep the current kernel owners and service topology intact.
- Keep service policy in init and the landed service helpers rather than
  moving it into a kernel-resident namespace framework.
- Keep the compiler, sema, MIR, backend, ABI, and `hal` surfaces closed.
- Do not widen into dynamic discovery, general namespaces, ambient authority
  lookup, broader init frameworks, or a service manager.
- Answer the current pressure mechanically with one fixed init-owned service
  directory audit over the real kernel artifact and one focused freestanding
  regression.

1. one canonical Phase 119 note recording the namespace-pressure audit and
   the decision to keep compiler work closed
2. one debug-owned Phase 119 audit that packages the landed Phase 118
   boundary together with one fixed init-owned three-entry service directory
3. one narrow audit builder and one root-proof helper that publish the stable

   only after the namespace-pressure audit succeeds

   Phase 119 freestanding regression

Chosen outcome
--------------
Chosen outcome: yes.

The repository should take this step now because the current namespace
pressure is real, but it is still narrow. The landed service slice already
has stable init-owned program capabilities, stable wait-handle slots, and one
shared control endpoint. The next honest gain is publishing that fixed
directory answer explicitly rather than inventing a broader namespace model
before a real kernel step requires one.

What this phase proves
----------------------
- init can own one fixed three-entry service directory over the landed log,
  echo, and transfer service set
- the current service directory answer is explicit and stable: each service
  keeps one distinct key, one distinct program capability, and one distinct
  wait-handle slot while sharing the existing init-owned control endpoint
- the current pressure does not yet require dynamic discovery, a general
  namespace model, or compiler work
- the repository can answer the next namespace question honestly while
  keeping both the kernel and compiler surfaces closed
