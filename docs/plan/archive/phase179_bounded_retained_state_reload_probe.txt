Phase 179 -- Bounded Retained-State Reload Probe
================================================

Status
------
COMPLETE

Outcome
-------
Phase 179 landed as one explicit retained-state reload path without opening a
filesystem, database, supervisor, or compiler-facing expansion.

The landed slice chooses the narrowest real retained-state owner already in
the system: `log_service`.

- `kernel/src/log_service.mc` now owns one explicit bounded save
  representation, `LogSnapshot`, plus two explicit helpers:
  - `log_snapshot(state)` saves the retained log bytes and length
  - `log_reload(pid, slot, snapshot)` reconstructs the service state from that
    saved representation
- `kernel/src/init.mc` remains the restart-policy owner and now uses the
  explicit snapshot path in `restart_log()` rather than reinitializing the
  log to an empty baseline
- the live integration path in `kernel/src/scenarios.mc` now exercises the
  retained reload result directly: the full retained log survives restart and
  remains full afterwards
- one new system fixture and workflow registration prove the more useful case
  that a partially filled retained log survives restart and can continue to
  grow after reload

What did not change:

- there is still no filesystem API, persistence namespace, or transaction
  model
- retained-state reload is still explicit and owner-local, not a framework
- only `log_service` has a reload path; `kv_service` still resets on restart
- no compiler, sema, MIR, backend, ABI, target, runtime, or hal expansion was
  required

Phase result
------------
The current service identity and init-owned restart model can carry one honest
restart-with-state path at this scale.

The key composition result is narrow and explicit:

- endpoint identity still stays stable across restart
- retained state survives only when init chooses an explicit save-and-reload
  path
- the first admitted retained-state reload path is log-only and bounded to
  the existing four-byte retained buffer

Implementation surfaces
-----------------------
The landed implementation changes are:

1. `kernel/src/log_service.mc`
   - add the explicit `LogSnapshot` representation
   - add `log_snapshot(...)` and `log_reload(...)`

2. `kernel/src/init.mc`
   - keep restart policy init-owned
   - change `restart_log()` from empty re-init to explicit snapshot reload

3. `kernel/src/service_identity.mc`
   - update the published restart contract so retained state is no longer
     described as universally lost

4. `kernel/src/scenarios.mc`
   - make the live restart probe check retained reload directly

5. `tests/system/kernel_reset_lane_phase179_retained_reload/`
   - prove the partial-log restart-with-reload path on the real reset-lane
     kernel path

6. `tests/system/kernel_reset_lane_service_identity/`
   - verify ServiceRef stability against the real restart-and-reload path

7. `tests/system/kernel_reset_lane_long_lived_coherence/`
   - narrow the older empty-baseline wording to fresh `kernel_init()` instead
     of every restart path

8. `tests/tool/tool_workflow_suite.cpp`
   - register the new Phase 179 fixture in the workflow suite

Why this landed shape is enough
-------------------------------
- It introduces real restart-with-state behavior, not just a proof-only note.
- The saved-state format is explicit and fixed-size.
- The reload owner is visible in source: `log_service` owns representation,
  `init` owns restart policy.
- The system stays Ken-Thompson-simple: one saved struct, one reload helper,
  one restart owner, no framework.

Validation
----------
- `mc_tool_workflow_unit`

Purpose
-------
Introduce the first minimal "state survives a restart" pressure without
opening a filesystem, database, or general persistence framework.

The goal is not durable storage as a feature family.
The goal is to discover whether the current service identity, restart truth,
and init-owned lifecycle model can carry one explicit retained-state reload
path honestly.

Repository status note
----------------------
- Current restart semantics are explicit and truthful precisely because restart
  behavior is caller-visible and owner-local.
- Before this phase, every admitted restart path reset service state to a
  known empty baseline.
- Current identity semantics are compile-time and runtime-local: a ServiceRef
  names a boot-wired endpoint, not a persisted object identity.
- The repository still makes no persistence, packaging, loader, or filesystem
  claim.
- Nevertheless, the next honest external pressure after service growth is not
  a full storage subsystem.  It is one bounded retained-state reload path.

Problem being solved
--------------------
- Without one retained-state reload probe, the repository cannot tell whether
  restart-with-state is merely deferred or whether it would immediately expose
  a deeper identity or lifecycle mismatch.
- Jumping straight to a filesystem would mix storage semantics, persistence
  format, boot loading, and service policy into one too-large step.
- This phase creates the pressure in the narrowest possible way: one explicit
  saved-state artifact or snapshot under explicit owner control, then one
  explicit reload on restart.

Core rule
---------
- Do not admit a filesystem API family.
- Do not admit general persistence or transaction semantics.
- Keep saved-state format explicit and bounded.
- Keep reload ownership visible: either service-owned or init-owned, but not
  hidden behind a framework.
- If retained-state reload exposes an identity mismatch, name that mismatch
  precisely instead of solving it with packaging or service registries.

Chosen slice for this phase
---------------------------
1. one retained-state candidate only, preferably the smallest current service
   whose retained behavior is already meaningful under restart pressure
2. one explicit save representation and one explicit reload path
3. one proof that restart-with-reload either preserves the intended visible
   state honestly or exposes the first real mismatch in service identity or
   lifecycle ownership
4. no broader storage namespace, file API, or general persistence abstraction

Why this is the right placement
--------------------------------
- After Phase 178: topology and restart structure have been re-judged under
  growth pressure first.
- Before any packaging or second-target investment: retained-state reload is a
  more immediate system pressure than broader artifact lifecycle policy.

Planning rule for this phase
----------------------------
- Choose the narrowest service whose state reload is both meaningful and easy
  to reason about.
- Prefer one explicit snapshot over any background persistence story.
- If the pressure still fits the current identity and restart model, say so.

What this phase should prove
----------------------------
- that the repository can admit one minimal restart-with-state path without
  lying about storage or packaging completion
- that service identity, restart truth, and retained state either compose
  honestly at this scale or expose one clearly named next boundary
- that persistence pressure can be introduced incrementally rather than as a
  subsystem leap
