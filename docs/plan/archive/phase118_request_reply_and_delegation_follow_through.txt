Phase 118 -- Request-Reply And Delegation Follow-Through
========================================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the completed Phase 117 init-orchestrated multi-service bring-up
by making the next running-system claim explicit: the repository now publishes
one bounded delegated request-reply path together with one explicit refused
source action after delegation rather than leaving that combined boundary
implicit inside the earlier transfer-service proof.

This document is both the Phase 118 execution note and its closeout.
It does not claim namespace policy, service discovery, multi-child service
topologies, dynamic loading, or a new compiler wave.
It records one narrow running-system follow-through inside the already
admitted first running Veya kernel slice.

Repository status note
----------------------
- Phase 106 already made one bounded request-reply service path real over the
  landed kernel slice.
- Phase 107 already made one bounded delegated endpoint-transfer path real,
  including one transferred reply-capable handle and one service-issued
  follow-through send.
- Phase 117 then published the log, echo, and transfer owners as one bounded
  init-orchestrated multi-service bring-up.
- That left the next running-system question narrower and clearer: the kernel
  already had the mechanism for a delegated request-reply path, but the
  repository still lacked one explicit phase-level proof that the delegated
  reply remains request-derived, that one retained sibling handle still owns
  reply reception, and that the invalidated source slot refuses later use
  without turning failure policy into a broader kernel framework.

Problem being solved
--------------------
- The speculative post-Phase-117 roadmap reserved Phase 118 for one bounded
  request-reply and delegation follow-through.
- The real kernel artifact already had enough explicit authority to express
  that step: init could retain one endpoint handle, delegate a sibling handle
  to a bounded service owner, and observe a reply routed back through the
  retained handle.
- But the repository still published that delegated interaction only as a
  capability-transfer proof, while the refused-action path on the invalidated
  source slot was not yet part of the running-system publication boundary.

Core rule
---------
- Preserve the current kernel owners and overall service topology.
- Keep service policy in `init` and the landed service helpers rather than
  moving it into a new kernel framework.
- Keep the compiler, sema, MIR, backend, ABI, and `hal` surfaces closed.
- Do not widen into namespace policy, dynamic loading, multi-child service
  orchestration, or a general service framework.
- Prove the boundary mechanically with one delegated-request-reply audit over
  the real kernel artifact and one focused freestanding regression.

Chosen outcome
--------------
Chosen outcome: yes.

The repository should take this step now because the delegated reply mechanism
is already real in the landed transfer-service slice. The next honest gain is
not a broader namespace or another compiler surface. It is publishing that
delegated reply route together with the refused invalidated-source action as
one bounded running-system truth.

What this phase proves
----------------------
- init can retain one local reply handle while delegating a sibling handle to
  a bounded service owner
- the delegated service path still returns a request-derived reply over the
  delegated handle rather than through ambient authority shortcuts
- the invalidated source slot refuses later send use explicitly with an
  `InvalidHandle` status after delegation
- the repository can widen its running-system claim without widening the
  compiler or introducing a broader service-management framework
