Phase 178 -- Topology And Restart Growth Re-Audit
=================================================

Status
------
COMPLETE

Outcome
-------
Branch B (one bounded structural follow-through) landed without reopening a
routing table, supervisor, retry framework, or any compiler-facing surface.

The re-audit result is: the flat topology and flat init-owned restart model
still hold at five services, but the first real post-Phase-176 growth pressure
does justify one small owner-local structural addition in each surface.

- `service_topology.mc` now owns three flat lookup helpers over the existing
  five slot constants:
  - `service_count()` returns the slot count as `usize`
  - `service_slot_at(index)` enumerates the static slots in named order
  - `service_slot_for_endpoint(endpoint)` maps a stable endpoint id back to
    its slot record
  - `service_slot_is_valid(slot)` makes the out-of-range sentinel explicit
- `init.mc` keeps `restart_log()`, `restart_kv()`, and `restart_echo()` as the
  explicit per-service policy surface, and now also owns:
  - `can_restart(endpoint)` for a flat yes/no policy check
  - `restart(state, endpoint)` for endpoint-driven caller-visible restart
- `scenarios.mc` now uses the endpoint-driven restart entry in the live kernel
  integration path, so the new shape is not test-only scaffolding
- one new system fixture and workflow registration prove the new boundary on
  the real reset-lane path: flat topology is enumerable, restartability stays
  explicit and selective, and echo pressure is recoverable through the new
  endpoint-driven restart entry

What did not change:

- topology is still fixed static wiring, not a routing table or service graph
- restart policy is still init-owned and caller-driven, not a supervisor
- serial and shell remain non-restartable in the current explicit policy model
- no compiler, sema, MIR, backend, ABI, target, runtime, or hal expansion was
  required

Why this is enough
------------------
- At five services, readers now have one flat way to enumerate or look up the
  static topology without inventing a data-driven router.
- Restart policy can now be driven from an observed endpoint id without
  widening into retry frameworks, service classes, or autonomous policy.
- The admitted shape stays Ken-Thompson-simple: explicit slots, explicit
  restartable endpoints, and explicit owner-local functions.

Purpose
-------
Re-audit the flat static topology and init-owned explicit restart policy after
the first post-Phase-174 growth pressure, and decide whether the Phase 170 and
Phase 172 deferrals still hold under that expanded service system.

This phase does not assume that topology or supervisor work is now required.
It asks whether the new pressure from Phases 176 and 177 is enough to justify
either one.

Repository status note
----------------------
- Phase 170 kept topology as flat slot constants with no iteration API,
  routing table, or supervisor-visible graph.
- Phase 172 kept restart policy in `init.mc` through flat explicit functions.
- The backlog trigger for both deferrals is now explicit: reopen only when the
  service count exceeds eight, when runtime slot iteration is required, when
  restart-frequency pressure requires per-service retry policy, or when
  caller-driven explicit restart is no longer the narrowest honest path.
- Phase 176 has now provided the first honest evidence beyond the earlier
  four-service, cooperative-user system.
- Phase 177 is expected to add hostile-user pressure on top of that growth.

Problem being solved
--------------------
- Once real growth pressure exists, the repository needs to know whether the
  current flat topology and flat restart policy are still stable or were only
  accidentally comfortable at four services.
- Without this re-audit, topology and restart convenience work could drift in
  piecemeal through local helpers instead of one explicit decision.

Core rule
---------
- Measure only from the pressure created in Phases 176 and 177.
- Do not admit a routing framework or supervisor by default.
- If the current flat forms are still sufficient, record that explicitly.
- If one bounded structural addition is needed, it must remain flat, named,
  and ownership-visible.

Chosen slice for this phase
---------------------------
Branch A -- explicit reaffirmation of current deferrals
- one audit confirming that flat ServiceSlot constants and flat init-owned
  restart functions remain sufficient after the new pressure
- one updated trigger statement if the evidence sharpens it

Branch B -- one bounded structural follow-through
- one small topology or restart-owner addition justified directly by the new
  pressure, such as a flat iterable slot list or a slightly less repetitive
  init-owned restart shape
- no routing table, supervisor service, watchdog, or retry framework

Why this is the right placement
--------------------------------
- After Phases 176 and 177: the repository finally has new evidence rather
  than speculation.
- Before any retained-state or packaging work: topology and restart ownership
  should be re-judged on that evidence first.

Planning rule for this phase
----------------------------
- Reuse the exact backlog triggers where possible.
- Prefer Branch A unless the new pressure clearly defeats the current flat
  forms.
- Keep policy in `init.mc` unless a narrower owner is clearly blocked.

What this phase should prove
----------------------------
- that the repository can distinguish real structural pressure from ordinary
  growth discomfort
- that Phase 170 and Phase 172 are either still correct deferrals or now have
  one specific bounded follow-through
- that topology and restart policy do not silently widen through convenience
  code
