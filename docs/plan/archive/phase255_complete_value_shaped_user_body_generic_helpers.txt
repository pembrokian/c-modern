Phase 255 -- Complete Value-Shaped User-Body Generic Helpers
============================================================

Status
------
COMPLETED

Purpose
-------
Complete the narrow value-shaped generic helper slice already framed in Phase
252 so the real `ServiceCell<T>` helper family can execute through the full
 compiler path and remove the current explicit wrapper noise in boot-state
 update code.

This phase exists now because the repository already named the exact blocked
consumer and the decision-log closure for Phase 252 already chose the smallest
honest implementation shape. The next honest move is to land that same-module,
direct-call helper family, not to reopen broader monomorphization.

Repository status note
----------------------
- Phase 252 already framed the bounded compiler problem and identified
  `service_cell_with_state<T>(...)` and `service_cell_restart<T>(...)` in
  `kernel/src/boot/boot_update.mc` as the real repository-owned spend site.
- The decision log now records and the compiler now carries the intended
  narrow owner: a MIR-owned executable specialization transform for same-module
  value-shaped generic bodies whose ABI depends on concrete type shape.
- `kernel/src/boot/boot_update.mc` now uses the real generic helper pair on
  the admitted path instead of retaining per-service inner-cell reconstruction
  noise.
- The repository now has owning MIR, executable, and reset-lane workflow proof
  coverage for this same-module helper slice while still leaving broader
  imported, inferred, and indirect-only shapes deferred.

Landed shape
------------
- Keep executable specialization MIR-owned in `compiler/mir/mir.cpp` and route
  the executable build path through that transform from
  `compiler/codegen_llvm/backend_host.cpp` before target validation and LLVM IR
  emission.
- Specialize only direct `symbol_ref` and direct `call` targets when MIR
  already carries a concrete procedure type and the callee ABI depends on
  concrete type shape.
- Keep erased-ABI generic bodies on the existing direct executable path rather
  than cloning them unnecessarily.
- Spend the admitted source-level cleanup in `kernel/src/boot/boot_update.mc`
  through the real `ServiceCell<T>` helper pair with explicit type arguments.
- Prove the slice through the existing MIR specialization regression, one
  focused executable regression on the `ServiceCell<T>` helper shape, and the
  maintained reset-lane workflow build path that compiles the real boot helper
  spend site.

Problem being solved
--------------------
- The repository already knows the exact helper family it wants, but the full
  compiler path still rejects user-body generic helpers whose instantiated ABI
  depends on concrete value shape.
- Leaving that gap unresolved keeps real boot-state update code noisier than it
  should be and leaves an already-chosen compiler repair stuck as planning
  rather than maintained capability.
- This is now a current-repo contradiction rather than speculative generic
  ambition: the repository has a real consumer, a real chosen narrow shape, and
  a real remaining compiler-owned blocker.

Non-goals
---------
- no whole-program generic monomorphization
- no imported or cross-module value-shaped generic-body follow-through
- no generic call-site inference
- no closures, lambdas, callback registries, or generic procedure constants
- no parser or sema redesign of generic meaning
- no silent widening to indirect-only generic call shapes without concrete
  direct-call metadata

Invariants
----------
- Generic legality and type meaning remain sema-owned.
- MIR owns any executable specialization required by value-shaped generic
  bodies; backend code lowers admitted MIR truth only.
- The already-admitted erased-ABI generic family remains unchanged.
- Artifact naming, emitted output, and MIR dumps remain deterministic.
- Unsupported broader generic-body shapes must continue to fail explicitly.

Required surface changes
------------------------
1. Land one MIR-owned executable specialization path for same-module
   value-shaped user-body generic helpers whose ABI depends on concrete type
   shape.
2. Keep the admitted slice tied to the real `ServiceCell<T>` helper pair in
   `kernel/src/boot/boot_update.mc` or an equivalently narrow checked-in
   helper family if the final spend site proves cleaner.
3. Remove the current per-service wrapper boilerplate only where the admitted
   helper pair now compiles honestly.
4. Add focused MIR and executable regressions for the admitted helper shape.
5. Preserve explicit rejection for imported, inferred, or indirect-only generic
   body shapes outside this slice.

Forbidden changes
-----------------
- do not widen into a general generic framework just because one helper family
  now works
- do not move executable specialization policy into sema or the driver
- do not silently accept imported or indirect-only value-shaped generic bodies
- do not bundle record-construction ergonomics or other source cleanup into the
  same phase beyond the direct `ServiceCell<T>` spend site
- do not weaken MIR validation just to admit unsupported generic bodies

Phase role in the larger sequence
---------------------------------
- This phase converts the Phase 252 chosen shape into a real landed compiler
  capability.
- It prepares Phase 256 by making the same-module helper family honest before
  any imported generic-body follow-through is considered.
- It also removes one concrete source distortion before Band A turns toward
  descriptor tables, dispatch tables, and later app-facing compiler pressure.

Preferred implementation shape
------------------------------
- Follow the Phase 252 decision-log shape: specialize only direct `symbol_ref`
  and direct `call` targets when the generic body ABI truly depends on
  concrete type shape.
- Keep erased-ABI generic bodies on the existing path.
- Keep parser and sema stable unless one narrow diagnostic or query hook is
  required by the specialization path.
- Spend the landing in `kernel/src/boot/boot_update.mc` using explicit type
  arguments rather than reopening inference.
- Keep backend work limited to lowering already-specialized MIR truth.
