Repository status note
----------------------
- The repository can now honestly claim one bounded running Veya kernel and
  one bounded interactive service system with explicit authority, bounded
  failure, bounded IPC composition, bounded UART ingress, one shell-owned
  command path, one retained log owner, one retained key-value owner, and one
  explicit init-owned restart path over the real kernel artifact.
- That result matters because it moves the repository past abstract model
  viability and into one small system that can be used repeatedly and can fail
  in a user-visible way.
- The strongest next question is no longer whether one more service can be
  added. It is whether the repository now understands how to build this kind
  of system cleanly, repeatedly, and without smuggling framework behavior back
  into the kernel or into ad hoc helpers.
- That pressure now includes the structural question of whether
  `kernel/src/main.mc` can be reduced to a thin orchestrator without turning
  the simplification into a framework or compiler-expansion excuse.
- The risk on one side is staying in one-off proof mode and never extracting a
  reusable service shape from the landed shell, log, and key-value work.
- The risk on the other side is jumping straight to filesystem, networking,
  scheduling richness, or other platform breadth and hiding structural flaws
  under complexity before the current system pattern is honest.

What this roadmap is trying to answer
-------------------------------------
- What should come after the completed Phase 145 restart-and-usage audit if
  the next goal is not more services but a repeatable way to build systems in
  Veya?
- Which next phases should classify message-shape, authority, restart, and
  composition friction before any broader subsystem ambition?
- How should the repository shift from "one small system is usable" to "one
  small system can be rebuilt more naturally the second time"?

Core rule
---------
- Treat the next wave primarily as system-pattern extraction, not as service
  accretion.
- Reuse the landed shell-plus-log-plus-key-value system as the evidence source
  rather than widening immediately to a second larger subsystem.
- Keep policy in init and service owners. Kernel changes remain justified only
  when repeated real usage exposes a concrete owner gap.
- Keep the Modern C core, ABI, target, runtime, and `hal` surfaces closed
  unless the repeated system-building work proves one specific owner
  deficiency.
- Prefer repeated use, repeated rebuild, and explicit friction classification
  over feature count.

Repeatability rule
------------------
Use this sanity check on every proposed post-Phase-149 change:

- helps rebuild the same small system more clearly, more explicitly, or with
  less user-space awkwardness while preserving ownership: admissible
- adds another service, discovery layer, broker, persistence framework, or
  orchestration layer before the current system shape is extracted: wrong phase
- widens to filesystem, networking, broader scheduling policy, or platform
  breadth before the current service pattern is stable: wrong phase

Kernel invariants
-----------------
These invariants apply across all phases in this roadmap:

- no subsystem may require knowledge of another subsystem's internal state in
  order to perform its normal owned behavior
- all cross-subsystem interaction must pass through explicit interfaces,
  ownership-visible helpers, or dependency-visible boundaries
- no new syscall, namespace hook, discovery hook, or reliability hook should
  be added without an explicit argument that the current owner is insufficient
- no phase in this wave should turn the kernel into a broker, shell runtime,
  restart supervisor, persistence layer, or policy engine
- no larger platform surface should be admitted until the repository can show
  that one small system has a stable shape that survives a clean rebuild

Chosen next direction
---------------------
The most practical next step is to turn the landed interactive-service slice
into a repeatable system pattern.

Chosen wave shape:

1. extract the common service structure from shell, log, and key-value owners
2. re-audit IPC under real repeated command and reply traffic rather than only
   under synthetic composition pressure
3. re-audit authority ergonomics under repeated service reuse rather than only
   under correctness pressure
4. formalize restartable-service contracts as an explicit user-space pattern
   rather than as a one-off audit result
5. rebuild the same small system cleanly using only the surviving lessons

The target is not a service framework.
The target is one repeatable way to build a small Veya system without
hiding ownership or failure under convenience abstractions.

Friction classification rule
---------------------------
When a later phase exposes awkwardness, classify it immediately:

- Category A - service design issue: fix in user space and keep the kernel
  unchanged
- Category B - missing small helper: admit one narrow library or helper only
  if repeated service code proves it is stable and ownership-visible
- Category C - kernel boundary gap: reopen the kernel only if the current
  owner is clearly insufficient under repeated real usage
- Category D - language ergonomics gap: reopen language or compiler work only
  after repetition proves the pain is structural rather than incidental

Primary friction fronts to watch
--------------------------------
These are the strongest likely post-Phase-145 signals:

- message shape friction in shell-to-service and service-to-service request or
  reply paths
- command syntax versus service semantics bleed between shell and service
  owners
- state-mutation clarity for retained log and key-value behavior over time
- ownership fatigue in repeated request, reply, and buffer construction paths
- awkward cross-service composition glue between shell, log, and key-value
- restart-semantics ambiguity around retry, idempotency, and visible reset
  truth
- addressing or identity awkwardness across fixed restarts without widening to
  discovery
- inconsistent reply shape between human-readable and structured service-owned
  answers
- backpressure or hidden queue pressure becoming user-visible under repeated
  use
- the general "this works but it is annoying" signal, which should be treated
  as design evidence rather than as noise

Roadmap bands
-------------

Band A - Extract The Minimal Service Shape
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
The first move should be to determine whether shell, log, and key-value now
share a stable structural pattern or whether the repository is still relying
on phase-local ad hoc designs.

Likely themes:
- lifecycle shape such as init, serve, fail, and restart consequences
- message-role shape such as request or reply and observation paths
- authority possession and transfer shape
- fixed separation between shell syntax ownership and service semantic
  ownership

Why this band comes first:
- after Phase 145, the repository has enough real usage truth to extract a
  pattern, and failing to do so now risks normalizing accidental complexity.

Phase 146 -- Service Shape Consolidation
- Phase 146 is now concrete as one bounded service-shape consolidation step with one published minimal service-shape statement, one explicit shell-syntax-versus-service-semantics boundary, one explicit intentional-difference classification, and compiler or framework surfaces kept closed.

Preferred proof points:
- one canonical service-shape description exists and is grounded in the landed
  shell, log, and key-value implementations
- at least one ad hoc difference is either justified as service-local or
  rejected as accidental complexity
- the shell syntax owner versus service semantic owner boundary is stated
  clearly rather than implied

Band B - Re-Audit IPC Under Real Usage
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Once the landed service shape is described, the repository should revisit IPC from
the perspective of repeated human-meaningful use rather than from synthetic
stress alone.

Likely themes:
- awkward request or reply construction
- mismatched request and reply shapes across services
- unnecessary copy or glue pressure
- compact command encoding pressure revealed by repeated use

Why this band comes next:
- message-shape friction is likely to be the first structural limit exposed by
  the landed interactive system.

Phase 147 -- IPC Shape Audit Under Real Usage
- Phase 147 is now concrete as one bounded IPC-shape audit under real usage with one repeated mixed shell workflow, one explicit service-local reply-shape judgment, and transport or compiler surfaces kept closed.

Preferred proof points:
- at least one concrete awkward request or reply path is classified honestly
- any admitted helper remains ownership-visible and service-local rather than
  transport-magical
- the repository either confirms the current IPC shape is sufficient or names
  one tightly bounded reason it is not

Band C - Re-Audit Authority Ergonomics Under Reuse
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Correct authority transfer has already been proved. The next question is
whether explicit authority remains comfortable when service code is written,
read, and repeated by humans over time.

Likely themes:
- authority passing fatigue
- overscoping pressure in helper paths
- reply construction and ownership handoff repetition
- small helper versus broader language-pressure classification

Why this band comes after IPC shape:
- it is easier to judge authority ergonomics once the message shapes they move
  through are already clearer.

Phase 148 -- Authority Ergonomics Under Reuse
- Phase 148 is now concrete as one bounded authority-ergonomics audit with one repeated shell-plus-restart workflow, one narrow capability-local authority-validation helper, and kernel or compiler surfaces kept closed.

Preferred proof points:
- at least one repeated authority-handling path is measured against the Phase
  145 system rather than against a synthetic micro-example
- any admitted convenience remains explicit about ownership transfer and scope
- the repository can name what is merely verbose versus what is structurally
  awkward

Band D - Formalize Restartable Service Contracts
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phase 145 proved one bounded restart path. The next move should be to turn
that one-off truth into an explicit user-space service contract rather than to
leave restart semantics implicit or phase-local.

Likely themes:
- what callers may assume after service death
- what post-restart state truth must remain visible
- retry, idempotency, and explicit reset classification
- fixed restart wiring versus broader rebinding pressure

Why this band matters:
- restart ambiguity is the most likely reliability fault line in a small
  stateful system, and it should be clarified before any broader subsystem is
  allowed to depend on the current service pattern.

Phase 149 -- Restart Semantics As A First-Class Pattern
- Phase 149 is now concrete as one bounded restart-contract audit with one
  explicit caller retry obligation, one explicit post-restart reset-state
  statement, one explicit init-owned restart-policy boundary, and compiler or
  recovery-framework surfaces kept closed.

Preferred proof points:
- one restartable-service contract is stated in terms of observable behavior
  rather than implementation accident
- the repository names at least one idempotency or retry assumption that must
  remain explicit
- the fixed-wiring restart result from Phase 145 is preserved as evidence, not
  abstracted away

Band E - Rebuild One System Cleanly
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
The last move in this wave should be to test whether the extracted pattern is
real by rebuilding the same small system with the surviving lessons and with
less ad hoc scaffolding.

Likely themes:
- remove experimental glue that no longer survives the earlier audits
- keep only the stable service pattern, helper, and contract decisions
- compare the rebuilt system against the first version for clarity and owner
  honesty

Why this band comes last:
- a clean rebuild is the strongest discriminator between a real pattern and a
  one-off pile of locally correct fixes.

Phase 150 -- One System, Rebuilt Cleanly
- Phase 150 is now concrete as one bounded clean rebuild of the same
  shell-plus-log-plus-key-value system, with the earlier manual five-state
  threading removed from the root proof path and replaced by one bounded
  rebuilt-system helper while compiler and framework surfaces stay closed.

Preferred proof points:
- the rebuilt system removes at least one earlier ad hoc pattern rather than
  merely rearranging it
- the rebuilt system preserves the admitted bounded behavior while improving
  explanation or structure
- if the second version is not simpler, the repository records that as a model
  problem rather than hiding it behind more features

What should naturally emerge from this wave
-------------------------------------------
If the repository follows this roadmap honestly, the next useful results
should be concrete evidence about:

- whether Veya now has one canonical service pattern rather than a set of
  phase-local service implementations
- whether the current IPC shape remains stable under repeated real usage or
  needs one narrow helper boundary
- whether the landed restart contract stays explicit enough for later
  subsystem work to rely on without hidden policy
- whether any language, library, or kernel ergonomics work is justified by
  repeated implementation pain rather than by taste
- whether the system can be built twice with the second version being simpler
  than the first

Likely ordering bias
--------------------
If the repository keeps following the same discipline, the safest order is:

1. extract the service pattern before widening service scope
2. revisit IPC only through real repeated usage of the existing small system
3. revisit authority ergonomics only after message-shape truth is clearer
4. formalize restartable-service contracts before allowing broader subsystem
   dependence on the current pattern
5. rebuild the same system cleanly before considering filesystem, networking,
   richer scheduling, or any broader platform wave

That ordering keeps the repository from confusing one-off implementation
success with a real system-building method.

Post-Phase-151 continuation
---------------------------
Phase 146 through Phase 151 answered the first repeatability question:

- whether the repository can extract one honest service pattern from the
  landed shell-plus-log-plus-key-value system
- whether that pattern survives one rebuild and one root-file
  simplification without smuggling framework behavior back into the kernel or
  into helper layers

If that wave is kept honest, the next question changes.
The repository should no longer ask only whether the same small system can be
rebuilt more cleanly.
It should ask whether the extracted pattern remains stable once the system is
varied slightly and then exposed to more realistic pressure.

The new center of gravity is therefore:

- first: stable under one controlled variation
- then: stable under real multi-actor, long-lived, failure, and observation
  pressure

New meta-rule
--------------
Treat Phase 152 through Phase 160 as a pressure wave, not as a breadth wave.

- do not answer this wave by adding more services just to increase surface
  area
- do not jump to filesystem, networking, multicore, dynamic discovery,
  broader scheduling policy, or a service framework
- do use one small difference or one honest pressure source at a time to
  expose whether the current model survives without new hidden rules

Recurring abstraction rule
--------------------------
Apply this invariant across every post-Phase-151 step:

- no hidden generalization

For any helper, abstraction, or reusable pattern admitted during this wave,
the repository should be able to show:

- what concrete duplication or friction it removes
- that it does not hide ownership
- that it does not hide failure
- that it does not smuggle policy into the kernel, shell, or helper layer

If a proposed abstraction fails that check, the phase should reject it and
record the friction directly instead of normalizing a second-system rewrite.

Band F - Controlled Variation
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
The first post-151 move should be one bounded variation over the rebuilt
system rather than immediate breadth.

Core question:
- does the extracted service pattern survive when the system is slightly
  different, or was the earlier result only a well-cleaned special case?

Admissible variation examples:
- one new service shape that is structurally different from the current trio,
  such as a stateless request-reply owner or a bounded multi-reply owner
- one deeper service-to-service chain rather than only shell-to-service
  interaction
- one simulated multi-client interaction over the existing fixed-wiring model

What this band should test:
- whether service shape still reads as one pattern rather than one frozen
  special case
- whether current IPC and authority handling remain honest under slightly
  different traffic
- whether the repository can add one difference without inventing new routing,
  discovery, or framework rules

Phase 152 -- Pattern Stability Under One Controlled Variation
- introduce one small system variation and classify whether the current
  service pattern still holds without new hidden rules

Preferred proof points:
- one specific variation is admitted and justified as the smallest useful
  discriminator
- the repository can name which existing pattern elements survive unchanged
  and which one difference becomes explicit
- any newly exposed awkwardness is classified as service-local, helper-local,
  kernel-boundary, or language-ergonomics pressure rather than hidden behind
  new abstraction

Band G - Composition Without Glue Explosion
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Once one variation exists, the next question is whether services compose
honestly or only through growing translation glue.

Core question:
- can services compose without becoming a mini-framework?

Likely signals:
- repetitive adapters between service-local request and reply shapes
- translation helpers that start to behave like implicit conventions
- shell-owned or init-owned orchestration glue that no longer reads as a
  clear owner-local policy decision

Phase 153 -- Service Composition Shape Audit
- measure whether cross-service composition still fits explicit owner-local
  glue or now exposes one honest need for a minimal composition primitive

Preferred proof points:
- at least one awkward composition path is classified concretely
- the repository either confirms that explicit composition remains the honest
  model or names one tightly bounded composition primitive candidate
- no broker, orchestration framework, or hidden routing policy is admitted

Band H - Identity And Addressing Semantics
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
The rebuilt interactive slice already hints at addressing and identity
awkwardness across fixed restarts.
The next move should be to clarify the semantics already being relied on,
without widening to dynamic discovery or registries.

Core question:
- what does it mean to refer to a service across time in the current model?

Likely themes:
- fixed wiring versus stable identity
- positional, capability-derived, or explicitly named reference meaning
- what survives restart and what must be reacquired explicitly

Phase 154 -- Service Identity And Addressing Semantics
- publish one explicit identity and addressing statement for the current
  fixed-wiring service model and classify any restart-time ambiguity

Preferred proof points:
- at least one current identity assumption is made explicit rather than
  remaining phase-local folklore
- the repository can state what remains stable across restart and what does
  not
- no discovery service, registry, or dynamic rebinding mechanism is admitted

Band I - Temporal Semantics And Backpressure Truth
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
After identity is clearer, the repository should make time and queue pressure
explicit rather than letting them remain accidental properties of the current
small system.

Core question:
- what happens when service latency, queued requests, delayed replies, or
  repeated use make temporal behavior visible?

Likely themes:
- ordering guarantees versus accidental ordering
- slow-service consequences
- hidden queue pressure becoming user-visible
- late or out-of-order replies and what the current model says about them

Phase 155 -- Temporal Semantics And Backpressure Audit
- classify the current model's ordering, delay, and queue-pressure truth
  without widening to a scheduler rewrite or an async framework

Preferred proof points:
- one temporal awkwardness is stated in user-visible terms rather than in
  implementation-only terms
- the repository can distinguish guaranteed ordering from accidental current
  behavior
- any visible backpressure consequence remains explicit and owner-local

Band J - Real System Pressure
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
If Phase 152 through Phase 155 remain honest, the next frontier is no longer
shape extraction.
It is whether that shape survives when the system stops being quiet.

The next steps should therefore expose the existing model to a small set of
real-system pressures without widening into platform breadth.

Band J1 - Multi-Client Reality
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
The current system is still effectively organized around one shell-owned flow.
The next question is what breaks first when two independent clients interact
with the same fixed-wiring services.

Phase 156 -- Multi-Client Interaction Audit

Preferred proof points:

- the repository names the first assumption that fails under interleaving
- no scheduler expansion, priority system, or wider concurrency model is

Band J2 - Long-Lived System Coherence
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
Restart has already been proved.
The next question is whether the current service meanings remain coherent over
time rather than only across one reboot boundary.

Phase 157 -- Long-Lived System Coherence Audit
- measure whether repeated long-lived use keeps state, logs, and restart

Preferred proof points:
- one long-lived interaction sequence is audited for semantic drift
- the repository can state whether restart means the same thing after many

Band J3 - Cross-Service Failure Semantics
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
The repository already has bounded restart contracts.
The next move is to check whether failure is understandable and usable across

Phase 158 -- Cross-Service Failure Semantics Consistency
- compare the visible failure contracts of the admitted services and classify

Preferred proof points:
- at least one cross-service failure mismatch is either justified or rejected

Band J4 - Observability Without Frameworks
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Phase 159 -- Observability As A System Property

Preferred proof points:
- one debugging workflow is supported by truthful system-visible observation

- the repository can distinguish implementation detail from user-meaningful

- no cross-cutting debug-print culture or broader telemetry platform is

Band J5 - First Model Boundary Identification
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
If the earlier pressure steps are honest, at least one problem should start to
look unnatural in the current model.
That is a success condition, not a failure.

Phase 160 -- First Model Boundary Identification
- name the first repeated problem that the current model cannot solve cleanly

Preferred proof points:
- the repository can point to one repeated pressure that the current model

- the boundary is stated before any fix is admitted
- any future expansion candidate is tied to repeated evidence rather than to

What Phase 160 should earn
--------------------------
Only after this pressure wave should the repository consider one carefully
chosen expansion axis, such as:

- state generalization pressure that might justify filesystem work
- identity or addressing pressure that might justify remote or network-facing
  work
- fairness or queue-pressure pressure that might justify scheduler work
- repeated authority-ergonomics pressure that might justify a capability-model

The admission rule stays the same:

- before any new subsystem is added, the repository should be able to say

Non-goals
---------
- treating this roadmap as a committed execution plan
- adding more services merely to increase surface area
- widening into filesystem, networking, richer timer or scheduling policy, or
  multicore work during this wave
- introducing dynamic discovery, a broker, a message bus, or a general shell
  or service framework
- turning the kernel into a supervisor, persistence layer, or policy engine
- reopening the Modern C core before repeated system-building work proves a
  concrete owner gap
- claiming broader platform maturity from one cleaned-up small system

Interpretation guide
--------------------
- Treat this as a rough map, not a contract.
- The readiness and support statements remain the canonical statement of
  present truth.
- Phase 141 through Phase 145 now form the historical bridge from stressed
  service composition to one small usable interactive system.
- The next honest question is not whether the repository can add another
  service. It is whether it now knows how to build the same kind of system
  better on the second pass.
- If this wave exposes a real kernel, language, ABI, or toolchain owner gap,
  reopen only the narrowest owning layer at that proof boundary.
