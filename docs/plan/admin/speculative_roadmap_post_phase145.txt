Repository status note
----------------------
- The repository can now honestly claim one bounded running Canopus kernel and
  one bounded interactive service system with explicit authority, bounded
  failure, bounded IPC composition, bounded UART ingress, one shell-owned
  command path, one retained log owner, one retained key-value owner, and one
  explicit init-owned restart path over the real kernel artifact.
- That result matters because it moves the repository past abstract model
  viability and into one small system that can be used repeatedly and can fail
  in a user-visible way.
- The strongest next question is no longer whether one more service can be
  added. It is whether the repository now understands how to build this kind
  of system cleanly, repeatedly, and without smuggling framework behavior back
  into the kernel or into ad hoc helpers.
- That pressure now includes the structural question of whether
  `kernel/src/main.mc` can be reduced to a thin orchestrator without turning
  the simplification into a framework or compiler-expansion excuse.
- The risk on one side is staying in one-off proof mode and never extracting a
  reusable service shape from the landed shell, log, and key-value work.
- The risk on the other side is jumping straight to filesystem, networking,
  scheduling richness, or other platform breadth and hiding structural flaws
  under complexity before the current system pattern is honest.

What this roadmap is trying to answer
-------------------------------------
- What should come after the completed Phase 145 restart-and-usage audit if
  the next goal is not more services but a repeatable way to build systems in
  Canopus?
- Which next phases should classify message-shape, authority, restart, and
  composition friction before any broader subsystem ambition?
- How should the repository shift from "one small system is usable" to "one
  small system can be rebuilt more naturally the second time"?

Core rule
---------
- Treat the next wave primarily as system-pattern extraction, not as service
  accretion.
- Reuse the landed shell-plus-log-plus-key-value system as the evidence source
  rather than widening immediately to a second larger subsystem.
- Keep policy in init and service owners. Kernel changes remain justified only
  when repeated real usage exposes a concrete owner gap.
- Keep the Modern C core, ABI, target, runtime, and `hal` surfaces closed
  unless the repeated system-building work proves one specific owner
  deficiency.
- Prefer repeated use, repeated rebuild, and explicit friction classification
  over feature count.

Repeatability rule
------------------
Use this sanity check on every proposed post-Phase-149 change:

- helps rebuild the same small system more clearly, more explicitly, or with
  less user-space awkwardness while preserving ownership: admissible
- adds another service, discovery layer, broker, persistence framework, or
  orchestration layer before the current system shape is extracted: wrong phase
- widens to filesystem, networking, broader scheduling policy, or platform
  breadth before the current service pattern is stable: wrong phase

Kernel invariants
-----------------
These invariants apply across all phases in this roadmap:

- no subsystem may require knowledge of another subsystem's internal state in
  order to perform its normal owned behavior
- all cross-subsystem interaction must pass through explicit interfaces,
  ownership-visible helpers, or dependency-visible boundaries
- no new syscall, namespace hook, discovery hook, or reliability hook should
  be added without an explicit argument that the current owner is insufficient
- no phase in this wave should turn the kernel into a broker, shell runtime,
  restart supervisor, persistence layer, or policy engine
- no larger platform surface should be admitted until the repository can show
  that one small system has a stable shape that survives a clean rebuild

Chosen next direction
---------------------
The most practical next step is to turn the landed interactive-service slice
into a repeatable system pattern.

Chosen wave shape:

1. extract the common service structure from shell, log, and key-value owners
2. re-audit IPC under real repeated command and reply traffic rather than only
   under synthetic composition pressure
3. re-audit authority ergonomics under repeated service reuse rather than only
   under correctness pressure
4. formalize restartable-service contracts as an explicit user-space pattern
   rather than as a one-off audit result
5. rebuild the same small system cleanly using only the surviving lessons

The target is not a service framework.
The target is one repeatable way to build a small Canopus system without
hiding ownership or failure under convenience abstractions.

Friction classification rule
---------------------------
When a later phase exposes awkwardness, classify it immediately:

- Category A - service design issue: fix in user space and keep the kernel
  unchanged
- Category B - missing small helper: admit one narrow library or helper only
  if repeated service code proves it is stable and ownership-visible
- Category C - kernel boundary gap: reopen the kernel only if the current
  owner is clearly insufficient under repeated real usage
- Category D - language ergonomics gap: reopen language or compiler work only
  after repetition proves the pain is structural rather than incidental

Primary friction fronts to watch
--------------------------------
These are the strongest likely post-Phase-145 signals:

- message shape friction in shell-to-service and service-to-service request or
  reply paths
- command syntax versus service semantics bleed between shell and service
  owners
- state-mutation clarity for retained log and key-value behavior over time
- ownership fatigue in repeated request, reply, and buffer construction paths
- awkward cross-service composition glue between shell, log, and key-value
- restart-semantics ambiguity around retry, idempotency, and visible reset
  truth
- addressing or identity awkwardness across fixed restarts without widening to
  discovery
- inconsistent reply shape between human-readable and structured service-owned
  answers
- backpressure or hidden queue pressure becoming user-visible under repeated
  use
- the general "this works but it is annoying" signal, which should be treated
  as design evidence rather than as noise

Roadmap bands
-------------

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
The first move should be to determine whether shell, log, and key-value now
share a stable structural pattern or whether the repository is still relying
on phase-local ad hoc designs.

Likely themes:
- lifecycle shape such as init, serve, fail, and restart consequences
- message-role shape such as request or reply and observation paths
- authority possession and transfer shape
- fixed separation between shell syntax ownership and service semantic
  ownership

Why this band comes first:
- after Phase 145, the repository has enough real usage truth to extract a
  pattern, and failing to do so now risks normalizing accidental complexity.

Phase 155 -- Temporal Semantics and Backpressure Audit (LANDED)
- Phase 155 made backpressure explicit across all fixed-capacity services:
  - `log_service.handle` returns `Exhausted` when the 4-entry buffer is full
    and an append is requested; state is unchanged on `Exhausted`
  - `kv_service.handle` returns `Exhausted` when the 4-key store is full
    and a new key is requested; overwriting an existing key is always `Ok`
  - the kv-write advisory log marker silently discards its own `Exhausted`
    reply at the boot dispatcher; the kv caller sees only the kv reply
- Ordering guarantee stated: `kernel_dispatch_step` is sequential and
  synchronous; a set on step N is always visible to a get on step N+1
- No scheduler, async framework, dynamic queue growth, timing API, or
  compiler surface was opened
- Validated by `tests/system/kernel_reset_lane_temporal_backpressure/`

Phase 156 -- Multi-Client Interaction Audit (LANDED)
- Phase 156 exposed the first ordering and isolation assumptions visible
  under two-client interleaving:
  - kv_service and log_service hold globally-shared state; a write from
    any source_pid is immediately visible to all subsequent dispatch steps
    regardless of which source_pid submits them (no per-client namespace)
  - reply routing is by call-return, not by source_pid; the effect returned
    by `kernel_dispatch_step` belongs to whichever call owns that return
    value; real concurrency would require per-client reply mailboxes
  - backpressure is per-service capacity, not per-client quota; a full
    buffer from client A leaves no room for client B
  - last writer wins with no read-your-writes isolation between clients
- The dispatcher remains purely sequential; these assumptions are only
  exposed by this audit, not fixed — they name what would fail first under
  real concurrent clients
- No scheduler, concurrency model, per-client session concept, reply
  mailbox, or compiler surface was opened
- `kernel/src/main.mc` gained `make_serial_command_pid` and one multi-client
  wiring step; no other kernel service files were changed
- Validated by `tests/system/kernel_reset_lane_multi_client/`

Phase 157 -- Long-Lived System Coherence Audit (LANDED)
- Phase 157 proved that service meaning remains coherent over repeated
  interactions, not just across one short proof path:
  - kv state is stable across repeated reads: no silent expiry or drift
  - log overwrite-on-full is permanent policy: tail still returns the
    original 4 entries; no silent loss or silent growth occurs
  - restart resets all service state to the empty baseline with no hidden
    residue from prior interactions
  - the kv-write advisory log marker is coherent: log saturation does not
    affect kv state or kv reply status
  - repeated overwrites of the same kv key never overflow the store
- No scheduler, persistence layer, service supervisor, or compiler surface
  was opened
- Validated by `tests/system/kernel_reset_lane_long_lived_coherence/`

Phase 158 -- Cross-Service Failure Semantics Consistency (LANDED)
- Phase 158 classified cross-service failure semantics across the admitted
  service set and named one intentional inconsistency:
  - Advisory log drop (intentional difference): when the log buffer is
    full, a kv write still returns Ok; the dispatcher silently discards
    the advisory log marker; the kv caller has no signal that the audit
    record was lost; this is documented as the one named cross-service
    failure inconsistency — kv success does not imply log record written
  - Exhausted is consistent: both log full and kv full return Exhausted
    with no state change; capacity overflow uses the same status code
    across service boundaries
  - InvalidArgument is consistent: kv get for a missing key and a shell
    invalid command both return InvalidArgument; the same status signals
    "request cannot be fulfilled as stated" across service boundaries
  - Exhausted is sticky: a full log stays full regardless of kv activity;
    services own their capacity independently
- No recovery framework, retry policy, two-phase commit, log-full callback,
  or compiler surface was opened
- Advisory log drop is explicitly intentional and classified in
  `docs/plan/decision_log.txt`
- `kernel/src/scenarios.mc` gained `run_cross_service_failure`; no service
  files were changed
- Validated by `tests/system/kernel_reset_lane_cross_service_failure/`

Phase 159 -- Observability As A System Property (LANDED)
- Phase 159 defined the minimum user-meaningful observation surface required
  by the admitted system:
  - kv_count (KC!!) is system-visible truth: the user can determine how many
    kv entries exist without knowing the stored keys; reply is Ok with
    count in reply_payload_len; this closed the one gap Phase 155-158 left
    open (log was already fully observable via LT!!, but kv occupancy was
    opaque without prior key knowledge)
  - kv_get (KG <key> !) and log_tail (LT!!) are confirmed as the only other
    system-visible observation queries
  - KV_WRITE_LOG_MARKER byte value and internal endpoint IDs are classified
    as implementation detail, not user contracts
  - observation is non-mutating: count and tail queries return the same
    values on repeated calls with no state change
- No telemetry stack, logging framework, tracing platform, or
  cross-cutting debug substrate was opened
- `kv_service.handle` already returned count from the zero-length payload
  path; the `serial_protocol.encode_kv_count()` function and
  `shell_service` route were added for the public-protocol surface
- `kernel/src/scenarios.mc` gained `run_observability_inspection`; no
  service module files were changed
- Validated by `tests/system/kernel_reset_lane_observability/`
- Phase 159 also confirmed the advisory-silencing gap visible since
  Phase 158: kv count is observable, but correlation between kv write
  count and log entry count breaks once the log is full; that gap is
  the named model boundary addressed in Phase 160

Phase 160 -- First Model Boundary Identification (LANDED)
- Phase 160 named the first repeated model failure from the Phase 152-159
  evidence and deferred its fix:
  - Named boundary: advisory audit silencing is invisible to the kv caller.
    When the log is at capacity, a kv write returns Ok but its advisory
    audit marker is silently discarded by the boot dispatcher.  The kv
    caller has no protocol-level signal that the audit trail was lost.
    The effect model has no witness for advisory send outcomes.
  - Repeated evidence: this gap was first visible in the Phase 152 restart
    variation (audit trail interrupted during selective restart), confirmed
    in Phase 156 (no per-client audit isolation), surfaced as an explicit
    inconsistency in Phase 158 (advisory log drop named as the one
    cross-service failure difference), and revealed as unobservable in
    Phase 159 (count diverges from log length but the caller cannot tell
    which writes were audited)
  - Narrowest next expansion axis named but not admitted: per-send delivery
    witnesses; a witness would let the dispatcher report Dropped to the kv
    caller when the advisory send was not delivered; this closes the gap
    without a recovery framework or platform; no implementation admitted
    in this phase
  - Compiler, sema, MIR, backend, ABI, target, runtime, and hal surfaces
    were kept closed
- `kernel/src/scenarios.mc` gained `run_model_boundary`; no service module
  files, dispatcher logic, or protocol layers were changed
- Boundary is grounded in the real running artifact:
  - `kernel/src/main.mc` calls `run_model_boundary` after
    `run_observability_inspection`; the running kernel proves the gap
    exists by confirming two kv writes succeed and log length stays at 4
  - `tests/system/kernel_reset_lane_model_boundary/` provides a standalone
    behavior-first fixture proving the same boundary without chaining on
    phase history
- Validated by `tests/system/kernel_reset_lane_model_boundary/`

Phase 161 -- Per-Send Delivery Witness (LANDED)
- Phase 161 implemented the narrowest expansion axis identified in Phase 160:
  per-send delivery witnesses.
  - Added `send_dropped: u32` to the Effect struct in service_effect.mc as
    the last field; all four Effect constructors carry send_dropped: 0.
  - Added `effect_mark_send_dropped()` and `effect_send_dropped()` to
    service_effect.mc.
  - In boot.mc kv dispatch: when the advisory log append returns Exhausted,
    the kv reply effect is marked with send_dropped = 1 before returning.
  - In kernel_dispatch_step: send_dropped propagates from the inner kv
    effect to the outer serial reply so callers routing through the serial
    path can observe advisory drop status.
  - The change is purely additive: existing callers that do not check
    effect_send_dropped() observe no behavior change.
  - No recovery, retry, two-phase commit, or callback mechanism admitted.

Phase 162 -- Orchestration Cleanup: Kill Mixed Envelopes and Per-Service Wrappers (LANDED)
- Phase 162 removed the two structural impurities that remained in boot.mc
  after Phase 161.
  - Removed kernel_dispatch_message: split into kernel_dispatch_kv (kv
    advisory dispatch as a named concept) and kernel_dispatch_leaf (routes
    log/kv/default; called by execute_effect).
  - Removed serial_path_post_dispatch: the serial arm now owns its complete
    reply assembly in kernel_dispatch_serial, which runs path_step, chases
    the shell→leaf chain, commits the resolved reply via path_commit_reply,
    and assembles the serial effect with event markers and send_dropped.
  - Removed message_to_observation: no longer needed.
  - execute_effect now calls kernel_dispatch_leaf instead of
    kernel_dispatch_message; this eliminates the circular forward-reference
    and reflects the topology invariant that all Send effects target leaf
    services only.
  - Added path_commit_reply to serial_shell_path.mc: pure function that
    commits a resolved inner effect into the path reply slot.
  - kernel_dispatch_step is now a thin router: no endpoint-specific
    post-processing at the step level.
  - No behavior change; all 31 tests pass.
  - No new modules admitted.  Compiler, sema, MIR, backend, ABI, target,
    runtime, and hal surfaces remain closed.
- `kernel/src/scenarios.mc` now owns one scripted event loop rather than a
  chain of named `run_*` helpers; the delivery-witness overwrite step is part
  of that loop and still proves `send_dropped = 1` when the log is full.
- `kernel/src/main.mc` now initializes state and calls `scenarios.run(&state)`.
- `tests/system/kernel_reset_lane_delivery_witness/` proves both cases:
  case A (log has space) → send_dropped = 0;
  case B (log full)      → send_dropped = 1.
- Validated by `tests/system/kernel_reset_lane_delivery_witness/`

What should naturally emerge from this wave
-------------------------------------------
If the repository follows this roadmap honestly, the next useful results
should be concrete evidence about:

- whether Canopus now has one canonical service pattern rather than a set of
  phase-local service implementations
- whether the current IPC shape remains stable under repeated real usage or
  needs one narrow helper boundary
- whether the landed restart contract stays explicit enough for later
  subsystem work to rely on without hidden policy
- whether any language, library, or kernel ergonomics work is justified by
  repeated implementation pain rather than by taste
- whether the system can be built twice with the second version being simpler
  than the first
- whether that rebuilt pattern still holds when one retained-state owner is
  varied without broader system growth

Likely ordering bias
--------------------
If the repository keeps following the same discipline, the safest order is:

1. extract the service pattern before widening service scope
2. revisit IPC only through real repeated usage of the existing small system
3. revisit authority ergonomics only after message-shape truth is clearer
4. formalize restartable-service contracts before allowing broader subsystem
   dependence on the current pattern
5. rebuild the same system cleanly before considering filesystem, networking,
  richer scheduling, or any broader platform wave
6. vary one retained-state owner before widening into composition, identity,
  or temporal follow-through

That ordering keeps the repository from confusing one-off implementation
success with a real system-building method.

Non-goals
---------
- treating this roadmap as a committed execution plan
- adding more services merely to increase surface area
- widening into filesystem, networking, richer timer or scheduling policy, or
  multicore work during this wave
- introducing dynamic discovery, a broker, a message bus, or a general shell
  or service framework
- turning the kernel into a supervisor, persistence layer, or policy engine
- reopening the Modern C core before repeated system-building work proves a
  concrete owner gap
- claiming broader platform maturity from one cleaned-up small system
