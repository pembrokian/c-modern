Veya Running-System Support Statement
========================================

Status
------
PUBLISHED

Purpose
-------
Record the current admitted Veya running-system slice separately from the
broader position note so the repository can state exactly what the work
through Phase 177 does and does not support.

This is the current refreshed running-system statement.  It supersedes the
earlier Phase 120 through Phase 121 statement and reflects the post-Phase-177
landing.

This note is intentionally narrower than the current-position assessment.
It does not claim a general Veya platform, dynamic loading, dynamic
service discovery, a service manager, broader freestanding workflow support,
multicore, or wider target-family admission.

Core rule
---------
- Keep this running-system support statement separate from the broader
  position note.
- State only the admitted Phase 109 through Phase 177 running-system slice,
  not a larger platform ambition.
- Keep service policy init-owned and explicit.
- Reopen broader running-system claims only when a later kernel proof lands
  them honestly.

Admitted support posture: current running-system support statement; not a
general Veya platform or broader freestanding workflow guarantee.

Usage and audit
---------------
Build from source on the supported host:

```sh
cmake -S . -B build/debug -DCMAKE_BUILD_TYPE=Debug
cmake --build build/debug -j4
```

The produced compiler entrypoint on the supported host is:

```sh
build/debug/bin/mc
```

Supported command pattern for the admitted running-system slice:

```sh
build/debug/bin/mc build --project kernel/build.toml --target kernel --build-dir <dir>
```

Repo-owned regression entrypoint for this support statement:

```sh
ctest --test-dir build/debug -R mc_tool_workflow_unit --output-on-failure
```

Aggregate public-cut smoke covering both the hosted and running-system
admitted slices:

```sh
make public-cut-smoke
```

Artifact expectations
---------------------
- `build/debug/bin/mc` is the produced compiler entrypoint for the admitted
  running-system slice on the supported host
- `build/` is disposable generated state
- command-local `--build-dir` trees are disposable generated state and should
  be treated as build outputs, not source inputs
- grouped kernel proof outputs should prefer semantic suite roots such as
  `build/debug/tool/freestanding/...`

Known limitations
-----------------
- the supported host target remains Darwin arm64 only
- the repository still publishes only one bounded kernel lane, not a broad
  product matrix
- the repository still does not claim dynamic discovery, a general namespace
  model, dynamic loading, a service manager, or a broader init framework
- the repository still does not claim public freestanding `mc run` or `mc
  test` workflow support
- the repository still does not claim broad device, filesystem, portability,
  packaging, or install policy
- struct embedding has been explicitly deferred; the decision is recorded in
  the decision log

Practical support statement
---------------------------
The current repository can honestly claim one bounded running-system support
statement over the repository-owned Veya kernel artifact: one interactive
serial-shell system with log, kv, echo, shell, and serial services; one named
neutral service topology module; one named event codes module; per-send
delivery witness on the Effect type; named dispatch arms in the boot
orchestrator; a scripted observation loop in scenarios.mc; a shell that
classifies shell-owned hostile-user invalid commands as unknown command or
malformed admitted shape; flat topology lookup helpers over the five boot
slots; an endpoint-driven init-owned restart entry for the admitted
restartable services; and a thin main.mc that is init plus one call.

The repository also provides the complete Phase 109–121 mechanism boundary
and the Phase 122–177 authority, service, orchestration, and model-repair
wave over that same artifact.

The repository still does not claim a general Veya platform, broader
freestanding workflow support, dynamic discovery, dynamic loading, a service
manager, a general namespace model, multicore, or a broader loader model.

Everything outside that slice should be treated as explicitly unsupported
until a later phase note and proof layer admit it.
