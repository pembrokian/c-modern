Phase 141 Through Phase 145 -- Speculative Post-Phase-140 Canopus Roadmap
=========================================================================

Status
------
SPECULATIVE

Purpose
-------
Sketch one plausible continuation after the completed Phase 140 serial-
ingress composed service graph.

This document is intentionally forward-looking and tentative.
It does not replace `docs/plan/plan.txt`.
It does not replace the concrete Phase 109 through Phase 140 execution and
closeout notes.
It gives one plausible ordering for the next Canopus wave if the repository
keeps following the same evidence-first rules used so far:

- one bounded real system pressure point at a time
- one real service-owned mechanism or follow-through at a time
- one explicit exit criterion per phase
- no broader platform claim before one bounded usable system is honest

Repository status note
----------------------
- The repository can now honestly claim one bounded running Canopus kernel and
  one bounded running service system with explicit authority, bounded failure,
  bounded IPC composition, bounded UART ingress, one compiler-known fatal
  path, and one richer serial-ingress route over the real kernel artifact.
- That result matters because it proves the current model survives stress
  without forcing a broker, recovery framework, or subsystem rebuild.
- The strongest next question is no longer whether the model can survive one
  more bounded proof. It is whether something small, real, and stateful can
  live on top of that model without fighting it.
- The risk on one side is staying in proof mode so long that the repository
  optimizes for sterile correctness work instead of usable system truth.
- The risk on the other side is jumping straight to a larger platform surface
  and hiding structural flaws under filesystem, networking, multicore, or
  performance work before the service model has been used honestly.

What this roadmap is trying to answer
-------------------------------------
- What likely comes after the completed Phase 140 service graph if the next
  goal is not another abstract stress proof but one bounded usable system?
- How should the repository shift from "the model works under stress" to
  "the model is worth using" without widening the kernel prematurely?
- Which next phases should add real service pressure, long-lived state,
  command handling, and restart follow-through before any broader subsystem or
  platform ambition?

Core rule
---------
- Treat the next wave primarily as one real system build, not as a feature
  land-grab and not as proof mode extended indefinitely.
- Reuse the landed serial-ingress path as the seed of one interactive service
  system rather than branching into many unrelated experiments.
- Keep policy in init and service owners. Kernel changes remain justified only
  when the real service system exposes a concrete owner gap.
- Keep the Modern C core, ABI, target, and `hal` surfaces closed unless the
  chosen system proves one specific language or toolchain deficiency.
- Prefer long-lived state, messy command input, partial failure over time, and
  explicit service restart pressure over new infrastructure abstractions.

System-building rule
--------------------
Use this sanity check on every proposed Phase 143 through Phase 145 change:

- making one small interactive service system more usable, more stateful, or
  more restart-tolerant while preserving explicit ownership: admissible
- adding framework machinery before the small system demands it: wrong phase
- widening to filesystem, networking, multicore, or throughput work before the
  service model shows a concrete gap: wrong phase

Kernel invariants
-----------------
These invariants apply across all phases in this roadmap:

- no subsystem may require knowledge of another subsystem's internal state in
  order to perform its normal owned behavior
- all cross-subsystem interaction must pass through explicit interfaces,
  ownership-visible helpers, or dependency-visible boundaries
- no new syscall, namespace hook, or service-discovery hook should be added
  without an explicit argument that the current owner is insufficient
- no phase in this wave should turn the kernel into a broker, shell runtime,
  recovery supervisor, persistence framework, or policy engine
- no larger platform surface should be admitted until one bounded interactive
  system is honest about real usage pressure

Chosen next direction
---------------------
The most practical next step is one bounded interactive service system rooted
in the already-landed serial-ingress graph.

Chosen system shape:

1. one serial shell service that accepts bounded commands and routes them to
   fixed known service owners
2. one long-lived log service that keeps state across multiple requests and
   supports bounded query or inspection paths
3. one long-lived key-value service that keeps explicit state across requests
   and can be used both directly and through another service path
4. the existing echo service retained as the simplest baseline consumer and
   composition leaf
5. one fixed composition graph where shell-driven requests can reach one
   service or one service plus one second service before replying

The target is not a broad shell environment.
The target is one small real system a human can interact with repeatedly over
serial and use to discover where ownership, IPC, failure, and authority become
awkward in practice.

Roadmap bands
-------------

Band A - Freeze One Interactive Service System Boundary
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
This first move is now landed.

Likely themes:
- serial shell as the single user-facing ingress
- fixed command vocabulary rather than dynamic service discovery
- explicit command-to-service routing owned by service and init code
- clear distinction between baseline command handling and later stateful work

Why this band comes first:
- after Phase 140, the repository already has the correct seed. The next risk
  is not missing infrastructure. It is losing focus and trying to widen into
  several systems at once.

Phase 141 -- Interactive Service System Scope Freeze
- Phase 141 is now concrete as one bounded interactive service system scope freeze with one concrete shell owner, one concrete key-value owner, one fixed graph rooted at serial ingress, and one explicit statement that the compiler, broker, and framework surfaces stay closed in this phase.

Preferred proof points:
- one canonical graph rooted at serial ingress is named explicitly
- one fixed shell vocabulary is defined without dynamic routing or discovery
- one long-lived system target is chosen so later phases cannot drift into
  unrelated experiments

Band B - Admit One Real User-Facing Shell Path
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Once the system boundary is frozen, the repository should make the serial path
meaningfully usable by a human rather than only auditable by a proof harness.

Likely themes:
- bounded command parsing over serial input
- shell-owned request routing to fixed services
- explicit reply formatting that remains service-owned and inspectable
- malformed-input handling that reflects real usage rather than test-only
  framing

Why this band comes next:
- command handling is the narrowest path from the already-landed serial graph
  to real usage pressure.

Phase 142 -- Serial Shell Command Routing
- Phase 142 is now concrete as one bounded serial shell command-routing step using a compact four-byte command encoding for the admitted echo, log, and key-value vocabulary while keeping payload width, compiler, broker, and framework surfaces closed in this phase.

Preferred proof points:
- at least one malformed command path is handled explicitly
- command parsing remains bounded and deterministic
- route ownership remains in shell and service code rather than in kernel
  helpers

Band C - Make State Survive Across Requests
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
The model has already survived transactional proofs. The next honest pressure
is long-lived state that must remain coherent across repeated requests and
service interactions.

Likely themes:
- log growth across multiple requests
- explicit log query or inspection commands
- key-value insert, lookup, and update paths
- state retention across command sequences without rebooting the whole system

Why this band matters:
- long-lived state is where ownership friction, message-shape awkwardness, and
  lifecycle assumptions become visible in a way transaction-only proofs cannot
  expose.

Phase 143 -- Long-Lived Log Service Follow-Through
- Phase 143 is now concrete as one bounded long-lived log-service follow-through with bounded in-memory retention, one service-owned `log tail` reply path, one explicit overwrite-on-full consequence, and compiler, persistence, and payload-width surfaces kept closed.

Phase 144 -- Stateful Key-Value Service Follow-Through
- Phase 144 is now concrete as one bounded stateful key-value service follow-through with bounded in-memory retained entries, one explicit missing-key reply, one explicit overwrite contract, one fixed key-value-write observation into `log_service`, and compiler, persistence, and middleware surfaces kept closed.

Band D - Make Real Failure And Restart Hurt The System
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
After shell routing and long-lived state are real, the next honest test is not
another synthetic pressure probe. It is whether the small system survives
service loss, restart, and messy repeated use without demanding ambient kernel
policy.

Likely themes:
- kill one stateful service mid-request or between requests
- explicit restart or replacement of that service through the existing model
- visible consequences at shell and peer-service call sites
- repeated human-driven interaction that exposes friction in protocols,
  authority transfer, or data ownership

Why this band comes last:
- failure pressure is most informative after the services are real enough that
  losing one actually affects useful behavior rather than only a proof graph.

Phase 145 -- Service Restart, Failure, And Usage Pressure Audit
- Phase 145 is now concrete as one bounded service restart, failure, and usage pressure audit with one explicit shell-visible key-value unavailable consequence, one init-owned fixed-wiring restart, one explicit post-restart empty-state truth, and compiler, supervision, and durability surfaces kept closed.

Preferred proof points:
- one service disappears during a real shell-driven workflow
- one restart path restores bounded usefulness without a kernel supervisor
- one resulting pain point is classified honestly as either user-space design
  work or a concrete kernel or language owner gap

What should naturally emerge from this wave
-------------------------------------------
If the repository follows this roadmap honestly, the next useful results
should not be abstract architecture claims. They should be concrete evidence
about:

- whether shell-to-service and service-to-service IPC shapes are awkward in
  real command flows
- whether authority transfer remains comfortable once stateful services are
  used repeatedly
- whether restart and replacement pressure stays service-owned or exposes one
  kernel boundary that is too thin
- whether one narrow library or helper should emerge from repeated user-space
  patterns rather than being invented first
- whether any future language or runtime ergonomic work is justified by real
  system implementation pain rather than by taste

Likely ordering bias
--------------------
If the repository keeps following the same discipline, the safest order is:

1. freeze one small interactive system before widening service scope
2. make the serial path usable by a human through one bounded shell
3. make log and key-value state survive across repeated requests
4. apply failure, restart, and repeated-use pressure only after the system is
   genuinely useful
5. reopen kernel, language, or library design only where the real system shows
   a specific owner gap

That ordering keeps the repository from confusing roadmap movement with
evidence while still forcing the model to justify itself through use.

Non-goals
---------
- treating this roadmap as a committed execution plan
- widening into filesystem, networking, multicore scheduling, or performance
  tuning during this wave
- introducing dynamic service discovery, a broker, a message bus, or a
  general shell runtime
- turning the kernel into a persistence layer, recovery supervisor, or policy
  engine
- reopening the Modern C core before one real system implementation proves a
  concrete owner gap
- claiming broader platform maturity from one small usable system

Interpretation guide
--------------------
- Treat this as a rough map, not a contract.
- The current readiness and support statements remain the canonical statement
  of present truth.
- Phase 131 through Phase 140 now form the historical bridge from bounded
  stress proofs to a richer serial-ingress service graph.
- The next honest question is not whether one more proof can land. It is
  whether one bounded interactive system can live on top of the current model
  without forcing hidden coupling or premature abstraction.
- If that small system exposes a real kernel, language, ABI, or toolchain gap,
  reopen the narrowest owning layer only at that proof boundary.
