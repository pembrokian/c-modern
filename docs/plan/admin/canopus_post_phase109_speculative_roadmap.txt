Phase 118 Through Phase 123 -- Speculative Post-Phase-117 Canopus Roadmap
=========================================================================

Status
------
SPECULATIVE

Purpose
-------
Sketch one plausible continuation after the completed Phase 117 init-
orchestrated multi-service bring-up.

This document is intentionally forward-looking and tentative.
It does not replace `docs/plan/plan.txt`.
It does not replace the concrete Phase 109 through Phase 117 closeout notes.
It gives one plausible ordering for the next Canopus wave if the repository
keeps following the same evidence-first rules used so far:

- one owner-local gap at a time
- one real mechanism, service, or boundary proof at a time
- one explicit exit criterion per phase
- no broader platform claim before the next running slice is honest

What this roadmap is trying to answer
-------------------------------------
- What likely comes after the first running-kernel statement, first ownership
  split, first lifecycle-owner clarification, first syscall-boundary
  thinness audit, first interrupt-boundary clarification, first MMU barrier
  follow-through, and first bounded multi-service bring-up if the
  repository keeps
  growing Canopus conservatively?
- Which follow-on fronts belong first to kernel ownership hardening,
  hardware-boundary truth, running-system follow-through, or image and target
  hardening rather than to new language machinery?
- In what order should the repository make the next kernel slice broader
  without drifting into full OS-completion, broad target promises, or a large
  in-kernel policy framework?

Core rule
---------
- Keep the current Modern C core closed unless a real next kernel step exposes
  a concrete parser, sema, MIR, backend, ABI, or target owner gap.
- Prefer subsystem ownership clarity, real hardware-boundary truth, and real
  running-system follow-through over new architecture prose alone.
- Keep the kernel small, explicit, table-driven, and inspectable while the
  owned tree is still young.
- Do not let namespace policy, loader ambition, device growth, or scheduler
  policy outrun the next honest running-system proof.

Kernel invariants
-----------------
These invariants apply across all phases in this roadmap:

- no subsystem may require knowledge of another subsystem's internal state in
  order to perform its normal owned behavior
- all cross-subsystem interaction must pass through explicit interfaces,
  ownership-visible helpers, or dependency-visible boundaries
- no new syscall should be added without either removing an older redundant
  path or explicitly justifying why the widened syscall surface is the right
  owner
- no kernel feature should encode user-space policy that could remain explicit
  in init or in ordinary services
- ownership-improvement work is only honest when at least one boundary becomes
  mechanically more enforceable, not merely better named

Roadmap bands
-------------

Band A - Harden Kernel Ownership Boundaries
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phase 112 is now concrete as one syscall-boundary thinness audit over the
landed running kernel slice.

Likely themes:
- keep `syscall` thin and prevent transition helpers from drifting back into
  decode paths
- answer the next remaining boundary question at the syscall edge rather than
  reopening task-state ownership ambiguity
- preserve current behavior while making future owners easier to name

Why this band comes first:
- after the landed Phase 111 clarification, the next highest-value step is
  proving that syscall decode and marshalling stay thin while IPC,
  capability, address-space, and lifecycle semantics remain in their owning
  subsystems.

Historical bridge into this band:
- Phase 110 is now concrete as `docs/plan/phase110_kernel_ownership_split_audit.txt`.
- Phase 111 is now concrete as `docs/plan/phase111_scheduler_lifecycle_ownership_clarification.txt`.
- Phase 112 is now concrete as `docs/plan/phase112_syscall_boundary_thinness_audit.txt`.
- Those phases made `sched`, `debug`, lifecycle slot-transition ownership, and
  syscall-boundary thinness explicit without widening the admitted running
  slice, and they leave the next open question centered on the hardware and
  interrupt-facing boundaries rather than on whether syscall decode still owns
  too much meaning.

Band B - Reify Narrow Hardware-Boundary Truth
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phase 113 is now concrete as one interrupt-entry and generic-dispatch boundary
over the landed timer-backed wake path.

Phase 114 is now concrete as one address-space and MMU ownership split over
the landed first-user and spawn translation-root path.

Phase 115 is now concrete as one timer ownership hardening step over the
landed timer-backed wake path.

Phase 116 is now concrete as one MMU activation barrier follow-through over
the landed translation-root activation path.

Likely themes:
- clearer trap or interrupt handoff shape
- first architecture-local versus generic MMU or address-space split if needed
- timer ownership that stays distinct from broader device policy
- barriers or low-level hooks only when real code requires them

Why this band comes next:
- the current kernel slice already proves timer-backed behavior, but it still
  carries a deliberately shallow hardware boundary, and the next honest timer
  and scheduler hardening work should come only after the mapping and memory-
  ownership split it depends on is clearer.

Historical bridge into this band:
- Phase 113 is now concrete as `docs/plan/phase113_interrupt_entry_and_generic_dispatch_boundary.txt`.
- Phase 114 is now concrete as `docs/plan/phase114_address_space_and_mmu_ownership_split.txt`.
- Phase 115 is now concrete as `docs/plan/phase115_timer_ownership_hardening.txt`.
- Phase 116 is now concrete as `docs/plan/phase116_mmu_activation_barrier_follow_through.txt`.
- Those phases turned one landed timer-backed wake path into an explicit
  architecture-entry plus generic-dispatch boundary, then separated generic
  address-space layout from translation-root mechanics, and then moved timer
  tick advancement plus single-delivery wake consumption fully into the timer
  owner, and then moved one translation-root activation barrier hook into the
  explicit `hal` plus MMU boundary without widening the admitted running
  slice. They leave the next open question centered on running-system growth
  rather than on whether one honest barrier-facing hook exists at all.

Band C - Grow From One Running Kernel To One Small Running System
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phase 117 is now concrete as one init-orchestrated multi-service bring-up
over the landed log, echo, and transfer service owners.

Phases 118 through 120 should widen the user-space side carefully from that
bounded aggregate bring-up toward a slightly broader init-owned system.

Likely themes:
- explicit init orchestration rather than ambient boot inheritance
- more than one service relationship without claiming a general namespace
- richer request-reply or authority-passing follow-through
- keep service policy in user space rather than in kernel helpers

Why this band is separate:
- the first running kernel statement is already honest, but the next real
  system question is how to widen a now-landed init-orchestrated service set
  while preserving the same explicit authority style.

Possible phases in this band:

Phase 118 -- Request-Reply And Delegation Follow-Through
- Exit criterion: one bounded service interaction widens beyond the current
  single-hop proofs to show explicit delegation or reply routing across more
  than one user-space participant without ambient authority shortcuts,
  including at least one bounded failure, invalidation, or refused-action path
  that crosses IPC and lifecycle boundaries without turning failure policy into
  a new kernel-owned framework.

Phase 119 -- Namespace-Pressure Audit
- Exit criterion: the repository records the smallest explicit answer to the
  next namespace or service-discovery pressure point, which may still be to
  defer a broader namespace model if the bounded running-system proof does not
  yet require one. An explicit decision that no namespace abstraction is yet
  the right answer is an acceptable and potentially preferred outcome.

Phase 120 -- Running-System Support Statement
- Exit criterion: if the prior phases land honestly, the repository can
  publish one updated support statement for a slightly broader running system
  while still naming all remaining deferrals explicitly.

Historical bridge into this band:
- Phase 117 is now concrete as `docs/plan/phase117_init_orchestrated_multi_service_bring_up.txt`.
- That phase turned the already-landed log, echo, and transfer service proofs
  into one explicit init-owned aggregate bring-up over the real kernel
  artifact without widening compiler work, dynamic loading, or namespace
  policy. It leaves the next open question centered on richer service
  interaction and bounded delegation rather than on whether init owns any
  honest multi-service bring-up at all.

Band D - Harden Image, Target, And Support Truth For The Next Plateau
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phases 121 through 123 should freeze the next slice around build, image,
target, and documentation truth after the broader running-system proof is
real.

Likely themes:
- emitted artifact ownership stays explicit
- target-family and build claims remain narrow and reviewable
- the next support statement is attached to real regression-owned artifacts

Why this band comes last:
- image, target, and support claims should move only after the underlying
  implementation slice actually widened.

Possible phases in this band:

Phase 121 -- Kernel Image Contract Hardening
- Exit criterion: the next running-system slice has one refreshed image-input,
  relink, or artifact-ownership audit that remains explicit about what the
  repository builds and what it still leaves to external tooling. No image-
  contract change is allowed here if it forces retroactive kernel ownership
  changes in lifecycle, VM, capability, or syscall owners.

Phase 122 -- Target-Surface Audit
- Exit criterion: the repository either keeps the bootstrap target-family
  claim unchanged with refreshed justification, or admits one narrowly wider
  target truth with explicit build, runtime, and testing boundaries.

Phase 123 -- Next Plateau Audit
- Exit criterion: the repository can publish one honest post-Phase-111 plateau
  statement naming the admitted kernel, running-system, image, and target
  truths exactly, together with the still-deferred areas that remain outside
  that plateau.

Likely ordering bias
--------------------
If the repository keeps following the same pattern, the safest order is:

1. keep kernel ownership explicit before widening kernel behavior
2. widen only the narrow hardware-boundary truths the next kernel step really
   needs
3. grow init-owned running-system behavior only after those boundaries are
   clearer
4. freeze image, target, and support truth last

That ordering keeps the repository from confusing a first running-kernel
statement plus landed ownership clarifications with permission to claim a
broader platform too early.

Non-goals
---------
- treating this roadmap as a committed execution plan
- claiming full Canopus Lite completion inside this horizon
- implying broad portability, packaging, install, or cross-compilation support
- adding a broad namespace or service-discovery framework before bounded
  running-system pressure proves one is honest
- adding a large driver-resident kernel or monolithic policy layer
- reopening the Modern C core before a real owner gap appears
