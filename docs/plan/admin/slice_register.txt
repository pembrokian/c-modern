Modern C Slice Register
=======================

Status
------
WORKING REGISTER

Purpose
-------
This file is the broad slice register for Modern C.

It is meant to answer two questions at once:

1. What is the full intended language, compiler, runtime, stdlib, and tool
   surface described by the current architecture docs?
2. Which of those slices are already landed, narrowly admitted, only partial,
   or still deferred in the repository?

This is therefore not just a near-term execution list.
It is an end-state map for the current documented v0.2 direction, with
repository status attached.

Scope rule
----------
This register follows what is actually described in:

- docs/arch/
- docs/plan/plan.txt
- docs/plan/*phase*.txt
- README.md

It does not invent extra future families that are not present in those docs.
If a family is intentionally excluded by the architecture, it is listed under
"explicit non-goals" rather than as missing work.

Reading key
-----------
- design: settled | directional | deferred
- repo status: landed | admitted | partial | deferred | published
- plan phase: original phase from docs/plan/plan.txt when applicable
- repo wave: repository-specific phase or closeout wave when applicable

Current position
----------------
- The repository has a real admitted hosted slice: parser, sema, typed MIR,
  LLVM backend on Darwin arm64, deterministic dumps, direct-source and
  project-mode workflow, narrow networking, narrow sync, and several admitted
  real projects.
- The repository does not yet claim the full long-term closure of every slice
  described by the architecture docs.
- The main difference between the current repo and the documented end state is
  no longer only "basic compiler missing pieces". It is also about broader
  module-family admission, fuller portability, fuller release hardening, and
  completing some compiler boundaries that still use bounded limited solutions.

Explicit Non-Goals And Things Not To Misread As Missing Work
------------------------------------------------------------

These are intentionally not part of the current documented v0.2 end state and
therefore should not appear as if the register forgot them:

- hidden runtime services
- garbage collection
- arbitrary compile-time execution
- compile-time reflection systems
- compile-time allocation or compile-time I/O
- if comptime style language branching
- template-style compile-time metaprogramming
- scheduler or green-thread runtime as a language requirement
- exceptions or hidden error propagation
- hidden module initialization hooks
- hidden prelude or omnibus core module
- large trait or type-class system
- test-specific language syntax
- package registry client as part of the core v0.2 build model
- remote dependency fetching as part of the core v0.2 build model

Short Gap Summary
-----------------

The biggest end-state families that remain materially open relative to the
current admitted repository slice are:

- fuller semantic closure wherever bootstrap-shaped gaps still exist beyond the
  currently admitted consumers
- broader imported generic procedure-call lowering beyond the bounded
  thread_spawn<T> rewrite
- full use of the broader target-kind and separate-compilation surface already
  described in the build docs
- freestanding startup and non-hosted execution support
- admission of time, os, and hal as honest module families once real consumer
  matrices exist
- growth of the testing companion module beyond the current minimal helper path
- fuller release hardening beyond the current supported-hosted-slice note
- any later multi-target or broader portability work beyond Darwin arm64
