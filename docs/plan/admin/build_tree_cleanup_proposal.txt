Modern C -- Build Tree Cleanup Proposal
=======================================

Status
------
PROPOSED

Purpose
-------
Restore `build/` to a cleaner and more trustworthy shape as disposable
generated state while preserving the repository's active bootstrap workflow and
ongoing phase-specific experimentation.

This proposal is not about source-control hygiene alone.  `build/` is already
ignored by Git.  The problem is workspace clarity, drift, and the growing gap
between the documented "disposable build tree" contract and how the directory
is currently used in practice.

Current Situation
-----------------
The repository documentation is clear about intent:

- `build/` is disposable generated state
- command-local `--build-dir` trees are also disposable generated state
- the canonical supported build path is `build/debug`

That contract is still good.

The current tree, however, has accumulated a large number of long-lived
phase-specific directories, manual probes, rerun builds, temporary projects,
and one-off validation outputs.  In practice, `build/` is currently serving
three different roles:

1. canonical compiler build output
2. automated workflow and test output
3. personal and historical experiment archive

The first two roles are healthy.
The third role is where drift has accumulated.

Assessment
----------
The current use of phase-labeled build directories is not inherently wrong.
For a staged compiler bring-up, phase-local build outputs are often the most
honest way to preserve context while a slice is active.

The issue is not phase naming by itself.
The issue is retention and mixing.

The tree currently makes it difficult to answer:

- which directories are part of the supported workflow?
- which directories are generated by automated tests?
- which directories are current manual probes?
- which directories are old historical residue that can be removed safely?

That ambiguity weakens the "disposable" contract and makes the build tree feel
more like an informal archive than a generated workspace.

Goals
-----
This cleanup proposal should improve the build tree along five axes:

1. preserve one obvious canonical supported build path
2. separate reproducible workflow output from ad hoc manual probes
3. make old experimental residue easier to delete without fear
4. keep current active phase work easy to run
5. avoid turning `build/` into a second documentation system

Non-goals
---------
This proposal does not aim to:

- remove all phase-named build directories immediately
- rewrite every README command that currently uses a phase-specific build dir
- forbid temporary probes
- move generated outputs into version-controlled directories
- preserve every historical local experiment forever

Core Policy
-----------
`build/` should remain disposable.

That means:

- anything under `build/` must be safe to delete without losing source truth
- if an experiment or proof matters long-term, its durable record belongs in:
  - `tests/`
  - `examples/`
  - `docs/`
  - or a real repository source directory
- `build/` may hold active work products, but it should not be the durable
  archive of repository history

Recommended Build Tree Shape
----------------------------
Keep one small number of stable top-level categories under `build/`:

- `build/debug/`
  The canonical local CMake build tree and the default supported developer
  workflow.

- `build/release/`
  Optional release-oriented CMake or packaging output if needed.

- `build/audit/`
  Reproducible workflow outputs tied to named repository-owned audits or smoke
  paths.
  Examples:
  - first-use smoke artifacts
  - public-cut smoke artifacts
  - release-readiness audit artifacts
  - release snapshot preparation outputs

- `build/probes/`
  Active manual experiments and temporary investigation work.
  These should be treated as short-lived and routinely deletable.

- `build/tmp/`
  Fast local scratch outputs with no expectation of stability.

This proposal does not require every existing command to change immediately,
but it should establish the direction for future additions.

Naming Guidance
---------------
Use naming that answers "what is this for right now?" rather than "which phase
originally introduced it?" whenever the directory is expected to survive more
than a single short session.

Prefer:

- `build/audit/first_use_smoke`
- `build/audit/public_cut_smoke`
- `build/audit/release_snapshot`
- `build/probes/freestanding_console`
- `build/probes/kernel_endpoint_queue`
- `build/tmp/manual_variant_probe`

Avoid adding new long-lived top-level directories such as:

- `build/phaseNN_some_probe`
- `build/phaseNN_some_probe_v2`
- `build/phaseNN_some_probe_v3`

Phase labels are still useful in documentation and plan files.
They are less useful as the long-term primary namespace of the build tree.

Classification Rules
--------------------
Use these rules when deciding where a new generated output should live:

Rule 1 -- Canonical toolchain build

- if the output is the normal compiled compiler and CTest tree, use
  `build/debug` or another clearly named CMake build root

Rule 2 -- Repository-owned reproducible audit

- if the output exists because the repository documents a repeatable smoke,
  audit, or gate path, place it under `build/audit/...`

Rule 3 -- Manual proof or investigation

- if the output exists to answer a local engineering question during active
  work, place it under `build/probes/...`

Rule 4 -- Throwaway scratch

- if the output is not expected to survive beyond a short debugging session,
  place it under `build/tmp/...`

Rule 5 -- Durable evidence

- if the output needs to remain part of repository truth, move the source,
  proof, or expectation into `tests/`, `examples/`, or `docs/` and stop using
  `build/` as the durable record

Recommended Cleanup Pass
------------------------
The first cleanup should be conservative.
Do not mass-delete active investigation directories blindly.

Phase A -- Identify stable vs active vs stale

- keep `build/debug` untouched
- keep any currently active manual probe directories that are still open in
  the working session
- identify reproducible repository-owned audit outputs that deserve a stable
  home under `build/audit`
- treat older phase-specific directories with no current owner as stale by
  default

Phase B -- Create a cleaner namespace

- create:
  - `build/audit/`
  - `build/probes/`
  - `build/tmp/`
- begin routing new manual work into `build/probes/...`
- begin routing new scratch work into `build/tmp/...`
- route any new documented audit outputs into `build/audit/...`

Phase C -- Stop adding fresh long-lived phase directories

- avoid introducing new top-level `build/phaseNN_*` directories except where
  changing an existing documented path would be more disruptive than useful
- if a phase label is helpful, use it after the functional namespace
  Example:
  - `build/probes/phase85_kernel_endpoint_queue`
  rather than
  - `build/phase85_manual_probe_v4`

Phase D -- Prune stale residue

- remove old top-level probe and temporary project directories once their
  lessons are captured in real tests, docs, or examples
- prefer deleting old versions rather than keeping `v2`, `v3`, `v4` variants
  indefinitely
- keep only the minimal currently active local probe set

Phase E -- Align docs and scripts gradually

- update repository docs to keep `build/debug` as the canonical build path
- adjust future examples and helper scripts toward `build/audit/...`,
  `build/probes/...`, and `build/tmp/...`
- do not churn historical docs unless the old path is now actively misleading

Acceptance Criteria
-------------------
This proposal should be considered complete enough when all of the following
are true:

- `build/debug` remains the obvious supported local build path
- new manual experiments stop creating long-lived top-level `phase*`
  directories by default
- reproducible audit outputs have a clearer stable home
- temporary and manual investigation outputs have an obvious short-lived home
- old experimental residue is being deleted rather than silently accumulated
- the repository no longer depends on `build/` as an informal historical
  archive

Do Not Over-Correct
-------------------
- do not forbid useful manual probes
- do not rewrite every older command path only for naming aesthetics
- do not move active source material out of currently used local probe
  directories until the owner is done with them
- do not treat local clutter as a source-control emergency; this is a
  workspace-clarity issue first
- do not confuse disposable generated state with durable proof artifacts

Recommended First Slice
-----------------------
The first slice should be:

1. establish `build/audit`, `build/probes`, and `build/tmp` as the intended
   future namespaces
2. keep `build/debug` as-is
3. route all new manual investigations into `build/probes/...`
4. route all new one-off scratch outputs into `build/tmp/...`
5. prune obviously stale old `phase*_manual_probe*`, `tmp_*`, and rerun
   directories that are no longer active

Bottom Line
-----------
The current `build/` tree reflects real bootstrap work, not chaos.  But it has
crossed the line from "useful phase-local evidence" into "accumulated legacy
workspace residue."  The right response is not to ban phase-specific outputs.
The right response is to restore `build/` as disposable generated state with a
small number of clear namespaces and a habit of deleting stale probes once
their lessons have been captured elsewhere.
