Modern C -- Veya Lite v0 Consumer Matrix
===========================================

Status
------
PROPOSED

Purpose
-------
Translate the Veya Lite v0 architecture into the smallest concrete Modern C
requirements set so the next work is chosen from real bring-up pressure rather
than from the full long-term architecture wish list.

This note is a consumer matrix.
It is not an admission note by itself.

Consumer summary
----------------
Veya Lite v0 wants to prove:

- the kernel boots and enters user space
- the kernel creates tasks and isolated address spaces
- user space exchanges structured messages through kernel-mediated endpoints
- user space holds and transfers explicit capability handles
- one small init process launches and coordinates a minimal service set

It explicitly does not want to prove on day one:

- the full graph model
- zero-copy IPC by default
- broad namespace rewiring
- sophisticated runtime scheduling
- policy-rich kernel semantics

Matrix rule
-----------
- Only derive requirements from the first bring-up slice.
- Treat kernel mechanisms, user-space services, and language/runtime support as
  separate layers.
- Prefer explicit library and build support over new language machinery.

Ranked Modern C pressure fronts
-------------------------------
1. freestanding startup and non-hosted build truth
2. narrow `hal` hardware boundary
3. target and linker truth for kernel and init artifacts
4. explicit struct and handle-based protocol expression in ordinary language
5. narrow observability path for boot and service diagnostics
6. optional later user-space helper libraries for endpoint and capability
   protocols

Front 1 -- Freestanding startup and non-hosted build truth
----------------------------------------------------------
Veya pressure:
- kernel boot and user-space entry are impossible to validate honestly through
  the current hosted-only repository slice

Required Modern C support:
- freestanding build mode that does not assume hosted startup
- explicit startup replacement or target-owned entry selection
- ability to omit hosted runtime pieces entirely
- documentation and tool behavior that make hosted versus freestanding choice
  explicit

Current repo status:
- architecture settled
- repository admission deferred

Why it ranks first:
- nothing else in the Veya bring-up stack is honest if the compiler still
  assumes process startup and hosted runtime support

Smallest useful next step:
- prove one freestanding executable or object path with replacement entry and
  no hosted startup assumptions

Front 2 -- Narrow `hal` hardware boundary
-----------------------------------------
Veya pressure:
- the first system needs explicit hardware-facing code for at least boot,
  timer, console, or one storage or transport path
- the Veya Lite note is explicit that MMIO, volatile access, and
  architecture-specific barriers belong in a hardware-facing boundary rather
  than in general `mem` or `sync`

Required Modern C support:
- `hal.mmio_ptr<T>`
- `hal.volatile_load<T>`
- `hal.volatile_store<T>`
- only the narrowest barrier or fence hooks that a real proof requires

Current repo status:
- architecturally named and exemplified
- repository admission deferred

Why it ranks second:
- Veya can be prototyped in pure software for a short distance, but the
  first real hardware-boundary proof appears quickly and should land in the
  right module family instead of through ad hoc backend or user-code escape
  hatches

Smallest useful next step:
- prove one polled UART-style MMIO write path or equivalent minimal device
  bring-up proof

Front 3 -- Target and linker truth for kernel and init artifacts
----------------------------------------------------------------
Veya pressure:
- kernel, init, and early service artifacts are not ordinary hosted
  executables

Required Modern C support:
- explicit target mode for freestanding artifacts
- predictable object or binary emission path suitable for kernel build
  integration
- explicit startup selection rather than hidden main-based assumptions
- enough link-shape control to integrate target-owned support objects

Current repo status:
- build architecture acknowledges hosted versus freestanding
- repository support is still hosted-only

Why it ranks third:
- even if the language and `hal` surface exist, the OS project still stalls if
  the compiler cannot reliably produce the artifacts the kernel build expects

Smallest useful next step:
- prove one kernel-side or init-side build artifact path that links without the
  hosted startup shim

Front 4 -- Ordinary-language protocol expression
------------------------------------------------
Veya pressure:
- endpoints, messages, handles, mapping requests, and service protocols all
  need explicit data representation

Required Modern C support:
- no new language feature is automatically required here
- existing structs, enums, arrays, pointers, distinct types, constants, and
  deterministic layout should carry most of the load
- explicit conversion and layout annotations remain the right pressure valves

Current repo status:
- largely satisfied on the admitted core-language slice

Why it ranks fourth rather than first:
- the language already appears well aligned with explicit protocol modeling
- this front should reopen only if a concrete Veya proof exposes a missing
  layout, ABI, or constant-evaluation owner gap

Smallest useful next step:
- model one endpoint request or reply protocol and one capability-handle type
  family using the existing core-language surface before proposing new syntax

Front 5 -- Observability and diagnostics
----------------------------------------
Veya pressure:
- the first system needs boot logs, service start or fail reporting, and phase
  markers

Required Modern C support:
- initially this should be solved by the same freestanding and `hal` work plus
  ordinary library code
- no broad tracing or reflection facility is justified

Current repo status:
- hosted observability is adequate for the current slice
- freestanding proof path not yet admitted

Why it ranks fifth:
- observability matters early, but it should ride on the same narrow console or
  endpoint proof surface rather than becoming its own capability family first

Smallest useful next step:
- include boot-marker output in the first freestanding proof and service-start
  logging in the first init-plus-service proof

Front 6 -- User-space endpoint and capability helper libraries
--------------------------------------------------------------
Veya pressure:
- service code will eventually want small libraries for message formats,
  handle wrappers, registry requests, and init orchestration

Required Modern C support:
- ordinary library code only unless repeated proofs show otherwise

Current repo status:
- not yet started

Why it stays last:
- these libraries should grow from real service code after the freestanding,
  target, and `hal` base exists

Smallest useful next step:
- keep first user-space protocols explicit and handwritten until a repeated
  shape appears

Explicitly deferred fronts
--------------------------
The following do not currently win the matrix and should stay deferred unless
the first proof set proves otherwise:

- hidden async runtime or task framework
- channel or actor abstractions in the language or stdlib
- broad hosted `os` growth
- broad `sync` growth beyond the admitted narrow hosted slice
- full capability algebra or derivation taxonomy
- zero-copy IPC as the default communication path
- generalized namespace rewiring
- richer scheduler policy as a language or stdlib concern

Derived Modern C requirement summary
------------------------------------
If Veya Lite v0 is the next real consumer, then the minimum honest Modern C
requirements are:

- real freestanding build and startup support
- narrow `hal` admission for hardware reality
- honest non-hosted artifact production
- reuse of the existing explicit core language for IPC and capability protocol
  structs
- no new language runtime policy unless a concrete proof fails without it

Bottom line
-----------
Veya Lite does not currently justify a broad language redesign.
It justifies making the already-described freestanding and `hal` boundaries
real, then validating that the current explicit core language can carry kernel
and user-space protocol work without hidden runtime machinery.
