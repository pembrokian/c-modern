Turn the current repository posture after the Phase 14 closeout into one

Problem being solved
--------------------
- The repository now has a real hosted compiler and toolchain slice, plus
  explicit pressure evidence, but the main remaining growth is no longer basic
  bring-up.
- The next work is spread across stdlib/runtime maturity, toolchain maturity,
  and pressure-program admission, so priority can drift unless one ranked
  sequence is written down.
- The main risk is not lack of ideas, but loss of leverage:
  - landing more examples before the owning library or workflow layer is honest
  - widening library breadth before current boundaries are pressure-tested
  - starting release hardening before the next capability stop-point is chosen
  - treating blocked networking, sync, or MMIO pressure as vague future work
    instead of named owner work

Core rule
---------
- Expand breadth only where the owning lower layer is already honest enough to
  support it.
- Use Phase 14 pressure to expose gaps, not to hide them.
- Prefer one narrow newly admitted capability family with real proof over many
  half-admitted module shells.
- Do not start the original bootstrap-plan release-hardening pass until the
  repository intentionally chooses the capability plateau it wants to ship.

Default strategic choice
------------------------
Unless near-term repository goals say otherwise, prefer a narrow networking
foundation before a narrow concurrency foundation.

Why this is the default:
- it unlocks the lead blocked evented-networking canonical entries rather than
  only one family of threaded pressure
- it exercises Phase 12 stdlib/runtime boundaries and Phase 13 project-mode
  workflow together
- it creates room for a bounded real utility family beyond the current text,
  traversal, hashing, and parser utilities

When to override the default:
- if current repository goals need local thread or atomic pressure sooner than
  networked I/O pressure
- if a credible hosted poller and net surface cannot be stated narrowly enough
  yet, while a narrow `sync` surface can

Deepen confidence in the already-admitted core before opening a new module
family.

Scope
~~~~~
Close out any remaining next-wave canonical programs that require no new Phase
12 module family.

Preferred candidates
~~~~~~~~~~~~~~~~~~~~

Deliverables

Exit criteria

  candidate as ambient deferred work
- any rejected candidate is recorded with a concrete owner-local reason rather
  than left as ambient future work

Step 2 -- Freeze The Next Capability Family
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Pick exactly one new Phase 12 capability family for the next expansion wave.

Chosen decision
~~~~~~~~~~~~~~~
- Option A is now the chosen next wave: portable `io` poller plus narrow `net`
- Option B remains the next fallback after Option A if networking admission
  stalls or proves less truthful than expected, but it is not started in
  parallel
- Option C remains deferred unless a concrete hardware-boundary consumer moves
  ahead of both networking and sync pressure

Decision options
~~~~~~~~~~~~~~~~
- Option A: portable `io` poller plus narrow `net`
- Option B: narrow `sync` plus hosted thread boundary
- Option C: narrow `hal` admission if an actual hardware-boundary consumer now
  exists

Required output
~~~~~~~~~~~~~~~
- a short owner document or bounded update to the relevant Phase 12 plan that
  names the consumer matrix, minimal API surface, proof layer, and explicit
  exclusions for the chosen family
- this document and the Phase 12 plan now treat Option A as the active next
  capability family and Option B as queued follow-up work rather than a second
  simultaneous start

Hard rule
~~~~~~~~~
- do not start two new capability families in parallel
- the repository may queue Option B after Option A, but must not implement
  both waves concurrently

Step 3 -- Execute The Chosen Phase 12 Second Wave
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Land one narrow new library and runtime family honestly, with proof.

If networking is chosen
~~~~~~~~~~~~~~~~~~~~~~~
- admit the smallest truthful poller surface needed for readiness-based I/O
- admit the smallest truthful `net` surface needed for one server path and one
  client path
- keep handle ownership, close behavior, and readiness transitions explicit
- keep unsupported portability or scaling claims out of scope

If sync is chosen
~~~~~~~~~~~~~~~~~
- admit the smallest truthful hosted thread spawn and join path
- admit mutex first, then condvar or atomic surfaces only if required by the
  chosen pressure proofs
- keep memory-order and blocking semantics explicit in the public surface

If hal is chosen
~~~~~~~~~~~~~~~~
- admit only the narrow MMIO boundary needed by the chosen proof
- keep hardware-facing semantics in `hal`, not in examples or backend-local
  conventions

Deliverables
~~~~~~~~~~~~
- implementation under `stdlib/` and `runtime/hosted/` as needed
- focused library and executable regressions under `tests/stdlib/`,
  `tests/codegen/`, and `tests/tool/`
- closeout-worthy documentation updates once the wave is stable

Exit criteria
~~~~~~~~~~~~~
- one new capability family is implemented, documented, and proven narrowly
- blocked Phase 8 entries tied to that family can now move forward honestly

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~

Expected work
~~~~~~~~~~~~~

Examples

Exit criteria
~~~~~~~~~~~~~

- the findings log records what the new pressure actually exposed

Ranked backlog
--------------

Rank 1 -- Current-core canonical pressure wave
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- owner: Phase 14
- why now: cheapest leverage; deepens truth without new library breadth
- first tasks: treat the current-surface wave as closed out and move to the
  next capability-family decision with the Phase 8 evidence now recorded
- proof target: consistent closeout between proof artifacts, admission matrix,
  findings log, and this ranked execution order

Rank 2 -- Narrow poller plus net admission
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- owner: Phase 12
- why now: best default unlock for blocked canonical entries and a new real
  utility family
- status: chosen next capability family
- first tasks: land the bounded Phase 12 owner plan for portable `io` poller
  plus narrow `net`, then implement it without starting `sync` work in
  parallel
- proof target: evented server or client canonical entry plus focused stdlib
  and tool regressions

Rank 3 -- Narrow sync admission
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- owner: Phase 12
- why now: queued fallback after the chosen networking wave if repository
  pressure later favors local concurrency or if the networking wave stalls
- status: active next capability family
- first tasks: land the bounded Phase 12 owner plan for hosted thread spawn or
  join plus mutex, then prove it through the shared-counter canonical entry
- proof target: shared counter with mutex canonical proof

Rank 4 -- Phase 13 second-wave workflow maturity
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- owner: Phase 13
- why now: the next capability family will expose real project and test-harness
  pressure that the first-wave toolchain slice only partly covers
- status: implemented for the hosted networking family
- first tasks: preserve the admitted evented-echo workflow proof as the project
  surface grows, and route any new multi-module workflow friction back to the
  smallest correct Phase 13 owner
- proof target: new capability-family projects continue to run cleanly through
  `mc run` and `mc test`

Rank 5 -- Text and allocation realism follow-up
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- owner: Phase 12
- why now: repeated parser, hashing, and future networking utilities will keep
  exposing owned-text and buffer-helper friction
- first tasks: audit repeated owned-text or allocator boilerplate before adding
  any new helper surface
- proof target: narrow stdlib regressions and one utility simplification that
  removes real boilerplate without hiding ownership

Rank 6 -- Narrow hal or MMIO admission
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- owner: Phase 12
- why now: meaningful only if a real hardware-boundary consumer exists; still
  worth keeping explicit so the capability does not vanish from view
- first tasks: confirm consumer matrix and minimal API spelling
- proof target: MMIO register access canonical proof

Rank 7 -- Release hardening and first public cut
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- owner: original bootstrap-plan Phase 9 work
- why now: needed for a public cut, but only after the desired capability
  plateau is chosen
- status: repository-bounded hosted support note published
- first tasks: keep the supported-host statement, known limitations, and usage
  docs aligned with the currently admitted slice as follow-on capability work
  lands
- proof target: reproducible documented workflow for the chosen supported slice

Parked until later
------------------
- multiple backend targets before the first hosted target is deeper
- package management or remote dependency fetching
- self-hosting
- broad stdlib shells without a consumer matrix
- replacing the current C++ proof harness wholesale

Acceptance criteria for this document
-------------------------------------
- The repository has one explicit post-Phase-14 execution order rather than
  relying on closeout notes to imply priority.
- The next work is ranked by leverage and owner rather than by whichever
  example or subsystem happens to be nearby.
- The plan keeps the repository focused on one new capability family at a time
  while still using Phase 14 pressure to validate each deeper layer.
