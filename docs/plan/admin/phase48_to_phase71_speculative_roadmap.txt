Phase 48 Through Phase 71 -- Speculative Roadmap
=================================================

Status
------
SPECULATIVE

Purpose
-------
Sketch one plausible continuation from the repository posture after Phase 47
without turning that sketch into a committed plan.

This document is intentionally forward-looking and tentative.
It does not introduce a new architecture phase after Phase 8.
It does not replace `docs/plan/plan.txt`.
It gives a rough shape for the next 24 phases if the repository continues
advancing in the same evidence-first style used so far:

- one owner-local gap at a time
- one real consumer or proof surface at a time
- one explicit exit criterion per phase
- no speculative widening without admitted pressure

Repository status note
----------------------
- The current repository has already moved through the bootstrap bring-up
  phases, the first broad hosted proof waves, and the planned full error-
  treatment follow-up captured in Phase 47.
- The admitted surface now includes sema, MIR, hosted runtime support, a
  hosted project workflow, bounded `sync`, bounded `io.pipe()`, a minimal
  hosted `os` subprocess slice, several real project proofs, and a clearer
  source-level error vocabulary.
- The remaining work is no longer about proving the compiler exists.
  It is about closing the last honest language-owner gaps, deciding which
  adjacent families are truly needed, and hardening the public-supported slice.

What this roadmap is trying to answer
-------------------------------------
- Which remaining semantic or MIR owner gaps are still real after the current
  admitted consumer set?
- Which adjacent runtime or stdlib surfaces are actually justified by pressure?
- Which workflow and release-hardening items need to land before a first public
  `v0.2` cut can be taken seriously?

Core rule
---------
- Prefer owner-local closure over convenience.
- Keep the next admitted surface narrow enough that one phase can still be
  audited by hand.
- Use real projects and canonical fixtures to justify widening.
- If a frontier has no real consumer, keep it deferred explicitly.

Roadmap Bands
-------------

Band A - Finish The Remaining Language Truth
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phases 48 through 57 likely close the last sema and MIR gaps that still matter
for the admitted v0.2 core.

Likely themes:
- imported generic procedure-call widening beyond the current narrow rewrite
- structural aggregate constant transport across sema, `.mci`, MIR, and backend
- any remaining layout, ABI, or conversion edge cases exposed by real consumers
- a final pass on compile-time truth, imported type identity, and owner-local
  lowering rules

Why this band comes first:
- the repository still gets the most leverage from making language meaning
  authoritative before widening more tools or more runtime surfaces
- the current plans already show repeated evidence that later layers should not
  compensate for missing semantic truth

Phases in this band:

Phase 48 -- Post-Plateau Frontier Audit
- Exit criterion: the next honest frontier after the Phase 47 error-treatment
  wave is named from current evidence, not from abstract completeness pressure.

Phase 49 -- Imported Generic Call Closure
- Exit criterion: a second admitted imported generic procedure-typed consumer
  works end-to-end without the current special rewrite boundary.

Phase 50 -- Structural Aggregate Constant Backend Emission
- Exit criterion: top-level aggregate constant globals emit honestly through
  the backend, matching the sema-plus-`.mci` truth already landed.

Phase 51 -- Imported Layout And ABI Re-Audit
- Exit criterion: any remaining layout or ABI edge cases found by real
  consumers are either closed or explicitly deferred with a named reason.

Phase 52 -- Compile-Time Truth Closure
- Exit criterion: the admitted compile-time subset is deterministic across the
  remaining static-storage and constant-expression cases.

Phase 53 -- Imported Identity Finalization
- Exit criterion: imported type identity and qualified nominal forms survive
  the supported workflow without reconstruction drift.

Phase 54 -- MIR Surface Completion For Remaining Owned Semantics
- Exit criterion: any sema construct still missing a dedicated MIR form has a
  clear owner-local lowering path or a named deferral.

Phase 55 -- Final Narrow Semantic Fixture Sweep
- Exit criterion: the remaining sema and MIR fixtures cover the final admitted
  edge cases without broadening the language surface.

Phase 56 -- Semantic Owner Re-Audit
- Exit criterion: no known sema rule is still being papered over in MIR,
  backend, or tooling.

Phase 57 -- Language Surface Plateau Decision
- Exit criterion: the repository has a stable answer on whether the admitted
  v0.2 core is now semantically complete enough for release hardening.

Band B - Runtime And Library Adjacent Closure
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phases 58 through 64 likely decide which adjacent runtime families are still
needed after the core language truth is stable.

Likely themes:
- `time` if and only if a real consumer needs deadlines, retries, or sleep
- `testing` only if repeated project helper pressure exposes a stable pattern
- `sync` follow-on only if atomic read-modify-write or broader coordination is
  clearly demanded
- a final answer on whether any additional `os` or `hal` surface is actually
  justified

Why this band is separate:
- the current repository already shows that adjacent families should be admitted
  only when a consumer matrix makes them honest
- the next question is not “what modules exist in the spec,” but “which of them
  still need to be implemented for the current admitted slice to feel real?”

Phases in this band:

Phase 58 -- Time Consumer Matrix
- Exit criterion: one real project or test family proves whether `time` is
  still deferred or should be admitted narrowly.

Phase 59 -- Minimal Time Admission
- Exit criterion: the smallest honest `time` slice lands, likely monotonic
  clock plus one adjacent behavior such as sleep or deadline support.

Phase 60 -- Time Follow-Through
- Exit criterion: a real admitted project uses the `time` slice through the
  normal workflow and tests.

Phase 61 -- Testing Surface Hardening
- Exit criterion: repeated ordinary-test helper patterns are absorbed into a
  stable `testing` surface without hidden compiler policy.

Phase 62 -- Sync Follow-On Gate
- Exit criterion: the repository decides whether any atomic or coordination
  widening is truly needed beyond the current admitted `sync` slice.

Phase 63 -- Minimal Sync Follow-On Or Explicit Deferral
- Exit criterion: either the smallest honest sync widening lands or the
  follow-on is explicitly parked again with a consumer-based reason.

Phase 64 -- Adjacent Runtime Deferral Audit
- Exit criterion: `os`, `hal`, and any other runtime-adjacent family still not
  justified by pressure remain named as deferred rather than ambient future
  work.

Band C - Workflow, Support, And Release Hardening
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phases 65 through 71 likely focus on making the supported slice feel like a
real toolchain that can be trusted by other humans.

Likely themes:
- package and target-model maturity without drifting into full package
  management
- stronger deterministic rebuild and selected-target reuse proof
- install and usage path hardening for the supported hosted slice
- support-matrix freeze, public-cut smoke audits, and README alignment
- a final decision on whether `v0.2` is ready to tag or still needs one more
  plateau

Why this band comes last:
- the repository already has enough breadth to expose workflow weaknesses
- the next risk is less “can it compile?” and more “can people trust the
  supported slice and the documentation around it?”

Phases in this band:

Phase 65 -- Package And Target Maturity
- Exit criterion: the admitted project graph stays honest across the real
  multi-target shapes the repository supports today.

Phase 66 -- Deterministic Rebuild And Cross-Target Reuse
- Exit criterion: selected-target rebuild reuse remains deterministic and does
  not churn unrelated targets across ordinary workflows.

Phase 67 -- Install And Usage Path Hardening
- Exit criterion: the supported path from source checkout to working `mc`
  usage is documented and repeatable.

Phase 68 -- Supported Slice Audit
- Exit criterion: the README and support notes accurately state what the
  repository does and does not support.

Phase 69 -- Public-Cut Smoke Refresh
- Exit criterion: the public-cut smoke commands and fixture set reflect the
  current admitted slice without stale examples or hidden assumptions.

Phase 70 -- Release-Readiness Re-Audit
- Exit criterion: the remaining gaps are either narrow enough to tolerate in a
  `v0.2` release or clearly large enough to keep the release deferred.

Phase 71 -- v0.2 Gate
- Exit criterion: the repository either tags `v0.2` or writes down the one
  remaining plateau that still blocks that tag.

Likely ordering bias
--------------------
If the repository keeps following the same pattern, the safest order is:

1. Finish the remaining semantic and MIR truth first.
2. Admit only the runtime or library families that real consumers force.
3. Harden workflow, support, and documentation last.

That ordering keeps the project from confusing “more features” with “more
confidence.”

Non-goals
---------
- claiming these phases are already approved
- widening multiple new families in parallel
- converting this note into a replacement for the phase-specific plans
- promising portability, package management, or self-hosting as part of this
  speculative arc

Interpretation guide
--------------------
- Treat this as a rough map, not a contract.
- If real pressure changes the shape of the remaining work, the right answer is
  to rewrite the next narrow phase docs, not to preserve this speculation.
- If the repository reaches a plateau earlier than Phase 71, that is a good
  outcome, not a failure of the roadmap.
