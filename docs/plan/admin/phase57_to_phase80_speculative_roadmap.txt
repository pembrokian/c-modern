Phase 67 Through Phase 79 -- Speculative Roadmap
================================================

Status
------
SPECULATIVE

Purpose
-------
Sketch one plausible continuation after the concrete Phase 59 imported-layout
and ABI re-audit decision without turning that sketch into a committed plan.

This document is intentionally forward-looking and tentative.
It does not introduce a new architecture phase after Phase 8.
It does not replace `docs/plan/plan.txt`.
It gives a rough shape for the next 20 phases if the repository continues
advancing in the same evidence-first style used so far:

- one owner-local gap at a time
- one real consumer or proof surface at a time
- one explicit exit criterion per phase
- no speculative widening without admitted pressure

What this roadmap is trying to answer
-------------------------------------
- Which remaining semantic or MIR owner gaps are still real after the current
  admitted consumer set?
- Which critique-driven stdlib and runtime follow-through items are now honest
  to land, and which ones are still blocked by current compiler or ABI
  boundaries?
- Which adjacent runtime or stdlib surfaces are actually justified by pressure?
- Which workflow and release-hardening items need to land before a first public
  `v0.2` cut can be taken seriously?

Core rule
---------
- Prefer owner-local closure over convenience.
- Keep the next admitted surface narrow enough that one phase can still be
  audited by hand.
- Use real projects and canonical fixtures to justify widening.
- If a frontier has no real consumer, keep it deferred explicitly.
- If the syntax docs already name a form such as `||`, `&&`, or `a[i]`, treat
  any missing parser, sema, MIR, or backend support as owner-local closure,
  not as justification for ad hoc library workarounds.

Short-Term Syntax Closure Note
------------------------------
The written grammar already names logical `||`, logical `&&`, and indexed
postfix access `a[i]`.
The short-term roadmap intent is therefore not to invent these forms late, but
to make sure parser, semantic, MIR, backend, fixture, and real-project support
for them are either admitted promptly in Band A or explicitly named as a still-
owned deferral.

Band C - Workflow, Support, And Release Hardening
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Phases 73 through 79 likely focus on making the supported slice feel like a
real toolchain that can be trusted by other humans.

Likely themes:
- package and target-model maturity without drifting into full package
  management
- stronger deterministic rebuild and selected-target reuse proof
- install and usage path hardening for the supported hosted slice
- support-matrix freeze, public-cut smoke audits, and README alignment
- a final decision on whether `v0.2` is ready to tag or still needs one more
  plateau

Why this band comes last:
- the repository already has enough breadth to expose workflow weaknesses
- the next risk is less “can it compile?” and more “can people trust the
  supported slice and the documentation around it?”

Phases in this band:

Phase 73 -- Package And Target Maturity
- Exit criterion: the admitted project graph stays honest across the real
  multi-target shapes the repository supports today.

Phase 78 -- Release-Readiness Re-Audit
- Exit criterion: the remaining gaps are either narrow enough to tolerate in a
  `v0.2` release or clearly large enough to keep the release deferred.
Phase 79 -- v0.2 Gate
- Exit criterion: the repository either tags `v0.2` or writes down the one
  remaining plateau that still blocks that tag.

Likely ordering bias
--------------------
If the repository keeps following the same pattern, the safest order is:

1. Finish the remaining semantic and MIR truth first.
2. Admit only the runtime or library families that real consumers force.
3. Harden workflow, support, and documentation last.

That ordering keeps the project from confusing “more features” with “more
confidence.”
