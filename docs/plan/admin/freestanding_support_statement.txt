Freestanding Proof Slice Support Note
=====================================

Status
------
HISTORICAL (retired proof-lane note; no active workflow or audit contract)

Purpose
-------
Record the historical admitted freestanding proof slice separately from the
current hosted and Veya running-system notes so the repository can preserve
what the landed Phase 81 through Phase 94 work proved without pretending that
the old freestanding workflow lane is still active.

Phase 202 retired the archived freestanding legacy suite and the old focused
freestanding support audit command.
This note now remains for historical context only.

This note is intentionally narrower than the hosted release-hardening note.
It does not claim general freestanding release readiness, broad non-hosted
workflow support, cross-compilation, kernel-image packaging, or a wider `hal`
surface than the proofs already require.

Repository status note
----------------------
- Phase 81 made the first freestanding boot-marker and narrow `hal` proof real.
- Phase 82 hardened that path around explicit freestanding `target` identity
  and manifest-owned `link.inputs`.
- Phase 83 then proved one deterministic polling-console path through the
  admitted `hal` MMIO surface.
- Phase 85 then proved one bounded endpoint queue and message-buffer
  representation with ordinary Modern C structs, enums, arrays, and functions
  on the same freestanding path.
- Phase 86 then proved one minimal create, ready, exit, and wait lifecycle
  path with an ordinary kernel-owned task table and task-state enum on that
  same freestanding path.
- Phase 87 then proved that kernel-owned aggregate const templates and
  zero-initialized mutable globals survive the freestanding build and artifact
  path on that same repository-bounded proof lane.
- Phase 88 then proved that the emitted freestanding entry object and
  generated runtime object compose with one minimal external relink step,
  so the current proof lane is no longer limited to driver-owned final linking
  alone.
- Phase 89 then proved that kernel boot can start one init context, pass one
  explicit log-service endpoint capability into that context, and complete
  one bounded request-and-acknowledgment flow on that same proof lane.
- Phase 90 then proved that one user-space component can transfer one
  explicit endpoint capability to another user-space component, lose local
  use of that transferred authority, and have the receiver complete one
  bounded acknowledged request through the transferred handle on that same
  proof lane.
- Phase 91 then proved that the repeated transfer-message, received-handle-
  install, request, and acknowledgment scaffolding from the first user-space
  proofs can move into one ordinary imported helper module without widening
  into a public `sys` wrapper or capability-runtime layer.
- Phase 92 then proved that init can launch one child from one explicit
  launchable program capability and observe one child exit through one
  explicit wait handle without widening into a public spawn or loader surface.
- Phase 93 then proved that one explicit timer-backed wait can fire at one
  deterministic tick, wake one blocked worker, and publish one explicit wake
  observation without widening into a public timer API, interrupt framework,
  or scheduler runtime.
- Phase 94 then proved one bounded first-system demo that composes kernel
  boot, init, one log service, one echo service, one client, one timer wake,
  one bounded request-reply exchange, and one explicit client-exit
  observation without widening into a public service framework, scheduler
  runtime, or broader support claim.
- Before this note, those truths existed in phase closeouts and tool tests,
  but not yet as one explicit published support statement.

Core rule
---------
- Keep this freestanding statement separate from the hosted `v0.2` support
  claim.
- State only the Phase 81 through Phase 94 proof slice, not a larger
  freestanding ambition.
- Keep target-owned support objects, startup choice, and `hal` scope explicit.
- Reopen broader non-hosted claims only when a later proof lands them.

admitted support-statement posture: first admitted freestanding `v0.3` slice statement only, not a broader release or portability guarantee.

Frozen support matrix
---------------------
- supported compiler host and produced proof-artifact target family: Darwin
  arm64 bootstrap target family only
- admitted runtime environment: freestanding project `exe` targets with
  explicit `env = "freestanding"`
- admitted project command path: `mc build --project <build.toml> --target
  <name> --build-dir <dir>`
- admitted manifest boundary: freestanding `exe` targets with explicit
  `target`, explicit `[targets.<name>.runtime] startup`, and optional explicit
  `[targets.<name>.link] inputs = [...]`
- admitted public stdlib boundary: narrow `hal` `mmio_ptr<T>`,
  `memory_barrier()`, `volatile_load<T>`, and `volatile_store<T>`
- admitted proof set: Phase 81 boot-marker executable proof, Phase 82
  explicit manifest-owned link-input proof, Phase 83 deterministic
  polling-console proof, Phase 85 kernel endpoint queue smoke proof, and
  Phase 86 task lifecycle kernel proof, and Phase 87 kernel static-data and
  artifact follow-through proof, and Phase 88 kernel build integration audit,
  and Phase 89 init-to-log-service handshake proof, and Phase 90
  capability-handle transfer proof, and Phase 91 early user-space helper-
  boundary audit, and Phase 92 user-space lifecycle follow-through proof, and
  Phase 93 timer wake proof, and Phase 94 first-system demo integration proof
- unsupported today: `mc run` or `mc test` freestanding workflow admission,
  general cross-compilation, non-bootstrap target families, linker scripts,
  arbitrary linker flags, shared libraries, kernel-image packaging, and
  broader `hal` taxonomy

Usage and audit
---------------
Build from source on the supported host:

```sh
cmake -S . -B build/debug -DCMAKE_BUILD_TYPE=Debug
cmake --build build/debug -j4
```

The produced compiler entrypoint on the supported host is:

```sh
build/debug/mc
```

Supported command pattern for the admitted freestanding proof slice:

```sh
build/debug/mc build --project <build.toml> --target <name> --build-dir <dir>
```

There is no current repo-owned audit entrypoint for this historical note.

Artifact expectations
---------------------
- `build/debug/mc` is the produced compiler entrypoint for the admitted proof
  slice on the supported host
- `build/` is disposable generated state
- command-local `--build-dir` trees are disposable generated state and should
  be treated as build outputs, not source inputs
- this note is retained as historical context only; active support claims now
  live in the hosted release-hardening note and the Veya running-system
  support statement

Known limitations
-----------------
- the supported host target remains Darwin arm64 only
- the freestanding side is still a repository-bounded proof path, not a broad
  non-hosted workflow guarantee
- the repository still does not claim `mc run` or `mc test` as freestanding
  command paths
- the repository still does not claim general cross-compilation, linker-script
  control, arbitrary linker-flag passthrough, or non-bootstrap target-family
  admission
- the repository still does not claim kernel-image assembly, init packaging,
  early-service packaging, general interrupt handling, broader timer
  facilities, broader barrier taxonomy, or richer hardware abstractions
- the public `hal` boundary remains intentionally narrow: `mmio_ptr<T>`,
  `memory_barrier()`, `volatile_load<T>`, and `volatile_store<T>` only

Practical support statement
---------------------------
The current repository can honestly claim one repository-bounded freestanding
executable proof slice on the bootstrap Darwin arm64 target family, with
explicit freestanding manifest identity, explicit startup selection, optional
manifest-owned extra link inputs, and the narrow public `hal` MMIO surface
proved through the Phase 81 through Phase 94 tool regressions.

the repository still does not claim full Veya Lite completion.

The repository still does not claim broader
freestanding workflow admission, namespace policy, scheduler policy, or
service-discovery generality beyond that first-system demo proof.

This remains the first admitted freestanding `v0.3` slice statement for the
repository, not a broader non-hosted release commitment.

Everything outside that slice should be treated as explicitly unsupported
until a later phase note and proof layer admit it.
