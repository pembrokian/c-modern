Phase 46 -- Pipeline And Stdio Follow-Through
============================================

Status
------
IMPLEMENTED (repository-bounded admitted follow-through)

Purpose
-------
Use the narrow subprocess slice from Phase 45 to land one real consumer-facing
follow-through wave: a bounded pipeline, relay, or filter utility that proves
why subprocess plus stdio composition belongs in the admitted hosted slice.

This document is a repository execution plan, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- Phase 44 must authorize subprocess admission through a real consumer matrix.
- Phase 45 must then land the minimal spawn plus wait plus stdio-pipe slice.

  the first `os` admission; it must be justified by the chosen consumer

Problem being solved
--------------------
- A narrow subprocess API only becomes meaningful if a real admitted project
  uses it to close an actual handle-first workflow gap.
- Without a follow-through phase, the repository could land spawn plus wait and
  pipe-backed stdio yet still fail to prove why they belong in the admitted
  slice.
- The repository needs one concrete consumer outcome: a bounded pipeline,
  filter, relay, or capture utility that is clearer with subprocess stdio than
  without it.

Core rule
---------
- Land one real subprocess or stdio consumer follow-through, not several broad
  `os` features.
- Prefer a small filter or relay utility with explicit pipe ownership before
  broader subprocess frameworks.
- Keep behavior explicit in project or stdlib code rather than hiding it in
  tooling.
- Record any remaining broader `os` desires as deferred instead of widening the
  family opportunistically.
- Preserve the handle-first story established by the admitted `io.pipe()`
  slice.

Chosen slice for this phase
---------------------------
Targeted proof shape: one admitted real project gains honest pipeline or stdio
behavior using the Phase 45 subprocess slice.

Strong candidate follow-through shapes
--------------------------------------
- a bounded line-filter relay that spawns one child process, feeds it explicit
  stdin, and reads stdout back through ordinary `io.File` handles
- a tiny pipeline utility proving one child-process composition step without
  opening shell-language semantics
- a capture-style helper utility only if it demonstrates the intended handle-
  first subprocess API more honestly than the filter or relay candidates

What this phase should try to prove
-----------------------------------
- the chosen project is materially more honest with subprocess stdio than with
  thread-only pipe proof or host-side scripting
- the subprocess API remains explicit and reviewable
- broader process policy stays deferred
- the repository can say clearly what its admitted hosted `os` slice now covers
  and what it still does not

Preferred pressure points
-------------------------
1. One admitted project with real child-process stdio composition.
2. Explicit separation between handle ownership in `io` and process ownership
   in `os`.
3. Deterministic project and ordinary-test proof for the new behavior.
4. Documentation updates that keep the hosted support statement honest.
5. Explicit preservation of deferred broader `os` features.

5. Record any still-deferred broader `os` desires explicitly.

Exit criteria
-------------
- one admitted project now proves the hosted subprocess slice through honest
  stdio composition
- the repository can describe the admitted subprocess surface concretely and
  narrowly
- any broader `os` growth remains explicit and deferred rather than implied
- the next plateau after this subprocess wave is chosen from clearer evidence

Non-goals

- environment management, cwd mutation, signals, or broader process control
- expanding into package-management, daemon control, or job supervision
- turning build-tool execution helpers into the public subprocess API
