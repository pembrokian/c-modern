Bootstrap Phase 5 Status

This repository has completed Stage 0 through Stage 5 of the narrower bootstrap Phase 5 plan, and it should still not claim that the full Phase 5 from plan.txt is finished.

The ordered implementation sequence for this status note now lives in `docs/plan/phase5_bootstrap_execution_plan.txt`.
The Stage 0 freeze note for that sequence now lives in `docs/plan/phase5_bootstrap_backend_slice.txt`.

What is in place now:
- typed MIR exists as a real compiler boundary after bootstrap semantic checking and before backend lowering
- `mc check` can parse, semantically check, lower to MIR, validate MIR, and emit deterministic MIR dumps for the currently supported bootstrap subset
- deterministic semantic and MIR fixture coverage exists under `tests/sema/` and `tests/mir/`, giving Phase 5 a stable frontend contract to lower from
- the tree now includes a real backend scaffold under `compiler/codegen_llvm/` plus a dedicated `mc_codegen_llvm` CMake target and focused backend unit coverage
- the current driver remains intentionally thin while now being able to emit LLVM IR, invoke the host toolchain, and produce one hosted executable through bootstrap `mc build`
- the current build stays green under the canonical local workflow, so later backend work can start from a known-good Stage 1 baseline

What Stage 0 has now fixed explicitly:
- the first backend choice is LLVM
- the first hosted bootstrap target is the current Darwin arm64 development host, recorded in repository terms as hosted `arm64-apple-darwin`
- the initial runnable backend slice is intentionally smaller than the full MIR surface and starts with direct-call scalar control flow rather than globals, checked faults, aggregate-heavy lowering, low-level memory operations, or atomics
- unsupported MIR outside that initial slice is now expected to fail with narrow backend diagnostics instead of being treated as implied future support
- the repository-specific meaning of Phase 5 bootstrap completion is now written down separately from the full `plan.txt` milestone

What Stage 1 has now fixed explicitly:
- `compiler/codegen_llvm/` is no longer a placeholder; it exposes `BootstrapTargetConfig()` and `LowerModule(...)` as the first backend entrypoints
- CMake now builds the dedicated `mc_codegen_llvm` library and links it into `mc` without turning `mc check` into a codegen command
- the backend scaffold rejects unsupported bootstrap inputs directly, including unsupported targets, globals, extern functions, non-function symbol refs, non-direct calls, and out-of-scope MIR instructions
- focused backend unit tests pin the frozen target contract and the Stage 1 rejection behavior

What Stage 2 has now fixed explicitly:
- the first inspectability surface is deterministic backend IR text, exposed through `DumpModule(...)` in the backend library and the optional `mc check --dump-backend` path
- backend dumps now use deterministic build-dir-scoped paths under `build/*/dumps/backend/`, and `mc dump-paths` reports them alongside AST and MIR dump locations
- backend naming is reviewable and stable for this slice: functions use `@name`, locals use `%local.name`, blocks use `bb<function>_<block>.<source-label>`, temporaries use `%t<function>_<block>_<instruction>`, and reserved future globals use `@g.name`
- `tests/codegen/` now contains focused backend snapshot coverage, and smoke coverage proves one or more checked modules can lower to backend output deterministically without linking

What Stage 5 has now fixed explicitly:
- the backend can render real LLVM IR for the runnable scalar control-flow slice and write deterministic `.ll` artifacts under `build/*/dumps/backend/`
- the backend now emits object files and links hosted executables through the ordinary host `xcrun clang` path on Darwin arm64, with deterministic object and executable paths under `build/*/obj/` and `build/*/bin/`
- the thin driver now exposes `mc build` while still remaining bootstrap-scoped; it reuses the existing parse -> sema -> MIR pipeline and only adds the backend, object, and link path required for one-module executable bring-up
- the hosted entry boundary is now explicit in codegen: an emitted wrapper bridges the C-runtime `main(argc, argv)` entry into bootstrap Modern C `main` procedures with either zero parameters or the canonical `Slice<cstr>`-shaped argument surface
- end-to-end executable coverage now exists in `tests/codegen/codegen_executable_tests.cpp`, and the canonical local workflow proves hello-style and direct-call programs build and run on the first hosted target

What a narrower bootstrap Phase 5 will need to satisfy:
- lower the supported typed MIR subset to backend IR without letting backend defaults redefine Modern C language semantics
- support enough code generation for ordinary procedures, globals, aggregates, calls, branches, and returns to produce runnable hosted executables
- carry the sema-owned layout and MIR-owned checked-operation semantics through code generation for the supported bootstrap subset
- provide a thin bootstrap path for object emission and linking without prematurely claiming the broader Phase 7 tool workflow
- add executable-focused tests under `tests/codegen/` that prove the supported bootstrap subset runs end to end on the first hosted target

What still blocks a fuller Phase 5 completion claim:
- checked runtime faults are not yet executable-supported in the Stage 5 build path even though backend inspectability already names them explicitly
- globals, aggregate-heavy lowering beyond the hosted entry boundary, and broader representation-sensitive cases are still outside the runnable executable slice
- Stage 6 still needs a broader executable evidence set before the repository can write a narrower `phase5_bootstrap_finish.txt`

Current structural debt worth tracking explicitly:
- the current driver surface is inspection-oriented around `mc check`; Phase 5 should extend it only as far as backend bring-up requires and should avoid collapsing Phase 7 workflow concerns into early backend work
- the repository currently has a MIR boundary but no backend-specific dump or validation layer, so Phase 5 will need to decide where backend-facing inspectability lives without weakening MIR as the semantic contract
- runtime startup and platform-linking concerns remain outside the current bootstrap compiler path, so executable bring-up must define a narrow ownership boundary between compiler output and hosted runtime support

Practical bootstrap interpretation:
- the repository is entering Phase 5 from a green Phase 4 bootstrap baseline rather than from a parser-only or sema-only bring-up state
- the immediate goal is not a full toolchain or build-system surface; it is a narrow, inspectable backend path that can turn the already-supported MIR subset into working executables on one hosted target
- Phase 5 should continue to distinguish a narrower repository bootstrap milestone from the broader Phase 5 target in plan.txt

Concrete next steps toward a narrower bootstrap completion note:
1. implement the initial MIR-to-backend lowering contract defined in `phase5_bootstrap_backend_slice.txt`
2. expand the backend text surface from structural inspectability into the first real runnable lowering slice
3. add the thinnest viable executable-emission path that can lower one checked module, emit an object, and link it with the hosted runtime boundary
4. stand up `tests/codegen/` with a small end-to-end executable suite before expanding backend surface area
5. only after those steps, consider writing a repository-specific `phase5_bootstrap_finish.txt` analogous to the earlier bootstrap milestone notes
