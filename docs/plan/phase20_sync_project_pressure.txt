Phase 20 -- Real Sync Project Pressure On The Admitted Hosted Slice
==================================================================

Status
------
PLANNED

Purpose
-------
Add one real project-mode consumer for the admitted hosted `sync` slice so the
repository can pressure concurrency under ordinary project workflow without
expanding the public `sync` API family opportunistically.

This document is a repository execution plan, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- The repository already admits a narrow hosted `sync` slice covering typed
  thread spawn or join, mutex, condvar, destroy, and atomic load or store
  publication, proved today by direct-source canonical executables.
- Phase 18 did not choose a broader `sync` follow-on as the immediate next
  plateau because the repository lacked a second real consumer strong enough to
  justify additional concurrency breadth.
- The current planning gap is therefore not missing low-level proof. The gap
  is missing project-mode pressure that uses the admitted `sync` surface as a
  real multi-module workflow consumer.

Problem being solved
--------------------
- The admitted `sync` slice is currently strongest at the direct-source
  canonical-program layer.
- Without one ordinary project consumer, it remains too easy to debate future
  `sync` widening from isolated canonical proofs instead of from a real
  project's graph, tests, helper modules, and workflow behavior.
- The repository needs one concurrency-shaped project that either justifies a
  second consumer for broader compiler work or proves that the current admitted
  slice is sufficient for now.

Core rule
---------
- Add one real sync project, not more sync API breadth.
- Stay inside the already-admitted hosted `sync` surface unless a new owner
  document explicitly widens it.
- Use the project to expose workflow and compiler pressure honestly rather than
  embedding example-local shims that bypass current owners.
- Treat any newly exposed compiler limitation as owner-local follow-up for the
  later compiler phase rather than as an excuse to widen `sync` opportunistically.

Chosen project shape
--------------------
Preferred candidate: one bounded threaded worker or queue project under
`/Users/ro/dev/c_modern/examples/real/`.

The project should:
- use ordinary project modules instead of a single-file concurrency demo
- exercise `thread_spawn`, `thread_join`, mutex, condvar, and optionally the
  already-admitted atomic publication surface only where the workflow actually
  needs it
- keep deterministic completion and reviewable tests as the main goal rather
  than benchmark-style throughput claims
- avoid reopening compare-exchange, fetch-add, condvar broadcast, or scheduler
  semantics that the current admitted slice does not own

Candidate pressure shapes
-------------------------
1. A bounded worker queue with one producer thread and one or more worker
   threads coordinated by one mutex and one condvar.
2. A small threaded job runner that publishes work items explicitly and joins
   deterministically before process exit.
3. A narrow sync-backed pipeline utility only if it pressures ordinary project
   modules more clearly than the worker-queue shape.

Preferred implementation bias
-----------------------------
- prefer one reviewable project whose concurrency story is obvious from the
  source over a more feature-rich utility with ambiguous scheduling behavior
- keep ordinary tests deterministic and serializable under `mc test`
- make the project useful enough to pressure imports, helper-module layout,
  and test organization, but not so large that example complexity hides the
  actual `sync` pressure

Deliverables
------------
- one new real project under `/Users/ro/dev/c_modern/examples/real/` that uses
  the admitted hosted `sync` slice honestly
- focused project workflow proof through `mc run --project ...` and
  `mc test --project ...`
- bounded tool regressions in
  `/Users/ro/dev/c_modern/tests/tool/phase7_tool_tests.cpp` proving ordinary
  run, test, and deterministic reuse on the new project
- updates to `/Users/ro/dev/c_modern/docs/plan/phase8_admission_matrix.txt`
  and `/Users/ro/dev/c_modern/docs/plan/phase8_findings.txt` so the bounded
  threaded project stops living as a deferred candidate once it is admitted or
  is explicitly re-deferred with a sharper owner-local reason
- decision-log or backlog updates if the project exposes a real second
  consumer for broader imported generic procedure-call lowering or another
  named compiler follow-up

Suggested execution order
-------------------------
1. Choose exactly one bounded threaded project shape and document the explicit
   exclusions up front.
2. Land the narrow project modules and direct ordinary tests without adding
   new `sync` surface area.
3. Add `mc run --project ...` proof first, then `mc test --project ...` proof,
   then deterministic build-directory reuse checks.
4. Route any exposed compiler or workflow gap immediately to the owning later
   phase instead of normalizing it inside the example.
5. Reassess whether the repository now has a real second consumer for broader
   imported generic procedure-call lowering.

Exit criteria
-------------
- the repository has one admitted real sync project rather than only
  direct-source canonical concurrency proofs
- the project proves ordinary multi-module workflow pressure on the current
  hosted `sync` slice without widening that slice opportunistically
- any newly exposed concurrency-related compiler debt is named precisely and
  handed to the correct owner
- the repository can make a sharper decision about future `sync` widening from
  real project pressure instead of from abstract completeness arguments

Non-goals
---------
- compare-exchange, fetch-add, condvar broadcast, or richer atomic API
  admission during this phase
- broader scheduler, thread-pool, or portability claims
- opening `hal`, `time`, or `os` as parallel capability families
- treating nondeterministic threaded behavior as acceptable just because the
  project is only an example

Acceptance criteria for this document
-------------------------------------
- Phase 20 stands alone as the plan for one real project consumer of the
  admitted hosted `sync` slice.
- The phase is explicit that project pressure, not API breadth, is its main
  purpose.
