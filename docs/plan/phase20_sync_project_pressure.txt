Phase 20 -- Real Sync Project Pressure On The Admitted Hosted Slice
==================================================================

Status
------
COMPLETED (repository-bounded real sync project pressure)

Purpose
-------
Add one real project-mode consumer for the admitted hosted `sync` slice so the
repository can pressure concurrency under ordinary project workflow without
expanding the public `sync` API family opportunistically.

This document is a repository execution plan, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- The repository already admitted a narrow hosted `sync` slice covering typed
  thread spawn or join, mutex, condvar, destroy, and atomic load or store
  publication through direct-source canonical executables.
- Phase 18 deliberately chose workflow hardening before any broader `sync`
  widening because the repository still lacked one real project-mode consumer
  for the already-admitted `sync` surface.
- Phase 20 therefore targeted project pressure only, not a new `sync` API
  family.

Problem being solved
--------------------
- The admitted `sync` slice was strongest only at the direct-source canonical
  program layer.
- Without one ordinary project consumer, future `sync` widening could still be
  debated from isolated canonical proofs instead of from a real project's
  module graph, helper modules, tests, and build-directory reuse behavior.
- The repository needed one bounded threaded project that either exposed a real
  owner-local gap or proved that the current admitted slice was already enough
  for one honest project-mode consumer.

Core rule
---------
- Add one real sync project, not more sync API breadth.
- Stay inside the already-admitted hosted `sync` surface unless a new owner
  document explicitly widens it.
- Use the project to expose workflow and compiler pressure honestly rather than
  embedding example-local shims that bypass current owners.
- Treat any newly exposed compiler limitation as owner-local follow-up for the
  later compiler phase rather than as an excuse to widen `sync` opportunistically.

Chosen project shape
--------------------
Landed project: one bounded worker queue under
`/Users/ro/dev/c_modern/examples/real/worker_queue/`.

The landed project:
- uses ordinary project modules instead of a single-file concurrency demo
- exercises `thread_spawn`, `thread_join`, mutex, condvar, and destroy without
  pulling in broader `sync` breadth that the workflow does not need
- keeps deterministic completion and reviewable ordinary tests as the main goal
  rather than throughput or scheduler claims
- avoids compare-exchange, fetch-add, condvar broadcast, and any broader async
  or scheduler model

What landed in this phase
-------------------------
- the real project `examples/real/worker_queue/` with a thin main module, one
  shared-state helper module, and ordinary checked tests
- a bounded producer-or-worker queue that uses one producer thread, one worker
  thread, one mutex, and one condition variable with deterministic shutdown
- Phase 7 tool regressions proving `mc run --project ...`, `mc test --project ...`,
  run-test-rerun reuse, and test-run-retest reuse on shared build directories
- documentation updates that move the bounded threaded project from deferred or
  blocked status into admitted repository truth

What the project exposed
------------------------
- The admitted hosted `sync` slice is now sufficient for one honest multi-
  module producer-or-worker queue under the ordinary project workflow.
- No new `sync` breadth was required beyond the already-admitted surface.
- The project did not justify broader imported generic procedure-call lowering:
  it stays within the already-admitted `thread_spawn<T>` path rather than
  creating a second independent generic-call pressure family.

Delivered outputs
-----------------
- one new real project under `/Users/ro/dev/c_modern/examples/real/` using the
  admitted hosted `sync` slice honestly
- focused project workflow proof through `mc run --project ...` and
  `mc test --project ...`
- bounded tool regressions in
  `/Users/ro/dev/c_modern/tests/tool/phase7_tool_tests.cpp` proving ordinary
  run, test, and deterministic reuse on the new project
- updates to `/Users/ro/dev/c_modern/docs/plan/phase8_admission_matrix.txt`,
  `/Users/ro/dev/c_modern/docs/plan/phase8_findings.txt`, and the repository
  summary docs so the bounded threaded project no longer lives as a deferred
  candidate

Exit criteria
-------------
- the repository now has one admitted real sync project rather than only
  direct-source canonical concurrency proofs
- the project proves ordinary multi-module workflow pressure on the current
  hosted `sync` slice without widening that slice opportunistically
- no new concurrency-related compiler debt needed to be normalized into the
  example
- the repository can make future `sync` widening decisions from real project
  pressure instead of from abstract completeness arguments alone

Non-goals
---------
- compare-exchange, fetch-add, condvar broadcast, or richer atomic API
  admission during this phase
- broader scheduler, thread-pool, or portability claims
- opening `hal`, `time`, or `os` as parallel capability families
- treating nondeterministic threaded behavior as acceptable just because the
  project is only an example

Acceptance criteria for this document
-------------------------------------
- Phase 20 stands alone as the plan for one real project consumer of the
  admitted hosted `sync` slice.
- The phase is explicit that project pressure, not API breadth, is its main
  purpose.

Closeout statement
------------------
Phase 20 is complete. The repository now includes one honest project-mode
