Plateau decision
----------------

the next highest-leverage plateau for the current admitted hosted slice.

Why Option A won
----------------
- the admitted evented-echo and evented-partial-write projects now expose real
  workflow pressure on `mc test`, reporting, and build-directory reuse without
  needing a third capability family to justify that work
- Phase 17 already narrowed the most visible compiler-shaped networking API
  compromise, so the next leverage lies more in workflow trust than in another
  immediate compiler widening slice
- the release-hardening note already exists, but the richer project fixtures
  still expose workflow-shaped pressure strongly enough that stopping capability
  growth for a harder public cut would be premature

Why the other options were not chosen

  currently justifies compare-exchange, fetch-add, condvar broadcast, or
  broader imported generic-call widening beyond the already-admitted slice
- Option C (`hal`) was not chosen because the repository still lacks a real
  MMIO consumer matrix strong enough to justify even a narrow hardware-boundary
  admission
- Option D (release hardening now) was not chosen because richer real-project
  workflow pressure is still exposing higher-leverage trust work inside the
  currently admitted slice
