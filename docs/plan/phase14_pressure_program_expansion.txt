Repository-specific scope note
------------------------------
- The architecture texts stop at Phase 8 and already require pressure
  findings to route back to the smallest correct owner.
- This repository uses the Phase 14 label only to package the next execution
  wave as a concrete plan document under `docs/plan/`.
- In substance, Phase 14 is the implementation plan for the unfinished
  Workstreams 8A through 8E from the non-bootstrap roadmap after the admitted
  first-pass Phase 8 slice.
- The first execution wave explicitly prioritizes refreshing the admission
  matrix, closing the next meaningful canonical-pressure gaps, and tightening
  findings-to-regression discipline before any broader utility-family growth.

Purpose
-------
Turn the broad non-bootstrap Phase 8 roadmap and the repository's first-pass
Phase 8 closure into an execution-ready second-wave plan that can be
implemented in reviewable slices without blurring the boundary between
pressure evidence and the owning compiler, stdlib, runtime, or toolchain
layers.

After Phase 14:

- the repository's admitted canonical pressure set is broader, more explicit,
  and less dependent on first-pass omissions or informal follow-up intent
- deferred or blocked canonical entries are either advanced honestly or kept
  as named enabling gaps with clear owner attribution
- the real-program suite expands one bounded family at a time instead of
  remaining clustered around the first-pass text, traversal, hashing, and
  parser utilities
- repeated friction discovered by pressure programs is promoted promptly into
  sema, MIR, backend, stdlib, runtime, or tool regressions instead of being
  normalized inside examples
- the admission matrix, findings log, and proof artifacts become the durable
  repository evidence for what the current hosted slice can and cannot yet do

Problem being solved
--------------------
- The repository already has a real first-pass Phase 8 slice: a bounded
  canonical set, several real utility projects, explicit admission and
  findings documents, and deterministic executable or project proof.
- That pressure layer is still incomplete relative to the long-term Phase 8
  contract in `plan.txt` and the non-bootstrap roadmap.
- The main remaining risk is not absence of pressure evidence, but partial
  closure and drift:
  - the first-pass canonical set can be mistaken for the full meaningful
    canonical pressure surface
  - deferred threading and evented-networking entries can remain ambiguous
    rather than being advanced or held as named enabling gaps
  - later real-program families can remain anecdotal ideas instead of explicit
    repository-backed execution slices
  - repeated findings can stay trapped in examples, README notes, or human
    memory instead of becoming earlier-phase backlog or stable regressions
  - proof can remain weighted toward the first landed projects without a clear
    second-wave plan for what comes next and why
- Without an explicit execution plan:
  - Phase 8 can overclaim from a successful first pass while meaningful
    remaining pressure is still deferred
  - examples can silently absorb awkward compiler, library, or workflow seams
    that should instead become owner-facing fixes
  - new program families can land opportunistically without a clear admission
    rule, findings taxonomy, or regression strategy
  - Phase 12 and Phase 13 follow-up work can deepen without Phase 8 applying
    renewed pressure back onto those boundaries

Core rule
---------
- Pressure programs are evidence, not a place to hide missing truth.
- Fix semantic meaning in sema, lowered truth in MIR, executable behavior in
  the backend, library contracts in stdlib, hosted startup or low-level seams
  in runtime, and workflow behavior in the driver or test harness.
- Keep the pressure suite explicit: admitted, blocked, deferred, and proven
  status must be recorded directly in repository artifacts.
- Prefer a narrower honest pressure artifact over a broader utility whose
  required owner truth is still missing or unclear.
- Reopen language, build-model, or library design only after repeated
  multi-program evidence rather than one awkward program.
- Preserve deterministic outputs, diagnostics, dumps, and test behavior so
  pressure evidence remains reviewable.

- The current repository already proves bounded real utilities including

- Deferred pressure remains explicit but unfinished: bounded threaded
  examples, evented-networking canonical entries, streaming-oriented I/O

- Some current pressure programs still intentionally avoid deeper pressure on
  concurrency, poller-driven I/O, richer diagnostics, or broader ownership
  helpers because the owning Phase 6 and Phase 7 surfaces were not yet honest
  enough to admit them.
- The repository therefore needs a second-wave Phase 8 plan that expands only
  where the enabling layers are now real enough to pressure honestly.

Phase 14A -- Admission Matrix Refresh And Canonical Re-Audit
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Refresh the repository's explicit Phase 8 admission truth so the next wave is
driven by the real remaining canonical set and current owner boundaries.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 8A -- Canonical-program closure

Tasks
~~~~~
1. Re-audit `canonical_programs.txt`, the current Phase 8 admission matrix,
   the Phase 8 findings log, and the landed first-pass examples against the
   repository's present compiler, stdlib, runtime, and toolchain posture.
2. Separate three classes explicitly: already-proven first-pass entries,
   next-wave candidates that now have honest enabling layers, and entries that
   remain blocked behind named Phase 6 or Phase 7 gaps.
3. Refresh the admission matrix so every meaningful canonical entry has a
   stated status, intended proof path, pressure focus, and named enabling gap
   if it is still blocked.
4. Reconfirm packaging decisions for each next-wave pressure program:
   direct-source, dedicated project fixture, or project plus ordinary tests.

Phase 14B -- Deferred Canonical Program Closure
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Advance the next meaningful canonical programs where the repository now has
enough truthful owner support, and record crisp enabling gaps where it does
not.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 8A -- Canonical-program closure

Tasks
~~~~~
1. Revisit the deferred canonical entries from `canonical_programs.txt`,
   especially the bounded threaded and evented-networking pressures, against
   the current stdlib, runtime, and toolchain boundaries.
2. Admit a deferred canonical entry only when the repository can state the
   required API, runtime, and proof surface honestly without speculative

4. Keep canonical programs meaningful acceptance pressure artifacts rather
   than replacing hard blocked entries with weaker surrogates.
5. Add deterministic executable or project proof for every newly admitted
   canonical entry so the admission matrix reflects repository truth, not
   planned intent.

Phase 14C -- Real-Program Family Expansion
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Expand the bounded real-program suite one family at a time so Phase 8 keeps
testing the deeper hosted slice through reviewable real utilities.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 8B -- Real-program family expansion

Tasks
~~~~~
1. Build a concrete next-wave family matrix for text processing, traversal,
   hashing, parsing, concurrency, networking, and other repository-relevant
   utility families.
2. Admit a new real utility only when it adds a distinct pressure dimension
   that the current suite does not already cover clearly.
3. Prefer bounded utilities with deterministic input and output over broader
   tools whose size would hide the actual pressure signal.
4. Use project-mode proofs whenever import roots, `.mci`, `mc run`, or
   `mc test` behavior are part of the intended pressure.
5. Keep new real programs self-contained by default, and only share helpers
   when the helper is itself intended as reusable repository-owned surface.

Phase 14D -- Pressure-Finding Routing And Design Signal Capture
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Strengthen the rule that repeated pressure findings become owner-facing work
items or design signals instead of lingering as example-local awkwardness.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 8C -- Pressure-driven finding routing
- 8D -- Design-signal capture

Tasks
~~~~~
1. Re-audit the current findings taxonomy so implementation bugs, capability
   gaps, stdlib or runtime gaps, tooling gaps, and true design signals remain
   sharply separated.
2. Route each repeated friction point to the smallest correct owner: sema,
   MIR, backend, stdlib, runtime, or driver and test harness.
3. Escalate a pattern to a design-signal or post-v0.2 follow-up only after
   repeated evidence across multiple programs, not from a single awkward case.
4. Keep Phase 12 and Phase 13 follow-up pressure explicit when Phase 8 exposes
   remaining library, runtime, `.mci`, graph, or ordinary-test weakness.
5. Update the findings log whenever a pressure program changes repository
   understanding of what is implemented, blocked, or still awkward.

Phase 14E -- Regression Hardening And Proof Maintenance
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Make the second-wave Phase 8 evidence durable by converting program pressure
into the narrowest stable proof layer that captures the real contract.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 8E -- Regression hardening from pressure findings

Tasks
~~~~~
1. For every newly admitted canonical or real program, add deterministic proof
   through the narrowest honest layer: executable tests, project workflow
   tests, sema fixtures, MIR fixtures, stdlib regressions, or tool tests.
2. Promote lower-layer failures discovered by programs into the owning test
   layer when that contract is clearer than a whole-program proof.
3. Keep the Phase 8 admission matrix, findings log, and repository proof in
   sync so no program is considered admitted without deterministic coverage.
4. Re-run earlier admitted pressure artifacts as new families land so later
   growth does not silently regress first-pass examples.
5. Preserve deterministic outputs, exit codes, diagnostics, and dump content
   across all pressure-driven regression layers.

Execution order
---------------
Recommended landing order inside Phase 14:

1. Phase 14A -- admission matrix refresh and canonical re-audit
2. Phase 14B -- next meaningful canonical closure or explicit enabling-gap
   naming
3. Phase 14E -- regression hardening alongside each newly admitted program
4. Phase 14C -- real-program family expansion once the next canonical wave is
   grounded
5. Phase 14D -- design-signal capture and backlog routing from repeated
   cross-program evidence

The order is intentional:

- the repository needs explicit admission truth before it widens the pressure
  suite further
- meaningful canonical closure should lead because it is the clearest
  acceptance pressure from the architecture texts
- regression hardening must accompany each admission step rather than wait for
  a larger batch of programs
- broader real-program growth should follow once the canonical and proof rules
  are sharper
- design questions should only reopen after repeated pressure across the
  refreshed suite, not before
