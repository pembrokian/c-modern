Problem being solved
--------------------
- The repository now has a technically acceptable narrow source-only hosted
  release claim, but technical readiness alone does not create a trustworthy
  release tag.
- A release tag still has to identify one final reviewed commit snapshot.
- The current repository work happens in a live working tree with ongoing
  phase-document edits and no commit-owned release snapshot cut as part of this
  phase.
- The repository therefore needs one explicit final decision rather than
  silently assuming that technical readiness alone implies an immediate tag.

Core rule
---------
- Separate technical readiness from maintainer-owned release execution.
- Do not treat a live working tree as equivalent to a final reviewed release
  snapshot.
- If the tag is not created here, the remaining blocker must be written down as
  one explicit plateau rather than left as implied hesitation.

Chosen slice for this phase
---------------------------
Targeted proof shape: one final repo-owned gate over the already landed
release-readiness audit plus one explicit documented tag-or-defer outcome.

Chosen outcome
--------------
Chosen outcome: defer the in-repository `v0.2` tag for one final explicit
plateau.

The one remaining plateau is:

maintainer-owned release execution.

More concretely:
- the repository still needs a maintainer to choose the final reviewed commit
  snapshot for the release
- the actual `v0.2` tag should be applied from committed repository state,
  not from a live working tree carrying in-flight edits

Why this plateau is the right blocker
------------------------------------
- Phase 78 already concluded that the current technical slice is ready enough
  for the narrow source-only hosted claim
- the remaining blocker is no longer parser, sema, MIR, backend, stdlib,
  workflow, or support-note drift inside the admitted slice
- tagging from a live working tree would blur release policy and technical
  proof in a way the earlier release-hardening waves deliberately avoided

What this phase should prove
----------------------------
- the repository now has a final explicit `v0.2` gate outcome rather than an
  implied next action
- the remaining blocker is operational and explicit, not hidden technical debt
  inside the admitted hosted slice
- a maintainer can now either cut the reviewed release snapshot and apply the
  tag or revisit only if the admitted slice changes materially

Exit criteria

- converting the current source-only hosted claim into a broader distribution

- treating a live working tree as the same thing as a final release snapshot
