Phase 301 -- Richer UI Slice Freeze And Audit
=============================================

Status
------
COMPLETED

Purpose
-------
State exactly what the post-Phase-294 richer UI slice now means, what evidence
supports it, and which tempting expansions still remain explicitly deferred.

This phase exists now because Band G should end with one durable freeze-and-
audit note rather than letting the richer-UI wave dissolve into scattered code
changes, implicit support claims, or silent framework pressure.

Repository status note
----------------------
- Phase 294 chooses richer UI composition and interaction as the next explicit
  branch.
- Phase 295 through Phase 300 spent that branch on `review_board` foreground
  interaction parity, one richer detail-selection slice, one tiny shared
  active-region truth, one bounded key-only filter-edit slice, hosted/reset-
  lane interaction parity, and one bounded persistence follow-through.
- Phase 293 already defines the previous user-facing support baseline and the
  major non-claims the repository keeps explicit.
- This phase now records that Band G succeeded on its admitted terms, what the
  richer UI slice now contains, and what still remains out of scope.

Problem being solved
--------------------
- Without a freeze-and-audit note, later phases can overstate what richer UI
  now means or forget which constraints kept the wave honest.
- That would weaken the support story, blur the loser branches, and make it too
  easy to reopen layout, input, packaging, or compiler work without a fresh
  evidence-based decision.
- The current-repo problem is therefore one missing durable account of the
  richer-UI landing, not a missing implementation feature by itself.

Non-goals
---------
- no new UI feature landing by default
- no new branch selection beyond recording the richer-UI result
- no broad roadmap rewrite unless the richer-UI outcome forces one narrow sync
- no desktop, toolkit, or package-platform claim
- no compiler reopening unless the audit proves a stronger blocker explicitly

Invariants
----------
- The freeze must describe only landed repository truth.
- The note must preserve the fixed-surface, key-only, launcher-selected,
  app-local owner map unless a prior Band G phase changed it explicitly.
- Deferred branches must remain explicit rather than becoming implied support.
- The audit must say whether Band G succeeded on its own admitted terms or hit
  an honest ceiling earlier than expected.
- The result must stay tied to maintained hosted and reset-lane evidence
  surfaces.

Required surface changes
------------------------
1. Record what richer interaction, detail, parity, and persistence behavior the
   repository now admits after Phase 295 through Phase 300.
2. Record which broader expansions remain deferred and why.
3. Sync only the narrow planning or support surfaces that must reflect the
   richer-UI freeze.
4. If Band G exposed a stronger next blocker, record that clearly enough for
   the next planning decision.

Forbidden changes
-----------------
- do not restate deferred layout, toolkit, packaging, or compiler branches as
  already-admitted capability
- do not widen the audit into a speculative wishlist
- do not hide an early stop or failed richer-UI assumption if Band G hit one
- do not rewrite support claims without tying them to maintained evidence
- do not create a second freeze note in a parallel planning surface

Phase role in the larger sequence
---------------------------------
- This phase closes Band G of the speculative post-294 roadmap.
- It determines whether the repository should next reopen the byte-literal
  compiler branch, broader packaging, or some newly exposed stronger pressure.
- It preserves the implementation evidence of the richer-UI wave in one place
  before the next branch decision or support update.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep richer UI claims bounded and evidence-backed
- preserve explicit loser branches after a successful or partial UI wave
- avoid silent drift from app-local bounded growth into framework ambitions

Questions this phase must answer
--------------------------------
1. What exact richer UI capabilities did Band G land on the maintained slice?
2. Which owner boundaries remained intact, and did any phase expose a tighter-
  than-expected ceiling?
3. Which next blocker is now strongest: compiler readability, packaging,
  another UI-local follow-through, or something else?
4. Which public or planning docs need narrow sync so the freeze stays durable?

Preferred implementation shape
------------------------------
- Start from the landed outcomes of Phase 295 through Phase 300 and the Phase
  293 support baseline.
- Prefer one compact freeze note with one loser-branch matrix over many small
  recap notes.
- Update only the narrow planning or support docs that future work will
  actually consult.
- If Band G stopped early, record that honestly rather than forcing an inflated
  “success” summary.

Implementation outcome
----------------------
- Band G landed cleanly on the admitted owner map rather than stopping early.
  The maintained richer UI slice is still one launcher-selected foreground app
  on one fixed four-cell visible surface with the existing optional two-region
  seam, one key-only input family, app-local interaction policy, and the same
  ordinary hosted and reset-lane workflow evidence surfaces.
- The additive richer interaction truth now carried by that slice is centered
  on `review_board`: Phase 295 put it fully on the admitted foreground-input
  path, Phase 296 added one app-local `Summary` or `Open` or `Closed` or
  `Urgent` detail-selection lane, Phase 297 extracted only the repeated
  `Body` or `Status` active-region truth into `stdlib/active_region.mc`,
  Phase 298 added one bounded key-only filter-edit lane with the fixed tokens
  `AL`, `OP`, `CL`, and `UR`, Phase 299 aligned hosted and reset-lane
  behavior on that same contract, and Phase 300 made the current detail view
  plus one in-progress filter-edit substate survive restart on the existing
  journal-backed lane.
- The richer UI wave therefore widened expressiveness without weakening the
  core owner split: `input_event.mc` remains key-only, `foreground_input_route.mc`
  remains a thin routing seam, `launcher_service.mc` remains the single-
  foreground owner, `display_surface.mc` remains the fixed-surface owner,
  `surface_composition.mc` remains the only admitted two-region helper, and
  app-local UI modules still own detail semantics, filter semantics, and
  present policy.
- The deferred branches remain explicit. Band G did not admit pointer or
  alternate input, a text engine, a shared control or editing framework beyond
  the tiny `active_region` value owner, richer layout or multi-surface policy,
  a desktop shell, a widget toolkit, a package platform, or broader retained
  UI-state machinery.
- The strongest next blocker exposed by the completed richer UI wave is still
  the byte-literal compiler readability slice already identified in Phase 291.
  Band G increased repeated `u8`-shaped key and cell pressure on the current
  admitted UI owners, while broader packaging still lacks a second installed-
  program consumer because `issue_rollup` remains the only installed path and
  `review_board` remains direct-launch.

Validation
----------
- Editor diagnostics on the touched planning and support documents.
- No executable validation was required because this phase did not change any

Phase review against spec
-------------------------
- Preserved invariants:
  the freeze records only landed repository truth, keeps the fixed-surface,
  key-only, launcher-selected, app-local owner map intact, and keeps deferred
  loser branches explicit.
- Ownership boundaries did not weaken:
  Band G added only one tiny shared value owner, `stdlib/active_region.mc`,
  while interaction semantics, persistence meaning, and render policy stayed on
  the existing app-local, journal, launcher, and display owners.
- Accidental centers of gravity were avoided:
  no toolkit, framework, registry, or new runner emerged from the richer UI
  wave, and the maintained proof surfaces remain the hosted UI real-project
  targets plus `mc_tool_workflow_kernel_reset_lane_unit`.
