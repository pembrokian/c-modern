Problem being solved
--------------------
- The speculative Canopus kernel roadmap reserved Phase 109 for the first
  running-kernel slice audit after Phase 108 closed image-input and program-
  cap contract pressure.
- The repository already had the bounded mechanisms and service proofs needed
  for that statement, but there was still no single canonical support note
  naming exactly what the repository-owned kernel now admits and what it still
  defers.
- Without this step, the repository could drift in two bad directions at the
  same time: undersell the landed kernel by treating it as still merely
  speculative, or oversell it by implying a general Canopus platform rather
  than one bounded first running kernel slice.

Core rule
---------
- Keep the phase to one support-statement owner only.
- Reuse the landed kernel artifact and prior proof validators rather than
  inventing a new mechanism path.
- State the admitted running slice explicitly in documentation and validate it
  against the same owned kernel artifact that Phase 108 already builds.
- Keep all non-goals explicit so the first running-kernel statement does not
  blur into platform, loader, or multi-service completion claims.

1. one canonical Phase 109 note that names the first running Canopus kernel
