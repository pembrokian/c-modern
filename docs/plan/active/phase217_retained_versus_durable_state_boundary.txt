Phase 217 -- Retained Versus Durable State Boundary
===================================================

Status
------
LANDED

Purpose
-------
Make the retained-versus-durable boundary explicit in both maintained planning
surfaces and executable runtime behavior, while keeping storage and persistence
mechanisms out of scope.

This phase is a real implementation phase, not a framework phase.
The landing is intentionally narrow and simple: one additive shell-visible
query over the existing lifecycle control family plus aligned planning updates.

Repository status note
----------------------
- The running-system slice already admits explicit retained reload and explicit
  coordinated retained restart for fixed targets.
- The running-system slice does not admit durable storage, naming persistence,
  durable logs, or recovery beyond explicit restart reload.
- Prior phases made retained summary, retained policy, authority inspection,
  identity taxonomy, and compact service-state inspection visible.

Problem solved
--------------
- Before this phase, the repository had retained-state truth but no direct,
  executable retained-versus-durable boundary query.
- That made it easier to over-read retained reload as if it were durable
  persistence.

Implemented slice
-----------------
1. Add one shell-visible boundary query to the existing lifecycle family:
   `XD<target>!`.
2. Keep ownership explicit:
   - protocol shape in `kernel/src/serial_protocol.mc`
   - shell command routing in `kernel/src/shell_service.mc`
   - reply execution in `kernel/src/kernel_dispatch.mc`
   - boundary encoding in `kernel/src/service_state.mc`
3. Keep the contract compact and deterministic:
   reply payload is four bytes: `<target, retained_or_none, durable_or_none,
   durable_requirements_mask>`.
4. Keep durable state unadmitted in implementation:
   `durable_or_none` is currently `none` for all targets.
5. Publish one bounded reset-lane executable follow-through for the new query.

Boundary contract admitted in this phase
----------------------------------------
Retained state means:
- explicit owner-local in-memory state
- optional reload across explicit restart in the current boot image
- no durability guarantee outside that bounded restart context

Durable state would require all of:
- persistence beyond in-memory restart context
- explicit durable naming or addressability
- explicit recovery semantics beyond current retained reload

Executable contract
-------------------
- `service_state.state_durability_payload(target)` now returns:
  - byte 0: target
  - byte 1: retained class (`none` or `retained`)
  - byte 2: durable class (`none` today)
  - byte 3: durable requirements mask
    - persistence requirement
    - naming requirement
    - recovery requirement
- This keeps the phase honest without introducing a storage subsystem.

Non-goals preserved
-------------------
- no filesystem implementation
- no durable logs or append-only storage subsystem
- no mountable namespace model
- no persistent object-id framework
- no compiler, sema, MIR, backend, ABI, target, runtime, stdlib, or `hal`
  expansion into storage mechanisms

Exit status
-----------
- one explicit retained-versus-durable boundary now exists in maintained docs
  and executable runtime behavior
- active support wording no longer blurs retained reload with durable
  persistence
- no storage framework was introduced
