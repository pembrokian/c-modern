Phase 151 -- Complete The Main Simplification
=============================================

Status
------
IMPLEMENTED

Purpose
-------
Continue from the landed Phase 150 rebuilt-system step by finishing the
structural simplification of `kernel/src/main.mc` so that it becomes a thin
bootstrap orchestrator rather than the owner of phase-local proof
choreography.

This document defines the next bounded cleanup phase after the service,
init-runtime, child-execution, and rebuilt-system extractions already landed.
It does not claim a new service wave, a wider kernel architecture, or a
generic framework layer.

Phase target
------------
Phase 151 should complete the main-file simplification by removing the
remaining large proof and workflow ownership bands from `kernel/src/main.mc`
without changing the admitted bounded kernel behavior.

The intended repository result is:

- `kernel/src/main.mc` owns boot sequencing, phase sequencing, marker
  emission, and final status checks only
- phase-local workflow construction moves to the nearest existing owner such
  as `bootstrap_services`, `bootstrap_proofs`, `bootstrap_audit`, `init`, or
  other already-admitted kernel modules
- the remaining runtime snapshots stay explicit, but their mutation and
  validation no longer force `main.mc` to host the full proof narrative
- the focused freestanding runtime surface continues to prove the same Phase
  105 through Phase 150 behavior after the refactor

Problem being solved
--------------------
- The repository has already simplified state ownership, but not yet control
  ownership.
- As long as `main.mc` still hosts large, phase-local workflow bodies, the
  kernel proof path remains difficult to audit, difficult to extend honestly,
  and vulnerable to reintroducing ad hoc coupling.
- Without one explicit completion phase, later work will keep building on a
  partially simplified root artifact and the repository will lose the chance
  to state clearly what `main.mc` is supposed to own.

Core rule
---------
- Simplify ownership, do not relocate clutter blindly.
- `main.mc` should keep orchestration responsibilities only: boot ordering,
  sequencing between admitted helpers, phase markers, and final return-code
  decisions.
- If logic computes service-local meaning, phase-local workflow state, or
  audit-local comparisons, it should move to the nearest existing owner
  rather than remain in `main.mc`.
- Do not answer this phase by inventing a generic runtime framework,
  scheduler layer, or dynamic registration model.
- Debug audits, MIR projections, and runtime descriptors remain acceptance
  artifacts only.
- Keep compiler, sema, MIR, backend, ABI, target, runtime, and `hal`
  surfaces closed unless a concrete extraction blocker forces a narrow change.

Planning rule for this band
---------------------------
Use this sanity check on every Phase 151 change:

- shrinking `main.mc` by moving phase-local workflow ownership to an already
  correct module: admissible
- replacing explicit ownership with a framework, callback table, or generic
  service runtime: wrong layer

The goal is a thinner root orchestrator, not a more abstract system.

Controlling simplification gaps
-------------------------------
Phase 151 treated these as the controlling gaps and closed them as follows:

1. Early boot helper ownership needed to stop dominating the root file
   - `bootstrap_main()` no longer hosts the full early boot branch ladder,
     and the live boot and first-user workflow now routes through the already
     admitted `bootstrap_runtime` owner surface.

2. Cross-phase validation state needed to stop dominating the root file
   - hardening flags, repeated config construction, and repeated audit-input
     assembly for the live mid-phase path now route through
     `bootstrap_proofs.execute_mid_phase_proofs(...)` and its grouped runtime
     and result records rather than through root-owned choreography.

Desired end-state for `main.mc`
-------------------------------
When Phase 151 is complete, `kernel/src/main.mc` should read as:

1. reset and seed kernel state
2. transfer into first-user setup
3. execute the admitted early IPC and child lifecycle bring-up
4. delegate mid-phase proof composition to one bounded owner-local surface
5. delegate late-phase service-system proofs to one bounded owner-local
   surface
6. emit late markers and perform final bounded status checks

The file should no longer be the primary home of service scenario assembly,
restart scenario assembly, or large audit-input choreography.
