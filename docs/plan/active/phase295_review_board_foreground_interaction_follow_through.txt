Phase 295 -- Review Board Foreground Interaction Follow-Through
===============================================================

Status
------
COMPLETED

Purpose
-------
Let `review_board` consume the admitted key-only foreground-input path through
its own app-local owner, so the second directly usable UI app stops depending
on a narrower interaction story than `issue_rollup`.

This phase exists now because Phase 294 chose richer UI as the strongest next
wave, and the first honest follow-through is to make both admitted apps live
on the same launcher-selected foreground-input contract before adding any
deeper interaction slice.

Repository status note
----------------------
- Phase 289 freezes the current bounded UI baseline: one fixed four-cell
  surface, one optional two-region seam, one key-only input family, and one
  launcher-selected foreground app.
- Phase 290 proves that `issue_rollup` and `review_board` both participate in
  one integrated small-system demo, but only `issue_rollup` has been the main
  consumer of the admitted foreground-interaction path.
- `kernel/src/review_board_app.mc` already owns app-local display and key
  handling, and `examples/real/review_board/` already proves hosted app logic.
- This phase lands the explicit maintained proof that `review_board` now
  participates as a first-class consumer of the same launcher-selected
  foreground-input contract on both the reset-lane and hosted surfaces.

Problem being solved
--------------------
- The current richer-UI direction is not yet honest if one admitted app still
  benefits from the foreground-input model while the second app effectively
  stays on a thinner interaction lane.
- Without this follow-through, later UI phases risk designing selection,
  filter, or persistence behavior on top of app-specific shortcuts rather than
  the actual admitted launcher-selected input path.
- The current-repo problem is therefore uneven use of the existing interaction
  substrate across the two admitted apps, not the absence of a broader input
  framework.

Non-goals
---------
- no new input family beyond the admitted key-only path
- no pointer, mouse, touch, IME, or text-engine work
- no shared UI controller between `issue_rollup` and `review_board`
- no launcher redesign, focus stack, or multi-foreground policy
- no layout or composition expansion beyond the current fixed surface

Invariants
----------
- `kernel/src/foreground_input_route.mc` remains the routing seam for admitted
  foreground input rather than becoming a shared interaction policy owner.
- `kernel/src/services/launcher_service.mc` remains the owner of foreground app
  truth rather than app-local key semantics.
- `kernel/src/review_board_app.mc` remains the owner of `review_board` key
  meaning, present behavior, and app-local state updates.
- `kernel/src/input_event.mc` stays key-only; this phase does not widen event
  families or transport vocabulary.
- Hosted and reset-lane proofs must continue to use the maintained real-project
  and grouped workflow surfaces rather than a new UI-specific runner.

Required surface changes
------------------------
1. Route `review_board` through the admitted launcher-selected foreground-input
   path on the maintained kernel path rather than through any app-specific
   shortcut.
2. Update the owning `review_board` app wrapper so the app-local key-handling
   path is exercised through that foreground route.
3. Add or update one narrow reset-lane scenario or fixture that proves
   `review_board` input travels through the same admitted route as the other
   foreground app.
4. Keep the hosted `examples/real/review_board/` proof aligned with the same
   interaction contract.

Forbidden changes
-----------------
- do not widen `foreground_input_route.mc` into a shared control policy layer
- do not move `review_board` key semantics into `launcher_service.mc`
- do not add a second foreground app path, focus manager, or app registry
- do not introduce a separate review-board-only input runner or workflow
- do not solve the phase by adding synthetic test hooks that bypass the real
  launcher-selected route

Phase role in the larger sequence
---------------------------------
- This phase is the first concrete spend of the Phase 294 richer-UI winner.
- It establishes one equal interaction baseline across the two admitted apps
  before the repository spends deeper UI pressure on selection, detail, or
  typed-input behavior.
- If it lands honestly, later phases can compare the two apps on the same
  input contract instead of mixing app-local shortcuts with true substrate use.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- richer UI growth on the existing fixed-surface, launcher-selected app model
- keep cross-app interaction work on app-local owners and the existing input
  route instead of widening into a UI framework
- preserve hosted and reset-lane evidence parity for user-facing features

Questions this phase must answer
--------------------------------
1. What is the smallest honest route that lets `review_board` consume the
   admitted foreground-input path?
2. Which owner should decide whether a key changes `review_board` state, and
   which owner should only route the key?
3. What reset-lane proof best shows that the path is real without inventing a
   second UI runner?
4. What hosted proof should remain the primary app-local behavior surface?

Preferred implementation shape
------------------------------
- Reuse the existing launcher-selected foreground-input route and keep the
  phase focused on bringing `review_board` onto that seam.
- Let `kernel/src/review_board_app.mc` continue to translate keys into app-
  local state and display updates.
- Keep `foreground_input_route.mc` and `launcher_service.mc` thin: they should
  select the app and forward keys, not interpret review-board behavior.
- Spend the proof on one narrow reset-lane scenario plus the existing hosted
  `review_board` real-project target.

Implementation notes
--------------------
- The controlling route and app-local owners were already correctly split:
  `foreground_input_route.mc` still selects the launcher-owned foreground app,
  and `review_board_app.mc` still decides what keys mean for `review_board`.
- The maintained reset-lane proof was re-homed under
  `tests/system/kernel_reset_lane_phase295_review_board_foreground_interaction_follow_through/`
  so the grouped workflow suite now records this interaction contract as a
  distinct Phase 295 owner instead of leaving it on the older Phase 286 label.
- The hosted `examples/real/review_board/` proof remained on the existing
  app-local `ui` target, which keeps parity on the same key-handling contract
  without adding a second runner or a review-board-specific input harness.

Phase review against spec
-------------------------
- Preserved invariants:
  `foreground_input_route.mc` stayed a thin routing seam, `launcher_service.mc`
  stayed the owner of foreground-app truth, `review_board_app.mc` stayed the
  owner of review-board key meaning, and `input_event.mc` remained key-only.
- Ownership boundaries did not weaken:
  no review-board key semantics moved into launcher or dispatch, and no shared
  UI controller or extra input family landed.
- Accidental centers of gravity were avoided:
  the implementation stayed on one existing app-local UI owner, one existing
  routing seam, one reset-lane fixture, and one hosted real-project proof.
