Phase 279 -- First Human-Facing App Proof
=========================================

Status
------
COMPLETED

Purpose
-------
Make one repository-owned app directly usable through the first admitted UI
surface so Band D proves a real human-facing outcome rather than isolated UI
mechanisms.

This phase exists now because the prior Band D slices should establish input,
display, routing, repaint, and one bounded rendering path. The next honest move
is to spend those owners on one app, not to widen the UI stack further.

Repository status note
----------------------
- `issue_rollup` is already the bounded app consumer used by the Band C
  installed-workflow demo.
- The app already has app-local parse, model, core, and render code under
  `examples/real/issue_rollup/src/`.
- Earlier Band D phases should establish the explicit UI path it needs.
- The repository still lacks one directly usable human-facing app proof on that
  path.

Problem being solved
--------------------
- Without one directly usable app, the first UI band would risk ending as a set
  of owner-local surface experiments.
- The repository needs one concrete user-facing proof that input, display,
  routing, repaint, and rendering compose into something legible.
- A bounded app proof is also the honest way to expose which UI pressures are
  still real before restart or audit work lands.

Non-goals
---------
- no second UI app in the same phase
- no widget library or shared control family
- no window manager, panels, or region splitting
- no broader app platform or session model
- no new package-management or multi-app install work

Invariants
----------
- The first app proof must remain one small repository-owned app.
- Launcher remains the foreground-app policy owner.
- The UI surface remains bounded to the earlier Band D contracts.
- The app proof must not claim a general toolkit or desktop framework.
- Band C install, manifest, and restart truths remain bounded to the admitted
  `issue_rollup` path.

Required surface changes
------------------------
1. Choose one repository-owned app as the first human-facing UI proof.
2. Wire that app through the admitted Band D input, display, routing, repaint,
   and rendering contracts.
3. Add one focused reset-lane fixture proving direct human-facing usability.
4. Add any narrow real-project validation needed for the chosen app.

Forbidden changes
-----------------
- do not admit a second app to justify framework growth
- do not widen the proof into widgets, menus, or a dashboard family
- do not move app-local state or render policy into launcher or the display
  owner
- do not use the app proof as a pretext for package or session work

Phase role in the larger sequence
---------------------------------
- This phase is the outcome checkpoint for the first half of Band D.
- It proves whether the admitted UI path is already strong enough to support a
  real app before restart-follow-through and audit work.
- It sets the baseline for the later UI audit.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep first-UI work tied to one concrete app result rather than framework
  growth
- keep the post-Phase-273 app path bounded to the admitted `issue_rollup`
  consumer unless a second real consumer proves otherwise

Questions this phase must answer
--------------------------------
1. Is `issue_rollup` still the smallest honest first UI app proof?
2. What visible interaction is sufficient for the app to count as directly
   usable?
3. Which app-local code must stay app-local rather than becoming UI framework
   surface?
4. What remaining friction becomes visible once the first app is actually
   usable on the admitted surface?

Preferred implementation shape
------------------------------
- Reuse `issue_rollup` as the first UI app proof unless a smaller existing app
  is clearly stronger.
- Keep the proof narrow: one app, one surface, one foreground path.
- Reuse the Band C install and launcher path instead of inventing another app
  activation story.
- Keep app logic inside `examples/real/issue_rollup/`; only the explicit UI
  contracts should be shared.

Implementation summary
----------------------
- Reuse `issue_rollup` as the first directly usable app and keep the new
  interaction model app-local in
  `examples/real/issue_rollup/src/app/issue_rollup_interactive.mc`.
- Keep the visible interaction bounded to one tiny key vocabulary over the
  admitted input path: `O` appends one ordinary open issue, `P` appends one
  priority open issue, and `R` resets the app back to an empty rollup.
- Recompute visible state through the existing repository-owned app path:
  app-local text buffer -> `rollup_parse.summarize_text(...)` ->
  `rollup_render.rollup_display_cells_with_manifest(...)` ->
  `display_surface.display_present(...)`.
- Keep launcher on foreground-app policy, display on visible-cell ownership,
  and the kernel-side `issue_rollup_app.mc` as a thin wrapper over the
  repository-owned app logic instead of a second app-policy owner.
- Add one app-local real-project regression and one focused reset-lane fixture
  that prove the visible `EMTY` -> `STDY` -> `BUSY` -> `ATTN` -> `EMTY`
  interaction path without widening into text editing, widgets, or another app.

Acceptance criteria
-------------------
- one repository-owned app is directly usable through the admitted UI path

Phase review against spec
-------------------------
- Met: `issue_rollup` remains the single repository-owned app proof and now has
  one directly usable visible interaction path on the admitted UI surface.
- Met: app logic stayed app-local. The new interaction reducer lives under
  `examples/real/issue_rollup/` and the kernel wrapper now delegates to that
  owner instead of carrying a second summary-state machine.
- Met: launcher remains the foreground-policy owner and display remains the
  fixed-cell surface owner. Neither owner absorbed app parsing, issue state, or
  render policy.
- Met: the proof stayed deterministic and bounded to one tiny key vocabulary,
  one four-cell surface, and one explicit installed-app path.
- Preserved invariants: no second app, no widget family, no window manager,
  no session model, and no package-management widening landed as part of the
  proof.
- Deferred intentionally: richer text entry, broader app-local editing,
  restart-follow-through for the first app, and any repeated UI friction that
  still needs a later audit.
