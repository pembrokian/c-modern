Phase 314 -- Universal Resource Model Freeze
============================================

Status
------
PROPOSED

Purpose
-------
Freeze the post-308 universal resource model as the current architectural truth
for the next durable implementation band, stating exactly what is now admitted,
what current owners must converge toward, and which tempting expansions remain
deferred.

This phase exists now because Phases 309 through 313 are expected to build the
full architecture checkpoint for the next durable implementation wave. The
repository needs one durable plateau note before implementation starts, not a
half-settled architecture that later phases silently reinterpret.

Repository status note
----------------------
- The roadmap now expects an architecture-first Band J before the first real
  app-facing durable slice.
- The new sys semantics notes publish the intended umbrella contract, resource
  model, and common service contract.
- Phases 309 through 313 are expected to audit divergence, commit identity and
  durability semantics, map current owners, and choose the first durable
  consumer and owner boundary.
- The repository still needs one explicit freeze note that turns those results
  into stable truth for the next implementation band.

Problem being solved
--------------------
- Without a freeze checkpoint, the architecture phases can still be treated as
  advisory commentary and later implementation work can reopen the same
  questions opportunistically.
- That would erase the main value of the architecture wave just before the
  first real durable slice tries to spend it.
- The current-repo problem is therefore one missing plateau note for the new
  universal resource model, not another design exploration phase.

Non-goals
---------
- no durable consumer implementation by default
- no new architecture branch selection beyond what the prior phases already
  decide
- no broader storage framework design
- no language-feature landing
- no roadmap rewrite beyond the narrow surfaces that must reflect the freeze

Invariants
----------
- The freeze must be driven by the actual results of Phases 309 through 313.
- The shared resource model must remain smaller than a filesystem or generic
  storage framework.
- The chosen first durable consumer and owner boundary must stay explicit.
- Deferred expansions such as discovery, directories, path graphs, multi-
  consumer policy, and broader syntax work must remain deferred explicitly.
- Planning churn must remain narrow and proportional to the checkpoint.

Required surface changes
------------------------
1. Record the admitted universal resource model as the current architectural
   truth.
2. Record the chosen first consumer and chosen durable owner boundary.
3. Record which expansions remain explicitly deferred.
4. Sync only the narrow roadmap, backlog, or architecture surfaces that must
   reflect the frozen result.

Forbidden changes
-----------------
- do not reopen identity, naming, durability, or owner-choice questions in the
  freeze note unless the earlier phases failed to answer them honestly
- do not land the first durable implementation slice inside the freeze
- do not widen the freeze into a broader storage or syntax wishlist
- do not let deferred expansions drift back in through vague language
- do not hide unresolved contradictions if they still remain

Phase role in the larger sequence
---------------------------------
- This phase closes Band J by turning the architecture sequence into one stable
  implementation checkpoint.
- It prepares Phase 315 and the rest of Band K by giving the first durable
  slice one explicit contract source.
- It also sets the baseline that later language work must justify against real
  implementation pressure rather than roadmap habit.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- the active backlog themes around one dominant resource, naming, durability,
  restart, and capability story
- the pressure to keep the first durable slice smaller than a filesystem or app
  framework
- the pressure to make later language work follow real implementation residue
  rather than planning momentum

Questions this phase must answer
--------------------------------
1. What exact universal resource model is now admitted as current truth?
2. Which current-owner classifications and deferred expansions are now stable?
3. Which first durable consumer and owner boundary are now the implementation
   checkpoint?
4. Which planning surfaces must reflect the freeze so later phases do not
   relitigate it immediately?

Preferred implementation shape
------------------------------
- Start from the actual landed results of Phases 309 through 313.
- Prefer one short plateau note with one admitted model, one chosen consumer,
  one chosen owner boundary, and one loser matrix for deferred expansions.
- Update only the architecture and planning surfaces that later work will
  actually consult.

Common breakage map
~~~~~~~~~~~~~~~~~~~
- Freeze note quietly reopens questions the earlier phases were meant to close
  -> the implementation checkpoint remains unstable.
- Freeze note hides still-unresolved contradictions -> later implementation
  phases discover architectural ambiguity too late.
- Roadmap or backlog sync widens too far -> planning churn exceeds the bounded
  checkpoint role.
- Deferred expansions are not recorded explicitly -> later phases reintroduce
  them by convenience.
