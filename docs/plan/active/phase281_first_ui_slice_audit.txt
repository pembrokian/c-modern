Phase 281 -- First UI Slice Audit
=================================

Status
------
COMPLETED

Purpose
-------
Close the first Band D UI wave by stating which input, display, routing,
presentation, rendering, and app-proof pressures are now maintained truth and
which still remain too weak to justify broader UI framework work.

This phase exists now because the repository should stop after one deliberately
narrow human-facing wave and classify what it actually proved before starting a
second layer of UI composition or alternate input work.

Repository status note
----------------------
- Phase 273 already classified the first UI band as the stronger next owner.
- Phases 274 through 280 now admit one bounded input owner, one display
  owner, one single-foreground input route, one explicit present contract,
  one fixed rendering path, one directly usable app proof, and one
  restart-or-update follow-through on that path.
- The repository will then have touched genuine UI work, but only through one
  narrow human-facing lane.
- The remaining question is whether any stronger still-open pressure outranks a
  second UI layer or whether Band E now honestly wins.

Problem being solved
--------------------
- Without an explicit UI closeout audit, the repository could slide from the
  first human-facing proof into windows, widgets, pointer input, or editor-like
  work without stating what the first UI slice actually solved.
- The repository needs one honest answer on which UI surfaces are now admitted
  truth and which remain explicitly deferred.
- This is a current planning problem, not a generic product-roadmap summary.

Non-goals
---------
- no new UI feature landing by convenience
- no implicit promotion of the first UI slice into a desktop or toolkit claim
- no hidden Band E planning inside the audit
- no silent reinterpretation of earlier Band D outcomes

Invariants
----------
- The audit must classify actual landed Band D outcomes only.
- Closed UI surfaces must remain bounded to what Phases 274 through 280 really
  proved.
- Still-open UI pressures must stay concrete and owner-local.
- The audit must not claim windows, widgets, pointer input, or composition
  layers unless the landed evidence actually supports them.

Required surface changes
------------------------
1. Re-audit the actual outcomes of Phases 274 through 280.
2. Publish which first-UI surfaces are now maintained truth.
3. Publish which still-open UI pressures remain deferred and why.
4. State whether Band E now honestly wins over more follow-through inside the
   first UI slice.
5. Tighten backlog or decision-log entries only where the audit exposes a real
   still-open pressure.

Forbidden changes
-----------------
- do not land new input, display, or widget features inside the audit
- do not treat one bounded human-facing app proof as a desktop or toolkit
- do not bury still-open blockers under vague polish language
- do not widen the audit into speculative graphics or product strategy

Phase role in the larger sequence
---------------------------------
- This phase closes the first Band D UI wave.
- It determines whether the repository is ready for Band E's next UI layer
  honestly.
- It preserves the evidence-first planning style by stopping after one narrow
  visible path and stating what is actually true.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep first-UI growth bounded to explicit input, display, and foreground-app
  seams
- keep composition, alternate input, and control-family growth deferred until a
  second real UI pressure demands them

Questions this phase must answer
--------------------------------
1. Which first-UI slices landed cleanly enough that later UI phases can rely on
   them as maintained truth?
2. Which still-open UI pressure, if any, remains stronger than Band E's next
   owner?
3. Did the first human-facing app proof expose one missing owner that still
   belongs in Band D rather than Band E?
4. What exact evidence now supports starting the next UI layer honestly?

Preferred implementation shape
------------------------------
- Keep the phase document-focused and evidence-based.
- Reuse the plateau and frontier-audit style already used by the repository.
- Prefer short explicit classifications over broad UI narrative.

Implementation outcome
----------------------
- Band D is now classified as complete for its admitted slice.
- The maintained first-UI truths are:
  - one explicit input-event owner, `input_event.mc`, for the admitted
    `I K <byte> !` path
  - one fixed four-cell display owner, `display_surface.mc`, with explicit
    present and query semantics
  - one single-foreground input-routing seam, `foreground_input_route.mc`,
    that consumes launcher-owned foreground truth instead of inventing a second
    focus owner
  - one app-owned present seam, `issue_rollup_app.mc`, that keeps render
    policy in the `issue_rollup` render modules instead of widening display or
    launcher ownership
  - one fixed readable four-cell rendering contract,
    `EMTY` or `STDY` or `BUSY` or `ATTN`, shared by the hosted and reset-lane
    `issue_rollup` paths
  - one directly usable human-facing `issue_rollup` app proof on that same
    bounded path
  - one launch-only restart-or-update overlay,
    `FRSH` or `RSUM` or `INVD`, that reuses launcher and update-store truth
    without introducing retained UI state
- The audit found no stronger still-open Band D owner than Band E's next
  bounded UI layer.
- The still-open UI pressures remain real but weaker than the completed first
  slice:
  - one bounded composition or region owner only if a second real visible
    region or app pressure appears
  - one second input family only if a real admitted app needs more than the
    current `I K <byte> !` path
  - one bounded focus or selection or shared control owner only if repeated
    app-local UI code proves it
  - one bounded document or view persistence follow-through only if a real app
    needs state beyond the current launch-only lifecycle overlay
- The first human-facing proof did not expose a missing Band D owner in input,
  display, routing, launcher, or the app-owned present path. It instead
  confirmed that the current bounded split is sufficient for one narrow UI
  lane.

Phase review against the spec
-----------------------------
- Preserved invariants:
  - the audit classifies only the landed Phase 274 through Phase 280 outcomes
  - closed surfaces stay bounded to one explicit input owner, one fixed
    display owner, one single-foreground route, one app-owned present seam,
    one fixed cell-rendering contract, one human-facing `issue_rollup` proof,
    and one launch-only lifecycle overlay
  - deferred pressures remain concrete and owner-local instead of being buried
    under vague polish or toolkit language
- Ownership boundaries did not weaken:
  - `live_receive.mc` remains the hosted polling and frame-assembly owner
  - `launcher_service.mc` remains the single-foreground and lifecycle-
    classification owner
  - `display_surface.mc` remains the fixed-surface owner only
  - `issue_rollup_app.mc` remains the app-local present consumer rather than a
    general UI framework owner
  - shell and dispatch remain routing seams instead of becoming event, render,
    or lifecycle policy owners
- Accidental centers of gravity were avoided:
  - no window system, widget toolkit, pointer stack, composition framework, or
    retained UI-state owner was introduced by the audit
  - no launcher-owned display policy, display-owned render policy, or
    update-store-owned app lifecycle policy was claimed
- Honest non-decisions and deferred work:
  - no second visible region or surface owner yet
  - no second input family yet
  - no shared control vocabulary or focus model yet
  - no document or view persistence beyond the launch-only overlay yet
