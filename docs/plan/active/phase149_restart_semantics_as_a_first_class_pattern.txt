Phase 149 -- Restart Semantics As A First-Class Pattern
=======================================================

Status
------
COMPLETE

Purpose
-------
Continue from the completed Phase 148 authority-ergonomics step by turning the
landed Phase 145 restart result into one explicit restartable-service
contract rather than leaving restart semantics as a one-off audit outcome.

This document records the landed Phase 149 slice.
It defines one bounded restart-contract phase that lands as explicit caller-
visible and init-owned service truth before later subsystem work is allowed
to depend on the current small-system pattern.
It does not claim a kernel supervisor, durable recovery, transparent retry,
or a broader reliability framework.

Phase result
------------
Phase 149 is complete as one bounded restart-contract audit over the landed
interactive-service slice.

The landed repository slice now includes:

- one explicit restartable-service contract over the existing key-value
  restart path rather than a phase-local restart anecdote
- one caller-visible statement that service death leaves retry obligation with
  the caller and that retry means issuing a fresh request rather than relying
  on transparent recovery
- one explicit statement that post-restart state truth remains visible,
  including the current reset-to-empty key-value state after restart
- one explicit statement that restart policy remains init-owned while request
  meaning and reset-state consequences remain service-owned and shell-visible
- one focused Phase 149 audit and one focused freestanding regression over
  the same published restart-contract boundary

Repository status note
----------------------
- Earlier phases already proved bounded service death, partial failure, and
  one explicit replacement path.
- Phase 145 then applied that truth to one user-meaningful key-value service
  and exposed manual retry and post-restart empty-state truth explicitly.
- Phase 148 then confirmed that restart-related authority handoff remains
  explicit and init-owned under repeated use.
- The repository therefore had restart evidence already, but it still needed
  one published statement about what a restartable service means in the
  admitted Canopus slice.

Problem being solved
--------------------
- Later subsystem work cannot depend on the current service pattern safely if
  restart meaning remains implicit or phase-local.
- Without one explicit restart contract, future work could blur caller
  obligations, idempotency assumptions, and post-restart truth into ad hoc
  retry behavior or hidden policy.
- The repository therefore needs one bounded phase that names the observable
  contract of service death and rebirth while keeping restart policy explicit
  in init and service owners.

Core rule
---------
- Formalize restart semantics; do not hide them.
- Restart policy remains explicit in init and service owners. The kernel may
  expose loss and replacement, but it must not absorb supervision,
  transparent recovery, or retry policy.
- Phase completion requires one real artifact path that demonstrates the
  published restartable-service contract.
- Debug audits, MIR projections, and tool regressions are acceptance artifacts
  only.
- Keep compiler, sema, MIR, backend, ABI, target, runtime, and `hal`
  surfaces closed.

Planning rule for this band
---------------------------
Use this sanity check on every Phase 149 change:

- making caller-visible failure, retry obligation, and post-restart state
  truth explicit: admissible
- hiding those consequences behind automatic rebinding, transparent retry, or
  kernel-owned recovery: wrong layer

Do not answer restart ambiguity by widening into a reliability subsystem. Keep
the phase bounded to one explicit restartable-service contract and the
smallest real workflow needed to demonstrate it.

   service contract for the admitted Canopus slice

   restart, including retry and reset-state consequences

   than service-owned or kernel-owned
4. one real repeated workflow that demonstrates the published restart contract
5. one audit-owned Phase 149 proof that the contract matches the real
   artifact rather than retrospective interpretation

Chosen outcome

Chosen outcome: landed with one explicit restartable-service contract and no
broader recovery or compiler reopening.

The landed key-value restart path was already strong enough to carry the Phase
149 contract honestly. The repository therefore published the contract over
that real artifact instead of adding new restart machinery, transparent retry,
or request-tracking infrastructure.
