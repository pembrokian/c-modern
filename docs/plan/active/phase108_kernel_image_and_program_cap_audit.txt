Problem being solved
--------------------
- The speculative Canopus kernel roadmap names kernel image and program-cap
  audit pressure as the next bounded owner after the landed user-to-user
  transfer proof.
- The repository already had most of the necessary truth, but it was spread
  across different places: `kernel/build.toml` carried explicit target,
  freestanding environment, package identity, and startup ownership; the
  build path emitted a module object, runtime object, and linked executable;
  and the kernel proof lane already seeded and consumed explicit program
  capabilities for init, log service, echo service, and transfer service.
- Without this step, later support notes could overclaim that the repository
  already has an honest kernel-image contract when the real emitted artifact
  ownership and program-cap provenance were still only implicit across build
  outputs and proof code.

Core rule
---------
- Keep the phase to one audit owner only.
- Reuse the landed kernel artifact and service proofs rather than inventing a
  new runtime path.
- Validate the actual emitted image-input artifacts the current build already
  produces.
- Keep program-cap provenance explicit in the root proof surface and bootstrap
  handoff rather than implying a general loader or ambient authority source.
- Stop before boot-image assembly, linker scripts, install trees, shared
  libraries, or general target-family promises.

Chosen outcome
--------------
Chosen outcome: yes.

The repository should cross this audit boundary now because the actual kernel
mechanisms and bounded service-policy proofs are already landed. The next
honest owner is not more mechanism. It is making the emitted image-input
artifacts and program-cap source contract explicit and reviewable.

What this phase proves
----------------------
- the real kernel manifest keeps target identity, freestanding environment,
  root module, package identity, and startup ownership explicit
- the real kernel build emits reusable image-input artifacts consisting of
  the lowered module objects and generated freestanding runtime object, and a
  target-owned relink step can reconstruct the running artifact from those
  emitted pieces without hidden extra driver state
- the program capability that bootstraps `init` remains explicit in the
  bootstrap handoff, while later service program capabilities remain seeded
  and consumed inside the owned proof lane rather than appearing ambiently
- the repository can now stop honestly at one kernel-image and program-cap
  audit without implying boot-image packaging, broad target support, or a
  final running-system claim
