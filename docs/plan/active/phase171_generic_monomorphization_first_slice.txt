Phase 171 -- Generic Monomorphization First Slice
==================================================

Status
------
PLANNED

Purpose
-------
Admit one bounded step of generic function monomorphization if and only if
real repository pressure justifies closing the gap left by the Phase 49
erased-ABI closure.

Phase 49 closed the admitted direct-call family needed by
`sync.thread_spawn<T>` through erased-ABI generic declarations.  It
deliberately stopped before general monomorphization: specialized codegen for
functions whose ABI depends on unresolved type parameters.

This phase is the first honest checkpoint for that deferral.

Repository status note
----------------------
- The backlog records: "reopen only when a second admitted imported generic
  procedure-typed consumer or another executable proof needs generic support
  beyond the erased-ABI family landed in Phase 49."
- The trigger condition is a second real admitted consumer, not a design
  preference for monomorphization.
- Before this phase begins, the repository should audit whether any kernel,
  stdlib, or real-project consumer currently requires ABI-specialized generic
  functions that are not already covered by the erased-ABI family.
- If no such consumer exists, this phase concludes as an explicit no-op gate.

Problem being solved
--------------------
- Generic support beyond the erased-ABI family would enable real consumers
  that today must work around the erased-ABI constraint (e.g., inline
  type-sized allocation, specialized layout, or performance-sensitive loops
  across typed generics).
- Without a checkpoint phase, the deferral can persist past the point where
  it is blocking real work without acknowledgment.

Core rule
---------
- Do not admit monomorphization without a real admitted consumer.
- If a consumer is found, implement only the narrowest specialization needed
  for that consumer's ABI requirements.
- Monomorphization lives in the MIR or backend layer, not in sema.
- The admitted erased-ABI family must remain correct after this phase.
- No implicit type reconstruction, and no monomorphization by inference.

Chosen slice for this phase
---------------------------
Conditional on evidence:

Branch A -- narrow monomorphization admission
- one audit identifying a specific real consumer that the erased-ABI family
  cannot serve
- one narrowly scoped MIR or backend monomorphization path for that consumer
- focused fixture and regression coverage
- erased-ABI family remains unchanged and correct

Branch B -- explicit deferral
- one audit confirming no current consumer requires ABI-specialized generics
- one decision-log entry refreshing the deferral condition
- no implementation

Why this is the right placement
--------------------------------
- After Phase 168: the support statement has been refreshed, so any claim
  about generic support can be grounded in current documentation.
- Before longer-horizon compiler work: monomorphization is a prerequisite for
  some of the deeper compiler expansions, so this checkpoint determines
  whether those horizons remain far or are now near.

Planning rule for this phase
------------------------------
- Start with the admitted consumer audit.
- Do not implement Branch A unless a specific ABI incompatibility is proved.

What this phase should prove
-----------------------------
- that the Phase 49 deferral is re-examined on evidence
- that generic expansion (if admitted) stays in the right compiler layer
- that the erased-ABI family remains correct after any change
