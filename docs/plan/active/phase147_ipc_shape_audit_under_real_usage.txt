Repository status note
----------------------
- Earlier phases already proved bounded IPC composition, backpressure,
  message-lifetime truth, and one compact four-byte shell route.
- Phase 146 extracted the minimal service shape, but it still reused a
  structural snapshot of the Phase 145 restart path rather than one repeated
  mixed-service shell workflow.
- The repository therefore needed one usage-grounded audit of whether the
  current message vocabulary remains sufficient under repeated real command
  traffic.

Problem being solved
--------------------
- Real shell usage can reveal awkward request construction, mismatched reply
  shapes, and glue code pressure that synthetic IPC composition did not need
  to address.
- Without one explicit audit, later growth could answer that awkwardness by
  reaching too quickly for dynamic message formats, wider frameworks, or a bus
  abstraction.
- The repository therefore needed one bounded phase that classifies current
  IPC-shape friction honestly and records whether the current vocabulary is
  already sufficient.

Core rule
---------
- Audit message shape under real use; do not widen transport casually.
- Request and reply shape remain owned by services and shell-visible routes,
  not by a new generic messaging framework.
- Phase completion requires one real running-system path that demonstrates the
  judged message-shape result.
- Debug audits, MIR projections, and tool regressions are acceptance artifacts
  only.
- Keep compiler, sema, MIR, backend, ABI, target, runtime, and `hal`
  surfaces closed.
