Phase 278 -- Fixed-Font Text Rendering Or Cell Rendering Path
=============================================================

Status
------
COMPLETED

Purpose
-------
Admit one stable text-or-cell rendering path for the first display surface so a
human-facing app can present deterministic readable state without widening into
font systems, rich text, or layout engines.

This phase exists now because the first UI slice needs one visible rendering
path after input, display, routing, and repaint are explicit. The smallest
honest move is one fixed rendering contract, not a general graphics stack.

Repository status note
----------------------
- `examples/real/issue_rollup/src/render/rollup_render.mc` and
  `rollup_render_table.mc` already own app-local render selection.
- Phase 275 and Phase 277 should establish one explicit surface and present
  contract.
- The repository still lacks one maintained text-or-cell rendering path tied to
  that surface.
- Current hosted app output remains closer to ordinary program output than to a
  UI-owned rendering contract.

Problem being solved
--------------------
- A first human-facing app proof needs a stable way to render readable content
  on the admitted display surface.
- Without one owned rendering path, the first UI app would rely on ad hoc
  output or app-local formatting that the display owner cannot interpret.
- The repository needs one rendering contract strong enough for a first app but
  still too small to justify font, layout, or widget machinery.

Non-goals
---------
- no proportional fonts, font loading, or font fallback
- no rich text, markup, or layout engine
- no graphics primitives beyond the admitted text-or-cell path
- no image decoding or asset-family widening
- no second rendering backend or acceleration layer

Invariants
----------
- The first rendering path must stay fixed and deterministic.
- App-local render policy remains app-local; the rendering path does not become
  a toolkit.
- The display owner remains the surface owner, not the global text-layout owner.
- The first rendering path is admitted for one surface and one initial app
  proof.
- No font system, layout engine, or general graphics API may be introduced.

Required surface changes
------------------------
1. Define one bounded text-or-cell rendering payload or contract.
2. Connect the first app's render path to the admitted display owner.
3. Add one deterministic visible-state fixture for the rendering path.
4. Add any narrow real-project validation needed for the first app consumer.

Forbidden changes
-----------------
- do not widen rendering into a font or layout framework
- do not treat the first app's renderer as a general widget library
- do not move manifest or app-state truth into the display owner
- do not add a second rendering family in the same phase

Phase role in the larger sequence
---------------------------------
- This phase is the last preparatory rendering slice before the first
  human-facing app proof.
- It turns the explicit repaint contract into one readable visible result.
- It keeps the first UI surface honest by staying text-or-cell bounded.

Relevant backlog items this phase intentionally brings under pressure
--------------------------------------------------------------------
- keep render policy app-local and surface ownership explicit
- keep asset and manifest growth deferred until a real second family appears

Questions this phase must answer
--------------------------------
1. Fixed-cell rendering is the more honest first contract.
2. The payload is one fixed four-byte readable cell contract.
3. Render classification and manifest-aware policy remain app-local in the
  existing `issue_rollup` render modules.
4. One reset-lane visible-state fixture plus narrow `issue_rollup`
  real-project validation prove the path without implying a richer graphics
  stack.

Preferred implementation shape
------------------------------
- Reuse the first app's existing render classification as the consumer-side
  decision point.
- Reuse the existing fixed four-cell readable payload that the display owner
  already stores.
- Prefer cells or equally bounded fixed-font text because they are easier to
  validate deterministically than richer drawing primitives.
- Keep manifest and render-policy decisions inside the app.

Implementation summary
----------------------
- Land the shared visible contract as the existing four-byte readable cell
  payload: `EMTY`, `STDY`, `BUSY`, and `ATTN`.
- Keep `display_surface.mc` as the fixed-surface owner and keep
  `issue_rollup_app.mc` on the app-owned present seam introduced in Phase 277.
- Keep render classification and manifest-aware render policy in
  `examples/real/issue_rollup/src/render/rollup_render.mc` and its adjacent
  table helper rather than widening display or launcher ownership.
- Align the hosted `issue_rollup` output path with the same four-cell visible
  contract by emitting the readable cell strings instead of a second hosted
  rendering family.
- Add one focused reset-lane proof for fixed-cell visible-state transitions and
  keep the real-project validation narrow to `issue_rollup`.
