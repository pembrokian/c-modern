Series summary

Core rule
---------
- Do not open a third capability family merely to create motion.
- Prefer more honest use of the admitted networking and `sync` slices before
  adding breadth elsewhere.
- Route repeated friction back to the smallest correct owner immediately.

Recommended execution choice
----------------------------
Prefer one more networking-pressure wave, then compiler widening for the
bootstrap-shaped imported-surface limits that wave still exposes, then a
plateau decision between deeper toolchain maturity, narrow `sync` follow-on,
or narrow `hal` admission.

Why this sequence is recommended
--------------------------------
- The networking slice is already real enough to carry more pressure.
- The current admitted networking boundary still carries deliberate bootstrap
  flattening that is a better compiler-widening target than speculative
  language-surface expansion.
- Phase 13 already has ordinary project proof for the real evented-echo
  project, so the next series can deepen deterministic project behavior using
  real pressure instead of abstract tool-only cleanup.
- The remaining `sync` widening and `thread_spawn<T>` rewrite removal remain
  explicitly conditional on new concrete consumers.

Phase summary
-------------
- Phase 16: planning-truth refresh plus the next networking-pressure artifact
- Phase 17: compiler widening for the admitted import-visible typed-surface
  compromises that real consumers justify
- Phase 18: workflow hardening and one explicit plateau decision

Suggested sequencing checkpoints
--------------------------------
1. Re-audit the stale pressure docs before landing additional examples.
2. Land the partial-write networking pressure artifact.
3. Decide whether the networking public surface still needs flattened bootstrap
  spelling after that proof is real.
4. Revisit imported generic procedure-call lowering only if a second consumer
  now exists beyond `thread_spawn<T>`.
5. Use the richer project fixtures to choose between more workflow maturity,
  narrow `sync`, narrow `hal`, or release hardening.

Non-goals for this series
-------------------------
- opening multiple new capability families in parallel
- broad `sync` expansion without a new concrete consumer
- self-hosting, package management, or additional backend targets
- using examples to absorb compiler, stdlib, or toolchain gaps that should be
  fixed at the owning layer
