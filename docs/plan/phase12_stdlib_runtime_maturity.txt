Purpose
-------
Turn the broad non-bootstrap Phase 6 roadmap into an execution-ready plan that
can be implemented in reviewable waves without blurring the boundary between
language-owned semantics, MIR-owned lowered truth, backend-owned executable
realization, stdlib-owned API contracts, and runtime-owned hosted entry.

After Phase 12:

- the existing hosted library modules reflect deliberate ownership and error
  contracts instead of bootstrap convenience seams being treated as the normal
  API shape
- hosted runtime support remains small, explicit, and outside the normal
  import surface rather than bleeding into compiler or backend-local behavior
- allocation, buffer, string, utf8, and file APIs are honest enough that
  real hosted utilities can use them without repeated workaround patterns
- newly admitted modules such as `time`, `os`, `sync`, or `hal`
  land only when a real repository consumer or spec obligation requires them
- deterministic stdlib and executable regressions under
  `/Users/ro/dev/c_modern/tests/stdlib/` and related proof layers become the
  authoritative evidence for supported library behavior on the first hosted
  target

Problem being solved
--------------------
- The repository already has a real hosted stdlib/runtime slice for `mem`,
  `io`, `fs`, `strings`, `utf8`, and a narrow `testing` companion surface.
- That library layer is still incomplete relative to the long-term Phase 6
  contract in `plan.txt` and the repository's own non-bootstrap roadmap.
- The main remaining risk is not total absence of library support, but partial
  boundary honesty: some admitted APIs remain bootstrap-shaped, some runtime
  seams still reflect bring-up convenience more than final ownership, richer
  string or utf8 utilities remain intentionally narrow, and deferred module
  families have not yet been admitted through executable proof.
- Without an explicit execution plan:
  - library convenience helpers can drift into de facto long-term contracts
    without an intentional review of ownership or allocation visibility
  - runtime support can slowly accumulate behavior that should live in stdlib
    code or build-tool wiring instead of the hosted startup seam
  - Phase 8 programs can normalize awkward allocation, text, or file flows
    instead of routing those repeated problems back into Phase 6 work
  - test coverage can remain weighted toward executable end-to-end proofs
    without enough narrow stdlib regressions to make library behavior stable

Core rule
---------
- Fix language meaning in sema, lowered-semantic truth in MIR, and executable
  realization in the backend, not by smuggling semantics into stdlib or
  runtime code.
- Keep the runtime boundary minimal and explicit.
- Prefer a narrower honest library API over a broader bootstrap convenience
  wrapper whose ownership or allocation behavior is unclear.
- Admit new modules only when a real consumer or explicit architecture
  obligation justifies them.
- Keep library tests, executable behavior, and fixture outputs deterministic
  so the library proof layer remains reviewable.

Phase 12A -- Existing Module API Honesty
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Move the already-admitted stdlib modules toward deliberate long-term ownership
and error contracts without widening them beyond what current repository proof
can support honestly.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 6A -- API honesty for existing modules

Tasks
~~~~~
1. Audit the current `mem`, `io`, `fs`, `strings`, `utf8`, and `testing`
   surfaces against `stdlib_spec.txt`, `runtime_spec.txt`, and the actual
   program patterns now present in the repository.
2. Separate honest temporary bootstrap choices from API shapes that should be
   frozen as real repository contracts.
3. Tighten allocator visibility, owned-buffer flows, borrowed-text boundaries,

   `strings` and `utf8`, unless an API explicitly owns allocation behavior.

   is backed by deterministic behavior rather than example-only usage.

Phase 12B -- Runtime Boundary Cleanup
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Objective
~~~~~~~~~
Keep hosted startup and low-level support in the runtime layer while reducing
compiler or backend knowledge of library implementation details.

Maps to roadmap workstreams
~~~~~~~~~~~~~~~~~~~~~~~~~~~
- 6B -- Runtime boundary cleanup

Tasks
~~~~~
1. Re-audit the hosted runtime seam for startup construction, argv handling,
   and any checked-support or helper symbols that the executable path depends
   on today.
2. Move or document remaining behavior so hosted entry remains an explicit
   runtime-owned seam instead of a backend-local convenience path.
3. Reduce compiler knowledge of stdlib implementation details where sema-owned
   layout, declarations, or library code can carry the truth directly.
4. Keep the runtime support object small and replaceable rather than growing it
   into a hidden service layer.
5. Add narrow proof that the runtime boundary remains deterministic and does
   not change library-visible behavior accidentally.

Execution order
---------------
Recommended landing order inside Phase 12:

1. Phase 12A -- existing module API honesty audit and first tightening pass
2. Phase 12B -- runtime boundary cleanup and ownership reduction in the
   compiler or backend where practical
3. Phase 12E -- stdlib-focused proof hardening alongside each admitted change
4. Phase 12D -- text, utf8, and allocation realism once repeated friction is
   concrete enough to guide narrow additions
5. Phase 12C -- deferred module admission only after real consumers or spec
   obligations clearly justify each module

The order is intentional:

- API honesty and runtime cleanup have the highest leverage for preventing
  bootstrap seams from becoming accidental long-term contracts
- proof hardening should accompany each admission step rather than wait until
  the end
- richer text or allocation helpers should be driven by repeated pressure,
  not by speculative breadth
- deferred module families should land only after the repository has clear
  evidence that they are needed
