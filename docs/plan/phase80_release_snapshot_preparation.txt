Phase 80 -- Release Snapshot Preparation
=======================================

Status
------
IMPLEMENTED

Purpose
-------
Turn the maintainer-owned release-execution plateau from Phase 79 into one
concrete repo-owned preparation path so a maintainer can rerun the final gate
from committed state, capture the exact reviewed commit snapshot, and see the
precise tag command to apply without having the repository create the tag
automatically.

This document is a repository execution plan and closeout note, not a new
architecture phase after Phase 8.

Repository status note
----------------------
- Phase 75 turned the narrow first-use path into one repo-owned smoke entry.
- Phase 76 re-audited the supported-slice statement and added a focused doc
  audit.
- Phase 77 refreshed the broader public-cut smoke path around the current
  admitted hosted proof set.
- Phase 78 concluded that the remaining deferred technical fronts were narrow
  enough to tolerate in a source-only Darwin arm64 hosted `v0.2` claim.
- Phase 79 then kept the tag deferred and wrote down maintainer-owned release
  execution from a final reviewed committed snapshot as the one remaining
  plateau.
- The next honest step is therefore not more release-slice debate. It is to
  make that maintainer-owned execution path concrete and repeatable without
  turning it into an automatic in-tree tag action.

Problem being solved
--------------------
- Phase 79 made the remaining blocker explicit, but the repository still left
  the actual release-snapshot preparation flow as maintainer folklore.
- Without a concrete preparation path, the repository still asks a maintainer
  to reconstruct the final steps by hand even though the technical gate is now
  repo-owned and repeatable.
- The repository needs one narrow helper that reruns the final gate, insists on
  clean committed state for real use, records the exact reviewed snapshot, and
  prints the tag command to apply later.

Core rule
---------
- Keep actual tag creation maintainer-owned and explicit.
- Require clean committed repository state for real release-snapshot
  preparation.
- Reuse the existing `mc_v0_2_gate` path rather than recreating a second
  technical readiness stack.
- Stop at preparation and exact command emission; do not mutate git refs from
  repository tests or convenience wrappers.

Chosen slice for this phase
---------------------------
Targeted proof shape: one repo-owned release-snapshot preparation helper over
the existing `mc_v0_2_gate`, one focused audit around that helper, and aligned
documentation telling maintainers exactly how to use it.

Chosen outcome
--------------
Chosen outcome: yes.

The repository should add a concrete release-snapshot preparation path.

That path should:
- rerun the existing final `v0.2` gate
- require a clean committed git worktree for real maintainer use
- emit the exact reviewed commit hash and suggested `git tag -a v0.2 ...`
  command
- stop short of creating the tag automatically

Why this is the right next step
-------------------------------
- the remaining blocker from Phase 79 was operational, not technical
- a preparation helper closes that operational gap without collapsing the
  deliberate policy boundary around tag creation
- this keeps the repository honest: the final gate remains repo-owned, the tag
  remains maintainer-owned, and the live-working-tree warning remains enforced
  on the real path

What this phase should prove
----------------------------
- the repository now has one explicit release-snapshot preparation path instead
  of leaving the post-gate steps implicit
- that path composes directly with the existing `mc_v0_2_gate`
- a maintainer can capture the exact reviewed commit snapshot and tag command
  from clean committed state without the repository auto-tagging on their
  behalf

Deliverables
------------
- one canonical Phase 80 note recording the release-snapshot preparation path
- one repo-owned `prepare_v0_2_snapshot.sh` helper under `tools/release/`
- one focused `mc_release_snapshot_prep` audit entry
- one top-level `make release-snapshot-prep` convenience entrypoint
- refreshed README, hosted release-hardening note, backlog, decision log, and
  speculative-roadmap wording aligned around the new preparation path

Suggested execution order
-------------------------
1. Add one helper that reruns `mc_v0_2_gate`, checks for clean committed git
   state on the real path, and prints the exact release tag command without
   creating it.
2. Add one repo-owned audit proving that helper stays aligned with the current
   gate and documentation.
3. Refresh README and the hosted release-hardening note so the new helper is
   documented at the same boundary as the existing release-hardening commands.
4. Record the operational decision in the backlog, decision log, and roadmap
   so the repository history no longer ends at an implicit post-gate manual
   step.

Exit criteria
-------------
- the repository has one explicit release-snapshot preparation path over the
  existing `mc_v0_2_gate`
- that path requires clean committed state for real use and emits the exact
  reviewed commit hash plus tag command
- the repository still does not create the tag automatically from tests or
  convenience wrappers

Non-goals
---------
- creating the `v0.2` tag automatically
- weakening the clean committed-state requirement for real release preparation
- reopening the release-readiness audit, support matrix, or broader hosted
  slice
