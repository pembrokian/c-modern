Phase 33 -- Checked, Debug, And Public-Cut Smoke Audit
======================================================

Status
------
COMPLETED (repository-bounded checked/debug and public-cut audit)

Purpose
-------
Record the checked-versus-debug posture, repeatable smoke validation, and the
 exact first public-cut claim the repository can defend after the completed
 Lane B and documentation-hardening waves.

This document is a repository closeout note, not a new architecture phase
after Phase 8.

Repository status note
----------------------
- The current hosted plateau already treats checked ordinary tests as the main
  safety-validation proof surface while real-project executable runs usually
  stay in debug mode.
- Before this phase, the release-hardening note published that posture, but it
  still stopped short of an explicit audited smoke set across the admitted
  examples and ordinary workflow.
- Phase 31 froze the supported slice and Phase 32 hardened the first-user
  command path, so this phase could focus on operational truth instead of
  wording drift.

What landed
-----------
- `README.md` and the hosted release-hardening note now distinguish debug-mode
  executable-run proof from checked-mode ordinary-test proof instead of
  treating them as one blended validation story.
- The repository now states the narrowest honest public-cut smoke set
  explicitly: one direct-source semantic check, one direct-source executable
  build, one richer project debug run through the admitted `staticlib`
  boundary, and checked ordinary-test passes for the richer grouped project,
  the admitted sync project, and the admitted networking project.
- The public-cut smoke set now names which workflow each proof covers:
  direct-source frontend and build entrypoints, debug executable-run workflow,
  checked ordinary-test workflow, grouped internal package layout, one
  admitted in-project `staticlib`, narrow hosted `sync`, and narrow hosted
  networking.
- The release note and README now carry one explicit first public-cut hosted
  statement tied to that smoke set rather than to accumulated capability prose.
- The slice register now records the release-hardening family as including the
  Phase 33 checked/debug and public-cut audit.

What did not land
-----------------
- This phase did not convert the repository into a broad release-certification
  effort or claim release-mode audit across the full canonical-program surface.
- This phase did not widen the admitted host set, target kinds, module
  families, linker policy, or packaging posture.
- This phase did not add backlog or decision-log entries because the audit
  exposed no new consciously accepted compromise beyond the already-documented
  supported-boundary limits.

Audited public-cut posture
--------------------------
- debug-mode proof: executable-run workflow on the admitted hosted slice is
  proved narrowly by executable runs, not by a broad release audit
- checked-mode proof: ordinary project tests are the main safety-validation
  surface and are proved by `mc test` on admitted projects
- public-cut smoke set:
  - `build/debug/mc check tests/cases/hello.mc`
  - `build/debug/mc build tests/codegen/smoke_return_zero.mc --build-dir build/debug/phase33_smoke_return_zero`
  - `build/debug/mc run --project examples/real/issue_rollup/build.toml --build-dir build/debug/phase33_issue_rollup -- examples/real/issue_rollup/tests/sample.txt`
  - `build/debug/mc test --project examples/real/issue_rollup/build.toml --build-dir build/debug/phase33_issue_rollup`
  - `build/debug/mc test --project examples/real/worker_queue/build.toml --build-dir build/debug/phase33_worker_queue`
  - `build/debug/mc test --project examples/real/evented_echo/build.toml --build-dir build/debug/phase33_evented_echo`
- first public-cut hosted claim: the repository can honestly claim a hosted
  Darwin arm64 bootstrap compiler and toolchain slice with direct-source
  semantic check and executable build, project-mode debug executable runs,
  checked ordinary tests, grouped internal package layouts, one admitted
  in-project `staticlib`, deterministic selected-target same-build-dir reuse,
  and the admitted `sync` and networking examples; it does not claim broad
  release-mode certification across the whole language surface
